The tool

One file, one job

The Explorer is a single self-contained HTML page. There is no build step and no server — open it in a browser and hand it a graph. It reads the pipeline's {nodes, edges, metadata} contract, computes a fresh layout in the browser, and draws the result on a GPU canvas. The panel in the corner reports node, edge, and group counts and a progress bar that tracks the layout cooling down. Three ways to load a graph:

load jsonPick a file with the button in the panel.
drag & dropDrop a .json anywhere on the window.
?src=grouped.jsonPass a URL query param and it fetches on load.
Input

The JSON it reads

A graph is one object with two arrays and a metadata block. Edges reference nodes by array index — source_index and target_index point into the nodes array — so the viewer never has to look an address up to draw a line. Every field below is one the viewer actually reads; anything else rides along untouched.

nodeone wallet address
// identity + group are what the picture is built from
{
  "address":    "0x9f…",   // shown in the tooltip
  "group":      3,        // → fill color (which cabal)
  "in_tx": 3, "out_tx": 4,     // → node radius (activity)
  "value_in": 15.0, "value_out": 9.5, // tooltip only
  "is_bridge":    false,    // → gold ring (cut vertex)
  "cabal_bridge": false,    // → white ring (two cabals, one wallet)
  "x": null, "y": null        // ignored here — see the note below
}
edgeone transaction
{
  "source_index": 41, "target_index": 30, // → endpoints, by node index
  "value_eth":   1.5,        // → line width (how much ETH moved)
  "collection": "MOON", "token_id": "7001", // tooltip
  "cabal_label": "bilateral_corridor"     // tooltip, if present
}
metadataprovenance + counts
{
  "source": "moon_collection.csv",   // panel title
  "grouping": { "group_count": 20 }  // panel "groups" readout
}
x / y → This viewer ignores baked positions. It seeds every node onto a small spiral and runs its own force layout, so the x/y fields in the file are overwritten. That is the whole point of the clustered viewer — it is the live-layout sibling of the prebaked tile viewer. Any post-grouping JSON will load; positions are not required.
Build

Two engines, one canvas

The page leans on two libraries with cleanly split jobs. D3 decides where things go; deck.gl decides how they look. D3's force simulation owns the layout and mutates each node's x/y in place. deck.gl owns the rendering and reads those same coordinates through its accessors onto the GPU. The seam between them is a single counter.

D3 mutates node objects in place; deck.gl will not notice unless told to. So each tick bumps a counter that the layers list as an updateTrigger — that one number is what makes the animation move. Layout runs on the CPU; drawing runs on the GPU; the OrthographicView handles pan and zoom and switches off auto-fit the moment you touch it. The forces that produce the clustering:

charge · −24

Many-body repulsion pushes every node off every other, capped at short range so it spreads clumps without flinging the frame apart.

link · dist 8

Springs along edges pull traded-together wallets tight. This is what makes each cabal a visible knot.

x / y gravity · 0.05

A gentle pull toward center keeps disconnected cabals — which share no edge — gathered in one frame instead of drifting off.

collide · 1.4

A small hard radius stops nodes from stacking, so dense clusters stay legible rather than fusing into a blob.

Re-run layout reseeds and cools again; the progress bar reads the simulation's alpha falling toward rest. Because edges carry no edge between disconnected components, space alone can't separate the cabals — which is why color does the heavy lifting.

Reading it

What the picture means

Five visual channels, each bound to one field. Nothing is decorative — position, color, size, and the rings all carry a specific attribute from the contract.

node position ← d3-force layout (not from file)

Where a node sits = who it trades with

Position is computed, not stored. Connected wallets clump because the link springs pull them together; everything else drifts apart. Proximity reads as "transacts with." A tight knot is a candidate cabal.

node fill ← group

Color = which cabal

Each group gets a hue stepped by the golden angle (137.5°) so adjacent group numbers land far apart on the color wheel and stay distinguishable. Color is what separates cabals that space can't — two unconnected clusters can sit near each other but never share a color.

node radius ← √(in_tx + out_tx + 1)

Size = how busy the wallet is

Radius grows with total transaction count, square-rooted so a hyperactive address doesn't swamp the view, and floored at 3px so nothing vanishes. Big dot = high-traffic wallet.

edge width ← √(value_eth) · edge color ← source group

Lines = transactions, thickness = ETH

Every edge is one transaction; its width tracks the ETH that moved and it wears its source wallet's group color at half opacity. A thick line is a large-value trade; a wall of thin same-color lines between two dots is the bilateral ping-pong of a wash corridor.

node ring ← is_bridge (gold) · cabal_bridge (white)

Rings = wallets that hold the network together

A gold ring marks a cut vertex — remove it and a cluster splits. A white ring marks a cabal bridge: a single wallet that links two otherwise separate cabals, the prime suspect for one hand wearing two gloves. No ring, no structural role.

Hovering any node prints its address, group, in/out counts, and ETH flow; hovering an edge prints its value, the collection and token it moved, and any cabal_label the pipeline attached. That is the whole tool: a contract in, a layout computed, five channels out, and the human in the loop doing the rest.
