Characters

In C, a character is just a small integer. The letter 'A' is the number 65. Understanding this identity — character = number — unlocks everything about how C handles text.

char Is an Integer Type

A char occupies exactly 1 byte (8 bits) and stores a number from 0 to 127 (or -128 to 127 if signed). That number is an index into a lookup table called ASCII — the American Standard Code for Information Interchange — which maps numbers to characters. The character 'A' is just syntactic sugar for the integer 65.

'A' and 65 are the same value in the same 1-byte storage. The format specifier in printf decides how to display it.
char c = 'A';
printf("%c\n", c);   // prints: A      (as character)
printf("%d\n", c);   // prints: 65     (as integer)

// Because char is an integer, arithmetic works:
char next = c + 1;        // 'B' — one step up in ASCII
char lower = c + 32;       // 'a' — uppercase to lowercase
int  digit_val = '7' - '0'; // 7  — char digit to integer value

The ASCII Table

You don't need to memorize every ASCII value, but the key ranges are worth knowing because they appear constantly in text processing code.

ASCII ranges. Digits, uppercase, and lowercase letters each occupy a contiguous range — enabling range-check arithmetic.

Character Classification with <ctype.h>

Rather than writing range comparisons by hand, use the functions in <ctype.h>. They correctly handle edge cases across all valid character values and make your intent clear.

#include <ctype.h>

char c = 'A';

isalpha(c)    // true if letter (a-z or A-Z)
isdigit(c)    // true if digit (0-9)
isalnum(c)    // true if letter or digit
isspace(c)    // true if whitespace (space, tab, newline, etc.)
isupper(c)    // true if uppercase letter
islower(c)    // true if lowercase letter
ispunct(c)    // true if punctuation

toupper(c)    // returns uppercase version of c
tolower(c)    // returns lowercase version of c

// Example: convert digit character to its integer value
char digit = '7';
int  val   = digit - '0';   // val = 7

Escape Sequences

Some characters can't be typed directly in source code — newline, tab, the null byte, backslash itself. C uses escape sequences: a backslash followed by a letter or code that the compiler translates into the actual byte value.

EscapeNameASCII valueMeaning
'\n'newline10Move cursor to next line
'\t'tab9Move cursor to next tab stop
'\0'null0String terminator — end of string

'\''single quote39Literal single quote inside char literal

'\0' is the most important escape sequence
The null character — ASCII value 0 — is how C marks the end of a string. It is not the character '0' (which is ASCII 48). Confusing these two is one of the most common string bugs in C. Every C string ends with a '\0' that is invisible when printed but essential for every string function to work correctly.

Reading and Writing Characters

// Reading one character at a time
char c;
scanf("%c", &c);       // reads next char from stdin (including whitespace)
c = getchar();         // same — reads one char, returns it as int

// Writing one character
printf("%c", c);        // prints as character
putchar(c);            // same — writes one char to stdout

// getchar returns int, not char — important for EOF detection
int ch;
while ((ch = getchar()) != EOF) {
    putchar(ch);         // echo every character until end of input
}
getchar() returns int, not char
getchar() returns an int so it can return both any valid character (0–255) and the special value EOF (-1). If you store the result in a char, the EOF value may be silently truncated to a valid character value, making the end-of-input check impossible. Always use int to receive the result of getchar().

Common Problems

Pitfall 01
Confusing '\0' with '0'
The null character '\0' has ASCII value 0. The digit character '0' has ASCII value 48. They are completely different. Using '0' as a string terminator leaves a visible '0' in the output. Using '\0' as a digit produces value 0, not "zero".
Pitfall 02
Storing getchar() in char
char c = getchar(); then checking c == EOF never works — EOF is -1, which either doesn't fit in an unsigned char or gets confused with a valid byte. Always declare the receiver as int: int c = getchar();.
Pitfall 03
Signed vs Unsigned char
Whether char is signed or unsigned is implementation-defined. On most x86 systems it's signed (-128 to 127), but ARM often defaults to unsigned. Code that relies on char being signed or unsigned will break across platforms. Use signed char or unsigned char explicitly when the signedness matters.
Pitfall 04
Character Arithmetic Going Out of Range
Adding to a char can overflow or go negative. 'z' + 1 is 123, which is still a valid ASCII character ('{'), not an error. But (char)127 + 1 on a signed char system wraps to -128. Arithmetic on characters works, but be aware of the boundaries.
Pitfall 05
Using == to Compare Strings
Characters compare fine with ==: c == 'A' works. But string literals do not: "hello" == "hello" compares pointer addresses, not content. For strings, always use strcmp(). This pitfall belongs on the Strings page too — it's common at both levels.
Pitfall 06
Passing char to ctype.h Functions
The <ctype.h> functions expect an unsigned char value or EOF. Passing a plain char that happens to be negative (on a signed-char platform) is undefined behavior. Cast first: isalpha((unsigned char)c).
Takeaway
A char is a 1-byte integer. 'A' is just 65. ASCII assigns characters to numbers — knowing the key ranges (0–9 at 48–57, A–Z at 65–90, a–z at 97–122) enables direct arithmetic. Use <ctype.h> for classification. Always receive getchar() in an int, not a char. And '\0' (zero) is not '0' (forty-eight).
