Variables & Assignment

A variable is a named location in memory. You give it a type, a name, and a value — and the program remembers that value for as long as the variable exists.

What a Variable Actually Is

When you declare a variable, you're asking the compiler to reserve a named slot in memory. That slot has four properties: a name (how you refer to it in code), a type (how many bytes it occupies and how those bytes are interpreted), a value (the bits currently stored there), and an address (where it lives in memory).

A variable named "score" of type int — 4 bytes at address 0x7fff1008 currently holding the value 42.
Analogy
A variable is like a labeled box. The type determines the size and shape of the box. The name is the label on the outside. The value is whatever's inside. The address is the box's location on the shelf. You refer to the box by label — the program finds it by address.

Declaration and Initialization

Declaration tells the compiler the variable's name and type — it reserves the memory. Initialization gives the variable its first value. These can happen separately or together.

int score;           // declaration only — value is undefined garbage
int score = 0;       // declaration + initialization — preferred

double gpa;          // uninitialized
gpa = 3.8;           // assignment after the fact

int x = 10, y = 20; // multiple declarations on one line
Uninitialized variables contain garbage.
In C, declaring a variable does not zero it out. The bytes at that memory location contain whatever was there before — a previous function call, leftover data, anything. Reading an uninitialized variable is undefined behavior. Always initialize at the point of declaration.
Always initialize variables at declaration. Uninitialized memory contains unpredictable values from previous use.

Assignment

The = operator copies a value into a variable. It is not a mathematical equality — it's an instruction: evaluate the right side, store the result in the left side. The right side can be a literal, another variable, or any expression.

int x = 5;
int y = 3;
int z;

z = x + y;        // z is now 8
x = x + 1;        // x is now 6 — fine, right side evaluated before assignment
y = z;             // y is now 8 — z is unchanged

// compound assignment operators — shorthand
x += 2;            // x = x + 2
x -= 1;            // x = x - 1
x *= 3;            // x = x * 3
x /= 2;            // x = x / 2
x ++;              // x = x + 1  (increment)
x --;              // x = x - 1  (decrement)
Assignment evaluates the right-hand side completely, then stores the result into the variable on the left.

Scope

A variable's scope is the region of code where it exists and can be accessed. In C, variables are scoped to the block (curly braces {}) in which they are declared. They come into existence when declared and cease to exist when execution leaves that block.

int main(void) {
    int x = 10;          // x lives in main's block

    {
        int y = 20;       // y lives only in this inner block
        printf("%d %d\n", x, y);  // both accessible here
    }

    // printf("%d\n", y);  ← COMPILE ERROR: y doesn't exist here
    printf("%d\n", x);   // x still accessible
    return 0;
}
Variables are scoped to their enclosing block. Inner variables shadow outer ones with the same name.

Constants

Sometimes a value should never change after being set. C gives you two ways to express this: the const qualifier and the #define preprocessor directive. They work very differently under the hood.

// const: a typed variable the compiler won't let you modify
const int MAX_SIZE = 100;
const double PI = 3.14159265358979;

MAX_SIZE = 200;   // ← COMPILE ERROR: assignment to const variable

// #define: textual substitution before compilation
#define MAX_SIZE 100
#define PI 3.14159265358979

// #define has no type — the preprocessor blindly replaces text
// Prefer const — it has a type and appears in the debugger
Prefer const over #define for constants
const variables have a type, obey scope rules, and show up in the debugger with their name. #define macros are invisible to the compiler and debugger — they're replaced before the compiler even sees the code. Use const for values; reserve #define for compile-time flags and macros.

Common Problems

Pitfall 01
Uninitialized Variable
Declaring without initializing leaves garbage in the variable. Reading it produces unpredictable results that change between runs. Always initialize at the point of declaration: int x = 0; not just int x;.
Pitfall 02
= vs == Confusion
Writing if (x = 5) assigns 5 to x and evaluates to 5 (always true). This is valid C — the compiler may not warn. The condition never does what you intended. Use == for comparison, = only for assignment.
Pitfall 03
Variable Shadowing
Declaring a variable with the same name in an inner scope hides the outer one. Code that looks like it's modifying the outer variable is actually modifying a separate inner copy. The outer variable is unchanged. This is legal but confusing — avoid reusing names in nested scopes.
Pitfall 04
Using a Variable Outside Its Scope
Variables declared inside a block cease to exist when the block ends. Attempting to use them after is a compile error. If you need a variable to outlive an inner block, declare it in the outer scope before the block begins.
Pitfall 05
Modifying a const Variable
Attempting to assign to a const variable is a compile error. But beware: casting away const and modifying through a pointer is undefined behavior — the compiler may place const variables in read-only memory where the write will crash at runtime.
Pitfall 06
Integer Division Truncation
int result = 7 / 2; gives 3, not 3.5. When both operands are integers, C performs integer division and truncates toward zero. If you need the fractional part, at least one operand must be a floating-point type: 7.0 / 2 gives 3.5.
Takeaway
A variable is a named, typed slot in memory. Always initialize at declaration — uninitialized values are garbage. Use = for assignment, == for comparison. Variables live and die with their enclosing block. Use const when a value should never change.
