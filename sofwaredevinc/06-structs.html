C Structs

A struct is how C lets you group related data under one name. It's the closest thing C has to an object — and understanding it unlocks every serious data structure in the language.

What Is a Struct?

C's primitive types — int, char, double — each hold one value. A struct lets you define a new type that holds multiple named values together, called fields (or members). Once defined, a struct behaves like any other type: you can declare variables of it, pass it to functions, put it in arrays, and point at it with pointers.

struct Point {
    double x;
    double y;
};

struct Student {
    char   name[64];
    int    id;
    double gpa;
    int    enrolled;   // used as a boolean
};

The struct keyword followed by a tag name defines the layout. Fields can be any type — including other structs — but not the struct itself (you can have a pointer to the same struct type, which is how linked lists work).

Analogy
Think of a struct definition as a blueprint — it describes what fields exist and their types, but uses no memory. Declaring a struct Student s is like constructing a building from that blueprint: now real memory is reserved with slots for name, id, gpa, and enrolled.

typedef — Losing the Keyword

Writing struct Student everywhere is verbose. Using typedef, you can create an alias so the word struct is no longer needed:

typedef struct {
    char   name[64];
    int    id;
    double gpa;
} Student;

// Now you can write:
Student s;          // instead of: struct Student s;

Both styles appear in real C code. System headers tend to use the struct Tag form; application code often uses typedef. Either is fine — pick one and be consistent.

Structs in Memory

The fields of a struct are laid out in memory in the order you declare them, one after another. The total size is at least the sum of the field sizes — but the compiler may insert padding bytes between fields to satisfy alignment requirements. Each type prefers to start at an address that is a multiple of its own size.

Struct fields are laid out in declaration order. Alignment rules may cause the compiler to insert invisible padding bytes between fields.
Key Concept
Never assume the size of a struct by adding up its fields. Always use sizeof(Student). Padding can make the real size larger — and it varies by platform and compiler.

Creating and Accessing Structs

Student s;                        // declare — fields uninitialized

// Designated initializer (C99+) — preferred, safe, clear
Student s = {
    .name = "Alice",
    .id   = 12345,
    .gpa  = 3.8,
    .enrolled = 1
};

// Access fields with the dot operator
printf("%s has a %.1f GPA\n", s.name, s.gpa);

// Modify fields directly
s.gpa = 3.9;
s.enrolled = 0;

Nested Structs

A struct field can itself be a struct. Access chained fields with repeated dot operators:

typedef struct {
    double x, y;
} Point;

typedef struct {
    Point  position;
    Point  velocity;
    double mass;
} Particle;

Particle p = { .position = {0.0, 0.0}, .velocity = {1.5, -2.0}, .mass = 1.0 };

p.position.x += p.velocity.x;   // advance position by velocity

Arrays of Structs

An array of structs is exactly what it sounds like — the structs are packed one after another in memory, each sizeof(StudentType) bytes apart. This is one of the most common patterns in C programs.

Student roster[35];               // 35 students, stack-allocated

// Initialize one by one:
strcpy(roster[0].name, "Alice");
roster[0].id  = 1001;
roster[0].gpa = 3.8;

// Or with a compound literal:
roster[1] = (Student){ .name = "Bob", .id = 1002, .gpa = 3.2 };

// Iterate:
for (int i = 0; i < 35; i++) {
    printf("%d: %-20s %.2f\n", roster[i].id, roster[i].name, roster[i].gpa);
}
Each element in the array occupies sizeof(Student) bytes. Pointer arithmetic steps by exactly that amount.

Dynamic Arrays of Structs

When you don't know the count at compile time, allocate on the heap with malloc:

int n = 35;  // or read from user / config

Student *roster = malloc(n * sizeof(Student));
if (!roster) { /* handle failure */ }

// Use exactly like a stack array:
roster[0].id = 1001;
roster[1].gpa = 3.2;

free(roster);

Passing Structs to Functions

C is a pass-by-value language. When you pass a struct to a function, the entire struct is copied — every field, every byte. The function works on its own private copy. This is safe but potentially expensive for large structs, and it means the function cannot modify the caller's original.

void print_student(Student s) {      // receives a COPY of the struct
    printf("%s — GPA: %.2f\n", s.name, s.gpa);
    s.gpa = 0.0;                      // only modifies the local copy
}

Student alice = { .name = "Alice", .gpa = 3.8 };
print_student(alice);
// alice.gpa is still 3.8 — untouched
Passing a struct by value copies every byte. The function's modifications never reach the caller's original.

Modifying Structs in Functions: Use a Pointer

To let a function modify the caller's struct — or to avoid copying a large struct on every call — pass a pointer to the struct. This is the idiomatic C pattern.

void update_gpa(Student *s, double new_gpa) {  // pointer — no copy
    s->gpa = new_gpa;    // arrow operator: dereference + field access
}

Student alice = { .name = "Alice", .gpa = 3.8 };
update_gpa(&alice, 3.95);
// alice.gpa is now 3.95 ✓

The arrow operator -> is the key. When you have a pointer to a struct, s->gpa means "follow the pointer s, then access the gpa field." It is exactly equivalent to writing (*s).gpa, just without the parentheses and dereference noise.

Operator Used when Example Equivalent to
. You have a struct value (not a pointer) alice.gpa Direct field access
-> You have a pointer to a struct s->gpa (*s).gpa
Passing a pointer copies only the 8-byte address. The function writes directly into the caller's struct through s->field.

Creating Structs Inside Functions

Sometimes a function's whole job is to build a struct and return it. C gives you two ways to do this, with very different lifetimes.

Return by Value (Stack)

You can return a struct directly. C copies the return value back to the caller. This is simple and safe for reasonably-sized structs, and modern compilers often optimize away the copy entirely (Return Value Optimization).

Student make_student(const char *name, int id, double gpa) {
    Student s = {
        .id       = id,
        .gpa      = gpa,
        .enrolled = 1
    };
    strncpy(s.name, name, sizeof(s.name) - 1);
    return s;    // struct is copied into the caller's variable
}

Student alice = make_student("Alice", 1001, 3.8);

Return by Pointer (Heap)

When the struct needs to outlive the function in a complex way, or when it's very large, allocate it on the heap and return the pointer. The caller is then responsible for free-ing it.

Student *make_student_heap(const char *name, int id, double gpa) {
    Student *s = malloc(sizeof(Student));
    if (!s) return NULL;

    s->id       = id;
    s->gpa      = gpa;
    s->enrolled = 1;
    strncpy(s->name, name, sizeof(s->name) - 1);

    return s;   // caller must free(s) when done
}

Student *alice = make_student_heap("Alice", 1001, 3.8);
// ...use alice...
free(alice);
The Classic Mistake: Returning a Stack Pointer
Never return a pointer to a local struct variable. The struct lives on the function's stack frame, which is destroyed the moment the function returns. The pointer immediately becomes dangling.

Student *bad() { Student s = {...}; return &s; } // ← UNDEFINED BEHAVIOR

If you need to return a pointer, malloc the struct. If you can return by value, do that instead.

Passing Arrays of Structs to Functions

Like all arrays in C, an array of structs decays into a pointer when passed to a function. The function receives a pointer to the first element and must be told the count separately.

void print_roster(Student *students, int count) {
    for (int i = 0; i < count; i++) {
        printf("%4d  %-20s  %.2f\n",
               students[i].id,
               students[i].name,
               students[i].gpa);
    }
}

void apply_curve(Student *students, int count, double bonus) {
    for (int i = 0; i < count; i++) {
        students[i].gpa += bonus;          // modifies originals directly
        if (students[i].gpa > 4.0) students[i].gpa = 4.0;
    }
}

Student roster[35];
// ... fill roster ...
print_roster(roster, 35);
apply_curve(roster, 35, 0.1);

Notice that apply_curve modifies the array elements directly — no extra & needed. That's because the array already decayed to a pointer, so students[i].gpa is writing through that pointer into the original memory.

Struct Pointers: The Full Picture

Struct pointers are used pervasively in C. Linked lists, trees, hash tables, file handles — every significant data structure in C is built with struct pointers. Here's a linked list node to see how it all fits together:

typedef struct Node {
    int          value;
    struct Node *next;   // pointer to same type — linked list!
} Node;

Node *make_node(int val) {
    Node *n = malloc(sizeof(Node));
    n->value = val;
    n->next  = NULL;
    return n;
}

// Build 3 → 2 → 1 → NULL
Node *head = make_node(3);
head->next = make_node(2);
head->next->next = make_node(1);

// Traverse
for (Node *p = head; p != NULL; p = p->next) {
    printf("%d ", p->value);
}
A linked list: each Node is heap-allocated. The next pointer chains them. head->next->next = 0x40C0.

Common Difficulties with Structs

Structs are not hard conceptually, but several specific patterns trip up almost every C developer — including experienced ones. Here are the ones you'll encounter most.

Pitfall 01
Dot vs. Arrow Confusion
Using . when you have a pointer, or -> when you have a value. The rule is simple: do you hold the struct itself, or a pointer to one? If you're ever unsure, read the variable's declaration.
Pitfall 02
Modifying a Copy Instead of the Original
Passing a struct by value and wondering why your changes don't stick. If a function must modify a struct, the parameter must be StructType *s, and callers must pass &myStruct.
Pitfall 03
Returning a Pointer to a Local Struct
return &localVar from a function is always wrong — the local variable's stack frame is destroyed on return. Either return by value, or malloc the struct and return that pointer.
Pitfall 04
Forgetting NULL Check After malloc
malloc can return NULL if memory is exhausted. Writing s->id = 5 immediately after a failed malloc is a segfault waiting to happen. Always check: if (!s) { /* handle */ }.
Pitfall 05
Using sizeof on a Pointer to a Struct
sizeof(Student *) is always 8 (pointer size), not the size of the struct. If you're allocating with malloc, write malloc(sizeof(Student)) — or the idiomatic and safer malloc(sizeof(*ptr)).
Pitfall 06
Shallow Copy of Pointer Fields
If a struct contains a char *name pointer field, assigning one struct to another copies the pointer — not the string it points to. Both structs now share the same string; freeing one corrupts the other.
Pitfall 07
String Fields: char[] vs char*
A char name[64] field stores the string inside the struct. A char *name field stores only a pointer — you need to separately allocate the string. Forgetting this leads to dangling pointers or writing to string literals.
Pitfall 08
Uninitialized Fields
Student s; without an initializer leaves all fields containing whatever bytes were previously on the stack. Always initialize — use designated initializers = {.id = 0} or memset(&s, 0, sizeof(s)) for a guaranteed zero state.

Pitfalls 1 & 2 in Code: The Canonical Mistakes

/* ── Pitfall 1: dot vs. arrow ─────────────────────────── */
Student *ptr = malloc(sizeof(Student));

ptr.gpa = 3.8;       // ✗ COMPILE ERROR — ptr is a pointer, use ->
ptr->gpa = 3.8;      // ✓

Student val = *ptr;
val->gpa = 3.8;      // ✗ COMPILE ERROR — val is a struct, use .
val.gpa = 3.8;       // ✓

/* ── Pitfall 2: copy vs. pointer ──────────────────────── */
void bad_curve(Student s)    { s.gpa += 0.1; }  // ✗ modifies copy
void good_curve(Student *s)  { s->gpa += 0.1; } // ✓ modifies original

Student alice = { .gpa = 3.8 };
bad_curve(alice);          // compiles but does nothing to alice
good_curve(&alice);        // alice.gpa becomes 3.9 ✓

Pitfall 6: Shallow Copy Deep-Dive

typedef struct {
    char *name;   // pointer field — not an array!
    int   id;
} BadStudent;

BadStudent a, b;
a.name = malloc(64);
strcpy(a.name, "Alice");

b = a;              // struct copy — copies the pointer, NOT the string!
                    // a.name and b.name now point to the SAME buffer

free(a.name);      // frees the buffer
printf("%s\n", b.name);  // ✗ use-after-free! b.name is now dangling

/* ── Safe deep copy ─────────────────────────────────── */
b.name = malloc(strlen(a.name) + 1);
strcpy(b.name, a.name);   // ✓ independent copy of the string
Pro Tip: Use char[] for Fixed-Size Strings in Structs
If you use char name[64] instead of char *name, the string is embedded directly in the struct — no separate allocation, no shallow-copy hazard, and struct assignment does the right thing automatically. The trade-off is a fixed maximum size. For most student/record-style structs, this is the right default.
Takeaway
A struct groups related fields into one named type. Use . to access fields on a value; use -> to access fields through a pointer. Pass structs by pointer when functions need to modify them or when the struct is large. Return by value for small structs; malloc and return a pointer for large or long-lived ones. The most common traps are using the wrong access operator, accidentally modifying a copy, and returning a pointer to a stack-allocated local.
