Linked Lists

A linked list is a chain of heap-allocated structs, each holding a value and a pointer to the next. It's the simplest data structure you can build from what you already know — and the foundation for nearly every other one.

Arrays vs. Linked Lists

You already know arrays. They're a contiguous block of memory — all elements side by side, instantly reachable by index. Linked lists solve a different problem: when you need to grow and shrink a collection at runtime without knowing the final size, and when you need fast insertion and deletion anywhere in the sequence.

Operation Array Linked List
Access element by index O(1) — direct address calculation O(n) — must walk from head
Insert at front O(n) — shift all elements right O(1) — rewire one pointer
Insert at arbitrary position O(n) — shift elements O(1) once you have the position
Delete from front O(n) — shift all elements left O(1) — rewire one pointer
Memory layout Contiguous — cache-friendly Scattered — pointer chasing
Size known at compile time? Required for stack arrays Grows and shrinks at runtime
Analogy
An array is a row of numbered seats in a theater — you can go directly to seat 12 in one step, but adding a new seat in the middle means moving everyone. A linked list is a treasure hunt — each clue tells you where the next one is. You can't jump to clue 5 without reading 1 through 4, but you can insert a new clue anywhere by changing two pieces of paper.

The Node: Building Block of Every List

A linked list is built from nodes. Each node is a struct with two things: the data it holds, and a pointer to the next node. That self-referential pointer — a struct pointing to another struct of the same type — is the key to the whole pattern.

typedef struct Node {
    int          value;       // the data this node holds
    struct Node *next;        // pointer to the next node (or NULL)
} Node;

Notice the tagged struct form here — struct Node *next must use the tag because the typedef name Node isn't complete yet when the field is declared. This is the one situation where the tagged form is not just stylistic — it's required.

The last node in the list sets its next to NULL. That's how you know you've reached the end. Every operation on a linked list eventually checks for NULL.

A three-node linked list: 3 → 7 → 2 → NULL. The head pointer is the entry point. NULL marks the end.

Creating a Node

Every node lives on the heap — it needs to outlive the function that creates it. The pattern is always the same: malloc, check for NULL, set fields, set next to NULL.

Node *make_node(int value) {
    Node *n = malloc(sizeof(Node));
    if (!n) return NULL;    // always check
    n->value = value;
    n->next  = NULL;
    return n;
}

Building a List

Prepend — Insert at the Front

Inserting at the front is the simplest and fastest operation: create a new node, point its next at the current head, then update head. Two pointer assignments. Order matters — if you update head first, you lose the list.

Node *prepend(Node *head, int value) {
    Node *n = make_node(value);
    if (!n) return head;   // allocation failed — leave list unchanged
    n->next = head;        // step 1: new node points at old head
    return n;              // step 2: new node IS the new head
}

Node *list = NULL;                 // empty list
list = prepend(list, 3);          // 3 → NULL
list = prepend(list, 7);          // 7 → 3 → NULL
list = prepend(list, 2);          // 2 → 7 → 3 → NULL
Prepend in two steps: point the new node at the old head, then return the new node as the new head.

Append — Insert at the End

Appending is slower — you have to walk to the last node first, then attach the new one. Always handle the empty list case separately.

Node *append(Node *head, int value) {
    Node *n = make_node(value);
    if (!n) return head;

    if (head == NULL) return n;   // was empty: new node is the list

    Node *cur = head;
    while (cur->next != NULL)     // walk to the last node
        cur = cur->next;

    cur->next = n;               // attach at the end
    return head;
}

Traversal

Traversal is the foundation of everything else you do with a linked list. The idiom is always the same: start at head, use a local pointer cur to walk, stop when cur is NULL.

void print_list(Node *head) {
    for (Node *cur = head; cur != NULL; cur = cur->next) {
        printf("%d ", cur->value);
    }
    printf("\n");
}

int length(Node *head) {
    int count = 0;
    for (Node *cur = head; cur != NULL; cur = cur->next)
        count++;
    return count;
}

Node *find(Node *head, int target) {
    for (Node *cur = head; cur != NULL; cur = cur->next)
        if (cur->value == target) return cur;
    return NULL;   // not found
}
Key Pattern
Never modify head during traversal. Use a separate cur pointer that walks the list. Losing head means losing the entire list — you have no other way to reach the first node.

Insertion in the Middle

To insert after a specific node, you need a pointer to the node before the insertion point — the predecessor. You re-wire two pointers: the new node points to the successor, and the predecessor points to the new node. The order matters.

// Insert 'value' after the node pointed to by 'prev'
void insert_after(Node *prev, int value) {
    if (!prev) return;
    Node *n = make_node(value);
    if (!n) return;

    n->next    = prev->next;  // step 1: new node points to successor
    prev->next = n;            // step 2: predecessor points to new node
}
Always set the new node's next pointer BEFORE updating the predecessor. Reverse the order and you lose the tail of the list.

Deletion

Deleting a node requires two things: unlinking it from the chain by re-wiring the predecessor's next, and then free-ing the node's memory. You need the predecessor again, which is why deletion often requires tracking both the current node and the one behind it.

Delete from the Front

Node *delete_front(Node *head) {
    if (head == NULL) return NULL;   // empty list
    Node *old_head = head;
    head = head->next;               // advance head first
    free(old_head);                    // then free the old head
    return head;
}

Delete by Value

Node *delete_value(Node *head, int target) {
    if (head == NULL) return NULL;

    // special case: target is the head node
    if (head->value == target) {
        Node *next = head->next;
        free(head);
        return next;
    }

    // general case: find the predecessor
    Node *prev = head;
    while (prev->next != NULL && prev->next->value != target)
        prev = prev->next;

    if (prev->next == NULL) return head;  // not found

    Node *to_delete = prev->next;
    prev->next = to_delete->next;        // bypass the node
    free(to_delete);                        // release its memory
    return head;
}
Free the node — but never free a node you still need to read from.
The classic mistake is writing free(cur); cur = cur->next;. After free(cur), reading cur->next is undefined behavior — the memory may have already been reused. Always save cur->next into a temporary variable before calling free.

Free the Entire List

When you're done with a list, every node must be freed. Walk the list, saving next before freeing each node.

void free_list(Node *head) {
    Node *cur = head;
    while (cur != NULL) {
        Node *next = cur->next;  // save next BEFORE freeing
        free(cur);
        cur = next;
    }
}

Passing Lists to Functions

A linked list is always handled as a pointer to its first node — Node *head. Passing this to a function gives the function access to the entire list by traversal. But there's a subtlety: if a function might change which node is the head (prepend, delete_front), the caller's head variable won't be updated unless you return the new head or pass a pointer-to-pointer.

// Pattern 1: Return the new head — clean and idiomatic
Node *list = NULL;
list = prepend(list, 5);      // caller re-assigns head
list = delete_front(list);

// Pattern 2: Pointer-to-pointer — lets the function update head directly
void prepend_pp(Node **head, int value) {
    Node *n = make_node(value);
    if (!n) return;
    n->next = *head;
    *head = n;                  // modifies caller's head variable
}

Node *list = NULL;
prepend_pp(&list, 5);         // list updated in-place
Which pattern to use?
Returning the new head (Pattern 1) is simpler to read and less error-prone. Use pointer-to-pointer (Pattern 2) when working with more complex structures like doubly-linked lists or when the API design requires it. Both are idiomatic C.

Variations

Doubly-Linked List

Add a prev pointer to each node and you can walk the list in both directions. Useful when you need to delete a node without finding its predecessor, or implement a deque. The cost is an extra pointer per node and double the pointer updates on every insert/delete.

typedef struct DNode {
    int           value;
    struct DNode *prev;
    struct DNode *next;
} DNode;

Circular List

The last node's next points back to the head instead of NULL. Used in round-robin schedulers, ring buffers, and anywhere you need to cycle continuously. The termination condition in traversal changes from cur != NULL to cur != head.

Circular list: the last node's next wraps back to head. Traversal stops when cur == head again.

Common Problems

Pitfall 01
Losing the Head Pointer
Using head itself as the traversal pointer — advancing it through the list — means you can never get back to the start. The entire list is now unreachable. Always use a separate cur pointer for walking.
Pitfall 02
Reading After free()
Writing free(cur); cur = cur->next; reads next from freed memory. Save cur->next in a temp variable before calling free. This applies equally in free_list and in deletion.
Pitfall 03
Wrong Insertion Order
When inserting in the middle, updating prev->next = n before n->next = prev->next severs the tail — you've lost every node after the insertion point. Set the new node's next first, always.
Pitfall 04
Not Handling the Empty List
Every list operation must handle head == NULL as a special case. Append without the empty-list check immediately dereferences a NULL pointer. Deletion without it crashes. Check NULL first — always.
Pitfall 05
Memory Leak on Deletion
Unlinking a node (prev->next = cur->next) without calling free(cur) leaks the node. The node is no longer reachable but its memory is never returned. Valgrind will catch this — and so will your heap at scale.
Pitfall 06
Not Updating Head After Deletion
Deleting the head node but not returning or updating the caller's head variable leaves the caller with a dangling pointer to freed memory. Functions that may change which node is the head must either return the new head or take a Node **.
Pitfall 07
Off-by-One in Traversal
Stopping the loop at cur->next != NULL versus cur != NULL gives you different end positions — the last node vs. past the last node. Which you want depends on the operation. Get it wrong and you'll either skip the last element or dereference NULL.
Pitfall 08
Forgetting to Free the Whole List
When a list goes out of scope or is no longer needed, every node must be individually freed. There's no garbage collector. Setting head = NULL without walking and freeing each node first leaks every node in the list.

The Two Worst Bugs Illustrated

/* ── Pitfall 1: clobbering head ──────────────────────── */
void bad_print(Node *head) {
    while (head != NULL) {       // ✗ advancing head itself
        printf("%d ", head->value);
        head = head->next;
    }
}   // caller's head is unchanged (pass by value saved us here)
    // but if head were Node**, this destroys the caller's list

void good_print(Node *head) {
    for (Node *cur = head; cur != NULL; cur = cur->next)  // ✓
        printf("%d ", cur->value);
}

/* ── Pitfall 2: reading after free ───────────────────── */
void bad_free(Node *head) {
    while (head != NULL) {
        free(head);               // ✗ freed!
        head = head->next;      // ✗ reading freed memory — UB
    }
}

void good_free(Node *head) {
    while (head != NULL) {
        Node *next = head->next;  // ✓ save next first
        free(head);
        head = next;
    }
}
Takeaway
A linked list is a chain of heap-allocated nodes, each containing data and a pointer to the next. Every node is individually malloc'd and must be individually free'd. The head pointer is your only entry to the list — protect it. Use a separate traversal pointer. Save next before freeing. Handle NULL at every operation. When inserting, set the new node's next pointer before rewiring the predecessor.
