Step-by-Step Explanation
// What actually happens when your program executes   sum(3, 4)
①

Evaluate the Arguments Caller side

Before calling sum(), the CPU first figures out what the argument values are. Here they are simple literals — 3 and 4 — so no extra work is needed. If the call were sum(a+1, b*2), those expressions would be computed here first and their results stored temporarily before moving to the next step.

💡
Analogy: Before handing someone a package, you first prepare what goes inside. Step ① is that preparation.
②

Load 1st Argument into RDI CPU Register

The x86-64 System V ABI — the calling convention used on Linux — has a strict rule: the first integer argument always goes into register RDI. So the value 3 is loaded into RDI.

movedi, 3; RDI = 3 (1st argument)
📌
The 6-register argument order on Linux x86-64 is: RDI, RSI, RDX, RCX, R8, R9. Arguments beyond 6 spill onto the stack. Our function only needs 2, so both fit in registers.
③

Load 2nd Argument into RSI CPU Register

Following the same convention, the second integer argument goes into RSI. The value 4 is placed there. At this point RDI = 3 and RSI = 4 — the CPU is ready to call the function.

movesi, 4; RSI = 4 (2nd argument)
④

CALL Instruction — Jump into sum() Stack

The CALL sum instruction does two things automatically in one instruction:

1. Pushes the return address onto the stack. This is the address of the instruction immediately after the CALL inside main() — so the CPU knows where to come back to when sum() finishes.

2. Jumps execution to the first instruction of sum().

callsum; push next-instr addr, then jump to sum
⚠️
Note on RSP: Pushing the return address automatically decrements RSP (the stack pointer) by 8 bytes, because the stack grows downward in memory toward lower addresses.
⑤

Function Prologue — Set Up the Stack Frame Inside sum()

The very first instructions inside every function are called the prologue. They establish a clean workspace for this function call on the stack:

push rbp — saves the caller's (main's) base pointer onto the stack. This is critical: it ensures main()'s frame can be fully restored after sum() returns.

mov rbp, rsp — sets RBP to the current stack pointer. This creates a stable frame pointer for sum() that doesn't move even if RSP changes during the function body.

pushrbp; save caller's (main's) base pointer
movrbp, rsp; RBP now points to sum()'s frame base
💡
Analogy: A chef clearing their workstation before starting a recipe, and placing a bookmark saying "my area starts here." The prologue is always generated automatically by the compiler.
⑥

Execute the Function Body Inside sum()

Now the actual work happens. The body of sum() adds the two parameters. Because both parameters are still sitting in registers RDI and RSI (they were never copied to the stack), the addition is a single fast register operation:

leaeax, [rdi + rsi]; RAX = 3 + 4 = 7

The result 7 is placed into RAX — the register designated for integer return values by the ABI. The computation is done.

📌
Key point: Parameters a and b are never copied to stack memory for a simple function like this. They stay in RDI and RSI the whole time. Copying to the stack ("spilling") only happens if the compiler runs out of registers or the address of a parameter is taken with &.
⑦

Function Epilogue — Tear Down the Stack Frame Inside sum()

The epilogue is the exact mirror of the prologue — it restores everything that was saved and collapses the stack frame before leaving:

pop rbp — pops the saved value of main()'s RBP back from the stack into the RBP register. The stack frame for sum() effectively disappears — RSP moves back up past it.

poprbp; restore main()'s base pointer
💡
Analogy: The chef wipes down and restores their workstation to exactly how it was before they started, then hands back control of the kitchen.
⑧

RET Instruction — Return to Caller Back in main()

The RET instruction also does two things automatically:

1. Pops the return address off the top of the stack (the one pushed in step ④) back into the instruction pointer register (RIP).

2. Jumps execution to that address — which is the line inside main() immediately after the original CALL.

ret; pop return address into RIP and jump there

Back in main(), the return value 7 sits in RAX ready to be used. For example: int result = sum(3, 4); stores RAX's value — 7 — into result.

🏁
Big picture summary: The entire call used no heap memory. The stack is now exactly back to how it was before the CALL in step ④. Every CALL is always balanced by exactly one RET — the stack is self-cleaning.
