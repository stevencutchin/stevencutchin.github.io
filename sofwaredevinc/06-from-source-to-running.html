Chapter 06

From Source to Running Program

You type gcc hello.c -o hello && ./hello and text appears on screen. But between your source code and that output, an extraordinary chain of transformations happens — spanning all four layers of the C world.

The Four Layers

Every step of turning a C file into a running program involves one or more of the four conceptual layers. Knowing which layer you're in helps you diagnose problems — a syntax error is a different animal than a linker error, and both are different from a segfault at runtime.

C Language
Syntax, semantics, types, keywords — the rules you write by
C Runtime
Program startup, the standard library, memory management
Machine Environment
CPU instructions, memory addresses, OS services
Dev Ecosystem
Compiler, linker, debugger, make — the tools you use

The Program

Let's trace the entire journey for a simple, self-contained C program. Here's what we're starting with:

#include <stdio.h>

int square(int n) {
    return n * n;
}

int main(void) {
    int x = 7;
    int result = square(x);
    printf("The square of %d is %d\n", x, result);
    return 0;
}

You save this as square.c and run:

$ gcc square.c -o square
$ ./square
The square of 7 is 49

Two commands. Looks simple. But behind those two commands, here's every step that actually happens:

The Build Pipeline

The full journey: source → preprocess → compile → assemble → link → load → startup → main(). Each stage operates at a different layer.

Step by Step

Preprocessing

Dev Ecosystem The preprocessor is a text-manipulation tool that runs before the compiler even sees your code. It handles every line that starts with #.

When it sees #include <stdio.h>, it literally copies the entire contents of the stdio.h header file into your source — hundreds of lines of function declarations, type definitions, and macros. Your 12-line program becomes thousands of lines.

// What you wrote:
#include <stdio.h>

// What the preprocessor produces (abbreviated):
// ... 800+ lines from stdio.h ...
typedef struct _IO_FILE FILE;
int printf(const char *__format, ...);
// ... more declarations ...

int square(int n) {
    return n * n;
}
// ...

You can see this yourself by running gcc -E square.c — the -E flag tells gcc to stop after preprocessing and print the result.

Compilation

C Language Now the compiler proper takes over. This is where the C language rules live. The compiler reads the preprocessed source and does three things:

Lexing & parsing: Breaks your text into tokens (int, square, (, n, )...) and checks that they form valid C syntax according to the grammar. If you wrote int 42x = ;, this is where you get a syntax error.

Semantic analysis: Checks that your code makes sense — types match, variables are declared before use, functions are called with the right number of arguments. If you wrote int x = "hello";, this is where you get a type error.

Code generation: Translates your C code into assembly language — a human-readable representation of machine instructions, specific to your CPU architecture (e.g., x86-64).

# Output of gcc -S square.c (x86-64 assembly, simplified)
square:
    imull   %edi, %edi       # multiply n * n
    movl    %edi, %eax       # put result in return register
    ret                      # return to caller

main:
    subq    $16, %rsp        # reserve stack space
    movl    $7, -4(%rbp)     # x = 7 on the stack
    movl    $7, %edi         # pass 7 as argument
    call    square           # call square function
    # ... printf setup ...
    call    printf           # call printf
    movl    $0, %eax         # return 0
    ret

You can see this output with gcc -S square.c, which stops after compilation and writes square.s.

Assembly

Machine Env The assembler translates the human-readable assembly into actual machine code — raw binary bytes that the CPU can execute. Each assembly instruction like imull %edi, %edi becomes a specific sequence of bytes (like 0F AF FF).

The output is an object file (square.o) — machine code, but not yet a complete program. It contains your compiled functions, but the call to printf is still unresolved — the assembler leaves a placeholder that says "I need the address of printf, but I don't know it yet."

The object file has your machine code and data, but printf is still unresolved — that's the linker's job.

Linking

Dev Ecosystem C Runtime The linker is the final build step. It takes your object file and combines it with other necessary pieces to produce a complete executable:

The C runtime startup code (crt1.o, crti.o) — provided by the system. This contains the actual entry point _start that the OS calls, which then sets things up and calls your main().

The C standard library (libc) — where printf actually lives. The linker finds printf's machine code and connects your call printf instruction to the right address.

The linker combines your object file with the C runtime startup code and the standard library to produce a runnable executable.

The output is an ELF binary (Executable and Linkable Format on Linux) — a file that the operating system knows how to load and run. Every symbol is resolved, every address is filled in (or marked for dynamic resolution at load time).

What gcc actually does
When you type gcc square.c -o square, gcc is not just a compiler — it's a driver that orchestrates all four stages automatically. It calls the preprocessor (cpp), the compiler proper (cc1), the assembler (as), and the linker (ld) in sequence. You can stop at any stage with flags: -E (preprocess only), -S (compile to assembly), -c (compile to object file).

Running the Program

Now you type ./square. The build is done — we're in the runtime world now.

OS Loading

Machine Env The operating system kernel takes over. It reads the ELF header of your executable, creates a new process with its own virtual address space, and maps the program's sections into memory: code goes into a read-only executable region, data into a writable region, and a fresh stack is allocated at a high address.

If your program uses shared libraries (like libc.so), the dynamic linker (ld-linux.so) loads those into memory too and patches up the remaining function addresses.

C Runtime Startup

C Runtime The OS doesn't jump directly to your main() function. It jumps to _start — a function provided by the C runtime that was linked into your program. This function:

Sets up the stack pointer properly. Initializes the C standard library (including stdio — so printf has somewhere to write to). Prepares argc and argv from the command line. And then calls main().

When main() returns, _start catches the return value and calls exit() with it, which flushes output buffers, runs cleanup handlers, and asks the OS to terminate the process.

Your Code Executes

C Language Machine Env C Runtime Finally, your main() function runs. Here's what happens at each line, across the layers:

int x = 7; — Machine: the CPU stores the value 7 in a stack slot (or register).

int result = square(x); — Machine: the CPU pushes a new stack frame for square, executes the multiply instruction, and returns. Language: the function call follows C's calling convention.

printf("The square of %d is %d\n", x, result); — Runtime: this calls into the C standard library. printf parses the format string, converts integers to text, and writes to an internal buffer. That buffer eventually gets flushed to file descriptor 1 (stdout) via an OS system call (write), which sends the bytes to your terminal.

return 0; — Runtime: main returns to _start, which calls exit(0). The OS reclaims all memory and resources.

The Full Picture

Knowing which build stage produced an error tells you which layer to investigate — and what kind of fix to look for.
Analogy
Think of building a C program like publishing a book. Preprocessing is the editor inserting cross-references and expanding abbreviations. Compiling is translating the manuscript from English into the printer's typesetting language. Assembling is converting those typesetting instructions into actual metal plates. Linking is binding all the chapters and the appendix together with a cover. Loading is putting the book on the shelf. And runtime startup is opening the book to page one and starting to read.

Try It Yourself

You can watch each stage in action using gcc flags:

# Stop after preprocessing — see the expanded source
$ gcc -E square.c -o square.i

# Stop after compilation — see the assembly
$ gcc -S square.c -o square.s

# Stop after assembly — produce the object file
$ gcc -c square.c -o square.o

# See what symbols are in the object file
$ nm square.o
# T square          ← defined (your function)
# T main            ← defined (your function)
# U printf          ← undefined (needs linker)

# Full build — all stages at once
$ gcc square.c -o square

# Run it
$ ./square
The square of 7 is 49
Takeaway
A C program goes through four transformations before it runs: preprocessing (text manipulation), compilation (C → assembly), assembly (assembly → machine code), and linking (combining pieces into an executable). Then three more things happen at runtime: OS loading, C runtime startup, and finally your main(). Each stage operates at a specific layer — and knowing which layer you're in is the key to diagnosing any problem you encounter.
