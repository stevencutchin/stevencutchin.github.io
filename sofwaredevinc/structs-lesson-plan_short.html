Lesson Plan — C Structs

CS253 Duration: 75 min Student page: 06-structs.html
Key: teach live code warn / pitfall ask the class
1 Motivation 0:00–0:08
Ask: how would you store 35 students without structs?
Let them answer. Write the four-parallel-arrays version on the board.
char   names[35][64];
int    ids[35];
double gpas[35];
int    enrolled[35];
Explain the problem: these four arrays belong together but the language doesn't know that.
What happens when you pass "one student" to a function? When you sort by GPA — all four arrays have to stay in sync. Nothing enforces it.
Introduce the struct: group the fields under one name and one type.
typedef struct {
    char   name[64];
    int    id;
    double gpa;
    int    enrolled;
} Student;
2 Declaration, Access, and Memory Layout 0:08–0:22
Explain: the struct definition is a blueprint — no memory yet. Declaring a variable is when storage is reserved.
Show declaration with a designated initializer. Explain why this is preferred.
Student s = { .name = "Alice", .id = 1001, .gpa = 3.8, .enrolled = 1 };
Order-independent, self-documenting, and unnamed fields zero-initialize automatically. Use this by default.
Show field access with dot. Read and write a field.
printf("%s: %.2f\n", s.name, s.gpa);
s.gpa = 3.9;
Print sizeof(Student). Ask the class what they expect. Show it's larger than the sum of the fields.
Explain padding: each type must start at an address divisible by its own size. The compiler inserts invisible bytes to satisfy that.
One-sentence takeaway: always use sizeof — never add up field sizes yourself.
Show the struct Tag form alongside typedef. Same layout, different syntax — typedef removes the need to write struct at every use site.
3 Arrays of Structs 0:22–0:35
Explain: an array of structs is a normal array where the element type is a struct. Same mechanics, stride is sizeof(Student).
Declare a small stack array, populate with compound literals, iterate and print.
Student roster[4];
roster[0] = (Student){ .name = "Alice", .id = 1001, .gpa = 3.8 };

for (int i = 0; i < 4; i++)
    printf("%s %.2f\n", roster[i].name, roster[i].gpa);
Show the heap version. Point out that usage after malloc is identical — same dot, same loop.
Student *roster = malloc(n * sizeof(Student));
roster[2].gpa = 3.9;
free(roster);
Call out the most common malloc mistake before they make it.
Students write malloc(n * sizeof(Student*)) — thinking "I'm making a pointer, use the pointer type." Wrong. The argument to malloc is the size of what you're storing. sizeof(Student*) is always 8 regardless of struct size.
Ask: given Student *roster, how does the CPU find roster[2] in memory?
Walk through: base address + 2 × sizeof(Student). Connects pointer arithmetic, sizeof, and struct access in one question.
4 Passing Structs to Functions 0:35–0:55
Write a function that takes a Student by value and modifies GPA. Call it, then print the original — unchanged.
void bad_curve(Student s) { s.gpa += 0.1; }

Student alice = { .name = "Alice", .gpa = 3.8 };
bad_curve(alice);
printf("%.2f\n", alice.gpa);  // still 3.8
Ask the class why the GPA didn't change. Let them diagnose before you explain.
Explain: C always passes by value. The entire struct was copied — every field. The function's copy was destroyed on return.
Draw two boxes: one in main's frame, one in the function's frame. The function's box is gone on return.
Rewrite with Student *s parameter, &alice at call site, s->gpa inside. Show it works.
void good_curve(Student *s) { s->gpa += 0.1; }

good_curve(&alice);
printf("%.2f\n", alice.gpa);  // 3.9 ✓
Explain the arrow operator: s->gpa is exactly (*s).gpa — dereference then dot, collapsed into an arrow.
Write both forms on the board. Then state the rule: dot when you have a struct, arrow when you have a pointer to a struct.
Quick drill — show these one at a time, class calls out dot or arrow to access gpa.
Student   s;           // → s.gpa
Student  *p;           // → p->gpa
Student   arr[10];     // → arr[3].gpa
Student  *heap = ...;  // → heap->gpa  or  heap[0].gpa
The last two rows are the tricky ones. arr[i] and heap[i] are struct values — indexing already did the dereferencing, so you use dot.
Show const Student *s for read-only access. Explain: callers know nothing will be modified.
void print_student(const Student *s) {
    printf("%s: %.2f\n", s->name, s->gpa);
}
5 Creating Structs Inside Functions 0:55–1:05
Show a factory function returning by value. The struct is copied to the caller on return.
Student make_student(const char *name, int id, double gpa) {
    Student s = { .id = id, .gpa = gpa, .enrolled = 1 };
    strncpy(s.name, name, sizeof(s.name) - 1);
    return s;
}
Student alice = make_student("Alice", 1001, 3.8);
Show the heap version: malloc inside, return the pointer. Caller owns the memory and must free.
Student *make_student_heap(const char *name, int id, double gpa) {
    Student *s = malloc(sizeof(Student));
    if (!s) return NULL;
    s->id = id;  s->gpa = gpa;  s->enrolled = 1;
    strncpy(s->name, name, sizeof(s->name) - 1);
    return s;
}
Show the bug students will definitely write: returning the address of a local variable.
Student *bad(void) {
    Student s = { .id = 1 };
    return &s   // ← ALWAYS WRONG
}
Often appears to work in debug builds because the stack memory hasn't been overwritten yet. Silently corrupts in release or under load. The frame is gone — the pointer is a lie.
State the decision rule: does the struct need to outlive the function? → heap. Otherwise return by value.
6 Pitfalls & Wrap-up 1:05–1:15
Run through the five bugs they will write — one sentence each.
Wrong operator: compiler error "not a structure." Used . on a pointer. Fix: use ->.
Modifying a copy: function ran, nothing changed. Passed by value. Fix: Student *s + &var.
Stack pointer return: works in debug, corrupts elsewhere. Returned &localVar. Fix: return by value or malloc.
Uninitialized fields: fields contain garbage. Fix: designated initializer or memset(&s, 0, sizeof(s)).
Shallow copy: one struct's string randomly changes. Struct has a char * field. Fix: use char name[64] inside the struct.
Close with the three decisions they make every time they write a struct function.
1. Does it need to modify the struct? → pointer parameter. Otherwise by value is fine.
2. Does the struct need to outlive the function? → heap. Otherwise return by value.
3. Do I have a struct or a pointer? → dot or arrow.
Bridge: structs with a pointer to the same struct type — linked lists, trees, graphs. Everything today is the foundation for every non-trivial data structure in C.
