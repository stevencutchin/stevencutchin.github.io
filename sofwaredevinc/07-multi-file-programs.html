Multi-File Programs

Real C programs are split across many files. Header files (.h) declare the interface. Source files (.c) define the implementation. Object files (.o) are compiled independently and the linker stitches them together. Here's how — and why.

Why Split a Program into Multiple Files?

A 200-line program fits comfortably in one file. A 20,000-line program does not. Splitting code into multiple files gives you three critical benefits:

Organization. Each file has a clear responsibility — one handles math utilities, another handles user input, another handles output formatting. You can navigate the project by looking at filenames rather than scrolling through one enormous file.

Independent compilation. If you change math_utils.c, you only need to recompile that one file and re-link. You don't need to recompile the other 50 files that didn't change. On large projects, this saves minutes of build time.

Reusability. A well-designed module (like a math library) can be compiled once and linked into many different programs.

An Example Project

Let's build a small program split into three source files: a math module, a display module, and a main driver.

project/
math_utils.h — declares the math functions (interface)
math_utils.c — implements the math functions (code)
display.h — declares the display functions (interface)
display.c — implements the display functions (code)
main.c — the program entry point

The Header Files (.h)

C Language A header file is a promise. It says "these functions exist, and here is how to call them" — but it does not contain the actual implementation. It's the menu, not the kitchen.

math_utils.h
#ifndef MATH_UTILS_H
#define MATH_UTILS_H

// Declarations only — no function bodies
int square(int n);
int cube(int n);
int sum_range(int lo, int hi);

#endif
display.h
#ifndef DISPLAY_H
#define DISPLAY_H

// Declarations only
void print_result(
    const char *label,
    int value
);
void print_banner(void);

#endif
Include Guards
The #ifndef / #define / #endif pattern is called an include guard. It prevents the same header from being included twice in the same compilation unit, which would cause "redefinition" errors. Without guards, if both main.c and display.c include math_utils.h, and one of those files also includes the other, the declarations would be duplicated. The guard ensures each header's contents are processed only once.

The Source Files (.c)

C Language Each .c file includes its own .h header (to ensure its definitions match its declarations) and then provides the actual function bodies.

math_utils.c
#include "math_utils.h"

int square(int n) {
    return n * n;
}

int cube(int n) {
    return n * n * n;
}

int sum_range(int lo, int hi) {
    int total = 0;
    for (int i = lo; i <= hi; i++)
        total += i;
    return total;
}
display.c
#include <stdio.h>
#include "display.h"

void print_result(
    const char *label,
    int value
) {
    printf("  %s: %d\n",
           label, value);
}

void print_banner(void) {
    printf("=== Results ===\n");
}

And the main driver that ties them together:

// main.c
#include "math_utils.h"    // so we can call square(), cube(), sum_range()
#include "display.h"      // so we can call print_result(), print_banner()

int main(void) {
    int x = 5;

    print_banner();
    print_result("square(5)", square(x));
    print_result("cube(5)", cube(x));
    print_result("sum(1..10)", sum_range(1, 10));

    return 0;
}

How #include Actually Works

Dev Ecosystem The #include directive is a preprocessor command — it literally copy-pastes the contents of the named file into your source before the compiler sees it. This is a critical distinction from how imports work in most other languages.

#include literally copies the header file's contents into the source. The compiler then sees one big file with all declarations at the top.

This is why the header contains only declarations (function signatures), not definitions (function bodies). If the header contained function bodies, those bodies would be compiled into every .c file that includes it, producing duplicate definitions — and the linker would complain.

Analogy
A header file is like a restaurant menu — it tells you what's available and how to order it (function name, parameters, return type). The source file is the kitchen — where the food is actually made. Every diner gets a copy of the menu (#include), but there's only one kitchen (one .c file defining each function).

Separate Compilation

Dev Ecosystem C Language Each .c file is compiled independently into its own .o object file. This is the key to the multi-file build process. The compiler only sees one .c file at a time — it has no knowledge of the other files.

# Compile each .c file separately into a .o file:
$ gcc -c math_utils.c -o math_utils.o
$ gcc -c display.c    -o display.o
$ gcc -c main.c       -o main.o

When the compiler processes main.c, it sees the declarations from the headers (so it knows square takes an int and returns an int), but it doesn't have the actual code for square. It compiles the call to square as "call a function named square — I trust it exists somewhere" and leaves a placeholder in the object file.

Each .c file is compiled independently into a .o file. The linker combines all .o files plus the C runtime into a final executable.
# Link all object files into the final executable:
$ gcc math_utils.o display.o main.o -o program

# Or, do it all in one command (gcc runs all stages):
$ gcc math_utils.c display.c main.c -o program

# Run it:
$ ./program
=== Results ===
  square(5): 25
  cube(5): 125
  sum(1..10): 55

Common Multi-File Errors

Now that you understand the process, the most common errors make sense:

"undefined reference to 'square'"
This is a linker error. It means main.o calls square, but you didn't provide math_utils.o on the link command. The linker can't find the symbol. Fix: make sure all .o files are included in the final gcc command.
"implicit declaration of function 'square'"
This is a compiler warning/error. It means main.c calls square() but hasn't seen a declaration for it — you forgot to #include "math_utils.h". The compiler doesn't know what arguments square takes or what it returns, so it guesses (badly). Fix: add the #include.
"multiple definition of 'square'"
This is a linker error. It means two .o files both define square. This often happens when you accidentally put a function body in a header file — then every .c file that includes that header gets its own copy of the function. Fix: headers should contain only declarations, never definitions (with rare exceptions for static inline).

Declaration vs. Definition

C Language This distinction is the heart of multi-file C. Getting it wrong causes the errors above.

// DECLARATION — goes in the .h file
// "This function exists somewhere and has this signature."
int square(int n);           // note the semicolon — no body

// DEFINITION — goes in the .c file
// "Here is the actual code for this function."
int square(int n) {         // note the curly braces — has a body
    return n * n;
}

A function can be declared many times (each .c file that includes the header gets a copy of the declaration, and that's fine). But it must be defined exactly once across all the .c files in the program. This is called the One Definition Rule.

Declarations are promises (many copies OK). Definitions are fulfillments (exactly one copy required).

The Build with Make

Dev Ecosystem Typing three gcc -c commands and one link command gets tedious. A Makefile automates this and — critically — only recompiles files that have changed.

# Makefile
CC = gcc
CFLAGS = -Wall -g

program: main.o math_utils.o display.o
	$(CC) $(CFLAGS) -o program main.o math_utils.o display.o

main.o: main.c math_utils.h display.h
	$(CC) $(CFLAGS) -c main.c

math_utils.o: math_utils.c math_utils.h
	$(CC) $(CFLAGS) -c math_utils.c

display.o: display.c display.h
	$(CC) $(CFLAGS) -c display.c

clean:
	rm -f *.o program

Now if you edit only display.c and run make, it only recompiles display.o and re-links. The other .o files are untouched because their source files haven't changed. On a project with hundreds of files, this saves enormous amounts of time.

Make tracks file modification times. Only changed source files get recompiled — the rest use their existing .o files.

The Big Picture

Here's how all the layers participate in a multi-file build:

Dev Ecosystem gcc orchestrates the pipeline, make decides what needs rebuilding, and the linker stitches everything together.

C Language The compiler enforces syntax and type rules within each translation unit. Headers provide the declarations needed to type-check across file boundaries. The One Definition Rule ensures functions are defined exactly once.

Machine Env Each .o file contains real machine code for the target CPU. The linker assigns final memory addresses. The OS loads the executable and creates a process.

C Runtime The startup code (crt1.o) and the standard library (libc) are linked in automatically. The runtime initializes the world your code expects — stdio, the stack, argc/argv — before calling main().

Takeaway
Multi-file C programs separate interface (.h files with declarations) from implementation (.c files with definitions). Each .c file compiles independently into a .o object file. The linker combines all .o files by matching undefined symbols to their definitions. Headers are the glue — they let the compiler type-check calls to functions defined in other files. This separation enables independent compilation, faster builds, and clean code organization — but it requires understanding the declaration/definition distinction and knowing which errors come from which stage of the build process.
