Lesson Plan — Linked Lists

CS253 Duration: 75 min Prereqs: Structs, malloc/free, pointers Student page: 07-linked-lists.html
Key: teach live code warn / pitfall ask the class
1 Motivation — Why Not Just Use an Array? 0:00–0:10
Ask: what's the cost of inserting an element at the front of an array of 10,000 ints?
Let them think. Answer: you shift every single element one position right — 10,000 moves. Ask what happens when the array is full and you need to grow it.
Establish what arrays are good at and what they aren't.
Arrays are fast at random access — arr[i] in one step. They're expensive at insertion and deletion in the middle because everything has to shuffle. And their size is fixed at allocation time.
Introduce the linked list as the trade-off: give up random access, gain O(1) insert and delete anywhere, and grow freely at runtime.
Neither structure is better — they solve different problems. The skill is knowing which one fits the situation.
2 The Node Struct 0:10–0:20
Explain that a linked list is just a chain of heap-allocated structs. Each struct holds the data and a pointer to the next one.
This is nothing new — it's structs and pointers, which they already know. The only new idea is that a struct contains a pointer to another struct of the same type.
Write the Node struct. Pause on the self-referential pointer and explain why the tagged form is required here.
typedef struct Node {
    int          value;
    struct Node *next;   // can't use typedef name yet — not complete
} Node;
The typedef name Node isn't available inside the definition itself. The tag struct Node is — so you use it for the pointer field. This is the one case where the tagged form isn't just style.
Write make_node(). Walk through each line — malloc, NULL check, set fields, return.
Node *make_node(int value) {
    Node *n = malloc(sizeof(Node));
    if (!n) return NULL;
    n->value = value;
    n->next  = NULL;
    return n;
}
Every node lives on the heap. It needs to outlive the function that creates it — stack allocation would be immediately destroyed on return.
Ask: what does NULL mean for the next field?
It means "there is no next node — this is the end of the list." NULL is the sentinel that terminates every traversal. Every list operation eventually checks for it.
Draw three nodes on the board — boxes connected by arrows — with the last one pointing to NULL. This diagram is the mental model for everything that follows. Keep it on the board all lecture.
3 Building a List — Prepend and Append 0:20–0:33
Write prepend(). Talk through the two steps out loud while you type.
Node *prepend(Node *head, int value) {
    Node *n = make_node(value);
    if (!n) return head;
    n->next = head;   // step 1: new node points at old head
    return n;         // step 2: new node IS the new head
}
Emphasize the order: set n->next before returning. If you returned first and then tried to set next, you've already handed off the pointer — it's too late.
Build a list live by calling prepend three times. Show the result on the board with arrows updating after each call.
Node *list = NULL;
list = prepend(list, 3);   // 3 → NULL
list = prepend(list, 7);   // 7 → 3 → NULL
list = prepend(list, 2);   // 2 → 7 → 3 → NULL
Make the caller pattern explicit: list = prepend(list, ...). The function returns the new head and the caller must reassign. Students who forget the assignment are a common source of "why doesn't my list change?" bugs.
Write append(). Show that it requires walking to the end first, and that the empty list is a special case.
Node *append(Node *head, int value) {
    Node *n = make_node(value);
    if (!n) return head;
    if (head == NULL) return n;       // empty: new node is the list

    Node *cur = head;
    while (cur->next != NULL)         // walk to last node
        cur = cur->next;

    cur->next = n;
    return head;
}
Ask: why does append stop at cur->next != NULL rather than cur != NULL?
Because you need to stop at the last node, not past it — you need to set cur->next, which requires cur to still be valid. Stopping at cur != NULL walks one step too far.
4 Traversal 0:33–0:42
Explain the traversal idiom: start a local pointer cur at head, advance it each iteration, stop when it hits NULL. Never touch head.
Write print_list() using the for-loop form. This is the canonical traversal — students should memorize this shape.
void print_list(Node *head) {
    for (Node *cur = head; cur != NULL; cur = cur->next)
        printf("%d ", cur->value);
    printf("\n");
}
Point out what happens if you use head directly as the traversal variable.
In a print function, pass-by-value means the caller's head is unaffected — so the bug is invisible here. But if someone does this pattern inside a function that takes Node **head and advances through *head, they destroy the caller's list. The habit of using cur must be reflexive.
Write find() to show traversal with an early return.
Node *find(Node *head, int target) {
    for (Node *cur = head; cur != NULL; cur = cur->next)
        if (cur->value == target) return cur;
    return NULL;
}
Returning NULL on failure is the standard C idiom for "not found." Callers must always check the return value before using it.
5 Insertion in the Middle 0:42–0:50
Explain: to insert between two nodes you need the predecessor — the node before the insertion point. You're rewiring two pointers.
Draw the two-step rewiring on the board before writing any code. Label the steps.
Step 1: n->next = prev->next — new node points at the successor. Step 2: prev->next = n — predecessor points at the new node. Draw arrows changing on the board for each step.
Now write the code, matching the diagram exactly.
void insert_after(Node *prev, int value) {
    if (!prev) return;
    Node *n = make_node(value);
    if (!n) return;
    n->next    = prev->next;  // step 1
    prev->next = n;           // step 2
}
Show what happens if you reverse the order of the two steps.
prev->next = n first means prev->next now points to the new node — so n->next = prev->next sets n->next to itself. The tail of the list is gone. Every insertion bug comes from getting this order wrong.
6 Deletion and Freeing Memory 0:50–1:03
Explain that deletion has two parts: unlink the node from the chain, then free its memory. Both must happen. Unlinking without freeing is a memory leak. Freeing without unlinking is a dangling pointer in the list.
Write delete_front(). Emphasize saving head->next before freeing.
Node *delete_front(Node *head) {
    if (head == NULL) return NULL;
    Node *old_head = head;
    head = head->next;    // advance first
    free(old_head);       // then free
    return head;
}
Show the classic read-after-free mistake right now, before writing free_list.
// WRONG:
free(cur);
cur = cur->next;   // reading freed memory — undefined behavior

// RIGHT:
Node *next = cur->next;   // save next BEFORE freeing
free(cur);
cur = next;
This bug often appears to work — the freed memory hasn't been overwritten yet. It blows up unpredictably later, usually far from the cause. Valgrind catches it immediately.
Write free_list() applying the correct pattern.
void free_list(Node *head) {
    while (head != NULL) {
        Node *next = head->next;  // save before freeing
        free(head);
        head = next;
    }
}
Run this under Valgrind after the demo to show zero leaks. Students are often surprised how clean a properly-freed list looks.
Write delete_value() to show the predecessor-tracking pattern for the general case.
Node *delete_value(Node *head, int target) {
    if (head == NULL) return NULL;
    if (head->value == target) {        // head is the target
        Node *next = head->next;
        free(head);
        return next;
    }
    Node *prev = head;                  // find predecessor
    while (prev->next != NULL && prev->next->value != target)
        prev = prev->next;
    if (prev->next == NULL) return head; // not found
    Node *to_delete = prev->next;
    prev->next = to_delete->next;        // unlink
    free(to_delete);                     // free
    return head;
}
The head special-case exists because deleting the head requires changing which node the caller holds — there's no predecessor inside the list to update. This pattern recurs constantly. It's worth calling out explicitly.
7 Pitfalls & Wrap-up 1:03–1:15
Run through the five bugs they will write — one sentence each.
Losing head: used head as the traversal pointer. Fix: always use a separate cur.
Read after free: free(cur); cur = cur->next. Fix: save next first, always.
Wrong insertion order: updated prev->next before n->next. Fix: set the new node's next first.
Not handling empty list: dereferenced head without checking NULL. Fix: check NULL at the top of every function.
Forgetting to free: set head = NULL without walking and freeing nodes. Fix: always call free_list() before discarding a list.
Close with the three rules that prevent most bugs.
1. Never advance head. Use a local cur for traversal.
2. Save next before free. Node *next = cur->next; free(cur); cur = next;
3. When inserting, wire the new node's next before updating the predecessor.
Bridge forward: everything built here — a node with a next pointer — is the foundation for trees, graphs, hash table chaining, and every other pointer-based data structure. The pattern scales.
