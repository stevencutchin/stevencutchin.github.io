Parsing flags with getopt

getopt is the standard POSIX way to parse single-letter options. You give it a string describing which letters are valid (a colon after a letter means "takes an argument"), and it returns one option at a time until it runs out:

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <ctype.h>
#include <dirent.h>
#include <unistd.h>
#include "ProcEntry.h"

static void print_usage(const char *prog)
{
    fprintf(stderr,
        "Usage: %s [-d <path>] [-p] [-c] [-z] [-h]\n"
        "\t-d <path> Directory containing proc entries (default: /proc)\n"
        "\t-p        Display proc entries sorted by pid (default)\n"
        "\t-c        Display proc entries sorted by command lexicographically\n"
        "\t-z        Display ONLY proc entries in the zombie state\n"
        "\t-h        Display this help message\n",
        prog);
}

int main(int argc, char **argv)
{
    const char *base_path = "/proc";
    int sort_by_comm = 0;        // 0 = pid (default), 1 = comm
    int zombies_only = 0;

    int opt;
    while ((opt = getopt(argc, argv, "d:pczh")) != -1) {
        switch (opt) {
            case 'd': base_path = optarg; break;
            case 'p': sort_by_comm = 0; break;
            case 'c': sort_by_comm = 1; break;
            case 'z': zombies_only = 1; break;
            case 'h':
                print_usage(argv[0]);
                return EXIT_SUCCESS;
            default:                          // unknown flag
                print_usage(argv[0]);
                return EXIT_FAILURE;
        }
    }

    /* ... directory walk and rest of program ... */
    return EXIT_SUCCESS;
}
The colon after d "d:pczh" means: -d takes an argument (the colon says so), -p, -c, -z, and -h don't. When getopt sees -d, it stuffs the argument into the global optarg for you to read.
Send usage to stderr, not stdout Notice fprintf(stderr, ...) in print_usage. This is intentional. Anything on stdout gets diffed against the reference output during grading. Errors and usage messages always go to stderr — that's a near-universal convention in Unix tools, and it keeps ./myps -h from polluting any pipeline you stick it in.

Filtering for PID directories

Now we need to walk the chosen directory and pull out only the entries that look like process IDs. That means two conditions:

  1. It's a directory — d_type == DT_DIR. Files and symlinks are not processes.
  2. Its name starts with a digit — isdigit(name[0]). PID directory names are integers; anything alphabetic is not a process.

Both conditions matter. /proc/sys/ is a directory but its name doesn't start with a digit. A regular file in /proc starts with a letter. The pair of checks together gives you exactly the PID directories.

scandir = readdir loop + filter callback + sort + allocation, all in one call.

scandir does the work for you

You could call opendir + readdir in a loop, allocate your own array, and copy each matching entry — and that's a perfectly valid solution. But the standard library already has the helper you want: scandir. It walks a directory, calls your filter callback on each entry, allocates an array of struct dirent* for the entries that pass, and hands the whole thing back to you.

// Filter callback — return non-zero to keep the entry, 0 to drop it.
static int pid_filter(const struct dirent *d)
{
    return d->d_type == DT_DIR
        && isdigit((unsigned char)d->d_name[0]);
}

Then in main, after option parsing:

struct dirent **dirents = NULL;
int n = scandir(base_path, &dirents, pid_filter, NULL);
if (n < 0) {
    perror("scandir");
    return EXIT_FAILURE;
}

// Optional debug — useful while developing, but goes to STDERR so it
// doesn't pollute stdout for the integration test.
for (int i = 0; i < n; i++) {
    fprintf(stderr, "  dirent[%d]: %s\n", i, dirents[i]->d_name);
}
The signature decoded int scandir(const char *path, struct dirent ***namelist, int (*filter)(const struct dirent *), int (*compar)(...)); — pass NULL for compar if you don't want scandir to sort (we'll sort the ProcEntry array ourselves later, so order here doesn't matter). The triple-pointer for namelist is so the function can write your pointer variable, which itself points to an array of pointers.
Why the cast on isdigit? isdigit takes an int but is undefined for negative values other than EOF. Plain char may be signed on some platforms — passing d_name[0] directly can sign-extend a high-byte character to a negative int. The cast to unsigned char first fixes that. Without the cast, you'll get a UBSan warning on some systems — and lose code-quality points.

Cleanup is your responsibility

scandir allocates two things that you have to free: each individual struct dirent it copied (one per entry), and the array of pointers itself. Forgetting either is a leak that valgrind will catch.

// At program exit, after you're done using the dirents:
for (int i = 0; i < n; i++) {
    free(dirents[i]);
}
free(dirents);
Two-level free This is the same pattern you'll use for myprocs on the next page. When you have an array of pointers to heap-allocated objects, you free the objects in a loop, then free the array. Inside-out, always.

Going deeper — the readdir alternative If you'd rather use opendir/readdir directly, the loop looks like this. It's more code but a useful pattern to know: 
DIR *dir = opendir(base_path);
if (dir == NULL) { perror("opendir"); return EXIT_FAILURE; }

struct dirent **dirents = NULL;
int n = 0, cap = 0;
struct dirent *d;
while ((d = readdir(dir)) != NULL) {
    if (!pid_filter(d)) continue;
    if (n == cap) {
        cap = cap ? cap * 2 : 16;
        dirents = realloc(dirents, cap * sizeof(*dirents));
    }
    dirents[n] = malloc(sizeof(struct dirent));
    memcpy(dirents[n], d, sizeof(struct dirent));
    n++;
}
closedir(dir);
Same end result. Pick whichever you find clearer — the grader doesn't care which one you used as long as the behavior is correct.
