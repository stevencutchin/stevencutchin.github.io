Part 0 — Orientation

The myps Project

A simplified ps command, built by reading the kernel's own bookkeeping out of the /proc filesystem. This is the capstone — every piece you've learned in the last four weeks lands here.

What you're building

Open a terminal on any Linux machine and run ps. The output is a snapshot of every process the kernel currently knows about: PID, parent PID, state, CPU time, command name. You're going to build a simplified version of that tool.

The real ps reads its information out of a special directory called /proc — a filesystem that doesn't actually exist on disk. The kernel synthesizes its contents on the fly so userspace programs have a uniform way to ask "what's going on right now?" Your job is to walk that directory, read each process's stat file, parse the fields, sort the results, and print them.

The shape of the problem Walk a directory. For each entry that looks like a process, open a file and parse seven fields out of one line. Build an array of structs. Sort the array. Print it. That's the whole project — but every step exercises a different concept from the course.

The /proc filesystem

Most directories on your machine are backed by storage — bytes that live somewhere on a disk. /proc is different. When you ls /proc, the kernel generates the listing on demand, exposing its internal data structures as if they were files. Every running process gets its own numbered subdirectory, and inside that subdirectory there are virtual files describing the process: its memory map, its open file descriptors, its scheduling stats, and so on.

/proc has hundreds of entries. We only care about the numbered ones — and inside each, only the stat file.
Why this is volatile Process directories appear and disappear constantly. By the time you finish reading /proc/1234/stat, PID 1234 may have exited and the directory may already be gone. Your code needs to handle file-open failures gracefully — they're not bugs, they're a fact of life on a running system.

Anatomy of a stat file

Each /proc/[pid]/stat file contains one line of space-separated fields. There are 52+ fields total, but you only need seven of them. Here's a real one with the relevant fields highlighted:

Out of 50+ space-separated fields, you only need fields 1, 2, 3, 4, 14, 15, and 20.
Watch the comm field Field 2 (the executable name) is wrapped in parentheses for a reason — the name itself can contain spaces, slashes, and other characters that would break a naive %s parse. The parens act as delimiters. We'll come back to this on the next page when we write the parser.

The four layers, all at once

This project is a capstone because it touches every layer of the framework we've used all semester. Here's the map:

▸ C Language
struct definitions, malloc/free, pointers, arrays, function pointers (qsort callbacks). The ProcEntry type itself and the array-of-pointers pattern.
▸ C Runtime
file streams (fopen, fscanf, fclose), directory streams (opendir, readdir), command-line argument parsing with getopt.
▸ Machine Environment
the kernel exposing process state through /proc. Understanding that "files" can be a kernel-synthesized API, not bytes on disk.
▸ Dev Ecosystem
unit testing in mytests.c, valgrind for memory correctness, diff-based integration testing, make targets, the Style Guide.

Project file layout

When you accept the GitHub Classroom link, you'll get a starter repository with this structure. Some files are provided and frozen (don't touch the headers or the Makefile), some are provided as starting points, and some are yours to write from scratch.

ProcEntry.h Frozen. Defines the ProcEntry struct and the function signatures you must implement. The grader compiles against this exact file — do not modify it.
ProcEntry.c You write. The four support functions: CreateProcEntry, DestroyProcEntry, CreateProcEntryFromFile, PrintProcEntry. Part 1 of the project.
mytests.c You write. Unit tests for every ProcEntry function. Must run clean under valgrind. Worth 40 of the 120 points — don't skip it.
myps.c You write. The main program: option parsing, directory walking, array building, sorting, printing. Parts 2 and 3 of the project.
Makefile Provided. Builds myps and mytests. Includes memtest-mytests and memtest-myps targets that wrap valgrind.
test_data.tgz Provided. Extract with tar -xzf test_data.tgz. Contains the onyx_proc reference dataset and the expected outputs you'll diff against.

The grading lens

The 120 points break down across four categories, and they're worth knowing about before you start coding so you can build with the rubric in mind:

  • Coding Style — 20 pts. The CS253 Style Guide we've been using all semester. Naming, indentation, function length, comments at the right places.
  • Code Quality — 20 pts. Zero compiler warnings. Zero valgrind errors. No leaked bytes at exit. The Makefile builds with -Wall -Wextra, so warnings will show up — fix them.
  • Unit Testing — 40 pts. Your mytests.c exercises the ProcEntry functions. The grader runs its own unit tests too — design your ProcEntry.c to behave correctly under both expected and unexpected inputs (NULL paths, missing files, malformed data).
  • Integration Testing — 40 pts. The grader runs your myps, redirects stdout to a file, and diffs against the expected output. Any extra debug output on stdout breaks this — push debug prints to stderr.
The most expensive bug you can make Anything you write to stdout that isn't the column header or a PrintProcEntry line will fail every integration test. fprintf(stdout, ...) for debug = 40 points lost. Use fprintf(stderr, ...) instead, or guard debug output behind a verbose flag. We'll cover this in detail on page 5.
