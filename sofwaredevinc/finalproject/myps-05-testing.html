Part 4 — Testing & Submission

Make the grade certain

Code that compiles and runs is half the job. Code that runs cleanly — no leaks, no diff failures, no surprise stdout — is the half that earns the points. This page is the difference between submitting and submitting confidently.

Two kinds of testing, two kinds of evidence

Your submission has to survive two completely different test regimes. Knowing which one is which keeps you from chasing the wrong fix when something breaks.

Unit tests live in mytests.c, exercise the ProcEntry functions in isolation, and run under valgrind. Their goal: prove that each function behaves correctly on every input it might see, including weird ones (NULL, missing files, malformed data). Evidence of success: every test prints "passed" and valgrind reports zero errors.

Integration tests run your myps binary end-to-end, redirect its stdout to a file, and diff that file against a reference output the instructor generated from a known-correct implementation. Evidence of success: diff exits with status 0 and produces no output.

Unit tests find logic bugs. Integration tests find output bugs. Most students who lose integration points do not have a logic bug — their parsing is fine, their sort is fine, their cleanup is fine. They have an extra newline, a debug print on stdout, or a column width that's off by one space. The bug is in the bytes, not the algorithm.

Reading valgrind output

The Makefile gives you two memtest targets: memtest-mytests wraps your unit-test binary in valgrind, memtest-myps does the same for the main program. Both must run completely clean before you submit. There are exactly three numbers to look for:

$ make memtest-myps
valgrind --tool=memcheck --leak-check=yes --show-reachable=yes ./myps -d test_data/onyx_proc -p > /dev/null

==23104== Command: ./myps -d test_data/onyx_proc -p
...

==23104==     in use at exit: 0 bytes in 0 blocks            ← this must be 0
==23104==   total heap usage: 1,847 allocs, 1,847 frees, 84,672 bytes allocated  ← these must match
==23104==

If any of the three is wrong:

==23104==     in use at exit: 240 bytes in 12 blocks           ← LEAK · 12 things weren't freed
==23104==   total heap usage: 1,847 allocs, 1,835 frees     ← confirms 12 missing frees
...
==23104== ERROR SUMMARY: 4 errors from 2 contexts            ← invalid reads/writes happened

For leaks, scroll up — valgrind will tell you which allocation site the leaked memory came from:

==23104== 12 bytes in 1 blocks are definitely lost in loss record 4 of 8
==23104==    at 0x4848899: malloc (in vgpreload_memcheck-amd64-linux.so)
==23104==    by 0x4914B41: strdup (strdup.c:42)
==23104==    by 0x10936A: CreateProcEntry (ProcEntry.c:24)     ← line in YOUR code
==23104==    by 0x1093F8: CreateProcEntryFromFile (ProcEntry.c:67)
==23104==    by 0x10921A: main (myps.c:84)

The bottom of the stack is your main. The top is the malloc/strdup that allocated the leaked memory. The line in your code that called the failing allocator is the one to investigate — was it stored, did the destroy free it, did an early-return path skip cleanup?

"still reachable" still counts Valgrind has four leak categories: definitely lost, indirectly lost, possibly lost, and still reachable. The Makefile's --show-reachable=yes flag means all four count against you for code-quality points. Anything you allocate must be freed before main returns.

The integration test pipeline

Each integration test is a single shell pipeline. The ones from the project description, verbatim — these are the exact commands the grader runs:

# Test 1 · sort by PID — straight diff
$ ./myps -d test_data/onyx_proc -p > myps-pid_sort.out
$ diff myps-pid_sort.out test_data/onyx_proc_expected/myps-pid_sort.out

# Test 2 · sort by command — strip BOTH sides to just the comm column, then diff
$ ./myps -d test_data/onyx_proc -c | awk '{print $7}' > myps-cmd_sort-stripped.out
$ cat test_data/onyx_proc_expected/myps-cmd_sort.out | awk '{print $7}' > \
    test_data/onyx_proc_expected/myps-cmd_sort-stripped.out
$ diff myps-cmd_sort-stripped.out \
       test_data/onyx_proc_expected/myps-cmd_sort-stripped.out

# Test 3 · zombies-only filter
$ ./myps -d test_data/onyx_proc -z > myps-zombie_only.out
$ diff myps-zombie_only.out test_data/onyx_proc_expected/myps-zombie_only.out
Why test 2 strips both sides Several processes can share a command name — for example, multiple (nfsd) entries. When the project sorts by command, the spec doesn't require any secondary sort on PID, so two correct implementations can legitimately disagree on the order of the duplicates. Stripping both sides down to just column 7 (the (comm) field) means the diff compares only what the -c sort actually controls. Without that step, you'd fail the test for being correct in a different way than the reference.

If diff prints anything, the test failed. If it prints nothing and exits with status 0, the test passed. Confirm exit status with echo $? after each diff — you want to see 0 three times in a row before you submit.

Three integration tests = ~13 points each = 40 of the 120 total. Diff is binary: pass or fail.

The stdout-discipline gotcha

This is the single most expensive bug a student can ship. It costs 40 points on a clean codebase and is invisible during local development if you haven't been paying attention. The story:

You added a print to debug your filter. Maybe printf("entry: %s\n", dirent->d_name). You forgot to take it out before submission. On your machine, the output looks fine to your eye — there's just one extra line at the top. But the grader doesn't read your output with their eye; they pipe it through diff. The first line is wrong. Every line after that is shifted. diff sees thousands of mismatches. You lose every integration point.

The discipline rule Never write debug output to stdout. Always use fprintf(stderr, ...). The integration test only redirects stdout — stderr stays on your terminal where you can read it. If you're disciplined about this from day one, you can keep all your debug prints in the code permanently and they'll never break a single test.

The pattern in practice

// BAD — kills your integration tests
printf("DEBUG: scanned %d dirs\n", n);
printf("about to call CreateProcEntryFromFile\n");

// GOOD — debug output goes to stderr, integration test only watches stdout
fprintf(stderr, "DEBUG: scanned %d dirs\n", n);
fprintf(stderr, "about to call CreateProcEntryFromFile\n");

If you want to silence the debug output without removing it, redirect stderr to /dev/null:

$ ./myps -d test_data/onyx_proc -p 2>/dev/null | head
$ ./myps -d test_data/onyx_proc -p 2>debug.log > myps-pid_sort.out
$ diff myps-pid_sort.out test_data/onyx_proc_expected/myps-pid_sort.out  # clean
The only things on stdout are the column header and PrintProcEntry rows. Everything else — usage messages, error messages, debug traces, "scanning %d entries" status, perror failures — goes to stderr. If you internalize this one rule, you will not lose integration points to formatting.

One last build-clean run, on a fresh checkout

Don't trust your dev tree. The single best protection against "but it works on my machine" is to clone the repo fresh and run everything end-to-end:

$ cd /tmp
$ git clone <your-repo-url> check
$ cd check
$ tar -xzf test_data.tgz
$ make clean && make
$ make memtest-mytests
$ make memtest-myps
$ ./myps -d test_data/onyx_proc -p > myps-pid_sort.out
$ ./myps -d test_data/onyx_proc -c | awk '{print $7}' > myps-cmd_sort-stripped.out
$ cat test_data/onyx_proc_expected/myps-cmd_sort.out | awk '{print $7}' > \
    test_data/onyx_proc_expected/myps-cmd_sort-stripped.out
$ ./myps -d test_data/onyx_proc -z > myps-zombie_only.out
$ diff myps-pid_sort.out test_data/onyx_proc_expected/myps-pid_sort.out
$ diff myps-cmd_sort-stripped.out test_data/onyx_proc_expected/myps-cmd_sort-stripped.out
$ diff myps-zombie_only.out test_data/onyx_proc_expected/myps-zombie_only.out

Every command succeeds silently. Every diff returns nothing. If anything breaks here, fix it on your dev tree, push, and rerun the whole sequence. Don't submit until this script is silent end-to-end.

Closing the loop

Four pages ago we mapped this project to four layers — C Language, C Runtime, Machine Environment, Dev Ecosystem. Look at what you've built, layer by layer:

  • A struct with paired Create/Destroy functions and unit tests — the C Language layer, end to end.
  • A program that uses file streams, directory streams, command-line argument parsing, and the qsort library function — the C Runtime in active use.
  • A pipeline that reads the kernel's process state through the /proc filesystem — the Machine Environment, exposed as files.
  • A workflow built around valgrind, diff-based integration tests, a Makefile, and git checkpoints — the Dev Ecosystem, used the way it's actually used in industry.

That's not bad for one project. You've integrated every layer of the framework, and the artifact you're submitting is a small but complete Unix tool that does something useful. When you finish the submission, take a moment to sit with that — it's harder than it looks.

One last note from your instructor Submit early enough that you have time to react if something breaks. The integration tests are deterministic — if they pass on your fresh-checkout run, they'll pass for the grader. The most painful losses on this project come from "I'll just push five minutes before the deadline" turning into "I can't figure out why diff is failing and the deadline just passed." Give yourself a buffer. You've earned it.
