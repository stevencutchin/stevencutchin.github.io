Linked Lists — Class Guide

CS253  ·  75 min  ·  07-linked-lists.html

Motivation

  1. Ask: what's the cost of inserting at the front of a 10,000-element array?
  2. Explain: arrays require shifting everything — O(n). Size is also fixed at allocation.
  3. Introduce the trade-off: linked lists give O(1) insert/delete anywhere, grow freely at runtime, but lose O(1) random access.

The Node Struct

  1. Explain: a linked list is just heap-allocated structs chained by pointers. Nothing new — structs and pointers they already know.
  2. Write the Node definition. Explain why the tagged form is required for the self-referential pointer. 
    typedef struct Node {
    int          value;
    struct Node *next;
} Node;
  3. Write make_node() — malloc, NULL check, set fields, return.
  4. Ask: what does NULL mean for the next field? Answer: end of list. Every traversal stops here.
  5. Draw three nodes on the board with arrows. Leave it up for the rest of class.

Building a List

  1. Write prepend(). Talk through the two steps: new node points at old head, return new node as head.
  2. Build a list live — call prepend three times, update the board diagram after each call.
  3. Call out the caller pattern: list = prepend(list, x). Forgetting the reassignment is a common bug.
  4. Write append(). Handle the empty list case. Walk to last node with cur->next != NULL.
  5. Ask: why stop at cur->next != NULL and not cur != NULL? Answer: you need to set cur->next — you can't do that if cur is already NULL.

Traversal

  1. State the rule: never advance head. Always use a local cur pointer.
  2. Write print_list() using the for-loop form. This is the canonical traversal shape — memorize it. 
    for (Node *cur = head; cur != NULL; cur = cur->next)
  3. Write find() to show traversal with early return and NULL on failure.
  4. Warn: using head as the traversal variable is invisible in read-only functions (pass by value protects you) but destroys the list if the function holds a Node **. Build the habit now.

Insertion in the Middle

  1. Explain: you need the predecessor node — the one before the insertion point.
  2. Draw the two rewiring steps on the board before writing any code.
  3. Write insert_after() matching the diagram: set n->next first, then update prev->next.
  4. Show what happens if you reverse the steps — prev->next = n first loses the tail of the list entirely.

Deletion

  1. Explain: deletion = unlink the node + free its memory. Both required. One without the other is a bug.
  2. Show the read-after-free mistake immediately, before writing any deletion code. 
    free(cur);              // WRONG: freed
cur = cur->next;        // reading freed memory — undefined behavior

Node *next = cur->next; // RIGHT: save next first
free(cur);
cur = next;
  3. Write delete_front(). Save head->next, free old head, return new head.
  4. Write free_list() using the save-next pattern.
  5. Write delete_value(). Handle the head special case first, then the predecessor-walk for the general case.
  6. Ask: why does deleting the head node need a special case? Answer: there's no predecessor inside the list to update — the change has to propagate to the caller.

Wrap-up

  1. Five bugs they will write: losing head · read after free · wrong insertion order · not checking NULL · forgetting to free.
  2. Three rules that prevent most of them: use cur not head · save next before free · wire new node's next before updating predecessor.
  3. Bridge: this node-with-a-next-pointer pattern is the foundation for trees, graphs, and hash table chaining. The shape scales to every pointer-based data structure in C.
