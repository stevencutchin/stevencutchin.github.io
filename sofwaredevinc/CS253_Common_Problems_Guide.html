Common Problems Guide

Organized by the Four Layers — find where you're stuck, understand why, fix it

When you hit a problem, figure out which layer it lives in. A syntax error is a Language problem. A segfault is a Runtime problem. "File not found" might be a Machine problem. A merge conflict is an Ecosystem problem. The "Crosses into" field shows when bugs span layers.

C Language Problems

Something wrong with what you wrote — illegal syntax, or legal syntax that doesn't mean what you think.

Pointer Confusion
Symptom
"Incompatible pointer type" or "assignment from incompatible type"
Why
Mixing up levels of indirection. An int* is not an int**, and neither is an int.
Fix
Trace the type from declaration to use. What type is this variable? What type does this expression produce? Do they match?
Crosses
Runtime — incorrect pointer types lead to reading memory as the wrong type
Symptom
Using & when you should use *, or vice versa
Why
& gives the address (up one level). * follows an address to the value (down one level). They're opposites.
Fix
Read & as "address of" and * as "thing at this address."
Symptom
Pointer arithmetic gives unexpected results
Why
Pointer math is scaled by the pointed-to type's size. p+1 on an int* moves by sizeof(int) bytes, not 1.
Fix
p[i] is exactly *(p + i). The compiler handles scaling. For byte-level access, cast to char*.
Crosses
Machine — scaling depends on type sizes on your specific system
Array Mistakes
Symptom
Off-by-one: accessing array[n] when array has n elements
Why
Zero-indexed. Size n means valid indices 0 through n-1. No bounds checking.
Fix
Use < n in loops, never <= n.
Crosses
Runtime — out-of-bounds access corrupts stack or heap, crashes far from the bug
Symptom
sizeof() on an array parameter gives 8, not the array size
Why
Arrays decay to pointers when passed to functions. sizeof measures the pointer.
Fix
Always pass array size as a separate parameter.
Symptom
arr1 = arr2 doesn't compile
Why
Arrays aren't assignable in C. They're memory regions, not first-class values.
Fix
Use memcpy() or copy element by element.
String Pitfalls
Symptom
Garbage at end of string, or crashes in string functions
Why
Missing null terminator. String functions walk memory until '\0'. Without it, they keep going.
Fix
Allocate n+1 for a string of length n. Verify '\0' is present after manual operations.
Crosses
Runtime — reading past a buffer is undefined behavior and a security vulnerability
Symptom
== to compare strings always fails
Why
== compares addresses, not contents.
Fix
Use strcmp(s1, s2) == 0.
Symptom
Buffer overflow writing into a fixed-size char array
Why
C doesn't check if data fits. Writing past the end corrupts adjacent memory.
Fix
Use strncpy or snprintf with explicit size limits. Never use gets().
Crosses
Runtime + Machine — buffer overflows corrupt the stack and are a classic exploit vector
Struct & Type Issues
Symptom
Using . when you need ->, or vice versa
Why
. is for struct values, -> is for struct pointers. p->field means (*p).field.
Fix
Is it a struct or a pointer to a struct? Pointer → ->. Struct itself → .
Symptom
Struct = copy doesn't copy what pointers point to
Why
Shallow copy. Both structs now point to the same heap memory.
Fix
Write a deep copy function that allocates new memory and copies contents.
Crosses
Runtime — shared heap pointer leads to double-free or use-after-free
Function & Scope Issues
Symptom
Returning a pointer to a local variable — works sometimes, crashes later
Why
Local variables die when the function returns. The pointer becomes dangling.
Fix
Allocate on the heap (caller must free), pass a buffer as parameter, or return by value.
Crosses
Runtime — stack memory is reused by the next function call
Symptom
Function changes to a parameter don't affect the caller's variable
Why
C is pass-by-value. The function gets a copy.
Fix
Pass a pointer: void modify(int *x) { *x = 42; } then call modify(&myVar).

C Runtime Problems

Your syntax is legal, but runtime behavior is wrong or dangerous. Crashes often appear far from the actual mistake.

Segmentation Faults
Symptom
"Segmentation fault (core dumped)"
Why
Accessed memory you don't own: NULL dereference, freed pointer, way-out-of-bounds, uninitialized pointer.
Fix
Run in gdb. At the crash, use bt. Inspect the pointer/index. The bug is often earlier than the crash.
Crosses
Machine — the OS enforces memory protection through page tables
Symptom
Segfault that only happens sometimes or with certain inputs
Why
Uninitialized or freed memory that sometimes still holds valid data. Undefined behavior.
Fix
Initialize all variables. Use valgrind to find uninitialized reads. Set pointers to NULL after free.
Memory Leaks
Symptom
Valgrind: "definitely lost" or "indirectly lost" bytes
Why
Allocated memory never freed, or lost your only pointer to it.
Fix
Every malloc needs a corresponding free. Trace each allocation's lifecycle.
Crosses
Language — often caused by losing pointer ownership across function calls
Symptom
Valgrind: "still reachable" bytes
Why
You still have a pointer but never freed before exit.
Fix
Add cleanup before exit. Free everything you allocated.
Heap Corruption
Symptom
Valgrind: "Invalid write" or "Invalid read"
Why
Accessing memory outside your allocation: past a buffer, freed pointer, out-of-bounds index.
Fix
Check malloc sizes, array indices, and pointer lifetimes.
Symptom
"double free detected" or valgrind "Invalid free"
Why
Calling free() on the same address twice. Corrupts the heap allocator.
Fix
After free(p), set p = NULL. Only one owner frees shared memory.
Crosses
Language — usually a shallow copy problem
Stack Overflow
Symptom
Segfault with deep recursion or very large local arrays
Why
Stack has a fixed size (~8 MB). Infinite recursion or huge local arrays exhaust it.
Fix
Ensure recursion has a terminating base case. Use malloc for large data.
Crosses
Machine — stack size is set by the OS (ulimit -s)
Build Pipeline Issues
Symptom
Linker: "undefined reference to 'function_name'"
Why
Called a function but the linker can't find the implementation. The .c file isn't in your build.
Fix
Include all .c files in gcc or Makefile. Check name matches exactly.
Crosses
Ecosystem — often a Makefile configuration issue
Symptom
Linker: "multiple definition of 'variable_name'"
Why
A global defined in a header included by multiple .c files.
Fix
Use extern in the header. Define in exactly one .c file.
Symptom
Preprocessor errors with #include
Why
Missing include guards cause double inclusion.
Fix
Always use: #ifndef FILE_H / #define FILE_H / ... / #endif

Machine Environment Problems

Problems from the physical system: hardware, the OS, and assumptions C makes about the machine.

Platform-Dependent Behavior
Symptom
Code works on one machine, fails on another
Why
int size, struct padding, byte order vary by platform.
Fix
Use sizeof() instead of hardcoded sizes. Use stdint.h types for exact sizes.
Symptom
Different results at -O0 vs -O2
Why
Undefined behavior that happens to work unoptimized. Compiler assumes UB never happens.
Fix
Compile with -Wall -Wextra -fsanitize=undefined to find the root cause.
File System & OS Issues
Symptom
"Permission denied" or "No such file or directory"
Why
Wrong path, wrong working directory, or missing permissions.
Fix
Always check fopen() return value. Print the path. Check working directory.
Crosses
Language — not checking fopen's NULL return leads to NULL dereference
Symptom
Works interactively but fails reading from file or pipe
Why
Different buffering modes. EOF handling may differ.
Fix
Always check return values of scanf/fgets for EOF.
Memory & Process Issues
Symptom
malloc returns NULL
Why
OS can't satisfy the request — possibly a size calculation bug asking for gigabytes.
Fix
Always check if malloc returned NULL. Verify allocation size makes sense.
Symptom
"Too many open files"
Why
File descriptor limit (~1024). Opening in a loop without closing exhausts it.
Fix
Close files when done. In loops, open and close within the loop body.

Development Ecosystem Problems

Your code may be fine, but you can't build it, track it, or find the bug.

Git Problems
Symptom
Merge conflict
Why
Two changes to the same lines. Git can't pick one.
Fix
Find the conflict markers, decide the correct code, remove markers, git add, git commit.
Symptom
Accidentally committed binaries or .o files
Why
No .gitignore.
Fix
Create .gitignore with *.o / *.out / a.out. Remove tracked: git rm --cached filename.
Symptom
"Your branch is behind" — push rejected
Why
Remote has changes you don't have locally.
Fix
git pull, resolve conflicts, then push.
Makefile Problems
Symptom
"missing separator. Stop."
Why
Recipes must start with a tab, not spaces.
Fix
Use real tabs. Check your editor's tab vs. spaces setting.
Symptom
"Nothing to be done" or doesn't rebuild after changes
Why
Dependencies are wrong.
Fix
List .h files as dependencies. Use make clean && make to force rebuild.
Debugging Tool Issues
Symptom
gdb: "No debugging symbols found"
Why
Compiled without -g.
Fix
Recompile with -g. Add it to Makefile CFLAGS.
Symptom
Valgrind output is overwhelming — hundreds of errors
Why
One bug cascades into many. Early errors propagate.
Fix
Fix the first error and rerun. One fix often eliminates dozens.
Compiler Warning Interpretation
Symptom
"Implicit declaration of function"
Why
Calling a function not yet declared. Missing #include or prototype.
Fix
Add the appropriate #include or function prototype.
Symptom
"Format '%d' expects 'int', but argument has type 'long'"
Why
Format specifier doesn't match argument type.
Fix
%d=int, %ld=long, %f=double, %zu=size_t, %p=pointer.
Symptom
"Variable may be used uninitialized"
Why
A code path reads the variable before assignment. Value will be stack garbage.
Fix
Initialize all variables at declaration. The compiler is almost always right.
Crosses
Runtime — uninitialized variables cause unpredictable behavior

Quick Diagnostic: Which Layer Is My Problem In?

Work through in order. The first "No" tells you where to start.

1. Does your code compile?
Language. Read the first error — file, line, and what's wrong. Fix it first; later errors are often caused by it.
Keep going ↓
│
2. Does it compile without warnings (-Wall -Wextra)?
Language/Runtime in disguise. Warnings predict bugs. Fix them.
Keep going ↓
│
3. Does it run without crashing?
Runtime. Run in gdb, bt at crash. Common: NULL, out of bounds, use-after-free.
Keep going ↓
│
4. Does it produce correct output?
Logic error (Language + Runtime). Step through in gdb. Compare values against expectations.
Keep going ↓
│
5. Does valgrind report zero errors and zero leaks?
Runtime/Machine. Memory errors are real bugs even if output looks correct. Fix every one.
Keep going ↓
│
6. Does it pass all provided tests?
Edge case. Empty input? Large input? Zero values? I/O format mismatch?
In good shape. Review for style and submit. ✓
