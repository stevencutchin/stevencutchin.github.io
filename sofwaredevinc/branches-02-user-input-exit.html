User Input & Exit Status

Programs that branch on hardcoded values aren't very useful. Real programs read input at runtime, validate it, and signal success or failure to the shell when they exit.

Reading User Input

C reads terminal input through the standard library. The two functions you'll use most are scanf for formatted values and fgets for lines of text. They work differently and have different failure modes — knowing both is essential.

Input flows: user → stdin buffer (on Enter) → scanf/fgets → your variable. The buffer is the source of most input bugs.

scanf — Reading Formatted Values

scanf reads from stdin, parses the input according to a format string, and stores the result in the variables you provide. You pass pointers to those variables — scanf needs an address to write into.

int age;
double gpa;

printf("Enter age: ");
int result = scanf("%d", &age);   // & gives scanf the address of age

if (result != 1) {               // scanf returns number of items read
    printf("Invalid input\n");
    return 1;
}

printf("Enter GPA: ");
scanf("%lf", &gpa);              // %lf for double, %f for float
Always check scanf's return value
scanf returns the number of items successfully read. If the user types "abc" when you expect a number, scanf returns 0 — it read nothing. If stdin is closed (EOF), it returns EOF (-1). Ignoring the return value means your variable contains garbage and your program silently produces wrong results.

The Leftover Newline Problem

When the user presses Enter, a \n character is left in the input buffer. A subsequent scanf("%c") or fgets() will immediately consume that newline instead of waiting for new input. This is one of the most common beginner bugs in C input handling.

scanf("%d") consumes the digits but leaves the newline in the buffer. The next read gets the leftover \n instead of new user input.
int age;
char initial;

scanf("%d", &age);

// flush the leftover newline before reading a char
while (getchar() != '\n');

scanf("%c", &initial);   // now waits for real input

fgets — Safer Line Reading

fgets(buffer, size, stdin) reads a full line of text into a fixed-size buffer. It is safer than scanf("%s") because it will never write more than size - 1 characters, leaving room for the null terminator. It reads and keeps the newline character at the end of the line.

char name[64];

printf("Enter your name: ");
if (fgets(name, sizeof(name), stdin) == NULL) {
    printf("Input error or EOF\n");
    return 1;
}

// strip the trailing newline fgets includes
name[strcspn(name, "\n")] = '\0';

printf("Hello, %s\n", name);
fgets vs scanf("%s"): always prefer fgets for strings
scanf("%s", buf) stops at whitespace so it can't read a full name like "Jane Smith", and has no built-in length limit — it will overflow your buffer if the user types too much. fgets reads the whole line and respects the buffer size. Use fgets whenever you're reading text.
fgets stores the newline in the buffer. Use strcspn(buf, "\n") to find and replace it with '\0' before using the string.

Branching on Input

The pattern for interactive programs is always: prompt, read, validate, then branch on the result. Validation — checking that the input is what you expect before acting on it — is the step most beginners skip.

int age;
printf("Enter age (1-120): ");

if (scanf("%d", &age) != 1) {
    fprintf(stderr, "Error: expected a number\n");
    return 1;
}

if (age < 1 || age > 120) {
    fprintf(stderr, "Error: age must be between 1 and 120\n");
    return 1;
}

// safe to use age now
printf("You are %d years old.\n", age);
Error messages go to stderr, not stdout
fprintf(stderr, ...) writes to the standard error stream. This keeps error messages separate from normal output — important when programs are composed together with pipes or redirects. Use stdout (or plain printf) for normal output, stderr for errors and diagnostics.

Exit Status

When a program finishes, it returns an integer called the exit status to the operating system. The shell captures this value and uses it to decide what to do next — whether to run the next command in a pipeline, whether a script should continue, whether a test passed or failed.

The exit status is the program's last communication with the shell. Zero always means success. Non-zero means something went wrong.

How to Return an Exit Status

#include <stdlib.h>

int main(void) {

    // Method 1: return from main
    return 0;    // success
    return 1;    // generic failure

    // Method 2: exit() from anywhere in the program
    exit(0);     // success — also flushes buffers and closes files
    exit(1);     // failure

    // Named constants from stdlib.h (preferred)
    return EXIT_SUCCESS;   // = 0
    return EXIT_FAILURE;   // = 1
}

Exit Codes in Practice

The Shell Uses Your Exit Status

The shell variable $? holds the exit status of the last command. Shell scripts and pipelines use this to make decisions — this is how Unix tools compose together.

# In a shell script:
./my_program
if [ $? -eq 0 ]; then
    echo "Success — run the next step"
else
    echo "Failed — stopping here"
fi

# Shorthand using &&  and  || :
./validate_input && ./process_data   # process only if validate succeeds
./compile || exit 1                  # stop script if compile fails

Putting It Together: A Complete Input Loop

Most interactive C programs follow a consistent pattern: print a prompt, read and validate input, branch on the result, repeat or exit. Here's a complete example that handles all the edge cases.

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

int main(void) {
    char buf[64];

    while (1) {
        printf("Command (q=quit, h=help): ");

        if (fgets(buf, sizeof(buf), stdin) == NULL) {
            printf("\nEOF reached\n");
            return EXIT_SUCCESS;         // Ctrl-D — clean exit
        }

        // strip newline
        buf[strcspn(buf, "\n")] = '\0';

        if (strcmp(buf, "q") == 0) {
            printf("Goodbye\n");
            return EXIT_SUCCESS;

        } else if (strcmp(buf, "h") == 0) {
            printf("Commands: q (quit), h (help)\n");

        } else {
            fprintf(stderr, "Unknown command: %s\n", buf);
        }
    }
}

Common Problems

Pitfall 01
Not Checking scanf's Return Value
If the user types "abc" when you expected a number, scanf returns 0 and your variable is uninitialized. The program continues with garbage data and produces wrong results silently. Always check if (scanf(...) != 1) before using the variable.
Pitfall 02
The Leftover Newline
After scanf("%d"), the \n from Enter is still in the buffer. A subsequent scanf("%c") or fgets() reads it immediately, skipping the user's next input entirely. Flush with while (getchar() != '\n'); between a numeric and a character/string read.
Pitfall 03
Buffer Overflow with scanf("%s")
scanf("%s", buf) writes until it hits whitespace with no length limit. If the user types more characters than your buffer holds, it overwrites adjacent memory. Use scanf("%63s", buf) with an explicit limit, or better yet, use fgets for all string input.
Pitfall 04
Not Stripping fgets Newline
fgets stores the \n in the buffer. strcmp(buf, "quit") will fail because the buffer actually contains "quit\n". Always strip with buf[strcspn(buf, "\n")] = '\0'; immediately after reading.
Pitfall 05
Forgetting to Return an Exit Status
A function defined as int main() that falls off the end without a return has undefined behavior in C89. In C99 and later, falling off main implicitly returns 0 — but relying on this is poor practice. Always explicitly return 0 or return EXIT_FAILURE.
Pitfall 06
Ignoring the Exit Status of Your Own Program
Returning 0 from a program that encountered errors misleads the shell into thinking everything succeeded. Scripts that call your program may continue when they should stop. Return a non-zero exit code whenever the program fails to do what it was asked to do.
Pitfall 07
Writing Errors to stdout
Using printf for error messages mixes errors into normal output. When the program's output is piped to another process or redirected to a file, the error disappears or corrupts the data. Always use fprintf(stderr, ...) for errors and diagnostics.
Pitfall 08
Infinite Loop on Bad scanf Input
When scanf fails to parse input, the offending characters stay in the buffer. If the program retries in a loop without flushing, scanf fails again on the same characters — forever. After a failed scanf, always flush the input buffer before retrying.
Takeaway
Read input with scanf for formatted values or fgets for strings — never scanf("%s") unguarded. Always check scanf's return value. Flush the buffer between numeric and character reads. Strip fgets's newline before comparing strings. Write errors to stderr. Return a non-zero exit status when your program fails — the shell is listening.
