Paging

Virtual memory is the idea. Paging is the mechanism. The address space is divided into fixed-size chunks called pages. The hardware translates virtual page numbers to physical frame numbers on every memory access.

Machine Environment Layer

Pages and Frames

Both virtual and physical address spaces are divided into fixed-size chunks. A chunk of virtual address space is called a page. A chunk of physical RAM is called a frame. On x86-64 systems the standard size is 4096 bytes (4 KB). Pages and frames are always the same size — the whole system is built on this alignment.

Translation is therefore simple in concept: the hardware needs to know which physical frame a given virtual page maps to. The offset within the page (the low 12 bits for 4 KB pages) is the same in both the virtual and physical address — only the page number changes.

An address splits into a virtual page number (which frame to use) and a page offset (where within that frame). Only the page number is translated.

The Page Table

Each process has a page table — a data structure maintained by the OS that maps virtual page numbers to physical frame numbers. The OS sets it up when the process starts and updates it whenever new memory is allocated or pages are swapped. The hardware MMU consults this table on every memory access.

A page table entry (PTE) contains more than just the frame number. It also carries permission bits that the hardware enforces automatically.

Each page table entry maps a virtual page number to a physical frame number, plus permission bits. P=1 means the page is resident in RAM. P=0 triggers a page fault.

Page Table Entry Fields

Each PTE is typically 8 bytes on x86-64 and packs several fields into those 64 bits:

// Simplified view of what a PTE contains:
Physical Frame Number   // which frame in RAM (or swap address if not present)
Present bit (P)         // 1 = in RAM, 0 = not present (triggers page fault)
Writable bit (W)        // 1 = read/write, 0 = read-only
User bit (U)            // 1 = user code can access, 0 = kernel only
Execute-disable (XD)    // 1 = cannot execute code from this page
Dirty bit (D)           // 1 = page has been written (needs flush to disk)
Accessed bit (A)        // 1 = page has been read or written recently
Permission bits enforce memory protection in hardware
The Read, Write, and Execute permissions are enforced by the MMU on every access — no software check needed. Writing to a read-only page triggers a hardware fault. Jumping to a non-executable page triggers a fault. This is how the OS isolates processes, how the kernel protects itself from user code, and how const globals can be placed in read-only memory.

The TLB — Translation Lookaside Buffer

Consulting the page table on every memory access would require reading from memory to find out where to read memory — a circular problem that would make programs 2–5× slower. The hardware solves this with a small, fast cache called the TLB.

The TLB stores recently used VPN→PFN translations directly in the CPU. A lookup takes 1 cycle. A page table walk takes 20–100 cycles depending on cache state. Most memory accesses hit the TLB; page table walks are the exception.

The TLB caches translations. A hit costs ~1 cycle. A miss forces a page table walk costing 20–100 cycles. The 99%+ hit rate is why virtual memory is fast in practice.

Page Faults

When the CPU tries to access a virtual address and the page table says the page is not present (P=0), the hardware generates a page fault — a CPU exception that transfers control to the OS. The OS must then handle it before the program can continue.

A page fault is a hardware exception. The OS examines the faulting address and either resolves it (load from disk, allocate new page) or terminates the process with a segfault.

The Three Outcomes of a Page Fault

1. Demand paging (swap-in): The page was previously swapped to disk to free RAM. The OS finds it, loads it into a free frame, updates the PTE, and restarts the faulting instruction. The program never knows this happened.

2. Anonymous page allocation: The program accessed a heap or stack page that hasn't been touched yet. The OS allocates a new frame, zero-fills it (for security — you must not see another process's data), marks it present, and restarts. This is how malloc'd memory actually gets physical RAM.

3. Segmentation fault: The virtual address isn't mapped at all, the access violates the permission bits (writing to a read-only page, executing a non-executable page), or the address is in kernel space. The OS sends SIGSEGV to the process, which typically terminates it.

Why zero-fill on fault?
When the OS allocates a new physical frame, it always fills it with zeros before handing it to your process. This is a security requirement: the frame may have previously belonged to another process. Without zeroing, your malloc call could return memory containing another process's passwords, keys, or data. The OS zeroes frames in the background when the CPU is idle so they're ready before you need them.

Swapping and Demand Paging

When physical RAM fills up, the OS must evict pages — write modified pages to the swap partition on disk and repurpose their frames for other processes. An evicted page has its PTE's present bit cleared. When the program accesses it again, a page fault triggers a swap-in from disk.

Accessing swap is catastrophically slow — disk I/O is 100,000× slower than RAM. When the working set of active pages exceeds available RAM and the system is constantly swapping back and forth, it is called thrashing — the machine appears to freeze as it spends all its time moving pages rather than executing instructions.

Performance 01
TLB Thrashing
Accessing memory in a random pattern across a large address space evicts TLB entries before they're reused. Every access pays the full page table walk penalty. Cache-friendly access patterns — walking arrays sequentially — maintain a high TLB hit rate. Random pointer chasing over large structures is the worst case.
Performance 02
Swap Thrashing
When a program's working set exceeds available RAM, the OS constantly swaps pages in and out. Each page fault involving disk I/O costs millions of cycles. Programs that allocate far more memory than they actively use contribute to system-wide thrashing. Free what you don't need; work in cache-sized chunks.
Correctness 01
Write to Read-Only Page → Segfault
String literals, code, and const globals are placed in read-only pages. Writing through a pointer to any of these triggers a write-protection fault → SIGSEGV. This is enforced in hardware by the PTE's write bit, not in software. The crash is immediate and unavoidable.
Correctness 02
Using Memory Before OS Allocates the Page
On Linux, mmap and large malloc calls succeed immediately but the physical pages aren't allocated until first write (demand paging). Code that checks the return of malloc but then gets OOM-killed mid-execution has been caught by this: the OS said "sure" but couldn't deliver later.
Takeaway
Paging divides address spaces into fixed 4 KB pages/frames. The page table maps virtual page numbers to physical frame numbers plus permission bits. The TLB caches recent translations for speed — a miss costs a page table walk. A page fault triggers when a page is not present; the OS either loads it from disk, allocates a new zeroed frame, or kills the process with SIGSEGV. Permissions are enforced in hardware on every access.
