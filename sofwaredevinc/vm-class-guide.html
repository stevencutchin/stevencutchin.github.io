Virtual Memory & Paging — Class Guide

CS253  ·  Three 75-min sessions  ·  vm-01 through vm-03
Session 1 — Virtual Memory

Opening Problem

  1. Open with a concrete puzzle: your machine has 16 GB of RAM and you're running 40 programs. Their combined memory use might be 30 GB. How? And how does a bug in one program not corrupt another? Let students guess. Don't answer yet — use the tension to motivate everything that follows.
  2. Ask: when you write int x = 42; and then printf("%p", &x), what does that address actually mean? Is it a location in a RAM chip? Most students assume yes. The answer is no — and that's the whole lecture.

The Abstraction

  1. State the definition plainly: a virtual address is a number in a private, per-process address space. A physical address is an actual location in RAM. Your program only ever sees virtual addresses. The hardware translates them to physical on every single memory access.
  2. Draw on the board: three processes on the left, each with "0x0000…FFFF" address space. An arrow from each to a box labeled "OS + MMU." An arrow from that box to a single bar of physical RAM divided into colored chunks. Each process thinks it owns everything; they actually share one physical resource.
  3. Introduce the MMU — Memory Management Unit. It's a piece of hardware inside the CPU. It performs the virtual→physical translation automatically, on every load and store, with no software involved. There is no speed penalty for the indirection in normal operation.

Three Problems Virtual Memory Solves

  1. Isolation. Process A's virtual address 0x1000 maps to a different physical location than Process B's virtual address 0x1000. A bug in A cannot corrupt B's memory — not because of software checks, but because the hardware simply routes them to different physical pages. This is why a crash in one process doesn't take down the system.
  2. More address space than physical RAM. Pages that haven't been accessed recently can be written to disk (swap) and their physical frames reused. When the program touches them again, they're loaded back transparently. The program never knows. This is why you can open more applications than your RAM can fit.
  3. Simplified loading. Because every process has its own address space, every program can be compiled to use the same virtual addresses. Without virtual memory, the OS would have to relink every binary at load time to avoid collisions. Ask students: why does every C program's main end up at roughly the same virtual address?

Address Space Size

  1. On a 64-bit system the virtual address space is 2⁶⁴ bytes in theory — 16 exabytes. In practice, current hardware only uses 48 bits (256 TB). Half goes to user processes (low addresses), half to the kernel (high addresses). Draw this split on the board as a tall bar.
  2. Point out the implication: a 64-bit pointer can address 256 TB of virtual space, but the machine may only have 32 GB of RAM. The virtual space is vastly larger than the physical backing. This is not a bug — it's the whole point.

Common Misconceptions to Address Head-On

  1. "A pointer is a RAM address." No — it's a virtual address. The physical location it corresponds to can change (e.g., after a swap), and the same virtual address in two processes points to different physical bytes.
  2. "malloc gives you raw memory." No — malloc carves from your process's heap, which is already a region of virtual address space the OS set up. The OS doesn't see individual malloc calls. It sees larger requests when the heap needs to grow.
  3. "Two processes can never share memory." Isolation is the default — but the OS can deliberately map the same physical page into two processes' address spaces. This is how shared memory IPC works. Isolation is a policy, not a physical constraint.

Pitfalls — Session 1

  1. Don't conflate virtual and physical addresses. In a C program, all addresses are virtual. Students will not be able to find the "real" location of a variable without OS tools like /proc/PID/maps.
  2. Optimistic allocation: malloc can succeed even when the physical page doesn't exist yet. On Linux, the OS may say "yes" and only actually assign physical RAM on first write. A program that checks malloc's return value but later crashes with OOM has been caught by this.
Session 2 — Paging

Dividing the Address Space

  1. Virtual memory is the concept. Paging is the implementation. Introduce the unit of management: a page is a fixed-size chunk of virtual address space, typically 4096 bytes (4 KB). Physical RAM is divided into equally sized frames. Pages map to frames.
  2. Write a 48-bit virtual address on the board. Divide it into two parts: the upper 36 bits are the Virtual Page Number (VPN) — which page. The lower 12 bits are the page offset — which byte within that page. The offset passes through unchanged; only the VPN gets translated.
  3. Ask: why 12 bits for the offset? Because 2¹² = 4096 — exactly one page. The page size and offset width are locked together.

The Page Table

  1. The OS maintains a page table for each process. It maps VPNs to Physical Frame Numbers (PFNs). Draw a two-column table on the board: left column is virtual page numbers 0, 1, 2, 3…; right column shows which physical frame each one maps to (or "not present").
  2. Show that virtual pages don't need to map to consecutive physical frames. Virtual page 0 might be in physical frame 7; virtual page 1 might be in physical frame 2. The virtual address space looks contiguous to the program; the physical layout can be scattered anywhere.
  3. Page Table Entry (PTE) fields worth knowing: Present bit (P) — is the page in RAM right now? Writable bit (W) — can it be written? User bit (U) — can user code access it, or kernel only? Execute-Disable (XD) — can code be executed from this page? Dirty bit (D) — has it been written since it was loaded?
  4. Emphasize: these permission bits are enforced entirely in hardware, by the MMU, on every access. There is no software check. Writing to a read-only page triggers a hardware fault before a single byte is changed.

The TLB

  1. Pose the obvious problem: if the page table is in RAM, looking it up on every memory access requires reading RAM to find out where to read RAM. That would make every memory access twice as expensive. How is this solved?
  2. Answer: the TLB — Translation Lookaside Buffer. A small, fast cache inside the CPU that stores recently used VPN→PFN mappings. A TLB hit takes ~1 cycle. A TLB miss requires a page table walk costing 20–100 cycles.
  3. TLB hit rate is typically >99% for well-behaved programs — programs that access memory in locality-friendly patterns. Ask: what kind of code would thrash the TLB? Answer: random access over a huge array, pointer-chasing linked lists scattered in memory, anything that jumps unpredictably across many pages.
  4. When the OS switches between processes (context switch), the TLB must be flushed or tagged — because the translations from Process A are meaningless to Process B. This is a real cost of context switching.

Page Faults

  1. When the CPU tries to access a page with P=0 (not present), it generates a page fault — a CPU exception that hands control to the OS page fault handler. The faulting instruction is suspended. The OS investigates the address and decides what to do.
  2. Walk through the three outcomes: 
    1. Swap-in:  page was on disk, OS loads it into a free frame,
             sets P=1, restarts the faulting instruction.
             Program never knew anything happened.

2. New page:  program accessed heap/stack memory for the first time,
              OS allocates a new frame, zero-fills it, sets P=1.
              This is demand paging — malloc doesn't touch RAM
              until you first write to the memory.

3. Segfault:  address is unmapped, permission violated (write to
              read-only, execute from non-exec page), or kernel
              address from user code. OS sends SIGSEGV. Done.
  3. Ask: why does the OS zero-fill new pages before handing them to a process? Security: the physical frame may have previously belonged to another process. Without zeroing, a program could read another process's passwords or keys through an uninitialized heap allocation. The OS pre-zeros frames in the background during idle time.
  4. Swapping: when RAM is full, the OS evicts cold pages to disk (swap partition). Accessing them later triggers a swap-in fault. Disk is ~100,000× slower than RAM. When the system is constantly swapping in and out — because the active working set exceeds RAM — it's called thrashing. The machine appears frozen. Students may have seen this on underpowered machines.

Connecting Page Faults to C Bugs

  1. Walk through each common segfault scenario in terms of page faults: 
    NULL dereference:    address 0x0 is intentionally unmapped
                     → page fault → no mapping → SIGSEGV

Wild pointer:        random address not in any mapping
                     → page fault → no mapping → SIGSEGV

Write to rodata:     string literal page has W=0
                     → page fault → permission violation → SIGSEGV

Stack overflow:      RSP crosses below the stack limit into
                     unmapped guard page → page fault → SIGSEGV
  2. The four-layer diagnostic: when a student gets a segfault, it lives in the Machine Environment layer. It is a virtual memory protection event — the hardware detected an illegal access and the OS terminated the process. Understanding this helps students locate the bug conceptually before reaching for gdb.

Pitfalls — Session 2

  1. TLB thrashing: code that accesses memory randomly over a large range pays a full page-table-walk penalty on nearly every access. Sequential array traversal is fast; random pointer chasing is slow — part of the reason is TLB miss rate.
  2. Write to read-only page: string literals and const globals live in read-only pages. Hardware enforces this. char *s = "hello"; s[0] = 'H'; is an immediate SIGSEGV — not a C rule violation but a memory protection fault.
  3. Optimistic allocation revisited: on Linux, a large malloc may succeed even when the OS can't back it with physical RAM at that moment. The physical page is assigned on first write. Programs that never write to every byte of a huge allocation may work fine on a machine with little RAM — until they do.
Session 3 — Process Layout

The Segments

  1. Draw the classic process address space diagram on the board — a tall rectangle, high addresses at top, low at bottom. Fill in the regions from top to bottom: kernel space, stack (grows ↓), mmap/shared libs region, heap (grows ↑), BSS, data, rodata, text. Students should draw this in their notes. They will refer to it for the rest of the course.
  2. Name the two growing regions and their directions. Stack starts at a high address and grows down toward lower addresses. Heap starts just above BSS and grows up. In theory they could collide in the middle under extreme use — in practice the address space is large enough that this is almost never an issue on 64-bit systems.

Text and Rodata — Read-Only Segments

  1. Text segment: compiled machine code. Permissions: read + execute, no write. Ask: why not writable? Self-modifying code would be a security disaster. The read-only permission prevents accidental and malicious modification of the program's own instructions.
  2. Rodata: string literals and const globals. Permissions: read-only, no write, no execute. Write on the board: char *s = "hello";. Ask: where does the string "hello" live? Answer: rodata. Ask: what happens if you write s[0] = 'H'? Answer: SIGSEGV from the write-protection fault. Now show the fix: char s[] = "hello"; — this copies the string onto the stack, which is writable.

Data and BSS — Global and Static Storage

  1. Data segment: initialized globals and static locals. The initial values are stored in the binary on disk and copied into these pages when the program loads. These pages are read/write and live for the entire process lifetime.
  2. BSS segment: uninitialized globals and statics. Not stored in the binary — the OS zero-fills these pages at startup. A 10 MB array of uninitialized globals adds nothing to the binary size on disk but is guaranteed to be all zeros in memory. 
    int scores[100000];   // BSS — 400 KB, zero-filled, no binary size cost
int count = 5;        // data — stored in binary, loaded at startup
  3. State the rule students will use forever: global and static variables are always zero-initialized. Local variables on the stack are never zero-initialized — they contain whatever bytes the previous function left there. This difference is a direct consequence of which segment they live in.

The Heap

  1. The heap is managed by the C library allocator (on Linux: glibc's ptmalloc). malloc and free do not talk directly to the OS on every call — they manage a pool of pages the allocator has already requested. When the pool is exhausted, the allocator calls brk() or mmap() to ask the OS for more pages.
  2. Draw fragmentation on the board: a sequence of alternating allocated and freed blocks. Free space is present but scattered. Ask: if you then call malloc(large), can it succeed? Maybe not — the total free bytes exist but no single contiguous block is large enough. This is heap fragmentation.
  3. Demand paging on the heap: when the allocator requests new pages from the OS, those pages are not immediately backed by physical RAM. Physical frames are assigned only when you first write to each page. This is why programs that allocate huge buffers but only touch a fraction of them don't actually use that much RAM.

The Stack

  1. Each function call pushes a stack frame: local variables, saved registers, the return address, and the function's arguments. When the function returns, the frame is popped — the stack pointer moves back up and that space is immediately reusable. No bookkeeping, no allocator, no free list. That's why stack allocation is so fast.
  2. Show the frame contents live: compile a simple function with -g and run under gdb. Use info frame and backtrace to show students the call stack. Make it concrete — this is what "the stack" actually looks like at runtime.
  3. Stack limit: typically 8 MB on Linux (ulimit -s). What happens when you exceed it? The stack pointer crosses into unmapped memory below the stack. The MMU detects a page fault to an unmapped address. The OS cannot grow the stack further. It sends SIGSEGV. This is a stack overflow.
  4. Show both overflow mechanisms: 
    // Deep recursion — many small frames
void inf(void) { inf(); }   // segfault after ~tens of thousands of calls

// One giant frame
void bad(void) {
    char buf[10000000];     // 10 MB in a single frame — instant death
}
  5. State the rule: if a local array is larger than a few KB, it belongs on the heap. Put anything big in malloc. The stack is for small, short-lived data.

Where Every Variable Lives

  1. Walk through the table with live code. Write the following on the board and ask students to place each variable in a segment before you reveal the answers: 
    int g_count = 0;           // data  — initialized global
int g_scores[100];         // BSS   — uninitialized global
const char *greeting = "Hi"; // ptr: data; "Hi": rodata
static int call_count = 0; // data  — static local
int x = 42;                // stack — local variable
int *p = malloc(40);       // p: stack; *p: heap
  2. Reinforce: it's the declaration form — global, static, local, malloc — that determines the segment, not the type. An int can live in any segment depending on how it was declared.

ASLR

  1. Ask students to run a simple program twice that prints the address of a local variable. Show that the addresses are different each run. This surprises students who expect deterministic output.
  2. Explain ASLR — Address Space Layout Randomization. Each time a process starts, the OS randomizes the base addresses of the stack, heap, and shared libraries. This makes it much harder to exploit memory bugs: an attacker cannot predict where code or the stack will land.
  3. Implication for debugging: pointer values printed with %p are not reproducible between runs. They're useful for comparing two pointers within one run, but you can't record them and compare across sessions. Sanitizers and tools like Valgrind disable ASLR to get reproducible addresses.

Pitfalls — Session 3

  1. Returning a pointer to a local variable: the local lives on the current stack frame. When the function returns, that frame is gone. The caller holds a pointer into recycled stack space that will be overwritten by the next function call. Compile with -Wall — GCC will warn about this.
  2. Large local arrays: char buf[10000000] as a local variable allocates 10 MB on the stack before the function body even runs. The stack limit is exceeded immediately. Put large buffers on the heap.
  3. Assuming locals are zero: stack memory contains garbage from previous calls. Only global/static variables are guaranteed zero. Always initialize locals. This is a virtual memory design fact, not just a C style preference.
  4. Heap-stack lifetime mismatch: storing a pointer to a local variable inside a heap-allocated struct is valid as long as the function hasn't returned. But if the heap struct outlives the function, the pointer dangles the moment the function returns. Lifetime discipline is required whenever mixing segment types through pointers.

Wrap-Up — The Full Picture

  1. Connect the three sessions into one chain: your C program uses virtual addresses (session 1) → the hardware breaks each address into a page number and offset, looks up the mapping in the page table, and translates it to a physical frame (session 2) → that physical frame holds bytes from one of the named segments your program was loaded into (session 3). Every memory access your program makes follows this complete path.
  2. Draw the diagnostic ladder on the board. When a student gets a segfault, they now have a mental model for what actually happened: a virtual address was accessed, the MMU checked the page table, something was wrong (no mapping, wrong permission, guard page), a page fault fired, the OS determined it was unrecoverable, SIGSEGV was delivered. The bug is wherever the bad address came from — null pointer, wild pointer, out-of-bounds access, dangling pointer, or stack overflow.
  3. Bridge to future topics: file I/O and mmap map file contents directly into the virtual address space — no explicit read/copy needed. Process creation with fork() copies the page table (with copy-on-write optimization). Shared libraries are mapped into the mmap region and their pages can be physically shared across every process that uses them. Virtual memory is the foundation all of these sit on.
