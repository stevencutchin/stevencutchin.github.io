Types

A type tells the compiler two things: how many bytes a variable occupies, and how to interpret those bytes. The same bit pattern means completely different things depending on the type.

Why Types Exist

Memory is just bytes. A type gives those bytes meaning. The 4-byte pattern 01000000 01001000 00000000 00000000 means 1,083,179,008 as an int, but it means 3.125 as a float. The bits are identical — the type determines what they represent.

Types are the interpreter. The bits don't change — the type determines what they mean.

Integer Types

Integer types store whole numbers. They vary in size (how many bytes) and signedness (can they hold negative numbers). The sizes shown here are typical for 64-bit systems — C only guarantees minimum sizes, not exact ones. Use <stdint.h> when exact sizes matter.

Each type occupies a fixed number of bytes. More bytes means wider range but more memory.

Floating-Point Types

Floating-point types store real numbers — values with fractional parts. They use a standardized binary encoding (IEEE 754) that trades exact representation for enormous range. Most decimal fractions cannot be represented exactly in binary floating-point.

TypeSizePrecisionRange (approx)
float4 bytes~7 significant digits±3.4 × 10³⁸

Floating-point numbers are approximations.
0.1 + 0.2 is not exactly 0.3 in binary floating-point. The result is 0.30000000000000004.... Never compare floats with ==. Instead, check whether the difference is smaller than a tolerance: fabs(a - b) < 1e-9. Use double by default — float has too little precision for most work.

Type Conversion

C converts between types in two ways: implicit conversion happens automatically when you mix types in an expression, and explicit casting is when you tell the compiler to convert with (type).

// Implicit: C promotes to the "wider" type automatically
int    i = 5;
double d = 2.5;
double r = i + d;    // i promoted to double → 5.0 + 2.5 = 7.5

// Integer division pitfall — both operands are int
int a = 7, b = 2;
double bad  = a / b;       // 3.0  ← integer division happened first!
double good = (double)a / b; // 3.5  ← cast a first, then divide

// Explicit cast: narrowing (can lose data)
double pi = 3.14159;
int    n  = (int)pi;        // n = 3 — fractional part silently dropped
Widening conversions preserve value. Narrowing conversions lose data — the compiler will not warn you.

sizeof

The sizeof operator returns the size in bytes of a type or variable at compile time. Use it instead of hardcoding sizes — this is especially important in pointer arithmetic and when allocating memory.

printf("char:   %zu bytes\n", sizeof(char));    // 1
printf("int:    %zu bytes\n", sizeof(int));     // 4
printf("double: %zu bytes\n", sizeof(double));  // 8

int arr[10];
printf("array:  %zu bytes\n", sizeof(arr));    // 40
printf("length: %zu\n", sizeof(arr) / sizeof(arr[0])); // 10

Exact-Size Types with <stdint.h>

The standard types like int and long have implementation-defined sizes that vary by platform. When you need a specific number of bits — for file formats, network protocols, or hardware registers — use the exact-size types from <stdint.h>.

#include <stdint.h>

int8_t   a;   // exactly 8 bits, signed
uint8_t  b;   // exactly 8 bits, unsigned
int16_t  c;   // exactly 16 bits, signed
uint32_t d;   // exactly 32 bits, unsigned
int64_t  e;   // exactly 64 bits, signed

Common Problems

Pitfall 01
Integer Division Truncation
7 / 2 is 3, not 3.5. When both operands are integers, C performs integer division. The fractional part is silently discarded. Cast at least one operand: (double)7 / 2 gives 3.5.
Pitfall 02
Integer Overflow
An int can hold up to 2,147,483,647. Adding 1 to that value wraps around to -2,147,483,648 — silently. The compiler will not warn. Use a larger type (long long) or check bounds before arithmetic when overflow is possible.
Pitfall 03
Comparing Floats with ==
0.1 + 0.2 == 0.3 is false. Floating-point arithmetic produces tiny rounding errors that make exact equality comparisons unreliable. Use fabs(a - b) < 1e-9 instead.
Pitfall 04
Signed/Unsigned Mismatch
Comparing a signed int with an unsigned int when the signed value is negative causes the negative value to be interpreted as a very large positive number. -1 > 1u is true. Enable -Wall -Wsign-compare to catch these.
Pitfall 05
Narrowing Cast Loses Data
Casting a large value to a smaller type silently discards the excess bits. (char)300 gives 44, not 300. (int)3.9 gives 3, not 4. The compiler does not warn about explicit casts — you asked for the conversion, so it assumes you know what you're doing.
Pitfall 06
Using float Instead of double
float has only ~7 significant digits of precision. Accumulated rounding errors in long calculations become visible. Use double by default. Only use float when memory or GPU performance demands it, and when you've measured that the precision loss is acceptable.
Pitfall 07
Assuming int Size
The C standard only guarantees int is at least 16 bits. In practice it's 32 bits on modern systems — but code that assumes a specific size will break when ported. Use <stdint.h> types like int32_t when the exact size matters.
Pitfall 08
Wrong printf Format Specifier
Using %d to print a long or %f to print a double with scanf produces undefined behavior. Match specifiers to types exactly: %d for int, %ld for long, %f/%lf for float/double, %zu for size_t.
Takeaway
A type determines how many bytes a value occupies and how those bytes are interpreted. Use int for general integers, double for real numbers, and <stdint.h> types when exact width matters. Integer division truncates — cast first to get a float result. Never compare floats with ==. Always match printf/scanf format specifiers to their types.
