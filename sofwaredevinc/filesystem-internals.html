Machine Environment  ·  File System Internals

Inodes &
Data Blocks

When you call stat() or readdir(), the OS is reading internal structures it manages on disk — inodes and data blocks. Understanding these structures explains everything: why filenames are separate from files, how hard links work, and why a 1 TB file doesn't need a 1 TB pointer array.

The Two Structures That Run Everything

A Unix filesystem stores its content in exactly two kinds of on-disk objects. Every file, every directory, every symlink — no matter its size or type — maps onto these two primitives.

An inode is a fixed-size record that holds all the metadata about a file: its permissions, owner, timestamps, size, and — critically — a set of pointers that locate its actual content on disk. An inode does not store the file's name. It doesn't even know it has a name.

A data block is a fixed-size chunk of raw disk space (typically 4 KB) that holds the actual bytes of a file's content, or in the case of directories, a structured list of name-to-inode mappings.

The Central Separation
The name of a file is stored in a directory, not in the file itself. The file's inode holds its metadata and data. This separation — name over here, content over there — is what makes hard links, renaming, and atomic file replacement possible without copying any data.

How a Filesystem is Laid Out on Disk

When you format a partition, the filesystem divides the raw block device into a fixed set of regions. Understanding this layout shows you where inodes live, where data lives, and how the OS keeps track of what's free.

Simplified ext2/ext4-style disk layout — real filesystems have multiple block groups, each with its own copies of these regions

The superblock is the filesystem's constitution — it records the block size, total inode count, total block count, and the counts of free inodes and blocks. If the superblock is corrupted, the entire filesystem is unreadable, which is why it is replicated in multiple block groups.

The bitmaps are compact free-space maps. One bit per block (or inode) — 1 means in-use, 0 means free. When you create a file the OS sets bits in both bitmaps. When you delete one, it clears them. The actual data is not zeroed — it just becomes "free."

Inside an Inode

Every file and directory has exactly one inode. Its size is fixed at filesystem-format time (128 or 256 bytes on ext4). An inode holds two categories of information: metadata about the file, and an array of block pointers that locate the file's content.

The block pointer design is one of the cleverest in Unix — it handles tiny files efficiently with direct pointers, then telescopes outward through layers of indirection to handle files up to several terabytes, all within those same 128 bytes.

The block pointer hierarchy — direct pointers reach 48 KB, single indirect reaches 4 MB, double indirect 4 GB. The cost of a small file is exactly three disk reads: one for the inode, zero for a pointer block, one for the data.

How Files Scale

The indirection levels exist because they make the common case cheap. A 1 KB config file needs only one direct pointer — no indirection block is ever allocated. The indirect pointer blocks are allocated on demand, only when the file grows large enough to need them.

Assuming 4 KB blocks and 4-byte block pointers, each indirect block can hold 1024 pointers:

Level Pointer path Pointer count Capacity added Disk reads to get data
Direct inode → data 12 48 KB 2 (inode + data)
1× Indirect inode → ptr block → data 1,024 ~4 MB 3
2× Indirect inode → L1 → L2 → data 1,048,576 ~4 GB 4
3× Indirect inode → L1 → L2 → L3 → data ~1 billion ~4 TB 5
Modern Filesystems Use Extents
ext4, APFS, and XFS replace the classical pointer array with extents — each extent records a starting block and a length, efficiently describing a contiguous run. A 1 GB file stored contiguously needs only one extent record, versus 262,144 individual block pointers in the old scheme. The inode still exists; only the pointer format changed.

Directories are Files Too

A directory is not a special OS object — it is a regular file whose data blocks contain a table of (inode number, name) pairs. When you call opendir() / readdir(), you are reading this table. When you create a file, the OS allocates an inode and appends one row to the parent directory's data block.

This design has one profound consequence: the filename is not stored in the inode. It is stored in the directory that contains the file. You can therefore have two directory entries — in the same or different directories — that both point to the same inode. That is a hard link.

Left: a directory's data block is a table of (inode number, name) pairs — the name lives here, not in the inode. Right: two directory entries pointing to inode #42 — a hard link. The inode's nlink field counts them; the data is freed only when nlink reaches zero.

The nlink counter

The nlink field in the inode counts how many directory entries point to it. When you create a file, nlink starts at 1. Each call to link() increments it. Each unlink() decrements it. The inode — and the data blocks it references — are freed only when nlink reaches zero.

This is why the system call for "delete a file" is called unlink(), not delete(). You are removing a name from a directory; the file's inode survives until all names pointing to it are gone.

Directories get nlink too
A fresh empty directory always has nlink = 2: one for its entry in the parent directory, and one for the . entry inside itself. Each subdirectory you create adds 1 more, because that subdirectory's .. entry also points back to the parent inode.

What stat() Actually Reads

When you call stat(path, &st) in your C program, the OS resolves the path — traversing directory data blocks to convert each name segment to an inode number — then reads the final inode off disk and copies its fields directly into your struct stat.

// Every field in struct stat maps directly to an inode field:
st.st_ino   // ← inode number (the index in the inode table)
st.st_mode  // ← type bits + permission bits from inode.mode
st.st_nlink // ← inode.nlink  (hard link count)
st.st_uid   // ← inode.uid
st.st_gid   // ← inode.gid
st.st_size  // ← inode.size  (bytes of file content)
st.st_mtime // ← inode.mtime (last data modification)
st.st_ctime // ← inode.ctime (last inode change — not "creation")
st.st_atime // ← inode.atime (last access)
st.st_blksize // ← filesystem preferred I/O block size
st.st_blocks  // ← 512-byte blocks actually allocated (≥ st_size/512)
st_ctime is not "creation time"
st_ctime is the inode's change time — the last time the inode metadata changed (permissions, owner, nlink count, etc.). Unix has no standard "creation time" field. macOS adds st_birthtime as an extension; Linux ext4 stores it internally but POSIX does not expose it.
⚠ Path resolution cost. Every component in a path like /home/amti/cs253/lab1/main.c requires reading the data block of each intermediate directory to find the next inode number. A stat() on a deeply nested file can take 5–10 disk reads before it even reaches the target inode. This is why keeping frequently-accessed files at shallow paths and using file descriptors (fstatat(), openat()) to avoid re-resolving paths matters in performance-sensitive code.
