The Basics

Write, Compile, Run

Getting a C program running takes three steps. Write your code in a text file, compile it into a program the computer can execute, then run it. That's the whole cycle.

1
Write
Write your C code in a text file. This is your source code — human-readable instructions.
hello.c
2
Compile
The compiler translates your source code into machine code the CPU understands.
gcc hello.c -o hello
3
Run
Execute the compiled program. The operating system loads it and your code runs.
./hello

Step 1: Write Your Code

C is a compiled language, which means you can't just run a .c file directly. You first write your code in a plain text file with the .c extension:

#include <stdio.h>

int main(void) {
    printf("Hello, world!\n");
    return 0;
}

This file is for you — it's written in a language humans can read and write. But the computer's CPU doesn't understand C. It only understands machine code — raw binary instructions specific to your processor. That's what the next step produces.

Analogy
Your .c file is like a recipe written in English. The computer is a robot that only reads barcodes. Before the robot can cook, someone has to translate the recipe into barcodes. That translator is the compiler.

Step 2: Compile

The compiler reads your .c file and translates it into an executable — a file containing machine code your CPU can run. We use gcc (the GNU C Compiler):

$ gcc hello.c -o hello

Let's break this command down:

gcc takes your human-readable C code and produces a machine-code executable the CPU can run.

If there are errors in your code — a missing semicolon, a misspelled function name, a type mismatch — the compiler will tell you now, before anything runs. This is one of C's greatest strengths: many bugs are caught at compile time, not while the program is running.

$ gcc hello.c -o hello
hello.c:4:5: error: expected ';' after expression
    printf("Hello, world!\n")
                             ^
                             ;
1 error generated.

No errors? No output. Silence from the compiler is good news — it means your executable was created successfully.

Key Concept
Compilation is a one-time translation. You compile once, and the resulting executable can be run as many times as you want without compiling again. You only need to recompile when you change the source code.

Step 3: Run

Now you have an executable file called hello sitting in your directory. Run it:

$ ./hello
Hello, world!

The ./ means "look in the current directory." The operating system loads your executable into memory, and the CPU starts executing the machine code the compiler generated. Your printf call sends text to the terminal, and the program exits.

That's it. Write, compile, run.

The Two Files

After compiling, you have two files — and understanding the difference between them is fundamental:

You edit the .c file. The computer runs the executable. The compiler bridges the gap between the two.

You edit hello.c when you want to change your program. You never edit the executable — it's regenerated every time you compile. Think of the .c file as the source of truth and the executable as a built product.

The Edit-Compile-Run Cycle

Programming in C is a loop. You write some code, compile it, run it, see what happens, then go back and change the code. Every time you change the .c file, you need to recompile before running again — otherwise you're running the old version.

The development cycle: edit → compile → run → think → repeat. You'll do this hundreds of times per project.
Common Beginner Mistake
You change your .c file but forget to recompile. Then you run the program and wonder why your changes didn't take effect. Always recompile after editing. The executable doesn't update itself — you have to rebuild it.

When Something Goes Wrong

Errors fall into two categories, and knowing which kind you're looking at tells you what to do:

Compile errors happen when you run gcc. The compiler can't understand your code — there's a syntax mistake, a type mismatch, or a missing function. No executable is produced. Fix the code and compile again.

$ gcc hello.c -o hello
hello.c:4:12: error: use of undeclared identifier 'prinft'
    prinft("Hello!\n");
    ^
1 error generated.

Runtime errors happen when you run ./hello. The code compiled fine, but something goes wrong while it's running — a division by zero, a bad memory access, or just wrong output. Think about the logic, fix the code, recompile, and run again.

$ gcc divide.c -o divide
$ ./divide
Floating point exception (core dumped)
Compile errors stop the build — no executable is created. Runtime errors happen after a successful build when you run the program.

Where the Layers Fit

Even in this simple three-step process, all four of our conceptual layers are at work:

C Language — The rules you follow when writing your .c file. Syntax, types, keywords. The compiler checks these rules.

Dev Ecosystem — The gcc compiler itself, your text editor, and the terminal you type commands into. These are the tools that make the cycle possible.

Machine Env — The executable contains machine code for your specific CPU. The operating system loads and runs it.

C Runtime — Behind the scenes, startup code initializes your program before main() runs, and the standard library provides printf. You'll learn more about this layer as the course progresses.

Takeaway
C programs can't be run directly from source code — they must be compiled first. The cycle is: edit your .c file, compile with gcc, run the executable. If the compiler reports errors, fix your source code and compile again. If the program runs but behaves wrong, think about the logic, fix the source, and go around the loop again. This edit-compile-run cycle is the heartbeat of C development.
