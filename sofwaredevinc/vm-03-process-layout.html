Process Layout

A running C program's address space is divided into distinct segments — each with different permissions, lifetimes, and growth behavior. Every variable you declare lives in one of them.

Machine Environment Layer

The Segments

When the OS loads your compiled C program, it maps several regions into the process's virtual address space. These are not arbitrary — each segment corresponds to a distinct kind of data with distinct rules about who can access it, whether it can grow, and how long it lives.

A Linux process address space. High addresses at top, low at bottom. Stack grows down from high addresses; heap grows up from low ones. The two meet in the middle under heavy use.

Each Segment in Detail

Text Segment — your code

The compiled machine instructions live here. It is mapped read-only and executable — the program can jump into it and execute instructions, but cannot modify it at runtime. This prevents accidentally (or maliciously) overwriting instructions. All calls to functions like main, printf, and your own functions land in this segment.

int add(int a, int b) {
    return a + b;   // these instructions live in the text segment
}                   // read-only, execute-only — cannot be written at runtime

Read-Only Data (rodata) — string literals and const globals

String literals and global const variables are placed here by the compiler. The pages are mapped read-only — no write permission. This is why writing to a string literal crashes: the page table entry has the write bit cleared, and the hardware enforces it.

const char *msg = "Hello";  // "Hello" lives in rodata — read-only pages
msg[0] = 'h';               // ← SIGSEGV: write to read-only page

const int MAX = 100;        // MAX may also be placed in rodata

Data Segment — initialized globals and statics

Global variables and static local variables that have an explicit initial value are stored here. The initial values are copied from the binary on disk into these pages when the process loads. They persist for the entire lifetime of the process and are readable and writable.

int counter = 0;            // data segment — initialized global
static double factor = 2.5;  // data segment — initialized static

int main(void) {
    static int call_count = 0; // data segment — persists across calls
    call_count++;
}

BSS Segment — uninitialized globals

Global and static variables that are declared without an initializer go into BSS (Block Started by Symbol). These are not stored in the binary — the OS zero-fills these pages when the process starts. A large array of uninitialized globals takes no space on disk but is zeroed automatically in memory at startup.

int    scores[1000];          // BSS — not in binary, zeroed by OS at start
double table[10000];          // BSS — 80 KB of guaranteed zeros, no disk cost

// Global variables you don't initialize are guaranteed to be 0.
// This is NOT true for local (stack) variables.
BSS zeros are guaranteed — stack zeros are not
Global and static variables start at zero because the OS zero-fills BSS pages. Local variables on the stack contain whatever bytes were left by the previous function that used that stack space. This is the source of the "uninitialized local variable contains garbage" rule — it is a virtual memory implementation detail, not an arbitrary C decision.

Heap — dynamic allocation

The heap is where malloc, calloc, and realloc allocate memory from. It starts just above the data/BSS segments and grows upward. The C library's allocator (glibc's ptmalloc, for example) manages a pool of pages it has requested from the OS and subdivides them for individual allocations. When the pool is exhausted, the allocator calls brk() or mmap() to request more pages from the OS.

The heap grows upward from the brk pointer. Freed memory returns to the allocator's free list. Fragmentation creates holes that may prevent future large allocations.

Stack — local variables and call frames

The stack grows downward from high addresses. Each function call pushes a stack frame onto the stack — a block of memory containing the function's local variables, its arguments, and the return address. When the function returns, its frame is popped: the stack pointer moves back up, reclaiming that space.

Each function call pushes a frame onto the stack. Frames contain local variables, saved registers, and return addresses. When the function returns, its frame is instantly reclaimed.

Where Every Variable Lives

Every variable in a C program lives in exactly one of these segments, determined by how it is declared — not by its type.

/* ── global scope ─────────────────────────────────── */
int         g_count = 0;      // data segment    — initialized global
int         g_scores[100];   // BSS             — uninitialized global, zeroed
const char *greeting = "Hi"; // ptr in data; "Hi" in rodata

int main(void) {
    /* ── local (stack) variables ───────────────────── */
    int  x = 42;               // stack           — gone when main() returns
    char buf[256];             // stack           — 256 bytes on the stack frame

    /* ── static local ──────────────────────────────── */
    static int call_count = 0; // data segment    — persists across calls

    /* ── heap allocation ───────────────────────────── */
    int *arr = malloc(100 * sizeof(int)); // heap — lives until free()
    free(arr);
}
Declaration form determines segment. Only BSS, data, and static variables are zero-initialized. Stack and heap (malloc) are not.

The Stack Limit and Stack Overflow

The stack is not infinite. The OS sets a limit (typically 8 MB on Linux, adjustable with ulimit -s). If your function calls recurse too deeply or declare very large local arrays, the stack pointer crosses into unmapped memory — a page fault with no recovery. The OS delivers SIGSEGV and the process dies. This is a stack overflow.

// Deep recursion — each call adds a frame
int factorial(int n) {
    return n == 0 ? 1 : n * factorial(n - 1);
}
factorial(1000000);  // stack overflow — ~1M frames × ~32 bytes each = ~32 MB

// Large stack allocation — one frame, instant death
void bad(void) {
    char buf[10000000];  // 10 MB on the stack — exceeds stack limit, segfault
}
Large buffers belong on the heap, not the stack.
The stack is fast and automatically managed, but it's limited to ~8 MB. Large arrays, matrices, and buffers should be allocated with malloc on the heap, which is limited only by available virtual address space. A common rule of thumb: if a local array is larger than a few KB, put it on the heap.

ASLR — Address Space Layout Randomization

On modern systems, the OS randomizes the base addresses of the stack, heap, and shared libraries each time a process starts. This is ASLR — Address Space Layout Randomization. It is a security feature that makes it much harder to exploit memory bugs: an attacker cannot predict where the stack or injected code will land.

ASLR is why the pointer values you print change between runs of the same program. The code segment address (text) is only randomized if the binary is compiled as position-independent (-fPIE), which is the default on most modern Linux systems.

// These addresses change on every run (with ASLR enabled)
int  x;
int *heap = malloc(4);

printf("stack: %p\n", (void*)&x);    // 0x7ffd3a12... (different each run)
printf("heap:  %p\n", (void*)heap);   // 0x5590a1c2... (different each run)

Common Problems

Pitfall 01
Returning a Pointer to a Local Variable
A local variable lives on the stack frame of the function that declared it. When the function returns, that frame is popped — the memory is no longer valid. Returning its address gives the caller a dangling pointer into reclaimed stack space. The next function call will overwrite that memory with its own frame.
Pitfall 02
Stack Overflow from Large Local Arrays
Declaring char buf[10000000] as a local variable puts 10 MB on the stack. The stack limit (typically 8 MB) is exceeded before the function body even runs. The result is immediate SIGSEGV. Large buffers must be heap-allocated with malloc.
Pitfall 03
Writing to a String Literal (rodata)
char *s = "hello"; s[0] = 'H'; attempts to write to a read-only page. The hardware enforces the page permission and delivers SIGSEGV. This is a virtual memory protection at work — not a C language rule. Use char s[] = "hello" to get a writable stack copy.
Pitfall 04
Assuming Local Variables are Zero
Stack pages may have been used by a previous function call. A local variable that "looks" uninitialized actually contains whatever bytes that earlier function left there. Only global and static variables are guaranteed zero. Always initialize locals at declaration.
Pitfall 05
Heap-Stack Pointer Confusion
It is legal to store a pointer to stack memory in a heap-allocated struct — but if the function that owns the stack variable returns first, the pointer dangles. The heap struct outlives the stack frame it points to. Lifetime must be carefully managed when mixing heap and stack pointers.
Pitfall 06
Assuming Pointer Values are Stable
ASLR means stack, heap, and library addresses change on every run. Code that hard-codes addresses, compares pointer values from different runs, or serializes pointers to disk and reads them back will break. Pointers are only valid within a single process execution.
Pitfall 07
Heap Fragmentation
Many small allocations and frees can fragment the heap — free space exists, but not in one contiguous block large enough to satisfy a big request. malloc returns NULL even though the system has enough total free bytes. Prefer fewer large allocations over many small ones in performance-critical code.
Pitfall 08
Buffer Overflow Crosses Segment Boundary
Writing past the end of a stack buffer can overwrite the saved return address in the same stack frame. When the function returns, execution jumps to whatever address the overflow wrote there. This is the classic stack smash exploit — and why modern systems use stack canaries, ASLR, and non-executable stacks.
Takeaway
A C process has five key segments: text (code, read+execute), rodata (string literals, read-only), data (initialized globals, read/write), BSS (uninitialized globals, zero-filled), heap (malloc, grows up), and stack (locals, grows down). Declaration form — not type — determines the segment. Globals and statics are zero-initialized; locals and malloc are not. The stack has a fixed size limit; large arrays belong on the heap. ASLR randomizes base addresses on every run.
