Session 1 — if / else / switch

Motivation

  1. Ask: what does a program with no branching do differently if you give it different input? Answer: nothing. Same steps every time.
  2. Explain: branching is how a program responds to its situation — different input, different path, different result.

Boolean Values in C

  1. State the rule: zero is false. Everything else is true. There is no boolean type in core C — conditions are just integers.
  2. Show a few examples live to make it concrete. 
    if (0)   { ... }   // never runs
if (1)   { ... }   // always runs
if (-42) { ... }   // always runs
if (ptr) { ... }   // runs if ptr is not NULL
  3. Mention stdbool.h gives you bool, true, false — but they're just aliases for 1 and 0. The underlying rule doesn't change.
  4. Show the comparison and logical operators. Call out && and || specifically — explain short-circuit evaluation and why if (ptr != NULL && ptr->value > 0) is safe.

if / else if / else

  1. Write a grade classifier — five lines of if/else if. Walk through how conditions are tested in order, first match wins, rest are skipped.
  2. Ask: what happens if score >= 60 is tested before score >= 90? Answer: every A student gets labelled passing — the narrower case can never be reached. Order matters, most specific first.
  3. Show the ternary operator as a shorthand for simple two-outcome decisions. Explain when to use it and when not to. 
    int max = (a > b) ? a : b;

The Dangling else

  1. Write the misleading indentation version on the board — else that looks like it belongs to the outer if.
  2. Ask the class which if the else belongs to. Let them answer. Show that C binds it to the nearest preceding if.
  3. Fix it with braces. State the rule: always use braces, even for one-line bodies. It costs nothing and prevents this entire class of bug.

switch

  1. Write a switch on a character command — q, h, r, default. Show it next to the equivalent if/else chain. Switch wins on readability when dispatching one variable to many known values.
  2. Ask: what happens if you remove the break from the 'q' case? Run it. Show it falls into the next case silently.
  3. Explain fall-through: without break, execution continues into the next case regardless of whether its label matches. Show the intentional stacked-case pattern for 'r' and 'R'.
  4. Warn: the compiler will not tell you about accidental fall-through by default. -Wimplicit-fallthrough helps. So do habits.
  5. State when to use switch vs. if/else: switch for one variable vs. many integer/character constants; if/else for ranges, floats, strings, or multi-variable conditions.
  6. Call out: switch (str) compares the pointer address, not the string content. Never switch on a string. Use strcmp() in an if/else chain.

Pitfalls — Session 1

  1. = instead of ==: if (x = 5) is always true and sets x. Legal C. Compiler may not warn. Enable -Wall.
  2. Missing break in switch: the most common switch bug. Silent. Fix: add break, or comment /* fall through */ if intentional.
  3. Comparing floats with ==: 0.1 + 0.2 == 0.3 is false. Use fabs(a - b) < 1e-9.
  4. Omitting braces: if (x) foo(); bar(); — bar always runs. Second line is never inside the if.
Session 2 — User Input & Exit Status

How Input Reaches Your Program

  1. Draw the path on the board: user types → stdin buffer (held until Enter) → scanf/fgets → your variable.
  2. Explain that the buffer is the source of most input bugs. Input doesn't arrive character by character — it arrives in a batch when the user presses Enter.

scanf

  1. Write a basic scanf for an int. Show the & — explain scanf needs an address to write into, not the value itself.
  2. Show the return value. Ask: what does scanf return if the user types "abc" when you expect a number? Answer: 0. The variable is untouched and contains garbage.
  3. State the rule: always check scanf's return value before using the variable. 
    if (scanf("%d", &age) != 1) {
    fprintf(stderr, "Error: expected a number\n");
    return 1;
}
  4. Show the format specifier table: %d for int, %lf for double (not %f), %c for char, %s for string.

The Leftover Newline Problem

  1. Write a program that reads an int then a char. Run it. Show that the char read skips — it gets the \n from the previous Enter immediately.
  2. Draw the buffer on the board: after scanf("%d") reads 42, the \n is still sitting there. The next read consumes it.
  3. Show the fix: flush with while (getchar() != '\n'); between the two reads.
  4. Ask: when does this problem not occur? Answer: when you use fgets for everything — it reads the whole line including the newline, leaving the buffer clean for the next call.

fgets

  1. Explain why fgets is safer than scanf("%s"): respects buffer size, reads the full line including spaces, returns NULL on EOF.
  2. Write fgets usage. Show the NULL check. Draw the buffer with "Alice\n\0" — fgets keeps the newline.
  3. Show the newline strip immediately after reading. 
    fgets(buf, sizeof(buf), stdin);
buf[strcspn(buf, "\n")] = '\0';
  4. Ask: what goes wrong if you compare the buffer with strcmp before stripping the newline? Answer: strcmp(buf, "quit") fails because buf contains "quit\n".

Branching on Input — Full Pattern

  1. Write the complete validated input pattern: prompt → fgets → NULL check → strip newline → strcmp branch.
  2. Point out: error messages go to fprintf(stderr, ...), not printf. Explain why — stdout and stderr are separate streams; errors should not mix with normal output, especially when piping.
  3. Build the complete interactive command loop live. Let students suggest what commands to add.

Exit Status

  1. Ask: when your program finishes, who is listening? Answer: the shell. The shell captures the exit status in $? and uses it to decide what to do next.
  2. State the convention: 0 = success. Anything else = failure. This is a universal Unix contract — every tool follows it.
  3. Show both ways to exit: return 0 from main, and exit(0) from anywhere. Show EXIT_SUCCESS and EXIT_FAILURE from stdlib.h.
  4. Show $? in the terminal after running a program. Then show && and || in shell — this is how programs compose. 
    ./validate && ./process   # process only runs if validate succeeds
./compile  || exit 1      # script stops if compile fails

Pitfalls — Session 2

  1. Not checking scanf's return: program continues with garbage. Always check.
  2. Leftover newline: mixed scanf/fgets reads break silently. Flush between them.
  3. Buffer overflow with scanf("%s"): no length limit. Use fgets instead.
  4. Not stripping fgets newline: strcmp always fails. Strip immediately after reading.
  5. Infinite loop on bad scanf: failed parse leaves bad input in buffer. Next attempt fails on same characters. Flush before retrying.
  6. Returning 0 on failure: misleads the shell. Return a non-zero code whenever the program fails to do what it was asked.
  7. Errors to stdout: corrupts piped output. Always use stderr for diagnostics.

Wrap-up

  1. The three decisions in every input-handling function: did scanf succeed? is the value in range? what do I do with a bad value?
  2. The two streams: stdout for output, stderr for errors. Non-zero exit for failure.
  3. Bridge forward: fgets + strcmp is the foundation of every command parser and config file reader they will write. File I/O uses the same pattern — open file, read line, branch, repeat.
