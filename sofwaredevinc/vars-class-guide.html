Variables, Types, Characters & Strings — Class Guide

CS253  ·  Four 75-min sessions  ·  vars-01 through vars-04
Session 1 — Variables & Assignment

What a Variable Is

  1. Ask: when you write int score = 42;, what actually happens in the computer? Let them guess. Answer: the compiler reserves 4 bytes of memory, gives that location the name "score", and stores 42 there.
  2. Draw a labeled box on the board with four properties: name, type, value, address. This is the mental model they keep for the whole course.
  3. Explain the address: every variable has a location in memory. &score gives you that address. The program finds variables by address — the name is for humans.

Declaration vs Initialization

  1. Show the difference explicitly: declaration reserves the slot, initialization fills it. They can happen together or separately. 
    int score;        // declaration only — value is garbage
int score = 0;    // declaration + initialization — always prefer this
  2. Ask: what's in an uninitialized variable? Answer: whatever bytes were at that memory location before — a previous function call, leftover data, anything. It is not zero.
  3. State the rule: always initialize at the point of declaration. There is no cost to this and it eliminates an entire class of bugs.

Assignment

  1. Write z = x + y on the board. Explain: = is not equality — it is an instruction. Evaluate the right side completely, then store the result in the left side. The right side is never modified.
  2. Show x = x + 1 — this is perfectly valid. Right side reads x, adds 1, result stored back into x.
  3. Walk through the compound operators briefly: +=, -=, *=, /=, ++, --. These are shorthand — nothing more.
  4. Warn: = in a condition (if (x = 5)) is an assignment that returns the assigned value — always true for non-zero. Legal C. Compiler may not warn. Enable -Wall.

Scope

  1. Explain: a variable lives in the block where it's declared — the enclosing { }. When execution leaves that block, the variable ceases to exist.
  2. Write a nested-block example live. Ask: can the outer block see the inner variable after the inner block closes? Answer: no. Compile and show the error.
  3. Show variable shadowing — inner variable with the same name as an outer one. Legal, but confusing. The inner one wins inside its block. The outer one is unaffected. Advise against it.

Constants

  1. Show const int MAX = 100; — the compiler enforces read-only. Show the compile error when you try to assign to it.
  2. Show #define MAX 100 — the preprocessor replaces the text before the compiler sees it. No type. No scope. Doesn't appear in the debugger.
  3. State the preference: use const. It has a type, obeys scope, and shows up in the debugger with its name. Reserve #define for compile-time flags.

Pitfalls — Session 1

  1. Uninitialized variable: garbage value, changes between runs. Always initialize.
  2. = vs ==: assignment in a condition is always true for non-zero. Enable -Wall.
  3. Integer division: 7 / 2 is 3. Cast first: (double)7 / 2 is 3.5.
  4. Variable shadowing: inner declaration hides outer — outer unchanged, inner goes away at block end.
Session 2 — Types

Why Types Exist

  1. Show the same 4 bytes of memory interpreted two ways: as an int it's a large number; as a float it's 3.125. Same bits, different meaning. Types are the interpreter.
  2. State the two things a type tells the compiler: how many bytes the variable occupies, and how to interpret those bytes.

Integer Types

  1. Walk the table: char (1), short (2), int (4), long/long long (8). Draw the relative widths on the board as bars.
  2. Explain signed vs unsigned: signed uses one bit for the sign, so half the range is negative. Unsigned gets double the positive range but can't go below zero.
  3. Show integer overflow live: add 1 to INT_MAX (2,147,483,647). Print it. Watch it wrap to a large negative number. The compiler will not warn.
  4. Ask: when should you use unsigned? Answer: counts that can never be negative, bit manipulation, matching external data formats.

Floating-Point Types

  1. Explain: floats use IEEE 754 — a binary encoding that trades exact representation for enormous range. Most decimal fractions have no exact binary representation.
  2. Show live: printf("%.20f\n", 0.1 + 0.2); — the result is not 0.3. It's 0.30000000000000004...
  3. State the rule: never compare floats with ==. Use fabs(a - b) < 1e-9. Show the alternative.
  4. State the default: use double. float has only 7 significant digits — not enough for most work. Only use float when memory or GPU performance genuinely demands it.

Type Conversion

  1. Show implicit widening: mixing int and double in an expression promotes the int to double automatically. No data lost.
  2. Show the integer division trap: 
    int a = 7, b = 2;
double bad  = a / b;         // 3.0 — division happened as int first!
double good = (double)a / b; // 3.5 — cast before dividing
  3. Show narrowing: (int)3.9 gives 3 — truncates toward zero, no rounding, no warning. The compiler assumes you know what you asked for.
  4. Ask: when does truncation differ from rounding? Answer: always for negative numbers. (int)-2.9 is -2, not -3.

sizeof and stdint.h

  1. Show sizeof live for several types. Emphasize: this is a compile-time operator, not a function. Use it instead of hardcoding sizes.
  2. Explain: the C standard only guarantees minimum sizes for int, long, etc. If exact size matters — network protocols, file formats, hardware — use <stdint.h>: uint32_t, int64_t, etc.

Pitfalls — Session 2

  1. Integer overflow: wraps silently. Use a wider type or check bounds before arithmetic.
  2. Float equality: use fabs(a - b) < epsilon. Never ==.
  3. Integer division truncation: cast before dividing, not after.
  4. Wrong printf specifier: %d for a long is undefined behavior. Match types exactly: %ld, %lf, %zu.
  5. Signed/unsigned comparison: -1 > 1u is true. Enable -Wall -Wsign-compare.
Session 3 — Characters

char Is an Integer

  1. Write on the board: 'A' is 65. 'a' is 97. '0' is 48. These are not conventions — they are the actual values stored in memory.
  2. Show the proof live: 
    char c = 'A';
printf("%c\n", c);   // A
printf("%d\n", c);   // 65
  3. Because char is an integer, arithmetic works. Show: 'A' + 1 is 'B'. Show converting digit char to int: '7' - '0' is 7.

The ASCII Ranges

  1. Draw the three key ranges on the board: digits 48–57, uppercase 65–90, lowercase 97–122. These are the only three they need to remember.
  2. Point out the gaps: uppercase-to-lowercase is always +32. Digit char to int value is always -48 (or - '0').
  3. Show range-check membership: ch >= '0' && ch <= '9' tests if ch is a digit. This is what the ctype.h functions do under the hood.

ctype.h

  1. Show the classification functions: isalpha, isdigit, isspace, isupper, tolower, toupper. These are cleaner than hand-rolled range checks and handle edge cases correctly.
  2. Warn: pass (unsigned char)c to ctype functions, not a plain char. On platforms where char is signed, a negative value is undefined behavior.

Escape Sequences and '\0'

  1. Walk the important escape sequences: '\n' (10), '\t' (9), '\0' (0), '\\' (92).
  2. Stop on '\0'. Write both on the board side by side: '\0' = 0, '0' = 48. They are completely different. Confusing them causes string bugs that are extremely hard to find.
  3. Ask: what does '\0' do in a string? Answer: it marks the end. Every string function walks forward until it hits this zero byte — that's how they know to stop.

Reading Characters

  1. Show getchar(). Ask: what type should you use to store the return value? Most say char. The answer is int.
  2. Explain why: getchar() returns EOF (-1) at end of input. If stored in a char, EOF either can't fit (unsigned char) or gets confused with a valid byte value (signed char). The EOF check silently breaks either way.
  3. Show the canonical read loop: 
    int c;
while ((c = getchar()) != EOF)
    putchar(c);

Pitfalls — Session 3

  1. '\0' vs '0': null is 0, digit zero is 48. Confusing them corrupts strings.
  2. Storing getchar() in char: EOF check breaks. Always use int.
  3. Comparing strings with ==: compares addresses, not content. Use strcmp(). (Relevant here and next session.)
  4. Signed vs unsigned char: platform-defined. Be explicit when signedness matters.
Session 4 — Strings

What a String Is

  1. State it plainly: C has no string type. A string is a char array with a zero byte at the end. That's the entire definition. Everything else follows from this.
  2. Draw "Hello\0" on the board as six labeled boxes: H(72), e(101), l(108), l(108), o(111), \0(0). Point to the last box — this is not optional. It is the contract every string function relies on.
  3. Ask: if strlen("Hello") returns 5, how big must the array be? Answer: 6. One extra byte, always, for the terminator.

Array Form vs Pointer Form

  1. Write both declarations side by side: 
    char  greeting[] = "Hello";    // array — modifiable copy on stack
const char *msg   = "Hello";   // pointer — points to read-only memory
  2. Show that modifying greeting[0] works fine. Then show that modifying through msg is undefined behavior — may crash, may silently corrupt.
  3. Explain the rule: if you need to modify the string, use the array form. If you're just reading or passing around, a pointer is fine — but mark it const.

Essential String Functions

  1. Walk through the unsafe versions first — strcpy, strcat, sprintf — and name the problem: no size limit, writes until null terminator regardless of your buffer size.
  2. Show the safe replacements: strncpy, strncat, snprintf. These take an explicit size argument. Make a habit: always use the n variants.
  3. Warn about strncpy's trap: if the source is longer than n, no null terminator is added. Always set the last byte manually: 
    strncpy(buf, src, sizeof(buf) - 1);
buf[sizeof(buf) - 1] = '\0';
  4. Show strcmp. Ask: what does it return when strings are equal? Most say true or 1. Answer: 0. Always check strcmp(...) == 0, not just strcmp(...).
  5. Show strlen vs sizeof. Write both on the board for the same array. strlen counts visible characters. sizeof counts total array bytes — including the '\0' slot. On a pointer, sizeof gives the pointer size (8), not the string length.

Strings Are Pointers

  1. Explain: passing a string to a function passes a char * — the address of the first character. The function knows nothing about the buffer size. This is why every write function needs a separate size argument.
  2. Show the traversal pattern with a pointer: 
    char *p = "Hello";
while (*p != '\0') {
    putchar(*p);
    p++;
}
  3. Ask: why does while (*p) work without the explicit != '\0' comparison? Answer: '\0' has value 0, which is falsy. The loop terminates naturally when the pointer reaches the null byte.

fgets and Input

  1. Show why scanf("%s", buf) is dangerous: no length limit, stops at whitespace, can't read full names. Never use it unguarded.
  2. Show fgets as the replacement: reads a full line, respects buffer size, includes the newline. 
    fgets(buf, sizeof(buf), stdin);
buf[strcspn(buf, "\n")] = '\0';   // strip the newline immediately
  3. Ask: what goes wrong if you forget to strip the newline before calling strcmp? Answer: the comparison always fails because the buffer contains "hello\n", not "hello".

Pitfalls — Session 4

  1. Buffer too small: char buf[5] = "Hello" — no room for '\0'. Always size + 1.
  2. Comparing strings with ==: compares addresses. Use strcmp() == 0.
  3. strcpy without size: overflows if source is longer than buffer. Use strncpy + manual termination, or snprintf.
  4. strncpy without manual termination: if source ≥ n, no '\0' added. Always set last byte.
  5. Modifying a string literal: undefined behavior. Use array form for mutable strings.
  6. Forgetting to strip fgets newline: all string comparisons silently fail.
  7. strlen vs sizeof on a pointer: sizeof(ptr) is 8, not the string length. Only use sizeof on the array itself.
  8. No null terminator when building strings manually: every string function will read past the end. Always append '\0' when constructing strings byte by byte.

Wrap-up

  1. Draw the chain on the board: a variable has a type → a type determines byte width and interpretation → char is a 1-byte integer → a string is a char array terminated by the zero byte. Each concept builds directly on the previous one.
  2. The three rules that prevent most string bugs: always size + 1 for the terminator · compare with strcmp, never == · always use fgets or the n-variants of string functions.
  3. Bridge forward: everything from here — file I/O, command-line arguments, parsing — involves reading text into buffers and processing it character by character or line by line. This is the foundation.
