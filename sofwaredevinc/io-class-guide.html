Input/Output — Class Guide

CS253  ·  ~85 min  ·  Three sessions
Student pages: io-01-terminal-io.html  ·  io-02-file-io.html  ·  io-03-parsing-csv.html  ·  io-cookbook.html
Session 1  ·  ~30 min  ·  Streams + Output

Opening Frame — What Is I/O?

  1. Ask: where does printf actually send its output? What is "the terminal"? Let students guess. Don't correct yet — just get the question alive.
  2. State the mental model: every C program starts with three pre-opened connections to the outside world. We call them streams.
    stdin (fd 0) ← keyboard / pipe stdout (fd 1) → terminal / file stderr (fd 2) → terminal (always)
  3. Key point to say aloud: stdin, stdout, stderr are just FILE* pointers — the same type you'll use for files. Terminal I/O and file I/O are the same machinery. This is the conceptual payoff of the whole lesson.
  4. Ask: why have stdout AND stderr? What's the difference in practice?
    Hint: run ./prog > output.txt and ask where errors go.
  5. Answer: they can be redirected independently. When stdout goes to a file, stderr still reaches the terminal. Errors written to stdout get silently buried in the file — a real bug in production programs.

printf — Formatted Output

  1. Write the basic call on the board. Point out the two parts: literal characters pass through unchanged, format specifiers get replaced. 
    printf("Name: %s, Age: %d, GPA: %.2f\n",
       name, age, gpa);
  2. Walk the specifier table — just the five they'll use constantly: %d, %f, %.Nf, %s, %c. Point out %p for pointers and %% for a literal percent. Skip the rest for now.
  3. Demo the mismatch bug live: 
    double gpa = 3.85;
printf("%d\n", gpa);   // wrong specifier
    This is undefined behavior — the output is garbage and the compiler may not stop you. The format specifier and argument type must agree exactly.
  4. Show fprintf. Emphasize: it's identical to printf with a stream prepended. 
    fprintf(stdout, "same as printf\n");
fprintf(stderr, "Error: %s\n", msg);
    Say: once you see this, printf is just shorthand. All output functions are really fprintf to some stream.
  5. Quickly show puts and putchar — simple cases, no format needed. Note puts appends '\n' automatically and most students forget it's there.
  6. Practice check — ask the class to predict output before running: 
    int x = 255;
printf("dec=%d  hex=0x%x  char=%c\n", x, x, x);
Session 2  ·  ~30 min  ·  Terminal Input + The Newline Problem

scanf — Reading Values

  1. Write the pattern. Pause on the & before saying anything. 
    int age;
scanf("%d", &age);
    Ask: printf takes a value. scanf takes... what? Why?
  2. Answer: printf reads. scanf writes. To write into a variable, it needs the variable's address — not its current value. That's the &. This is the single most common scanf mistake.
  3. Show the check pattern immediately after: 
    int n = scanf("%d", &age);
if (n != 1) {
    fprintf(stderr, "Expected an integer\n");
    return 1;
}
    scanf returns the number of items successfully matched. If the user types "hello" and you expected %d, the return is 0 and the variable is uninitialized. Always check it.
  4. Ask: what can scanf NOT read? Elicit "a sentence with spaces." Demonstrate: typing "Alice Smith" — %s stops at the space, name gets only "Alice".

fgets — Reading a Full Line

  1. Introduce fgets as the safe line reader: 
    char buf[128];
fgets(buf, sizeof(buf), stdin);
    Three arguments: buffer, size limit, stream. The size limit is the reason this is safe — it cannot overflow.
  2. Immediately show what fgets puts in the buffer. Type "Alice" and hit Enter. Ask: what is in buf right now?
    buf = ['A','l','i','c','e','\n','\0', ...]
    The '\n' is in there. This surprises everyone.
  3. Show the strip pattern every time — make it muscle memory: 
    size_t len = strlen(buf);
if (len > 0 && buf[len-1] == '\n')
    buf[len-1] = '\0';
  4. Show the consequence of not stripping: 
    fgets(buf, sizeof(buf), stdin);
if (strcmp(buf, "quit") == 0) ...  // never true
    buf holds "quit\n". strcmp("quit\n", "quit") is not zero. This is the #2 most common fgets bug after not stripping the newline from a CSV field.
  5. Show getchar briefly — return type is int, not char. Ask: why int?
    If getchar returns a char, what happens when EOF (-1) gets stored in an unsigned char? It becomes 255 — and the EOF check never triggers.

The scanf / fgets Newline Problem

  1. This is the most important live demo of the session. Write this and run it: 
    int n;
scanf("%d", &n);
char name[64];
fgets(name, sizeof(name), stdin);
printf("n=%d, name=[%s]\n", n, name);
    Type 42 and press Enter. Output: n=42, name=[]. The name is empty.
  2. Explain what happened. Draw the buffer on the board:
    After scanf reads "42": buffer contains: '\n' fgets grabs '\n' immediately — returns ""
  3. Show the fix: 
    scanf("%d", &n);
/* Consume the rest of the line */
int c;
while ((c = getchar()) != '\n' && c != EOF);
/* Now safe to call fgets */
fgets(name, sizeof(name), stdin);
    Alternatively: avoid mixing scanf and fgets entirely. Use fgets for everything and convert numbers with atoi/atof. Avoids the problem completely.
  4. Point to the cookbook page — the scanf/fgets recipe has a diagram of exactly this. Good reference when they hit it in a lab.
Session 3  ·  ~25 min  ·  File I/O + CSV Capstone

FILE* and fopen

  1. Connect to what they already know: the stream they've been writing to (stdout) is a FILE*. When you open a file, you get back the same type. 
    FILE *fp = fopen("data.txt", "r");
if (fp == NULL) {
    perror("fopen");
    return 1;
}
  2. Draw the open/use/close lifecycle on the board. Write it as three numbered steps. This is the shape of every file operation they'll ever write.
    1. fopen — get a FILE*, check for NULL 2. use — fgets / fprintf / fscanf 3. fclose — flush buffer, release handle
  3. Ask: what happens if you skip fclose?
    The data you wrote may still be in a memory buffer — not on disk yet. Your output file could be empty or truncated.
  4. Walk the mode string table — focus on the gotcha: mode "w" destroys the existing file immediately. Not when you write. Not when you close. When fopen succeeds.
    Opening the wrong file with "w" destroys it silently. There is no warning and no undo. Always double-check the filename in write mode.
  5. Introduce perror — show the output format: 
    fopen: No such file or directory
    Convention: pass the function name as the prefix. The OS fills in the reason. No need to look up errno manually.

The File Reading Loop

  1. Write the canonical pattern: 
    char line[256];
while (fgets(line, sizeof(line), fp) != NULL) {
    /* process line */
}
fclose(fp);
    Point out: loop condition is the fgets return — not feof. This is important.
  2. Show the feof loop bug right away — don't let them encounter it in the lab: 
    while (!feof(fp)) {          // WRONG
    fgets(line, sizeof(line), fp);
    process(line);            // processes last line twice
}
    Explain: feof only becomes true after a read returns NULL. So the loop runs one extra time with stale data in line.
  3. Show fscanf for structured files — same as scanf but file instead of stdin: 
    char name[64]; int score;
while (fscanf(fp, "%63s %d", name, &score) == 2) {
    printf("%s: %d\n", name, score);
}
    Check for exactly the count you expect (2), not != EOF.

CSV Capstone

  1. Frame CSV as the payoff: it's not a new concept — it's everything from today applied to a real format. File I/O for the outer loop, string splitting for the inner loop, type conversion for the fields.
  2. Write the file on the board:
    name,major,gpa,year Alice,CS,3.85,3 Bob,EE,3.40,2
    Ask: what do we need to do to turn line 2 into a struct? (fgets → strip '\n' → strtok → atof)
  3. Introduce strtok. Emphasize the unusual pattern: 
    char *tok = strtok(line, ",");    // first call: pass string
while (tok != NULL) {
    printf("[%s]\n", tok);
    tok = strtok(NULL, ",");      // subsequent: pass NULL
}
    Ask: what does strtok actually do to the string?
  4. Answer: it writes '\0' over each delimiter. The original string is destroyed. Tokens are pointers into the modified buffer. Never call strtok on a string literal.
    This is the most common strtok crash: char *s = "Alice,CS"; strtok(s, ",") — string literals are read-only. Use a char array.
  5. Walk the full parser from io-03-parsing-csv.html. Don't retype from scratch — open the page and read through parse_student() together. Ask: why does it check each tok for NULL before using it?
  6. Build the main loop together — fopen, fgets outer loop, skip header, call parse_student, fclose. Let students supply the pieces.
  7. Show what happens when a field is missing. strtok returns NULL. What does atof(NULL) do?
    Crash — segfault. Every strtok result must be checked before use.

Wrap-up

  1. The three functions they'll use in 90% of labs: fopen / fgets / fclose. Everything else is a variation on this pattern.
  2. Point to the cookbook (io-cookbook.html). Walk the mistake grid briefly — name each one. The top four that will bite them on the next lab: no NULL check after fopen, missing fclose, feof loop, fgets keeps '\n'.
  3. Name the two things that make CSV parsing tricky that we didn't cover: quoted fields (a field can contain a comma if it's in double quotes) and multi-line values. Reassure them: for lab data files, the simple approach is correct and sufficient.
  4. Connection to the four-layer framework:
    C Language layer: printf / scanf / fopen / fgets / strtok C Runtime layer: streams opened at startup, fclose flushes buffers Machine Environment: file descriptors 0/1/2, OS handles actual disk I/O Dev Ecosystem: redirect with > and < in shell; pipe with |
  5. Ask: if we redirect stdin from a file — ./prog < data.txt — what changes in the code? Answer: nothing. fgets(stdin) reads from the file automatically. The same code works for keyboard and file input. This is why streams matter.
