Chapter 02

File Input & Output

Files are just streams that point to disk instead of the terminal. The same functions, the same mental model — and the same responsibility to clean up when you're done.

FILE* — The Stream Handle

In C, every open file is represented by a FILE* — a pointer to an opaque structure maintained by the standard library. You never look inside it directly. You hand it to I/O functions and they do the right thing.

Analogy
A FILE* is like a bookmark in a book, combined with a description of how you opened the book (read-only? read/write? start from the beginning?). The library keeps track of where you are and handles the actual disk access. You just pass the bookmark around.

The three standard streams you already know — stdin, stdout, stderr — are all FILE* values that the runtime pre-opens for you. When you open your own files, you get the same type.

The Open / Use / Close Pattern

Every file operation follows the same three-step lifecycle. Skipping step 3 is one of the most common resource leaks in C programs.

1
Open — call fopen() to get a FILE*. Check for NULL before proceeding.
2
Use — read from or write to the file using the FILE* you received.
3
Close — call fclose() to flush any buffered data to disk and release the file handle back to the OS.
#include <stdio.h>

int main(void) {
    // 1. Open
    FILE *fp = fopen("data.txt", "r");
    if (fp == NULL) {
        perror("fopen");   // prints the system error message
        return 1;
    }

    // 2. Use
    char line[256];
    while (fgets(line, sizeof(line), fp) != NULL) {
        printf("%s", line);    // line already contains '\n'
    }

    // 3. Close
    fclose(fp);
    return 0;
}

The second argument to fopen is the mode string — it specifies what you want to do with the file and what happens if it doesn't exist yet.

Opening a file with "w" immediately truncates it to zero bytes — even before you write anything. If you open the wrong file by accident, the original content is gone. There is no warning. Always double-check the filename before using write mode.

Reading from Files

fgets — Line by Line

fgets is the standard way to read a text file line by line. It reads until it encounters a newline, fills the buffer, or reaches EOF — whichever comes first.

FILE *fp = fopen("names.txt", "r");
if (!fp) { perror("fopen"); return 1; }

char line[256];
while (fgets(line, sizeof(line), fp) != NULL) {
    // Strip trailing newline
    size_t len = strlen(line);
    if (len > 0 && line[len-1] == '\n')
        line[len-1] = '\0';
    printf("Read: [%s]\n", line);
}
fclose(fp);

fscanf — Formatted File Reading

fscanf is the file version of scanf. It reads formatted data from a FILE* instead of stdin. Useful for files with a consistent record structure.

// Suppose scores.txt contains lines like: "Alice 95"
FILE *fp = fopen("scores.txt", "r");
if (!fp) { perror("fopen"); return 1; }

char name[64];
int  score;

while (fscanf(fp, "%63s %d", name, &score) == 2) {
    printf("%s scored %d\n", name, score);
}
fclose(fp);

fread — Binary / Raw Reads

For binary files — images, structs, custom formats — use fread. It reads a specified number of items each of a specified size and returns how many items were actually read.

// Read up to 100 integers from a binary file
int buffer[100];
FILE *fp = fopen("data.bin", "rb");
if (!fp) { perror("fopen"); return 1; }

size_t n = fread(buffer, sizeof(int), 100, fp);
printf("Read %zu integers\n", n);
fclose(fp);

Writing to Files

fprintf — Formatted File Writing

Just like printf but directs output to a FILE*. This is the most common way to produce structured text output to a file.

FILE *fp = fopen("report.txt", "w");
if (!fp) { perror("fopen"); return 1; }

fprintf(fp, "Student Report\n");
fprintf(fp, "Name: %s, Score: %d\n", "Alice", 95);
fprintf(fp, "Name: %s, Score: %d\n", "Bob", 87);

fclose(fp);   // flush and close — don't skip this!

fputs — Write a String

fputs(str, fp) writes a string to a file without a trailing newline (unlike puts which adds one to stdout).

fwrite — Binary / Raw Writes

// Write an array of structs to a binary file
typedef struct {
    int   id;
    float score;
} Record;

Record records[3] = {{1, 95.5f}, {2, 87.0f}, {3, 91.2f}};

FILE *fp = fopen("records.bin", "wb");
if (!fp) { perror("fopen"); return 1; }

fwrite(records, sizeof(Record), 3, fp);
fclose(fp);

Detecting End-of-File and Errors

When a read function returns a sentinel value — NULL for fgets, EOF for fscanf — it could mean one of two things: either you reached the end of the file, or an error occurred. Use feof and ferror to tell them apart.

FILE *fp = fopen("data.txt", "r");
if (!fp) { perror("fopen"); return 1; }

char line[256];
while (fgets(line, sizeof(line), fp) != NULL) {
    process(line);
}

// After the loop, determine why fgets stopped
if (ferror(fp)) {
    fprintf(stderr, "Error reading file\n");
} else if (feof(fp)) {
    printf("Reached end of file normally\n");
}

fclose(fp);
Don't loop on feof
A classic bug is writing while (!feof(fp)) as the loop condition. feof only becomes true after a read has failed — so the loop always executes one extra iteration and processes stale data. Loop on the return value of the read function instead.

perror — Printing System Error Messages

When an I/O function fails, the C library sets a global variable called errno to indicate the reason. perror(label) prints that reason as a human-readable message to stderr:

FILE *fp = fopen("nonexistent.txt", "r");
if (fp == NULL) {
    perror("fopen");
    // Output: "fopen: No such file or directory"
    return 1;
}

The string you pass to perror becomes a prefix before the system message. Use the function name so you know where the failure happened.

A Complete Read/Write Example

This program reads student names from one file and writes a formatted report to another — showing the full open/use/close pattern for both input and output:

#include <stdio.h>
#include <string.h>

int main(void) {
    FILE *in  = fopen("students.txt", "r");
    FILE *out = fopen("report.txt",   "w");

    if (!in  || !out) { perror("fopen"); return 1; }

    fprintf(out, "=== Enrollment Report ===\n");

    char name[64];
    int  count = 0;

    while (fgets(name, sizeof(name), in) != NULL) {
        // Strip trailing newline
        size_t len = strlen(name);
        if (len > 0 && name[len-1] == '\n')
            name[len-1] = '\0';

        count++;
        fprintf(out, "%3d. %s\n", count, name);
    }

    fprintf(out, "\nTotal students: %d\n", count);

    fclose(in);
    fclose(out);
    printf("Report written to report.txt\n");
    return 0;
}

Pitfalls

⚠ Pitfall 1 — Not checking fopen for NULL

If fopen fails and you ignore the return value, the first read or write call will dereference a NULL pointer and crash. Every fopen must be followed by a NULL check.

⚠ Pitfall 2 — Forgetting fclose

Data written to a file is often held in an internal buffer and not flushed to disk until fclose (or until the buffer fills). Skip fclose and your output file may be empty or truncated — even if no error was reported.

⚠ Pitfall 3 — Opening for write when you meant append

Mode "w" silently destroys the existing file contents the moment you call fopen. If you want to add to an existing file, use "a".

⚠ Pitfall 4 — Mixing text and binary mode

On Windows, text mode ("r", "w") translates \r\n to \n. Binary mode ("rb", "wb") does no translation. For portability with binary files — images, structs — always use binary mode explicitly.
