Chapter 03

Parsing CSV Data

CSV is the universal format for tabular data. Parsing it in C puts everything together: file I/O, string splitting, type conversion, and error handling — all at once.

What Is CSV?

CSV — Comma-Separated Values — is a plain-text format for tables. Each line is one row. Within a line, fields are separated by commas. The first line is usually a header row that names the columns.

name,major,gpa,year
Alice,CS,3.85,3
Bob,EE,3.40,2
Carol,Math,3.95,4
Dave,CS,2.90,1

CSV is ubiquitous: spreadsheets export it, databases dump to it, data-science pipelines read it. Knowing how to parse it in C means you can process real-world data with nothing but the standard library.

The Two-Level Structure

Parsing CSV is a two-level problem. At the outer level you handle lines — one row per iteration. At the inner level you handle fields — one value per comma-separated token within the line.

fgets handles the outer loop; strtok handles the inner field extraction

strtok — Splitting a String on a Delimiter

strtok breaks a string into tokens separated by delimiter characters. Its interface is unusual because it uses a stateful, two-call pattern:

char line[] = "Alice,CS,3.85,3";

// First call: pass the string
char *token = strtok(line, ",");

// Subsequent calls: pass NULL to continue from where we left off
while (token != NULL) {
    printf("field: [%s]\n", token);
    token = strtok(NULL, ",");
}
// Output:
// field: [Alice]
// field: [CS]
// field: [3.85]
// field: [3]
How strtok works
strtok modifies the original string in place: it writes '\0' over each delimiter it finds, turning the single string into a sequence of substrings. The tokens it returns are just pointers into the modified original. Do not pass a string literal — it must be a writable character array.
strtok modifies its input
Because strtok overwrites delimiters with null terminators, the original string is destroyed. If you need to keep the original, copy it with strcpy or strncpy before calling strtok.

Type Conversion — Strings to Numbers

All CSV fields arrive as strings. To use numeric fields mathematically, you must convert them. The standard library provides conversion functions for the most common types:

#include <stdlib.h>

char *field = "3.85";

int    year = atoi(field);      // string → int  (simple but no error detection)
double gpa  = atof(field);      // string → double

// Safer alternatives with error detection:
char *end;
long   n = strtol(field, &end, 10);   // base-10 integer; end points past last digit
double d = strtod(field, &end);       // double; end points past last digit

if (end == field) {
    fprintf(stderr, "Not a valid number\n");
}

For simple labs, atoi and atof are fine. In production code, prefer strtol/strtod because they tell you whether conversion succeeded and where it stopped.

A Complete CSV Parser

Here is a full program that reads the student CSV file, parses every field, and computes the class average GPA:

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

#define MAX_LINE  256
#define MAX_FIELD  64

typedef struct {
    char   name[MAX_FIELD];
    char   major[MAX_FIELD];
    double gpa;
    int    year;
} Student;

/* Parse one CSV line into a Student struct.
   Returns 1 on success, 0 if the line couldn't be parsed. */
int parse_student(char *line, Student *s) {
    char *tok;

    tok = strtok(line, ",");  if (!tok) return 0;
    strncpy(s->name, tok, MAX_FIELD - 1);
    s->name[MAX_FIELD - 1] = '\0';

    tok = strtok(NULL, ","); if (!tok) return 0;
    strncpy(s->major, tok, MAX_FIELD - 1);
    s->major[MAX_FIELD - 1] = '\0';

    tok = strtok(NULL, ","); if (!tok) return 0;
    s->gpa = atof(tok);

    tok = strtok(NULL, ","); if (!tok) return 0;
    s->year = atoi(tok);

    return 1;
}

int main(void) {
    FILE *fp = fopen("students.csv", "r");
    if (!fp) { perror("fopen"); return 1; }

    char    line[MAX_LINE];
    Student s;
    int     count  = 0;
    double  gpa_sum = 0.0;
    int     lineno = 0;

    while (fgets(line, sizeof(line), fp) != NULL) {
        lineno++;

        /* Strip trailing newline */
        size_t len = strlen(line);
        if (len > 0 && line[len-1] == '\n')
            line[len-1] = '\0';

        /* Skip the header row */
        if (lineno == 1) continue;

        /* Skip blank lines */
        if (line[0] == '\0') continue;

        if (!parse_student(line, &s)) {
            fprintf(stderr, "Warning: bad line %d, skipping\n", lineno);
            continue;
        }

        printf("%-12s %-6s  GPA: %.2f  Year: %d\n",
               s.name, s.major, s.gpa, s.year);
        gpa_sum += s.gpa;
        count++;
    }

    fclose(fp);

    if (count > 0)
        printf("\nAverage GPA: %.2f  (%d students)\n",
               gpa_sum / count, count);
    else
        printf("No student records found.\n");

    return 0;
}

Bob          EE      GPA: 3.40  Year: 2
Carol        Math    GPA: 3.95  Year: 4
Dave         CS      GPA: 2.90  Year: 1

Average GPA: 3.53  (4 students)

Walking Through the Key Decisions

Why strip the newline before parsing?

fgets includes the '\n' in the buffer. If you don't strip it, the last field in every row gets a hidden newline appended — "3\n" instead of "3". atoi("3\n") still works, but strcmp(s.major, "CS") would fail because the stored value is "CS\n", not "CS".

Why skip line 1 (the header)?

The first line contains column names, not data. If you tried to parse "name,major,gpa,year" into a Student struct, atof("gpa") would return 0.0 silently and you'd get a garbage record. Skipping it is simpler than detecting and rejecting it after the fact.

Why pull parse_student into its own function?

Separating parsing from the main I/O loop keeps each piece testable and readable. You can call parse_student from a unit test with a hard-coded string — no file I/O needed. This is the separation of concerns principle in practice.

Limits of This Approach

Full CSV Has More Rules
The simple parser above handles the common case. Real CSV (RFC 4180) also allows fields to be quoted — "Smith, John" contains a comma but is one field. Quoted fields can span multiple lines. Quotes inside a quoted field are escaped by doubling them: "He said ""hello""". For production code that must handle arbitrary CSV, consider a dedicated parsing library.

For coursework and data files you control, the simple approach is exactly right — and understanding its limitations is just as valuable as knowing how to use it.

Pitfalls

⚠ Pitfall 1 — Calling strtok on a string literal

strtok writes into its first argument. String literals are read-only — writing to them is undefined behavior. Always pass a character array: char line[256], not char *line = "...".

⚠ Pitfall 2 — Forgetting to strip the trailing newline

The last field of every line parsed with fgets will have a hidden '\n' appended. String comparisons and numeric conversions may still work, but the stored string won't match what you expect. Always strip it before parsing.

⚠ Pitfall 3 — Trusting atoi / atof with invalid data

atoi("hello") silently returns 0. atof("") silently returns 0.0. If your CSV might have missing or malformed fields, use strtol/strtod and check whether the end pointer advanced.

⚠ Pitfall 4 — strtok skips consecutive delimiters

strtok treats consecutive delimiters as a single delimiter, so empty fields — "Alice,,3.85,3" — are silently swallowed. If empty fields are valid in your data, you'll need a character-by-character approach or a dedicated CSV library instead.

⚠ Pitfall 5 — strtok is not re-entrant

strtok stores its position in a hidden static variable. Calling it on a second string before finishing the first will corrupt the first parse. Never interleave two strtok parses. If you need that, POSIX provides strtok_r, a re-entrant version.
