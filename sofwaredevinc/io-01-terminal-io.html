Chapter 01

Terminal Input & Output

Every program communicates. Before files and networks, there are three standard streams your program gets for free — and the functions to use them.

The Three Standard Streams

When your program starts, the operating system opens three streams automatically. A stream is an ordered sequence of bytes — you can think of it as a pipe connecting your program to something in the outside world.

The three standard streams — automatically open when main() starts

Key Concept
stdin, stdout, and stderr are just FILE* pointers — the same type used for file I/O. Terminal I/O and file I/O use the same underlying mechanism; the only difference is where the stream is connected.

The distinction between stdout and stderr matters in real programs: they can be redirected independently. If your program writes errors to stderr, the user can redirect stdout to a file without losing error messages in the terminal.

Output: printf and Friends

printf writes a formatted string to stdout. The format string is a template — literal characters are copied as-is, and format specifiers beginning with % are replaced with values from the argument list.

#include <stdio.h>

int main(void) {
    int    age   = 21;
    double gpa   = 3.75;
    char   name[] = "Alice";

    printf("Name: %s\n", name);          // %s — string
    printf("Age:  %d\n", age);           // %d — signed int
    printf("GPA:  %.2f\n", gpa);         // %.2f — float, 2 decimal places
    printf("Hex:  0x%x\n", age);         // %x — hexadecimal
    printf("Addr: %p\n", (void*)&age);   // %p — pointer address

    return 0;
}

fprintf — Writing to Any Stream

fprintf is the same as printf but takes a stream as its first argument. This lets you write to stderr or to a file with the same format-string syntax.

// Write an error message to stderr
fprintf(stderr, "Error: could not open file '%s'\n", filename);

// printf is literally just fprintf(stdout, ...)
fprintf(stdout, "Hello, world\n");
Analogy
printf is just a shorthand for fprintf(stdout, ...). Once you see that all these functions work with streams, the whole I/O system clicks into place — terminal and file I/O are the same machinery.

puts and putchar

For simple output that doesn't need formatting, there are simpler functions:

puts("Hello, world");   // writes string + '\n' to stdout
putchar('A');           // writes one character to stdout

puts automatically appends a newline — handy for whole lines, but surprising if you forget it's there. putchar is the lowest-level character output; use it when you need precise control over exactly what bytes are written.

Input: scanf and Friends

scanf reads formatted input from stdin. The format string uses the same specifiers as printf, but instead of producing output it matches and converts input. The key difference from printf: arguments must be pointers to where the values should be stored.

int   age;
float score;
char  name[64];

printf("Enter your name: ");
scanf("%63s", name);          // %63s limits to 63 chars (leaves room for '\0')

printf("Enter age and score: ");
scanf("%d %f", &age, &score);  // & because scanf needs addresses
Why the & ?
printf receives values — it just needs to read them. scanf receives addresses — it needs to know where to write the result. Pass a plain variable and scanf writes to garbage memory. Pass &variable and it writes to the variable.

scanf Return Value

scanf returns the number of items it successfully matched. Always check it:

int x;
int n = scanf("%d", &x);

if (n != 1) {
    fprintf(stderr, "Error: expected an integer\n");
    return 1;
}
printf("You entered: %d\n", x);

Reading a Whole Line: fgets

scanf("%s", buf) stops at whitespace, so it can't read a sentence. Use fgets instead — it reads an entire line up to a size limit:

char line[256];

printf("Enter a sentence: ");
fgets(line, sizeof(line), stdin);
// line now holds the full input, INCLUDING the '\n' character

// Strip the trailing newline if present
size_t len = strlen(line);
if (len > 0 && line[len - 1] == '\n')
    line[len - 1] = '\0';
Mental model
fgets(buf, size, stream) reads at most size - 1 characters, then appends '\0'. The -1 leaves room for the null terminator. This is the safe pattern for reading lines — you control the buffer size, so there is no overflow.

Character-Level Input: getchar

At the lowest level, getchar() reads one character at a time from stdin. It returns an int — not a char — because it also returns the special value EOF (typically −1) to signal end-of-input:

int c;
while ((c = getchar()) != EOF) {
    if (c == '\n') {
        putchar('\n');
        break;
    }
    putchar(c);  // echo the character
}
Common Mistake — char vs int for getchar
If you store getchar()'s return value in a char instead of an int, the EOF comparison may never trigger — EOF (−1) gets truncated to fit in a char. Always use int c = getchar().

The scanf Newline Problem

This is the #1 source of confusion when mixing scanf and fgets. After scanf reads a value, it leaves the newline character in the input buffer. The very next fgets call then reads that leftover newline and immediately returns an empty string.

scanf leaves the newline in the buffer — the next fgets sees it immediately

int n;
scanf("%d", &n);

// Flush the rest of the line (including the '\n') before calling fgets
int c;
while ((c = getchar()) != '\n' && c != EOF);

char name[64];
fgets(name, sizeof(name), stdin);  // now safe to call

Pitfalls

⚠ Pitfall 1 — Passing a value where scanf needs a pointer

Writing scanf("%d", x) instead of scanf("%d", &x) passes the integer value of x as an address and writes into garbage memory. The compiler may warn about this; listen to it.

⚠ Pitfall 2 — No size limit on %s

Writing scanf("%s", buf) with no length limit will overflow the buffer if the user types more characters than the array can hold. Always specify the length: scanf("%63s", buf) for a 64-byte array.

⚠ Pitfall 3 — Ignoring scanf's return value

If the user types "abc" when you expected an integer, scanf matches nothing and returns 0. The variable is uninitialized and the "abc" stays in the buffer, poisoning every subsequent read. Always check the return value.

⚠ Pitfall 4 — fgets keeps the newline

fgets stores the '\n' as part of the string. If you then compare the string with strcmp, the newline will cause the comparison to fail even when the content matches. Strip it after reading.
