Virtual Memory

Your program thinks it has 64-bit address space all to itself. That's a lie — a very useful, carefully maintained lie told by the operating system and the hardware working together.

Machine Environment Layer

The Problem Virtual Memory Solves

Imagine a system with 8 GB of physical RAM running thirty programs simultaneously. Their combined memory demands exceed what physically exists. How does each program get memory addresses that don't collide with every other program? How does a bug in one program not corrupt another? How can a program use more memory than the machine has RAM?

The answer to all three questions is virtual memory — an abstraction layer between the addresses your program uses and the physical RAM chips on the motherboard.

Every process believes it has its own full address space. The OS and MMU cooperate to map those virtual addresses to different physical locations.

Virtual vs Physical Addresses

Every address in your C program — every pointer, every variable, every function pointer — is a virtual address. It exists in your process's address space, which is a contiguous range of numbers from 0 to some maximum (2⁶⁴ − 1 on 64-bit systems, though only the lower 48 bits are typically used).

A physical address is an actual location in RAM — a real row and column in a real chip. Your program never sees or uses physical addresses. The translation from virtual to physical happens automatically in hardware, every single time your program reads or writes memory.

Virtual-to-physical translation is performed by the MMU on every single memory access. Your program only ever deals in virtual addresses.
Analogy — The Hotel Key Card
Every hotel guest is handed a key that opens "Room 101." But Room 101 on the 3rd floor isn't Room 101 on the 7th floor — they're different physical rooms. The front desk (the OS) issues the keys and maintains the mapping. Each guest thinks they have Room 101 to themselves. Virtual memory works the same way: every process has address 0x400000, but they map to completely different physical locations.

The Three Problems Virtual Memory Solves

1. Isolation

Each process has its own private address space. Process A cannot read or write memory belonging to Process B — those virtual addresses either don't exist in A's mapping or map to different physical locations. This is why a bug in one program can't corrupt another program's data, and why a crash in one process doesn't take down the system.

Two processes can have identical virtual addresses without conflict. Their mappings point to completely different physical pages.

2. More Memory Than You Have

Virtual memory allows programs to use more address space than physical RAM exists. Pages of memory that haven't been accessed recently can be written to disk (the swap space), freeing physical RAM for other uses. When the program accesses them again, the OS loads them back. This is transparent — the program never knows it happened. From its perspective, the memory was always there.

3. Simplified Linking and Loading

Because every process has its own address space starting from the same base addresses, the compiler and linker can produce code with fixed addresses. The program's code always loads at the same virtual address. Without virtual memory, every program would need to be relinked every time it ran to avoid colliding with whatever else happened to be in RAM.

Address Space Size

On a 64-bit system, the virtual address space is enormous: 2⁶⁴ bytes = 16 exabytes. In practice, current processors only use 48 of those bits (256 terabytes) due to hardware limitations, split evenly between user space (low addresses) and kernel space (high addresses). Your programs live in the lower half; the OS kernel lives in the upper half.

The 64-bit virtual address space. User programs occupy low addresses; the kernel occupies high addresses. The canonical hole in the middle is unmapped by hardware.
The Core Idea
Every address in your C program is a virtual address — a number in a private, process-specific address space. The hardware (MMU) translates these to physical RAM locations on every access, using a mapping maintained by the OS. Programs are isolated from each other. The address space can exceed available RAM. This all happens transparently, with no action required from your program.

Common Misconceptions

Misconception 01
"Pointers are RAM addresses"
Pointers hold virtual addresses, not physical ones. The number 0x7ffd2a40 in a pointer tells you where in your process's virtual address space the data lives — not which physical chip or which row of DRAM. The physical location can change (e.g., after swapping) while the virtual address stays constant.
Misconception 02
"malloc gives you raw memory"
malloc carves space out of your process's heap, which is already part of the virtual address space the OS set up when the process started. The OS doesn't see individual malloc calls — it sees larger requests when the heap needs to grow. malloc manages virtual memory your process already owns.
Misconception 03
"Two processes can share memory"
By default, processes are completely isolated — the same virtual address in two processes maps to different physical pages. However, the OS can deliberately map the same physical page into multiple processes' address spaces (shared memory). This is how processes communicate via mmap and shm_open.
Misconception 04
"If malloc succeeds, the memory exists"
Many OSes use optimistic allocation — malloc and mmap can succeed even when no physical page is immediately available. The physical page is only assigned when you actually write to it (demand paging). A program can receive a valid pointer and then crash with OOM much later when it first touches that memory.
