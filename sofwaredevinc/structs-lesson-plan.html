CS253 — Instructor Guide

Lesson Plan: C Structs

Unit: Memory Concepts Series Follows: Pointers, Arrays, malloc/free Duration: 75–80 min (one lecture) Student resource: 06-structs.html

OverviewAt a Glance

Prerequisite Knowledge
Pointers, arrays, malloc/free, stack vs. heap
Central Skill
Grouping data; struct pointer patterns
Follow-on Topics
Linked lists, function design, file I/O records

This lesson introduces structs as the mechanism by which C programs organize related data into named, typed units. Students have already met pointers, arrays, and heap allocation — structs are the natural next move: now they can group the pieces together. The hardest conceptual shift is not the struct syntax itself but the dot-vs-arrow discipline and the consistent habit of passing structs by pointer when mutation is intended.

Students who struggle with this lesson almost always have one of two underlying gaps: they haven't fully internalized pass-by-value (so they don't understand why the pointer pattern is needed), or they're confused about what & does in different contexts. Surface these early.

ObjectivesLearning Objectives

By the end of this lesson, students should be able to:

  • Declare a struct type with struct and typedef, and explain the difference
  • Declare struct variables, initialize them with designated initializers, and access fields with .
  • Explain why struct fields are not necessarily contiguous and why sizeof is required
  • Declare and iterate over arrays of structs, on both the stack and the heap
  • Pass a struct to a function both by value and by pointer, and articulate when each is appropriate
  • Use the -> operator correctly and explain its equivalence to (*p).
  • Write a factory function that returns a struct by value and one that returns a heap-allocated pointer
  • Identify the five most common struct-related bugs by their symptoms

Part 1 · 13 minDeclaration, Memory, and Access

Hook The Problem Structs Solve 0:00–0:07

Open with a concrete, slightly uncomfortable scenario. Ask the class: suppose you need to track 35 students. Each student has a name, an ID, a GPA, and an enrollment status. Write on the board what that looks like without structs:

char  names[35][64];
int   ids[35];
double gpas[35];
int   enrolled[35];

Ask: what happens when you write a function to "work on one student"? You have to pass four separate parameters. What happens when you need to sort by GPA? All four arrays have to move together. The data belongs together but the language has no way to express that — yet.

Then write the struct version. Students will visually see the grouping. That's the entire conceptual case for structs made in two minutes.

Pedagogy

Don't frame structs as "making things easier." Frame them as making the relationships in the problem visible in the code. Students who understand the conceptual point will generalize to every data structure they encounter for the rest of their careers.

Teach Declaration, typedef, and Field Access 0:07–0:14

Show the two syntactic forms side by side — the tagged struct and the typedef form. Explain that the definition is a blueprint: it describes layout but uses no memory. Declaring a variable is when storage is reserved. This maps directly to the blueprint/building analogy in the reference page.

Demonstrate designated initializers (.field = value syntax). Emphasize this is C99 and it should be their default: it's self-documenting, order-independent, and initializes unnamed fields to zero automatically.

Show field access with .. Also show sizeof(Student) and print the result — students are typically surprised it isn't the naive sum of the fields.

Common confusion

Students often think typedef struct {...} Student and struct Student {...} are the same thing. They produce the same layout, but with the tagged form you still need the struct keyword at every use site unless you also add a typedef. Show both; then pick one for the rest of the lesson and stick with it.

Teach Memory Layout and Padding 0:14–0:20

Draw the memory layout diagram on the board (or show the one in the reference page). The key points:

  • Fields are laid out in declaration order.
  • Each type has an alignment requirement — it must start at an address divisible by its own size (double at multiples of 8, int at multiples of 4, etc.).
  • The compiler inserts invisible padding bytes to satisfy those requirements.
  • Therefore sizeof(struct) can be larger than the sum of field sizes.

Demonstrate live: print sizeof and the offsetof of each field. Then reorder the fields (largest first) and show the size shrink. You don't need to teach this as an optimization — the goal is that students understand why they must use sizeof and not hand-calculate struct sizes.

Depth calibration

Don't go deep into alignment rules — this isn't a systems architecture course. The takeaway is one sentence: "Always use sizeof. Never add up field sizes yourself." If a student asks why, the padding explanation is the answer.

Part 2 · 13 minArrays of Structs

Teach Stack Arrays, Heap Arrays, and Iteration 0:20–0:33

An array of structs is conceptually nothing new — it's the same array mechanics they already know, with a struct as the element type. Make this explicit. The elements are contiguous; the stride between them is sizeof(Student); indexing still works identically.

Show the stack version first: Student roster[35]. Then show initialization using a loop with compound literals. Then show the heap version using malloc(n * sizeof(Student)). Emphasize that the usage after malloc is identical — the same dot notation, the same loop — just with a different origin of the pointer.

Live demo

Declare a small array of 4–5 students. Use a range-based for loop to print them. Then do a linear search by ID. Then sort them by GPA using qsort with a comparator. The comparator forces students to cast void* back to Student* — good foreshadowing of the pointer discipline they'll need throughout the course.

The sizeof trap

Students almost always write malloc(n * sizeof(Student *)) instead of malloc(n * sizeof(Student)) — they're thinking "I'm going to use a pointer" and reach for the pointer type. Point this out preemptively: the argument to malloc is the size of what you're storing, not the size of the pointer to it. On a 64-bit system, sizeof(Student*) is always 8 regardless of how large the struct is.

Discussion prompt

"If I have a Student *roster pointing to malloc'd memory, and I do roster[2].gpa = 3.9, how does the CPU know where roster[2] starts in memory? Walk me through the arithmetic." — A good two-minute group question that forces students to connect pointer arithmetic, sizeof, and struct field access into one mental model.

Part 3 · 19 minFunctions: Passing, Modifying, and Creating Structs

This is the heart of the lesson. Expect to spend the most time here and to get the most questions.

Teach Pass by Value — The Copy Problem 0:33–0:39

Write a function that takes a Student by value and tries to modify the GPA. Call it, then print the original. The GPA hasn't changed. Ask students why before explaining. Most will know theoretically ("because it's a copy") but haven't seen it this starkly.

Draw the memory diagram: two separate struct-sized boxes, one in main's frame and one in the function's frame. The function's box is destroyed on return. This is the diagram in the reference page — reproduce it on the board, or project the page.

Pedagogy

Pass-by-value is the reason the pointer pattern exists. Students who understand the reason will remember the pattern; students who just memorize "pass a pointer" will forget it or apply it inconsistently. Take 60 extra seconds to make the causal connection explicit: "The only reason we're about to use a pointer here is that we want to reach across the function boundary into the caller's data."

Teach Pass by Pointer — The Arrow Operator 0:39–0:47

Rewrite the function to take Student *s. Show that the call site needs &student. Show the modification with s->gpa = .... Demonstrate that the original now reflects the change.

Explain the -> operator carefully. It is not new syntax — it is exactly (*s).gpa with the parentheses and dereference collapsed into an arrow. Write both forms on the board and circle the equivalence. Then make the rule concrete:

"Dot when you have the struct. Arrow when you have a pointer to the struct."

Write three one-line examples on the board and have students call out "dot" or "arrow" before you type each one. This takes 90 seconds and is the single most effective exercise for ingraining the rule.

Quick drill

Show these declarations one at a time. Ask the class which operator to use to access gpa:

Student   s;       // → ?
Student  *p;       // → ?
Student **pp;      // → ? (trick: (*pp)->gpa)
Student   arr[10]; // → arr[3].gpa
Student  *heap = malloc(sizeof(Student)); // → ?

The double-pointer row is for the curious students who'll ask — not required curriculum.

Teach Creating Structs Inside Functions 0:47–0:52

Cover the two patterns for factory functions:

  • Return by value. The struct is local to the function; C copies it to the caller on return. Modern compilers optimize this to zero copies (RVO), but the semantics are "caller gets a fresh copy." This is fine for structs of reasonable size.
  • Return a heap pointer. malloc inside the function, return the pointer. The struct outlives the function. The caller owns the memory and must call free.

Write both side by side, then write the one pattern students must never use:

Student *bad(void) {
    Student s = { .id = 1 };
    return &s;   // ← ALWAYS WRONG — stack frame destroyed on return
}
Why this hurts so much

The "return stack address" bug often appears to work in debug builds because the stack memory hasn't been overwritten yet by the time the caller uses it. It silently fails in optimized builds, under heavier load, or in a slightly different call sequence. Students will encounter it, be baffled that it "worked before," and waste hours. Inoculate them now with the explanation: the frame is gone. The pointer is a lie.

Part 4 · 13 minLive Coding: Student Roster

This section ties all the pieces together in a single 50–80 line program written from scratch. Build it incrementally so students can see each concept in context. Keep your editor font large. Narrate everything you type.

Demo Full Roster Program 0:52–1:05

Build order:

  • Declare the Student typedef with char name[64], int id, double gpa, int enrolled.
  • Write make_student() that returns by value. Show the designated initializer and strncpy for the name field.
  • Write print_student() that takes a const Student *. Emphasize const — it says "I'm looking, not touching," and the caller knows nothing will be modified. Use -> to access fields.
  • Write apply_curve(Student *students, int n, double delta). Loop, modify students[i].gpa, clamp to 4.0. Point out: the array-of-structs accessed through the pointer uses . on each element because students[i] is a Student value, not a pointer.
  • In main: allocate a small heap array of 5 students, populate it using make_student() calls, print, curve, print again, free.
Anticipated stumble

When you write students[i].gpa in apply_curve (after receiving a Student *), students who just learned -> will expect you to use it. Stop and explain: students is a pointer, but students[i] is the struct value at that index — the indexing has already done the dereferencing. students[i].gpa and (students + i)->gpa are identical. Write both forms side by side.

Deliberate mistake technique

At some point during the demo, intentionally write the function to take Student s by value, call it, and show the output didn't change. Ask the class what's wrong. Let them diagnose it before you fix it. This embeds the "I need a pointer" decision as a debugging reflex rather than a rule to memorize.

Part 5 · 10 minCommon Pitfalls & Wrap-up

Synthesis The Bugs They Will Definitely Write 1:05–1:13

Close with a rapid-fire bugs walkthrough. Don't spend more than 90 seconds per row. The goal is recognition, not deep diagnosis. Students should leave able to say "oh, that's the copy problem" or "oh, that's the stack pointer bug."

Symptom the student sees The actual cause One-line fix
Compiler error: "request for member in something not a structure" Used . on a pointer to a struct instead of -> Change s.field to s->field
Function ran but the struct wasn't modified Passed struct by value; function modified the copy Change param to Student *s, pass &var
Works in debug, crashes or gives garbage in release Returned & of a local struct variable Return by value, or malloc and return the pointer
Segfault immediately after malloc malloc returned NULL (or wrote sizeof(Student*)); no NULL check Check if (!p) return; and verify sizeof argument
One struct's string field randomly changes Struct with char * field was copied; both point to same buffer Use char name[64] inside the struct, or implement a deep copy
Fields contain garbage on first use Struct declared but not initialized; stack contents are arbitrary Use designated initializer or memset(&s, 0, sizeof(s))
Close Summary and Bridge to Next Topic 1:13–1:15

Close with the three decisions a programmer makes every time they write a struct function:

  • Does this function need to modify the struct? → Pass by pointer. Otherwise, by value is fine for small structs.
  • Does the struct need to outlive this function? → Put it on the heap, return the pointer. Otherwise, return by value.
  • Am I holding a struct or a pointer to one? → Dot or arrow. Read your declaration if you're unsure.

Bridge forward: structs become far more powerful when combined with pointers to other structs. The Node pattern — a struct with a next pointer to the same type — is how every linked list, tree, and graph in C is built. That's the next step.

AnticipateFrequently Asked Student Questions

What's the actual difference between struct Student and the typedef version? When should I use which?
They produce identical memory layout. The typedef is stylistic convenience — you skip the struct keyword at every use site. In your own code, typedef is usually cleaner. In system headers (POSIX, kernel code) you'll see the tagged form everywhere because it's more explicit about what type it is. Either is correct; the important habit is consistency within a codebase.
Why does students[i].gpa use dot when students is a pointer?
Because students[i] is the element at position i — a Student value, not a pointer. The pointer arithmetic is already done by the indexing. students[i].gpa and (students + i)->gpa compile to identical machine code. The rule is about what you have at the moment of access: if the expression to the left of the dot/arrow is a struct value, use dot; if it's a pointer, use arrow.
Can I compare two structs with ==?
No. C does not define == for struct types. You must compare field-by-field, or use memcmp if (and only if) there is guaranteed to be no padding and no pointer fields. In practice, write an explicit comparison function. This is a common surprise for students coming from Python or Java.
Can a struct contain a pointer to another struct of the same type?
Yes — and this is exactly how linked lists work. You use the tagged form: struct Node { int val; struct Node *next; }. You cannot use the typedef name inside the definition because the typedef isn't yet complete, but the tag is available immediately. This is one of the rare cases where the tagged form is mandatory.
If structs are copied when passed by value, should I always pass pointers for efficiency?
For the sizes we deal with in this course — a few hundred bytes at most — the copy is nearly free on modern hardware. Correctness matters far more than micro-optimization. Use by-value when the function should not modify the caller's data and the struct is small. Use pointers when modification is intended. Never pass a pointer to avoid a copy unless profiling has shown the copy is actually a bottleneck.

LabsLab Ideas & Exercises

Student Records CRUD
Implement make_student, print_student, find_by_id, and update_gpa for a heap-allocated array. Forces all four function patterns and the dot/arrow distinction within a single coherent program.
Struct Sorting with qsort
Given an array of structs, sort by different fields (GPA ascending, name alphabetical) by writing comparator functions. Reinforces pointer-to-void casting and the array-of-structs mental model.
sizeof Detective
Given 4–5 struct definitions with varying field orderings, predict their sizeof before running. Print offsetof for each field. Students learn padding intuitively and develop the "never hand-calculate struct sizes" habit.
Bug Hunt
Provide a 40-line program with 4 seeded struct bugs (wrong operator, copy instead of pointer, stack pointer return, uninitialized field). Students must find and fix all four. Effective consolidation exercise.
