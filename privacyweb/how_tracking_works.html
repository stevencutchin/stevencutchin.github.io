Understanding the Mechanics

When you visit a website, your browser and the website's server engage in a complex dance of data exchange. This guide explains exactly how various tracking mechanisms are created, stored, and transmitted between your computer and remote servers.

Each tracking method has a specific technical implementation. Some live entirely on your computer (client-side), some are controlled by the server, and some involve both. Understanding where and how these mechanisms operate is crucial to understanding privacy implications.

HTTP Cookies

Small text files stored by your browser

Client & Server Server-Initiated

How Cookies Are Created and Transmitted:

1
Your Browser Requests a Web Page

When you type a URL and press Enter, your browser sends an HTTP request to the website's server saying "I'd like to see this page, please."

2
Server Sends Response WITH Instructions

The server sends back the web page, but also includes a special header called "Set-Cookie" with instructions like: "Store this piece of data: user_id=abc123, and send it back to me every time you visit."

HTTP Response: Set-Cookie: user_id=abc123; Expires=Wed, 12-Nov-2025; Domain=example.com; Path=/
3
Browser Stores Cookie on Your Computer

Your browser obediently creates a small text file in a special cookies folder on your hard drive. This file contains: the cookie name, value, expiration date, and which website owns it.

4
Browser Automatically Sends Cookie Back

Every subsequent request your browser makes to that website includes the cookie in the HTTP headers. The server now "remembers" you because it sees its own ID coming back.

HTTP Request: Cookie: user_id=abc123
üé´ Real-World Analogy:

Imagine visiting an amusement park. At the entrance, they stamp your hand with invisible ink (the cookie). Every time you want to enter a new ride, staff shine a UV light on your hand to verify you paid admission. The stamp lives on your hand (your computer), but the park gave it to you and checks for it.

Key Technical Points:
  • Cookies are plain text files‚Äîyou can actually open and read them
  • The server CANNOT access cookies it didn't create (domain restrictions)
  • Third-party cookies are created when your page loads content from other domains
  • JavaScript can also create cookies directly on your computer (document.cookie)
Why This Matters for Privacy: Because cookies are automatically sent with EVERY request to that domain, the server can track every page you visit, every action you take, and build a complete profile of your behavior‚Äîall without you clicking anything or taking any action.

Canvas Fingerprinting

Creating unique device signatures using graphics rendering

Client-Side Only Results Sent to Server

How Canvas Fingerprinting Works:

1
Website Loads JavaScript Code

When you visit the page, JavaScript code downloads and runs in your browser. This is completely invisible to you‚Äîno windows pop up, nothing changes on screen.

2
Script Creates Invisible Canvas

The JavaScript creates an invisible HTML canvas element (like a digital drawing board) that's never shown to you. Think of it as drawing on a piece of paper that's inside a closed box.

// JavaScript running in your browser var canvas = document.createElement('canvas'); var ctx = canvas.getContext('2d');
3
Script Draws Complex Graphics

The code draws text, shapes, and emojis on this invisible canvas using various fonts and colors. It might write "Hello, World!" in multiple fonts, draw geometric shapes with gradients, and render emoji characters.

ctx.textBaseline = "top"; ctx.font = "14px 'Arial'"; ctx.fillText("Hello üòä üåç!", 2, 2);
4
Script Reads Pixel Data

Here's the clever part: The script reads back the exact pixel values of what it just drew. It gets a long string of numbers representing every pixel's color.

var imageData = canvas.toDataURL();
5
Creates Unique "Fingerprint" Hash

The script runs these pixel values through a mathematical function (hash) that converts them into a unique identifier. This becomes your device's fingerprint.

var fingerprint = hash(imageData); // Result: "a8f5c9d2e7b1f4a3"
6
Fingerprint Sent to Server

The JavaScript sends this fingerprint to the tracking server. The server stores it and uses it to recognize you on future visits‚Äîno cookies needed.

Why This Creates a Unique Identifier:

Different devices render graphics slightly differently based on: graphics card model, driver version, installed fonts, operating system, screen resolution, and color calibration. These tiny variations create a unique pattern‚Äîlike a fingerprint‚Äîthat's remarkably stable and hard to change.

‚úçÔ∏è Real-World Analogy:

Imagine asking 100 people to write "Hello World" in cursive. Everyone writes the same words, but each person's handwriting is slightly unique‚Äîspacing, slant, pressure, loops. Even if they try to disguise it, handwriting experts can still identify them. Canvas fingerprinting works the same way with your device's "handwriting" of graphics rendering.

Why This Matters for Privacy: This tracking method requires NO cookies, NO login, and persists even if you clear your browser data. It's nearly impossible for regular users to prevent because it exploits fundamental differences in how hardware and software render graphics. You can't easily change your graphics card or installed fonts without breaking other software.

Session Recording

Capturing and transmitting user interactions in real-time

Monitored Client-Side Streamed to Server

How Session Recording Captures Your Actions:

1
Recording Script Loads

When the page loads, a JavaScript library (like FullStory, Hotjar, or LogRocket) downloads and initializes. This script is typically 50-200KB in size.

2
Script Attaches "Event Listeners"

The JavaScript registers event listeners for EVERY possible user action. Think of these as invisible tripwires that trigger whenever you do anything:

// Event listeners attached to document document.addEventListener('mousemove', recordMousePosition); document.addEventListener('click', recordClick); document.addEventListener('scroll', recordScroll); document.addEventListener('keydown', recordKeypress); document.addEventListener('change', recordFormInput);
3
Every Action Triggers Data Capture

When you move your mouse, the script records: X and Y coordinates, timestamp. When you click: element clicked, timestamp, page location. When you type: which field, which keys (sometimes masked, sometimes not), timing between keystrokes.

4
Data Buffered in Browser Memory

Events are stored temporarily in your browser's RAM (not written to disk). The script builds a timeline of everything you did: [{time: 0.5s, action: "mouse_move", x: 234, y: 567}, {time: 1.2s, action: "click", element: "button#submit"}]

5
Periodic Transmission to Server

Every few seconds (typically 5-10 seconds), the script bundles all captured events and sends them to the recording company's servers via HTTP POST requests. This happens in the background‚Äîyou see nothing.

// Data sent to server POST https://collect.hotjar.com/events { "session_id": "xyz789", "events": [...1000+ captured actions...] }
6
Server Reconstructs Your Session

The recording service's servers reassemble all these events into a video-like playback. Company employees can literally watch your mouse move around their website like a screen recording.

What Gets Captured:
  • Mouse movements: Every pixel your cursor travels across
  • Clicks: What you clicked, where, and when
  • Scrolling: How far you scrolled, how fast, where you paused
  • Keystrokes: Sometimes character-by-character (including passwords if not properly configured)
  • Form inputs: Everything you type into forms, even if you delete it
  • Page navigation: Every page you visit in sequence
  • Time on page: Exactly how long you viewed each section
üé• Real-World Analogy:

Imagine someone following you around a store with a camera, recording every item you pick up, every aisle you walk down, how long you read each label, what you put back, and what you ultimately buy. They even record when you picked up your phone to check prices elsewhere. That's essentially what session recording does‚Äîexcept it's invisible and follows you across every page.

Why This Matters for Privacy: Session recording captures things you explicitly CHOOSE NOT to submit. If you type a message and delete it, they still captured it. If you hover over a "premium" option but choose "basic," they know you considered upgrading. This level of monitoring would be unacceptable in physical spaces, yet happens routinely online.

Facebook Pixel Events

Sending your activity directly to Facebook's servers

Triggered Client-Side Data to Facebook

How Facebook Tracks You Across the Web:

1
Website Embeds Facebook Pixel Code

The website owner adds Facebook's tracking JavaScript to their site. This is a small snippet that looks innocent but connects every visitor to Facebook's tracking infrastructure.

<script> !function(f,b,e,v,n,t,s) {/* Facebook Pixel Code */} fbq('init', '1234567890'); fbq('track', 'PageView'); </script>
2
Pixel Loads from Facebook's Servers

Your browser downloads the actual tracking library from connect.facebook.net. This is Facebook's code running on your computer, not the website's code.

3
Automatic PageView Event

The moment the page loads, Facebook's code automatically fires a "PageView" event. This sends to Facebook: The URL you're visiting, Your IP address, Your browser fingerprint, Timestamp, Any Facebook cookies you have (linking to your profile).

4
Website Triggers Custom Events

The website owner's code triggers specific events for actions they want Facebook to know about. Each event sends detailed information:

// When you view a product fbq('track', 'ViewContent', { content_name: 'Red Shoes', content_category: 'Footwear', value: 99.99, currency: 'USD' }); // When you add to cart fbq('track', 'AddToCart', { content_ids: ['SKU123'], value: 99.99 });
5
Data Sent to Facebook's Servers

Each event triggers an HTTP request to Facebook's tracking servers. The request includes all event data PLUS Facebook's tracking cookies if you've ever logged into Facebook, Instagram, or visited any site with Facebook integration.

POST https://www.facebook.com/tr/ { "event": "ViewContent", "fbp": "fb.1.1234567890.987654321", // Your tracking ID "data": {...product details...} }
6
Facebook Links Data to Your Profile

If Facebook can identify you (through cookies, login state, or device fingerprinting), they add this behavior to your advertising profile. Now Facebook knows you're interested in red shoes, even though you're not on Facebook.

üëÇ Real-World Analogy:

Imagine if every store you entered had a Facebook employee following you around with a clipboard, writing down everything you looked at, touched, or considered buying‚Äîthen immediately calling Facebook headquarters to report it. Later, when you're at home watching TV, Facebook uses this information to show you commercials for those exact products. That's what the Pixel does, but most people don't realize the Facebook employee is there.

Why This Matters for Privacy: You might think you're just browsing a shoe store's website, but Facebook is secretly watching and recording your every action. This data feeds into Facebook's advertising profile of you‚Äîwhich is sold to thousands of advertisers. You're being tracked across the web by Facebook even when you're not using Facebook.

IP Address & Browser Information

Automatic data transmission with every web request

Automatic Transmission Captured Server-Side

How Servers Automatically Learn About You:

1
Every Request Includes Technical Metadata

This is unavoidable: For the internet to work, every HTTP request MUST include certain information. Your browser automatically sends this data with every page load, image request, or API call.

2
IP Address (Your Internet "Home Address")

Just like mail needs a return address, internet packets need to know where to send responses. Your IP address is automatically included in every request. The server logs show: Your approximate location (city/region), Your internet service provider, Often enough to identify your household.

IP Address: 203.0.113.42 Location: Los Angeles, California ISP: AT&T Internet Services
3
User-Agent String (Browser Fingerprint)

Your browser announces itself with a "User-Agent" string that includes: Browser brand and version, Operating system and version, Device type, Rendering engine version.

User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Safari/537.36

This seemingly technical string actually reveals: You're using Chrome 120 on Windows 10, 64-bit processor, That you're probably on a desktop or laptop.

4
HTTP Headers (Additional Automatic Data)

Every request includes many headers that browsers send automatically:

Accept-Language: en-US,en;q=0.9 Referer: https://google.com/search?q=red+shoes Accept-Encoding: gzip, deflate, br DNT: 1 (Do Not Track - often ignored)

These headers reveal: Your language preference, Where you came from (referrer), That you probably speak English, What compression your browser supports.

5
Server Logs Everything

Web servers automatically write this information to log files:

203.0.113.42 - - [12/Nov/2025:10:15:30] "GET /products/shoes HTTP/1.1" 200 4523 "https://google.com/search?q=red+shoes" "Mozilla/5.0 (Windows NT 10.0; Win64; x64)..."

This single log line captures: Your IP address, Exact date and time of visit, What page you requested, That you came from a Google search for "red shoes", Your browser details.

6
JavaScript Can Access More Details

Beyond HTTP headers, JavaScript running in your browser can query for additional system information:

// JavaScript that runs in your browser navigator.platform // "Win32" navigator.hardwareConcurrency // 8 (CPU cores) screen.width √ó screen.height // 1920√ó1080 navigator.languages // ["en-US", "en", "es"] Intl.DateTimeFormat().resolvedOptions().timeZone // "America/Los_Angeles"
The Unavoidable Nature of This Data:

Unlike cookies or JavaScript tracking, you CANNOT prevent your IP address and basic browser information from being transmitted. It's fundamental to how the internet works. The only way to hide this is through technical tools like VPNs (which give you a different IP address) or privacy-focused browsers (which normalize User-Agent strings).

üì¨ Real-World Analogy:

When you send a letter, the envelope must have your return address‚Äîotherwise the post office can't send replies. Similarly, when you visit a website, your "return address" (IP) must be included. However, in the physical world, we have privacy protections: mail carriers can't open your letters or create profiles of your correspondence. Online, there are fewer protections‚Äîservers can log, analyze, and indefinitely store every "envelope" you send.

Why This Matters for Privacy: This "free" data is combined with other tracking methods to build comprehensive profiles. Your IP address + browser details + timestamps + browsing patterns = a fairly unique fingerprint, even without cookies. Law enforcement and civil litigants can often subpoena ISPs to connect IP addresses to specific households.

Local Storage & Session Storage

JavaScript-controlled databases in your browser

Stored Client-Side JavaScript Controlled

How Websites Store Data Directly in Your Browser:

1
JavaScript Gets Storage Permission

Unlike cookies (which require server instructions), JavaScript running in your browser can directly write to browser storage. No permission prompt appears‚Äîit just works automatically.

2
Data Written to Browser's Database

The script stores key-value pairs, like a simple database on your computer:

// JavaScript code running on the page localStorage.setItem('user_id', 'abc123'); localStorage.setItem('preferences', JSON.stringify({ theme: 'dark', language: 'en', last_search: 'red shoes' })); sessionStorage.setItem('shopping_session', 'xyz789');
3
Data Persists on Your Hard Drive

Local Storage: Stays forever (until manually deleted or code removes it). Session Storage: Deleted when you close the browser tab. Both are stored in your browser's profile folder as SQLite database files.

4
JavaScript Reads Data on Future Visits

When you return to the website, JavaScript immediately reads the stored data:

var userId = localStorage.getItem('user_id'); var prefs = JSON.parse(localStorage.getItem('preferences'));

The website now "remembers" you without needing to ask the server for anything.

5
Data Optionally Sent to Server

The stored data lives on your computer, but JavaScript can send it to servers whenever it wants:

fetch('https://analytics.example.com/track', { method: 'POST', body: JSON.stringify({ user_id: localStorage.getItem('user_id'), session: sessionStorage.getItem('shopping_session'), stored_searches: localStorage.getItem('search_history') }) });
Key Differences from Cookies:
  • Size: Can store ~5-10MB per domain (vs. 4KB for cookies)
  • Automatic transmission: NOT sent automatically with HTTP requests (unlike cookies)
  • Control: Entirely controlled by JavaScript (no server involvement needed)
  • Visibility: Not visible in HTTP headers (harder to monitor with network tools)
  • Lifespan: Local Storage persists indefinitely; Session Storage lasts until tab closes
üìù Real-World Analogy:

Imagine a store clerk giving you a notepad and saying "Please write down your name, shopping history, and preferences on this notepad. Keep it in your pocket. Every time you come back to our store, we'll ask you to show us your notepad so we can 'remember' you." The notepad lives with you (local storage), but the store reads it whenever you visit. Unlike store loyalty cards (cookies), which the store keeps, this notepad is entirely in your possession‚Äîbut that doesn't mean you have privacy.

Why This Matters for Privacy: Because Local Storage isn't automatically sent with requests, many privacy tools and cookie blockers miss it entirely. Websites increasingly use it to circumvent cookie restrictions. The large storage capacity (5-10MB) means websites can store detailed behavioral profiles locally, then transmit them whenever they want. Users often don't realize this "database" exists on their computer.
