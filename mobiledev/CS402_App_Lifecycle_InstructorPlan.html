The CRUD Model — Introducing the Framework
Concept framing · State machine diagram · Two-layer model
0:00 – 0:10 · 10 min

Opening

  • Ask: "What does CRUD stand for in databases?" Students know this. "The app lifecycle has its own version. It's not identical, but the structure rhymes — and once you see it, the whole thing snaps into place."
  • Write on the board: C — Create · R — Render · U — Undisplay · D — Delete. Tell students to use this as their mental anchor for the entire session.
  • Emphasize the two levels immediately: "This model applies to a single React component AND to the entire app at the OS level. Both at the same time, in parallel."
🗣 Activation question: "Think of an app on your phone — say, Spotify. When you press the home button and come back ten minutes later, what do you expect? Your place in the song? The same playlist? What if it was gone?" → This frames why lifecycle management matters. Students understand the user expectation immediately.

Walk through the state machine diagram

  • Open the student document and display the two-layer state machine SVG on the projector. Walk each phase left to right.
  • Point out the curved back arrows — Undisplay is reversible. The app comes back. The component gets re-focused. This is critical: Undisplay is not a death sentence.
  • Point out the dashed vertical connectors — the React layer and the OS layer move in parallel. When the OS enters background, the components are still mounted. When the OS kills the process, everything vanishes.
📝 CRUD as scaffold: The CRUD framing isn't in the official React docs — it's a teaching model for this course. Students should understand it as an analogy to build intuition, not as a term they'll find in job interviews. The actual API terms (mount, unmount, effect, cleanup, AppState) are what they'll encounter in the wild. Name this explicitly so students don't get confused if they Google it later.
2
C — Create
Mount order · useState initialization · useEffect(fn, []) · First paint
0:10 – 0:22 · 12 min

The three-step mount sequence

  • Step 1: Component function runs. useState slots are created with initial values. No UI yet.
  • Step 2: JSX is evaluated and committed. The user sees the first paint — often a loading spinner or skeleton.
  • Step 3: After paint, useEffect(fn, []) fires. Data fetching starts here — never before the first paint.
  • Ask: "Why does React commit the UI before running effects?" → So the screen is never blank while a fetch is in-flight. The user sees something immediately.
🖥 Demo 1 (6 min) — See the Create sequence: Open a Snack. Add a console.log('function ran') at the top of the component and a console.log('effect ran') inside useEffect(fn, []). Show the console output order: function → function (StrictMode double-invokes) → effect. Then disable StrictMode to show the clean sequence. Students are often surprised that the function runs before the effect and before the user sees anything.
StrictMode note: In Expo's development mode, React runs component functions twice intentionally (StrictMode) to surface side effects in the function body. This doubles console logs and makes the Create sequence look weird. It only happens in development — production runs once. Worth mentioning so students aren't alarmed.
✅ Quick Check
  1. You have useEffect(fn, []). The function inside fetches data and calls setData(result). How many times does the fetch run?
    → Once — the empty array means "Create phase only." The fetch fires after the first paint, sets state, triggering one re-render into the Render phase.
3
R — Render
Active phase · Re-render triggers · AppState 'active' · Render ≠ visible
0:22 – 0:32 · 10 min

The Render phase is normal operation

  • The Render phase is where the component spends most of its life. State updates trigger re-renders; each re-render runs the component function again.
  • useEffect(fn, [dep]) is the Render-phase hook — it fires on Create AND whenever a listed dependency changes.
  • At the OS level, this corresponds to AppState === 'active': screen on, app visible, user interacting.
🚨 Address "Render ≠ visible" explicitly: This is the insight most students miss before it bites them in an assignment. A component in a tab they're not looking at is still mounted, still in the Render phase, and its effects are still running. If that component has a polling interval, it's polling even when the user is on a different tab. This is why useFocusEffect exists — and why we cover it in the Undisplay segment.
🗣 Discussion: "I have a stock ticker that polls every 5 seconds. My app has three tabs. The ticker is on Tab 1. The user switches to Tab 3. Is the ticker still polling?" → Yes, if you used useEffect. The component is mounted. The interval is registered. Tab visibility is not the same as component mount state. How would you fix it? → useFocusEffect.
4
U — Undisplay
AppState API · Background behavior · Save now · Three "not visible" scenarios · useFocusEffect
0:32 – 0:47 · 15 min

Why Undisplay is the most important phase to get right

  • The user presses home. Your app is still mounted in memory. State is intact. But the OS is now managing your CPU and power allocations aggressively.
  • This is the moment to save. Not when the app is killed — that happens without warning. Now, while you still have execution time.
  • It's also the moment to release expensive resources: GPS, camera, microphone, frequent network polling. The user isn't looking at your app; there's no reason to keep burning battery on their behalf.
🖥 Demo 2 (7 min) — AppState listener live: Add the useAppStateChange hook from the student document to a Snack. Add console.log calls in onBackground and onForeground. Run on a physical device (or ask a student with their phone on the Snack URL). Press the home button — show "background" log. Return to app — show "active" log. This is viscerally satisfying; students immediately see the hook firing in real time.

The three "not visible" scenarios — nail this distinction

Draw this on the board or point to the table in the student document:

  • Navigation blur: User went to another screen inside your app. Component still mounted. AppState still 'active'. Effects still running. Most common source of "ghost" intervals.
  • AppState background: User pressed home or switched apps. Component still mounted. AppState is now 'background'. This is the save-your-data moment.
  • Component unmount: Component was removed from the navigation stack or conditionally rendered away. Cleanup runs. State gone. This is the Delete phase.
📝 useFocusEffect timing: Walk through the useFocusEffect example in the student document. Emphasize that it takes a useCallback wrapper — without it, the effect re-registers on every render. Students commonly forget the useCallback and then wonder why the effect fires too often.
✅ Check for Understanding
  1. The user presses the home button on an iPhone. What is AppState.currentState now?
    → On iOS, it briefly becomes 'inactive' during the transition animation, then 'background'. On Android, it goes directly to 'background'. The correct check compares previous state to current state, not just current state.
  2. You have a setInterval in a useEffect(fn, []). The user navigates to a different tab. Is the interval still running?
    → Yes. Navigation blur does not unmount the component (in a tab navigator). The effect is still registered. Use useFocusEffect with a cleanup to pause it when the screen blurs.
5
D — Delete
Cleanup functions · OS kill without warning · What causes unmount
0:47 – 0:57 · 10 min

The cleanup function is the Delete hook

  • The function returned from useEffect runs on two occasions: before the next effect (when deps change) and on component unmount. The unmount run is the Delete phase.
  • Think of it as a contract: "If I started something, I stop it here." Sensor subscriptions, intervals, event listeners, network connections — all have a counterpart teardown in the cleanup.
  • Missing cleanups are the root cause of the most common React Native production bugs: ghost timers, memory leaks, and the "can't perform state update on unmounted component" warning.
🚨 The OS kill has no callback: Emphasize this strongly. There is no AppState event when the OS decides to kill your app. No "last chance to save." The process simply ends. This is the single most important reason to save data in the Undisplay phase — you cannot rely on getting a signal at Delete.

A helpful analogy: imagine your computer's power cord being yanked out. You don't get a "shutting down" dialog. You get nothing. That's what OS kill looks like for your app.

What triggers Delete at the component level

  • The user presses the back button and the screen is popped from the stack
  • A conditional renders null where the component used to be
  • A FlatList scrolls and a virtualized item leaves the visible window
  • The root component unmounts because the process ends
✅ Quick Check
  1. A student says "I'll save the user's draft when the app is killed." What's wrong with this plan?
    → There is no AppState event for OS kill. The process ends without notice. Drafts must be saved during the Undisplay phase (AppState → 'background'), not at Delete.
  2. Name two things that should always be in a useEffect cleanup function.
    → Any of: interval/timeout clearance, subscription removal, event listener removal, connection close, ref flag (isMounted = false).
6
Synthesis — LocationTracker Annotation Activity
Full component walkthrough · Pairs activity · Final CFU
0:57 – 1:15 · 18 min

Instructor-led walkthrough (8 min)

Paste the LocationTracker component from the student document into Snack or display it on screen. Walk through it section by section, asking students to name the phase before you label it.

  • Point at const [coords, setCoords] = useState(null) — "What phase is this?" → Create. State initialization always happens in Create.
  • Point at useEffect(() => { startTracking(); return () => sub.remove(); }, []) — "What phase does the body belong to? What phase does the return belong to?" → Body: Create. Return: Delete.
  • Point at the AppState listener — "The onBackground branch — what phase?" → Undisplay. "The onForeground branch?" → Render resuming.
  • Point at the JSX return — "What phase?" → Render. Always. Every paint is Render.
🏋 Pairs Activity (8 min) — Lifecycle Annotation

Share the following stripped component via Snack or a code paste tool. Each pair annotates every line with its lifecycle phase (C / R / U / D). The goal is to label every useState, every useEffect body, every cleanup, and the JSX. 

function ChatRoom({ roomId }) {
  const [messages, setMessages] = useState([]);
  const [draft, setDraft]    = useState('');

  useEffect(() => {
    const sub = subscribeToRoom(roomId, msg =>
      setMessages(prev => [...prev, msg])
    );
    return () => sub.unsubscribe();
  }, [roomId]);

  useEffect(() => {
    const appSub = AppState.addEventListener('change', state => {
      if (state !== 'active') saveDraft(draft);
    });
    return () => appSub.remove();
  }, [draft]);

  return <MessageList messages={messages} />;
}

Expected annotations: useState calls → C. First useEffect body → C (mount) and re-runs on R when roomId changes. First cleanup → D. Second useEffect body → C (mount). Second useEffect listener body → U (fires when leaving active). Second cleanup → D. JSX return → R.

✅ Final Check for Understanding
  1. In the ChatRoom component above, what is the bug if draft is missing from the second effect's dependency array?
    → The AppState listener captures a stale closure of draft. When the app goes to background, saveDraft() receives the value of draft from when the effect last ran — possibly an empty string or outdated text, not what the user typed most recently.
  2. The user is on the ChatRoom screen and presses the home button. Which lifecycle transitions fire, in order?
    → (1) AppState changes to 'inactive' (iOS) then 'background'. (2) The AppState listener fires → saveDraft() is called. (3) Component remains mounted (Undisplay). No cleanup runs. No unmount.
  3. The user now presses Back and the screen is removed from the navigation stack. What happens next?
    → Component unmounts (Delete phase). Both useEffect cleanup functions run: sub.unsubscribe() and appSub.remove(). State is gone.

Common Student Mistakes

MistakeWhat goes wrongFix
No cleanup for subscriptions Listener accumulates on every render. Ghost callbacks fire on unmounted component. "Can't perform state update on unmounted component" warning. Always return a cleanup from effects that subscribe, start timers, or add event listeners.
Saving data "on app kill" OS kill has no callback. Data is lost. The plan can never work. Save during Undisplay (AppState → 'background'), not at Delete.
Assuming navigation blur = unmount Interval or subscription keeps running on a "hidden" screen. Stale data, ghost updates, battery drain. Use useFocusEffect for effects that should pause when the screen blurs.
Missing useCallback in useFocusEffect Effect re-registers on every render, not just on focus. Causes rapid subscription cycling. Always wrap the useFocusEffect callback in useCallback.
Treating 'inactive' as 'background' on iOS Save logic fires during transient system overlays (calls, swipe gesture). Can cause duplicate saves or incorrect state resets. Compare previous state to current state. Trigger on the transition from 'active', not on seeing any non-'active' value.
Putting side effects in the function body (not in useEffect) Side effect runs on every single render, not just on Create. Can cause infinite loops if the effect updates state. All side effects belong in useEffect. The component function body should only compute and return JSX.

Grading & Assessment Notes

When lifecycle management appears in assignments, focus on whether the student understands the intent of each phase. Credit generously for correct structure; deduct for systematic absence of cleanup or data loss patterns that would materially affect the user.

  • Full credit: Correct phase structure — data fetched on mount, effects cleaned up, AppState handled if the assignment involves background behavior.
  • Partial credit (comment, not deduction): A single missing cleanup in a component that otherwise demonstrates phase understanding. Note it in feedback.
  • Significant deduction: No cleanups anywhere (systematic), or a component that causes the "unmounted component" warning to fire repeatedly, breaking the required behavior.
  • Do not penalize: Imperfect handling of the 'inactive' state on iOS, or using useEffect instead of useFocusEffect when the assignment doesn't explicitly require focus-aware behavior.
📝 The CRUD model in feedback: When leaving written feedback on assignments, using the phase language ("your Delete phase cleanup is missing the interval teardown") gives students a vocabulary to act on. It's more actionable than "you have a memory leak" alone.
