Building a Camera App in React Native

View, Capture, Browse, and Share — from Scratch
📷 Project 4 Companion ⏱ Lecture + Lab 🔗 Prerequisite: useEffect, useState, Expo permissions

What We Are Building

A single-screen React Native app that opens the camera, lets the user snap photos, and displays them in a browsable list. The whole thing lives in one file — CameraApp.js — plus two small sibling components for the list views. By the end of this walkthrough you will understand every line.

The app does exactly four things, no more:

1
Asks for permission. The OS will not open the camera without an explicit user grant. We request it on mount, once.
2
Shows a live viewfinder. Expo's CameraView renders a real-time camera preview inside your layout, like a native camera app.
3
Captures photos. Tapping Snap saves a JPEG to the device and adds it to an in-memory list. The list grows with each shot.
4
Switches list views. A Switch button toggles between a preview strip (PreviewList) and a thumbnail grid (ListGrid).

The App at a Glance

Three visual layers stacked inside a single orientation-aware container. The viewfinder, button row, and photo list are each independent — swappable without touching the others.

Chapter 1 — What the App Does

1
Overview

CameraApp is a single functional component — no class syntax, no Redux store, no navigation stack. Everything it needs to know lives in five local state variables. The JSX it returns changes based on those variables, and the buttons update those variables. That's the whole model.

Reading the file top-to-bottom tells the story in the right order: what we import, what we remember, how we ask permission, how we take a photo, and finally what the screen looks like. Each section is short enough to hold in your head while reading the next.

Chapter 2 — Imports

2
Imports

Good imports tell you at a glance what a file depends on. Here's the complete import block, and the reason each item is present: 

import React, { useState, useEffect, useRef } from 'react';
import {
  StyleSheet, Text, View,
  TouchableOpacity,
  useWindowDimensions,
} from 'react-native';
import { CameraView, Camera } from 'expo-camera';
import PreviewList from './PreviewList';
import ListGrid    from './ListGrid';
ImportWhat it provides
useStateThe five pieces of memory we carry across re-renders (permission, camera type, current photo, photo list, view mode).
useEffectThe "run once on mount" hook where we request camera permission.
useRefA stable handle to the CameraView instance so we can call takePictureAsync imperatively.
useWindowDimensionsLive width and height — used to detect portrait vs landscape so the layout adapts on rotation.
CameraViewThe Expo widget that renders a live camera preview on screen. (Renamed from Camera in SDK 50.)
CameraThe static API object — we only call Camera.requestCameraPermissionsAsync() from it.
PreviewList, ListGridSibling components for the two browsing modes. We pass them the photo list and the current photo URI.
⚠ SDK 50 rename: Many tutorials and Stack Overflow answers still use <Camera> as the viewfinder component. That was renamed to <CameraView> in Expo SDK 50. If you see <Camera ref={...}> in an example, treat it as out of date. The static API object (Camera.requestCameraPermissionsAsync) kept the same name.

Chapter 3 — State: What the App Remembers

3
State

A functional component re-runs from top to bottom every time the UI updates. useState is the mechanism that lets values survive those re-runs. Think of each state variable as a named slot in memory that persists between renders. 

const [hasPermission, setHasPermission] = useState(null);
const [type,          setType         ] = useState('front');
const [aphoto,        setPhoto        ] = useState('assets/snack-icon.png');
const [photolist,     setPhotoList    ] = useState([]);
const [viewmode,      setVMode        ] = useState(false);

Five state slots. Each one answers a single question the component needs to make a UI decision. hasPermission has three meaningful values (null / true / false) — not two — because "not asked" and "asked and denied" are different situations.

Chapter 4 — The Camera Ref

4
useRef

There is a fundamental difference between data that drives the UI and a command you send to a component instance. Taking a photo is a command — "fire the shutter right now" — not a state change. React's tool for commands is useRef. 

const cameraRef = useRef(null);

When we write ref={cameraRef} on the CameraView element, React wires cameraRef.current to the live camera widget as soon as it mounts. From then on, inside snap() we can call cameraRef.current.takePictureAsync() directly — no state involved, no re-render triggered, no delay.

useState — for data that drives rendering

Changing state schedules a re-render. Use it when the UI needs to look different as a result of the change. Example: which camera is active (type), or the list of photos (photolist).

useRef — for imperative commands

Changing a ref does NOT trigger a re-render. Use it when you need to call a method on a component instance directly. Example: cameraRef.current.takePictureAsync().

⚠ The class component mistake: An older pattern creates camera refs with this.SnapCamera = React.createRef() inside a class component. In a functional component there is no this, so that pattern breaks silently — the ref appears to work but is recreated on every render and never actually connects to the hardware. Always use useRef(null) in functional components.

Chapter 5 — Asking for Permission

5
Permissions

The OS will not allow any app to access the camera until the user explicitly grants permission in a system dialog. We must ask before doing anything else — and we only need to ask once. useEffect with an empty dependency array is the correct place. 

useEffect(() => {
  (async () => {
    const { status } = await Camera.requestCameraPermissionsAsync();
    setHasPermission(status === 'granted');
  })();
}, []);

Permission has three states, not two. null means "we haven't asked yet." The UI shows a loading message during this brief window. Once the user responds, the state moves to true or false permanently for that app install.

There are two syntax things happening at once here worth understanding separately:

Immediately-invoked async function: useEffect's callback cannot itself be async — it would return a Promise, which React would incorrectly treat as a cleanup function. The workaround is to define and immediately call an async function inside the effect: (() => { ... })().

Destructuring the result: requestCameraPermissionsAsync() returns an object like { status: 'granted', ... }. Writing const { status } = await ... pulls out just the status field — a common JavaScript shorthand.

Chapter 6 — Taking a Photo

6
Capture

The Snap button calls snap(), an async function because image processing takes a moment. Here is the complete function: 

const snap = async () => {
  if (cameraRef.current) {
    const options = { quality: 0.5, base64: true };
    const photo   = await cameraRef.current.takePictureAsync(options);
    photo.name    = 'photo_' + photolist.length;
    setPhoto(photo.uri);
    setPhotoList(prev => [photo, ...prev]);
  }
};

The function does four things in order:

1
Guard check. if (cameraRef.current) verifies the camera widget is actually mounted before trying to use it. If the camera isn't ready, we do nothing rather than crash.
2
Capture. takePictureAsync() fires the shutter. The options set JPEG quality to 50% (fine for a list thumbnail) and include a base64 encoding alongside the file URI. The returned photo object contains uri, width, height, and base64.
3
Label it. We add a name field to the photo object using the current list length as a counter. This is how list components can label each entry.
4
Update state. setPhoto refreshes the "current photo" preview. setPhotoList prepends the new photo to the existing array so the newest shot appears first.
The functional update form: setPhotoList(prev => [photo, ...prev]) uses the functional form of the setter. This guarantees we are reading the current value of the list, not a stale copy from the closure. Whenever an update depends on the previous value of the same state variable, use the functional form. Writing setPhotoList([photo, ...photolist]) would work most of the time but can silently drop updates in concurrent rendering scenarios.

Chapter 7 — Switching Views

7
View Toggle

The Switch button flips between two list presentations. This is a pattern you will use constantly in React: store the user's intent as a boolean, and derive the actual UI from that boolean. 

const switchMode = () => {
  setVMode(prev => !prev);
};

And in the JSX, the conditional rendering reads the state and picks a component: 

{viewmode
  ? <ListGrid    photolist={photolist} source={{ uri: aphoto }} />
  : <PreviewList photolist={photolist} source={{ uri: aphoto }} />
}

Both components receive the same two props: photolist (the full array) and source (the URI of the most recent photo). The parent doesn't care how they display those — it just hands over the data.

Chapter 8 — Permission Guards

8
Guard Clauses

We check permission state before rendering the camera. Two early returns handle the two non-happy paths: 

if (hasPermission === null) {
  return <View><Text>Requesting camera permission…</Text></View>;
}
if (hasPermission === false) {
  return <View><Text>No access to camera.</Text></View>;
}

Early returns like these — often called guard clauses — are a React convention that keeps the main rendering code clean. By handling the error and loading states up front and exiting early, the happy-path JSX below them can assume everything is fine and be written without defensive nesting.

Note the triple equality === null, not == null. The difference matters here: hasPermission starts as null (JavaScript's "no value") and can become false (a boolean). Both are falsy, so a loose if (!hasPermission) check would conflate them — showing the "no access" error before the dialog even appears.

💡 Design note: These guard messages are placeholders — good enough for development. For a real app, the "null" state is so brief (milliseconds) that a splash screen or skeleton loader is more appropriate, and the "false" state should include a button that opens the device's Settings app to the permission screen using Linking.openSettings().

Chapter 9 — Responsive Layout

9
Orientation

Mobile apps run on phones held in portrait and landscape orientations, and sometimes on tablets with much wider screens. We adapt the layout using a single comparison: 

const { width: SCREEN_WIDTH, height: SCREEN_HEIGHT } = useWindowDimensions();
const mainstyle = SCREEN_WIDTH > SCREEN_HEIGHT ? styles.widebox : styles.tallbox;

useWindowDimensions() re-runs every time the device rotates and provides the new dimensions, which causes the component to re-render with the updated mainstyle. tallbox stacks children in a column (portrait); widebox lays them out in a row (landscape).

In portrait, children stack vertically (column). In landscape, they sit side by side (row). The same three children render in both orientations — only the container's flexDirection changes.

⚠ useWindowDimensions vs. Dimensions.get(): useWindowDimensions() returns the window height, which shrinks when the software keyboard opens. On some phones, when the keyboard is open, window.height < window.width can become true even in portrait, causing a spurious layout flip. For camera-adjacent UIs where the keyboard isn't shown, this is not a problem — but keep it in mind if you later add a text entry field to this screen. The stable alternative is Dimensions.get('screen'), which ignores the keyboard.

Chapter 10 — The UI

10
JSX

The rendered JSX, annotated to show how each section maps to the state and functions we have built: 

<View style={mainstyle}>               // ← portrait or landscape container

  <CameraView
    ref={cameraRef}              // ← connects the shutter handle (Chapter 4)
    facing={type}               // ← 'front' or 'back' (state: type)
    style={styles.camera}
  />

  <View style={styles.buttonRow}>

    <TouchableOpacity onPress={snap}>
      <Text> Snap </Text>              // ← calls snap() (Chapter 6)
    </TouchableOpacity>

    <TouchableOpacity
      onPress={() => setType(prev => prev === 'back' ? 'front' : 'back')}
    >
      <Text> Flip </Text>              // ← toggles 'front'/'back' inline
    </TouchableOpacity>

    <TouchableOpacity onPress={switchMode}>
      <Text> Switch </Text>            // ← calls switchMode() (Chapter 7)
    </TouchableOpacity>

  </View>

  {viewmode                          // ← conditional: state drives which list
    ? <ListGrid    photolist={photolist} source={{ uri: aphoto }} />
    : <PreviewList photolist={photolist} source={{ uri: aphoto }} />
  }

</View>

Notice that the Flip button's handler is written inline rather than as a named function. Both styles work — inline is appropriate when the logic is a single expression, as it is here. The Snap and Switch buttons are named functions because they contain multiple lines of logic.

Chapter 11 — Styles

11
StyleSheet

StyleSheet.create() compiles the style objects once at module load time instead of recreating plain JavaScript objects on every render. It also provides IDE autocomplete for valid React Native style properties, which helps catch typos early. 

const styles = StyleSheet.create({
  tallbox: {
    flex: 4, marginTop: 50,
    flexDirection: 'column',          // portrait stack
    borderWidth: 5, backgroundColor: '#fff',
    alignItems: 'center', paddingTop: 50,
  },
  widebox: {
    flex: 4, paddingTop: 0, paddingLeft: 10,
    flexDirection: 'row',             // landscape side-by-side
    borderWidth: 5, backgroundColor: '#fff',
    alignItems: 'center',
  },
  camera: {
    flex: 4, height: 300, width: '80%',
    padding: 5, borderWidth: 6,
    objectFit: 'cover',               // fill the box, no letterboxing
  },
  buttonRow: {
    flex: 1, flexDirection: 'row',
    alignItems: 'center',
    borderWidth: 2, margin: 5,
  },
  behind: {
    alignItems: 'center',
    justifyContent: 'flex-start',    // ← 'top' is not a valid RN value
    position: 'absolute',
    left: 0, top: 50, width: '100%', height: 248,
    borderWidth: 6,
  },
});
Style propertyWhat it does / common gotcha
flex: 4 on containerTells the parent flexbox to give this container 4 units of available space. With flex: 1 on the button row, the camera gets 4× as much height.
objectFit: 'cover'Scales the camera preview to fill its box without distorting the aspect ratio. The equivalent of CSS object-fit: cover. Without it, the preview may letterbox.
justifyContent: 'flex-start''top' is not a valid React Native value for justifyContent. It will be silently ignored, and the default (flex-start) will apply anyway — but it's a source of confusion in code reviews.
position: 'absolute' on behindThe behind style sets up an overlay layer — reserved for future use like focus indicators or AR overlays. Absolute positioning removes the element from the normal flex flow.

The Complete Flow, End to End

The startup path (top row) runs once. The capture path (bottom row) runs every time the user taps Snap. Both end in a state update that causes a re-render, which is how React keeps the UI in sync with the data.

Common Mistakes and How to Fix Them

MistakeSymptomFix
Using this.SnapCamera or React.createRef() Camera ref is always null; Snap button does nothing Use const cameraRef = useRef(null) and pass ref={cameraRef} to CameraView
Making useEffect callback itself async React warning about returning a Promise from useEffect; cleanup may silently fail Define and immediately invoke an async function inside useEffect: (() => { ... })()
Missing [] dependency array on useEffect Permission dialog appears on every re-render, including every keypress in a nearby TextInput Always pass [] as the second argument when the effect should run only once
Using result.uri instead of result.assets[0].uri Photo URI is undefined; images show a broken placeholder (this applies to launchCameraAsync, not takePictureAsync) In Expo SDK 48+, ImagePicker results nest assets. Check the SDK docs for the version you are on.
Flip button always returning one value Flip button has no effect Use the toggle form: prev => prev === 'back' ? 'front' : 'back'. A common mistake is writing a constant return instead of reading prev.
justifyContent: 'top' Silently ignored — layout looks wrong but no error is thrown Use 'flex-start'. Valid values are: flex-start, flex-end, center, space-between, space-around, space-evenly.
Not checking cameraRef.current before calling takePictureAsync App crashes if Snap is tapped before the camera widget has mounted Guard with if (cameraRef.current) { ... } inside snap()
