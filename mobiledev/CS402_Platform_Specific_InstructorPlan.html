Introduction — The Detection Spectrum
Two platforms, one codebase, and where the seams show
0:00 – 0:08 · 8 min

Opening framing

  • "React Native's pitch is write once, run everywhere. That's mostly true. But iOS and Android are not the same OS — they just speak enough of the same language that we can write shared code for most things. Today is about the exceptions."
  • Display the detection spectrum from the student document: Platform.OS → Platform.select() → .ios.js/.android.js → native modules. This is the tool ladder. Bigger differences need bigger tools.
  • The two-part structure of this lecture follows the ladder: Part 1 is everything you can handle in JS; Part 2 is what needs to go below JS.
🗣 Quick opener: "Name something you've noticed looks or works differently between the same app on iPhone vs Android." Students usually mention shadows, back button behavior, notification style, or swipe gestures. All of these are either Part 1 or Part 2 topics — use their answers to motivate the two sections.
📝 Frame the principle up front: State the underlying principle early and return to it throughout: program to an interface, not an implementation. Whether the platform split is a Platform.OS ternary or a native module in Swift, the consuming component should see a stable, named API. The platform handling is an implementation detail — it belongs behind the interface, not scattered through components. Students who internalize this write much cleaner code.

Part 1 — Platform.OS, Platform.select(), Platform.Version
Inline branching · Style differences · Version checks
0:08 – 0:20 · 12 min

Platform.OS — the simplest tool

  • Platform.OS is a string evaluated at runtime. It never changes in a session. Use it for one or two conditional values anywhere — labels, heights, behavior flags.
  • Platform.Version: a number on Android (API level), a string on iOS (version string). The most common use is the Android API level check — permissions behavior changed at API 23, storage at 29, granular media at 33.
🖥 Demo 1 (5 min) — Shadow vs. elevation: The most immediately visible platform style difference. Open a Snack with a white card on a gray background. Add iOS shadow props — show nothing happening on Android. Then add elevation — show nothing happening on iOS simulator. Then show the ...Platform.select() spread pattern that handles both. The visual difference is immediate and students remember it.

Platform.select() — multiple values at once

  • Platform.select() takes an object, returns the matching platform's value. The default key catches anything not explicitly listed.
  • The spread pattern inside StyleSheet.create is idiomatic: write shared styles once, spread in only the differences. Don't duplicate the whole style object per platform.
  • Works for style objects, strings, numbers, functions — any value at all.
✅ Quick Check
  1. A student adds shadowColor: '#000' and shadowOpacity: 0.2 to a card style. It looks perfect in the iOS Simulator. Why does the Android emulator show no shadow?
    → Android ignores shadow* props. It uses elevation instead. The fix is to use Platform.select() to provide elevation for Android alongside the iOS shadow props.

Part 1 — Permissions
iOS one-shot rule · Android API level branching · Settings fallback
0:20 – 0:34 · 14 min

Why permissions are the most important Part 1 topic

  • Permissions are where the platforms diverge most sharply in behavior. Getting them wrong produces either a crash, a denied user experience, or an App Store rejection.
  • Expo packages abstract most of this — but students need to understand what's happening underneath, especially the iOS one-shot rule and the Android version ladder.
🚨 Burn the iOS one-shot rule into memory: On iOS, the permission dialog appears exactly once per permission per app install. If the user taps "Don't Allow" — even accidentally — the dialog never appears again. They must go to Settings → [App] → enable manually. This is not a bug you can fix in code. The only defense is to ask at the right moment — contextually, when the user already understands why you need it.

Tell students: "If you ask for camera permission when the app first launches, before the user has ever tried to take a photo, a significant percentage of users will tap Don't Allow out of reflex. You've permanently lost that permission on their device."

Android API level branching

  • API < 23 (Android 5): Permissions granted at install time. Runtime requests not needed and will crash.
  • API 23–28 (Android 6–9): Runtime dangerous permissions. The standard flow.
  • API 29+ (Android 10): Background location requires a separate, follow-up request after foreground is granted. You cannot ask for both at once.
  • API 33+ (Android 13): Granular media permissions. READ_MEDIA_IMAGES and READ_MEDIA_VIDEO replace READ_EXTERNAL_STORAGE.
🖥 Demo 2 (7 min) — The photo permission flow: Walk through the requestPhotoAccess() function from the student document. Show: (1) the Android API level branch for API 33+, (2) the iOS denied → Settings redirect with Linking.openURL('app-settings:'). Run on iOS Simulator — tap "Don't Allow," then show the Alert with "Open Settings" appearing. This is the exact experience users will have in real apps.
📝 Expo abstracts most of this: The expo-* permission packages handle platform differences internally. Students don't need to write this branching code for every permission — they need to understand it exists so they know what the packages are doing and can handle the edge cases (like the iOS Settings redirect) themselves.
✅ Check for Understanding
  1. A user's Android device has API level 21. Your app calls Location.requestForegroundPermissionsAsync(). What should happen?
    → On API 21 (Android 5), permissions are granted at install time. The correct behavior is to skip the runtime request and return 'granted'. Calling the request on API < 23 can crash on some devices.
  2. iOS user taps "Don't Allow" on the camera permission dialog. What can your app do?
    → It cannot re-prompt. It should detect the denied status and show an Alert explaining why the permission is needed, with a button that opens app-settings: via Linking.openURL(). The user must manually re-enable it.

Part 2 — File-Extension Routing
Metro resolution · .ios.js / .android.js / .native.js · HapticEngine build
0:34 – 0:46 · 12 min

When to move from inline to file-level

  • The signal: when the platform difference is large enough that having both implementations in one file makes the file hard to read, test, or maintain.
  • File-extension routing moves the branching to build time. No runtime check. No dead code shipped to either platform. The bundle for iOS never contains the Android implementation, and vice versa.
  • Metro's resolution order: .ios.js → .native.js → .js (for iOS). Android checks .android.js first. Web checks .web.js first.
🖥 Demo 3 (8 min) — HapticEngine build: This is the most satisfying demo in the session. Create three files in a Snack: HapticEngine.ios.js, HapticEngine.android.js, and a component that imports from './HapticEngine'. In the ios file, add a console.log "iOS haptic." In the android file, add "Android haptic." Run on iOS Simulator — show the iOS log. Switch to Android emulator — show the Android log. Same import. Different code. Metro did the routing.

Then evolve it to the actual HapticEngine from the student document — expo-haptics with the full/limited implementation on each side.
📝 The shared interface principle in action: Point explicitly to the consuming component and note: it just calls impact('light'). It doesn't know which file ran. This is the principle from the intro — program to the interface, not the implementation — made concrete. Students who haven't fully internalized it will see it clearly here.
Always provide a fallback: Ask students: "What happens if someone builds your HapticEngine module for web and there's no HapticEngine.web.js?" → Metro falls back to HapticEngine.js. If that doesn't exist, the import fails at build time. Always provide a fallback — either a .js or a .native.js that covers the non-web case.

Part 2 — Native Modules
What they are · Installing · Using expo-local-authentication · Writing your own (overview)
0:46 – 1:00 · 14 min

What a native module is

  • Some device capabilities have no JavaScript layer at all — Face ID, HealthKit, Android BiometricPrompt, background geofencing. These are accessible only through native APIs (Swift/Kotlin).
  • A native module is a package that contains both JS and native code. The Expo Modules API generates the bridge — the JS side calls a named function, the bridge calls native code, the result comes back via a Promise or callback.
  • From the consuming side, it looks like any other module: import * as LocalAuth from 'expo-local-authentication'. The native plumbing is invisible.
  • The cost: native modules require a full native build. They can't run in Expo Go — they need a development build. This is the tradeoff for getting access to deeper platform APIs.
🖥 Demo 4 (7 min) — expo-local-authentication: Walk through the authenticateUser() function from the student document. Show: (1) supportedAuthenticationTypesAsync() to check hardware, (2) Platform.select() to customize the prompt string, (3) authenticateAsync() with the full options object. Run on a device with biometrics if available. If not, show the simulator's simulated Face ID (iOS Simulator → Features → Face ID → Enrolled, then Matching Face). Students are always surprised that the simulator can simulate Face ID.

Writing your own — awareness only

  • Writing a custom native module is out of scope for CS 402, but students should know it's possible and understand the pattern at a high level.
  • Expo Modules API: you write a Swift class and a Kotlin class, each with exported functions. Expo generates the JS interface. You get TypeScript types for free.
  • Use case: a capability that exists on device hardware but no package exposes it. In practice, this is rare — the Expo and community ecosystems are comprehensive.
📝 Don't get pulled into native module implementation details: Students curious about writing native modules can pursue Expo's Modules API documentation independently. The goal here is awareness — they should understand what a native module is, why it exists, and roughly how the bridge works. Not how to write Swift.

Fallback Stubs, Pairs Activity, and Wrap-Up
Stub pattern · Graceful degradation · Strategy selection activity · Final CFU
1:00 – 1:15 · 15 min

The fallback stub (3 min)

  • Every platform-specific module should have a fallback. The fallback can be a no-op, a stub that returns a safe value, or a function that throws a descriptive error. Any of these is better than a missing-module crash.
  • Show the BiometricAuth.web.js stub from the student document. It exports the same function names, returns false, and logs a console.warn. The consumer never crashes — it just doesn't get biometric authentication on web, which is the correct behavior.
  • Key insight: the stub's return value should be the same type as the real implementation. A function that returns Promise<boolean> in the native version should also return Promise<boolean> in the stub.
🏋 Pairs Activity (8 min) — Strategy Selection

Give pairs a list of platform differences and ask them to choose the correct tool from the spectrum for each. Speed matters — most pairs should finish in 5–6 minutes.

  1. The submit button says "Done" on iOS and "Submit" on Android.
  2. Card shadows work on iOS but not Android.
  3. You need to access iOS HealthKit step count data.
  4. Your app uses a tab navigator — on Android, pressing the hardware back button from Tab 2 should go to Tab 1, not exit the app.
  5. You want a shared Analytics module that calls the native SDK on iOS/Android and does nothing on web.
  6. Location permission behavior needs to handle the Android API 29 background location rule.

Answers: (1) Platform.OS ternary · (2) Platform.select() spread in styles · (3) Native module (expo-health or custom) · (4) BackHandler API with Platform.OS === 'android' check · (5) .ios.js / .android.js / .web.js stub · (6) Platform.OS + Platform.Version branch in permission logic

✅ Final Check for Understanding
  1. You import ./ShareSheet in a component. You have ShareSheet.ios.js and ShareSheet.android.js but no ShareSheet.js. A teammate adds a web build. What happens?
    → The web build fails — Metro cannot find a matching file (no .web.js, no .js fallback). Fix: add ShareSheet.web.js as a stub that either does nothing or opens a browser-native share mechanism.
  2. When should you reach for a native module instead of a Platform.select() branch?
    → When the capability requires platform-native code that doesn't exist in JavaScript — e.g. Face ID, HealthKit, BiometricPrompt. If the capability can be handled entirely in JS with different values or behavior, Platform.OS and Platform.select() are sufficient.
  3. What is the rule about the return type of a platform fallback stub?
    → It must match the real implementation's return type. If the real function returns Promise<boolean>, the stub returns Promise<boolean> (typically Promise.resolve(false)). This ensures the consumer's code — which checks the return value — works identically on all platforms.

Common Student Mistakes

MistakeWhat goes wrongFix
Using shadow* props expecting Android shadows Android ignores all shadow* style properties. Card looks flat on Android. Use Platform.select() to provide elevation for Android.
Asking for iOS permission too early User hasn't established context; denies reflexively. Dialog never appears again. Ask at the contextually obvious moment — when the user first tries to use the feature.
No platform fallback file Build succeeds on iOS and Android but fails when a web build is added later. Always provide a .js or .web.js fallback for every platform module.
Fallback stub returns the wrong type Consumer code calls result.success on the stub's return value; stub returns false (not an object); crash. Stub must return the same shape as the real implementation: if real returns {success: bool}, stub returns {success: false}.
Platform detection scattered through components Platform branches duplicated everywhere. Changing behavior for one platform requires hunting through all files. Isolate all platform-specific logic in a module. Components call the module interface; the module owns the branching.
Requesting Android background location with foreground in one call Android 10+ requires background location to be requested as a separate, sequential permission after foreground is granted. Combined request is silently rejected. Check Platform.Version >= 29 and make the background request as a follow-up after foreground succeeds.

Grading & Assessment Notes

Platform handling shows up most in assignments that involve styling (shadow/elevation), permissions (camera, location), or cross-platform navigation behavior. Grade the pattern, not the polish.

  • Full credit: Correct API for the platform difference (select() for styles, version branch for permissions), working on both platforms, fallback present for file-extension modules.
  • Partial credit: Correct on one platform only (e.g. shadow props work on iOS, nothing on Android) — the concept is there, the cross-platform follow-through is missing.
  • Comment, not deduction: Shadow on iOS without elevation on Android (if Android styling isn't the assignment's focus). Note it in feedback.
  • Significant deduction: Runtime crash due to missing module fallback, or Android background location requested simultaneously with foreground causing silent permission failure.
