CS 402 — Mobile Application Development

Building a Camera App: View, Capture, Browse, Share
🕐 Total time: 75 minutes 📅 Semester: Spring 2026 📦 SDK: Expo SDK 52 📄 Student doc: CS402_Camera_App_Walkthrough.html 🔗 Prereqs: useState, useEffect, useRef, Expo permissions

Lesson Overview

This lecture walks students through a complete, working React Native camera app line by line, using the literate-programming structure of CameraApp.js as the spine of the lesson. Each of the eleven chapters in the student document corresponds to a cohesive concept; the seven instructor segments group those chapters into teachable arcs with appropriate pacing and demos.

The primary teaching challenge here is density: five state variables, a ref, an async permission flow, a capture pipeline, orientation-aware layout, and conditional rendering all coexist in a single component. The goal is not to cover every line at equal depth, but to build a clear mental model of how these pieces connect — students can re-read the document for detail. Prioritize the ref vs. state distinction (Chapter 4) and the permission state machine (Chapter 5); those are where confusion most often takes root.

Pedagogical framing: This is a code reading lecture as much as a code writing one. The student doc is written in literate style for a reason: students who read it top-to-bottom before class will follow the lecture well; those who haven't will still be able to follow because the narrative structure is identical. Explicitly tell students at the start: "The document and this lecture tell the same story. You can use one to fill gaps in the other."

Learning Objectives

Understand
Explain the difference between useRef and useState and when each is the right tool.
Understand
Describe the three-state permission model (null / true / false) and why all three states require distinct handling.
Apply
Write a correct useEffect permission request using an immediately-invoked async function and an empty dependency array.
Apply
Use takePictureAsync() via a ref and prepend the result to state using the functional update form.
Apply
Implement orientation-aware layout using useWindowDimensions() and conditional style selection.
Analyze
Identify and explain at least four common mistakes in camera component code and articulate the correct fix for each.

Hook & Orientation — What We Are Building
Show the finished app first; establish the four-function contract
⏱ 5 min

Open by running the finished app on a physical device. Let students watch it work before explaining a single line. This 60 seconds of live demo sets the target and makes everything that follows concrete.

🖥 Open demo (no narration yet): Launch the app. Tap Snap — a photo appears in the list. Tap Snap again — two photos. Tap Switch — the view changes to a grid. Tap Flip — the camera swaps front to back. Rotate the device — the layout reflows. Then stop and say: "One file. 150 lines. Let's read it together."

Talking Points

  • This is the simplest possible camera app — no navigation, no server, no persistence. Four things: ask permission, show viewfinder, snap photos, browse the list. That's the whole spec.
  • The entire app is one functional component. No classes, no Redux, no navigation stack. All state is local. This is intentional — Project 4 will ask you to compose this into something larger; today we understand the core.
  • Point to the student doc: "The document and this lecture are the same story told twice. Use whichever one fills gaps."
📝 Pacing note: Resist the urge to explain the app during the demo. The silence while it runs is productive — students are forming questions. You want those questions to surface naturally as you walk through the code.

Chapters 2 & 3 — Imports and State
What the file depends on; what the component remembers
⏱ 14 min

Imports (4 min)

Walk through the import block. The key teaching point is that imports are a promise about what the file will use — nothing extra, nothing missing. Have students predict which imports they would expect before you show the list.

💬 Before showing the imports, cold-call: "Without looking at the file — what do you think this component will need to import from React? From React Native? From Expo?" Let two or three students answer. Then reveal the actual imports. Note what they predicted correctly and what they missed (useRef is the one most students forget).
  • We import only what we use — useState, useEffect, useRef from React. Nothing else. This is a readability signal: every hook you see in the imports, you will see used below.
  • Two separate things come from expo-camera: CameraView (the JSX widget — the live viewfinder) and Camera (the static API object — we only call requestCameraPermissionsAsync from it). Students often confuse these.
  • SDK 50 renamed <Camera> to <CameraView>. Any tutorial that uses <Camera ref={...}> as the viewfinder is out of date. This matters because students will Google and find old examples.
  • PreviewList and ListGrid are sibling components — we don't build them today, but we see how CameraApp communicates with them via props.

State — The Five Slots (10 min)

Display the state diagram from the student doc (or draw it on the board: five columns, each with the variable name and its possible values). Walk each one.

  • hasPermission is the most important state variable in the file. It starts as null — not false, not undefined — because "not asked" and "denied" are genuinely different situations that require different UI responses. Conflating them would be a bug.
  • type is just a string: 'front' or 'back'. It goes straight into the facing prop on CameraView. One state variable, one prop, zero logic in between.
  • aphoto starts as a placeholder asset path. After the first snap, it holds a file:// URI. The child list components use this to show the most recent capture.
  • photolist starts empty and grows with every snap. Newest first — we prepend, not append. This is an array of raw Expo photo objects.
  • viewmode is a boolean. false = PreviewList, true = ListGrid. The Switch button flips it. Students sometimes ask "why not a string like 'preview' and 'grid'?" — a boolean is correct here because there are exactly two states. If a third view were added later, you'd refactor to a string then.
✅ Check for Understanding
  1. "Why does hasPermission start as null rather than false?"
    → Null means "not asked yet." False means "asked and denied." They require different UI responses — a waiting message vs. an error message. Starting as false would show an error before the dialog has even appeared.
  2. "Which import lets us actually fire the shutter?"
    → Neither import on its own. CameraView renders the viewfinder, but the shutter is fired via cameraRef.current.takePictureAsync() — which requires the ref introduced in Chapter 4.
3
Chapters 4 & 5 — useRef and Permissions
The remote-control mental model; the three-state permission machine
⏱ 15 min

useRef — Commands vs. Data (7 min)

This is the conceptual crux of the segment. Take the time to establish the distinction cleanly before showing any code.

📝 Board moment: Draw two boxes side by side. Label one "Data that drives the UI" (useState) and one "Commands sent to a component" (useRef). Ask students to place examples from the app into each box before revealing the answer. Most will correctly place type, photolist, and viewmode in the useState box. Almost none will independently suggest that taking a photo belongs in the useRef box — that's the insight to make explicit.
  • Think of a ref as a remote control. You don't re-render the camera to take a photo — you press its shutter button directly. State is for values that make the UI look different. Refs are for imperative actions on mounted component instances.
  • Changing a ref does NOT trigger a re-render. That's the key technical distinction. A ref is just a mutable container — { current: null } — that React fills in when the component mounts.
  • The old class-component pattern was this.SnapCamera = React.createRef(). Two problems: there's no this in a functional component, and createRef() inside the function body creates a new ref on every render, which never stably connects to the hardware. Always use useRef(null).
  • Wiring it up: const cameraRef = useRef(null) creates the slot. Writing ref={cameraRef} on <CameraView> tells React to fill that slot when the widget mounts. After mount, cameraRef.current is the live camera instance.

Permissions — The State Machine (8 min)

Walk through the permission flow using the state-machine diagram in the student doc (or draw the three-node diagram on the board).

  • useEffect with [] fires once, right after first render. This is where startup work lives — analogous to componentDidMount in class components. The empty array is not optional; without it the effect runs on every re-render, which would show the permission dialog on every state change.
  • Why the immediately-invoked async function? useEffect's callback must be synchronous — if it returns a Promise, React treats it as a cleanup function and things break silently. The IIFE lets us use await inside the effect without making the callback itself async.
  • Once a user denies permission, the OS will not show the dialog again. Calling requestCameraPermissionsAsync() on a denied permission returns { status: 'denied' } immediately. The only recourse is directing the user to Settings.
🖥 Live demo — show both permission paths: Reset the app permission in device Settings to "not determined." Run the app — students see the system dialog. Tap Allow. Now reset to denied in Settings. Run again — no dialog, state goes directly to false, the error message appears. This 90 seconds makes the state machine viscerally real.
✅ Check for Understanding
  1. "The useEffect dependency array is []. What happens if you accidentally leave it out?"
    → The effect runs after every re-render. The permission dialog fires every time any state changes. In practice it would appear, be dismissed, appear again — an infinite loop of dialogs.
  2. "Why can't we just write useEffect(async () => { ... }, [])?"
    → useEffect expects its callback to return either nothing or a cleanup function. An async function always returns a Promise. React would try to call the Promise as a cleanup function, which fails silently and can cause memory leaks.
4
Chapters 6 & 7 — Capture and View Toggle
The snap() pipeline; boolean intent driving conditional rendering
⏱ 14 min

snap() — The Four-Step Pipeline (9 min)

  • The guard check first: if (cameraRef.current) before calling anything. If the widget isn't mounted yet, we do nothing rather than crash. This is defensive programming — write it habitually.
  • takePictureAsync() is awaited because image processing takes real time on the hardware. quality: 0.5 means 50% JPEG compression — fine for thumbnails, much smaller files. The returned object has uri, width, height, and base64 fields.
  • We mutate the photo object to add a name field: photo.name = 'photo_' + photolist.length. This is a common pattern — Expo gives you what the hardware returns, and you augment it with app-specific metadata before storing.
  • The functional update form: setPhotoList(prev => [photo, ...prev]). Ask students: "Why not just setPhotoList([photo, ...photolist])?" The closure over photolist could be stale if two snaps happen in rapid succession. The functional form always reads the current value.
✏ Think-Pair (2 min)
"If you removed the guard check and called takePictureAsync() directly, when exactly would it crash — and what would the error say?" Have students think for 60 seconds, then pair-share. Expected answer: it would crash if Snap is tapped before CameraView has finished mounting, throwing a TypeError trying to call a method on null.

switchMode() — State-Driven Conditional Render (5 min)

  • switchMode is the simplest function in the file: setVMode(prev => !prev). The functional toggle form is correct here — same reason as the photo list. It reads the current value rather than closing over a potentially stale one.
  • The payoff is in the JSX: {viewmode ? <ListGrid /> : <PreviewList />}. This is a React pattern called conditional rendering. Store intent as a boolean in state; derive the actual UI from that boolean. The component doesn't manage two UIs — it manages one piece of state and lets the JSX compute the rest.
  • Both child components receive the same two props. The parent doesn't care how they display the photos — it just hands over the data. This is the separation of concerns from the composable components lecture applied here.
✅ Check for Understanding
  1. "Why does snap() use the functional form of setPhotoList rather than spreading photolist directly?"
    → The closure over photolist in the snap function could be stale if two snaps happen before React re-renders. The functional form (prev =>) is guaranteed to use the current list value.
  2. "PreviewList and ListGrid receive the same two props. What are they?"
    → photolist (the full array of photo objects) and source (the URI of the most recently captured photo as an object: {{ uri: aphoto }}).
5
Chapters 8 & 9 — Guard Clauses and Responsive Layout
Early returns as error handling; orientation-aware containers
⏱ 10 min

Guard Clauses (5 min)

  • The two early returns before the main JSX are guard clauses — a React convention for handling loading and error states before reaching the happy path. They keep the main return clean and readable because everything below the guards can assume a camera is available.
  • Point out the triple equality: === null, not == null. Both null and false are falsy, so if (!hasPermission) would conflate them — showing the "no access" error message before the permission dialog has appeared. Use strict equality when the difference between null and false matters.
  • These guard messages are deliberately minimal. In a real app, the null state is so brief that a splash screen is more appropriate. The false state should include a button using Linking.openSettings() to send the user to the correct Settings panel.

Responsive Layout with useWindowDimensions (5 min)

  • useWindowDimensions() is a hook that re-runs whenever the device rotates, providing updated width and height. The comparison SCREEN_WIDTH > SCREEN_HEIGHT determines landscape vs. portrait. That single boolean drives the entire layout switch.
  • tallbox uses flexDirection: 'column' — children stack vertically. widebox uses flexDirection: 'row' — children sit side by side. The viewfinder, button row, and photo list are the same three children in both orientations.
  • The keyboard gotcha: useWindowDimensions returns the window height, which shrinks when the software keyboard opens. On some phones this shrinkage can flip width > height from false to true even in portrait — causing a phantom landscape switch. For this camera app there's no keyboard, so it's safe. But if you ever add a text field to this screen, switch to Dimensions.get('screen').
📝 Live rotation demo: If you haven't already, rotate the physical device during this segment while the app is running. Students seeing the layout reflow in real time is worth 30 seconds of explanation. Point out that zero code in the component changes — only the hook's return value changes, which triggers a re-render with the new mainstyle.
6
Chapters 10 & 11 — The UI and StyleSheet
Reading JSX back to state; StyleSheet.create() and common gotchas
⏱ 10 min

Reading the JSX (5 min)

Walk through the JSX by tracing each element back to the state or function that drives it. The goal is not to read every prop — it's to show students that every piece of JSX is a direct consequence of something they've already seen.

  • <View style={mainstyle}> — the container uses the orientation-computed style from Chapter 9. One variable, one style, zero conditionals in the JSX itself.
  • <CameraView ref={cameraRef} facing={type}> — two props, two state/ref connections from Chapters 3 and 4. The ref wires the shutter; facing controls which lens.
  • The Flip button's handler is written inline: prev => prev === 'back' ? 'front' : 'back'. This is appropriate because it's a single expression. Snap and Switch are named functions because they contain multiple statements. Both styles are idiomatic — choose based on complexity.
  • The conditional at the bottom: {viewmode ? <ListGrid> : <PreviewList>}. Point back to viewmode in the state block. The parent manages one boolean; the JSX computes which component to render. React unmounts one and mounts the other when the boolean flips.

StyleSheet (5 min)

  • StyleSheet.create() compiles style objects once at module load time. React Native can validate property names and values against the native style system at that point, which means typos throw warnings immediately rather than silently doing nothing at runtime.
  • The flex values: the container has flex: 4; the button row has flex: 1. In a flex column, the camera gets 4 units of space and the buttons get 1 — roughly 80/20 split. flex is proportional, not a fixed size.
  • objectFit: 'cover' on the camera style: fills the box without letterboxing. This is what camera apps feel like on a phone. Without it, the preview may have black bars.
  • Point out the comment on behind.justifyContent: 'top' is not a valid React Native value. It is silently ignored. The correct value is 'flex-start'. Valid options are: flex-start, flex-end, center, space-between, space-around, space-evenly. This is the most common style typo in student code.

Common Mistakes, Q&A, and Exit Ticket
Rapid-fire bug table; open questions; bridge to Project 4
⏱ 7 min

Rapid-Fire Bug Review (4 min)

Project the common mistakes table from the student doc. Read down it quickly — one sentence per row. The goal is pattern recognition, not deep explanation. Students should leave knowing the symptoms; they can read the fixes in the doc.

  • this.SnapCamera / createRef() in a functional component — ref is always null, Snap does nothing. Fix: useRef(null).
  • Async useEffect callback — silent warning, potential memory leak. Fix: IIFE inside the callback.
  • Missing [] dependency array — permission dialog on every keystroke. Fix: always add the array.
  • Flip button returning a constant — always goes to one camera. Fix: prev => prev === 'back' ? 'front' : 'back'.
  • justifyContent: 'top' — silently ignored, layout looks wrong. Fix: 'flex-start'.
  • No guard on cameraRef.current — crash if Snap tapped before camera mounts. Fix: if (cameraRef.current) { ... }.

Bridge to Project 4 (2 min)

Connect what they just learned to the assignment waiting for them.

🔜 Project 4 connection: Project 4 asks you to extend this app in three directions: (1) persist photos so they survive an app restart — today's photolist is in-memory only; (2) wrap FlatList into a PhotoGallery module using the composable patterns from last lecture; (3) save captured photos to the device's MediaLibrary using createAssetAsync(). Today's code is the camera layer. You are building everything around it.
💬 Exit ticket (1 question, written or verbal): "In your own words: what is the difference between useState and useRef, and which one do we use to fire the camera shutter — and why?"

Expected answer: useState stores data that drives re-renders; useRef holds a stable handle to a mounted component instance for imperative commands. We use useRef for the shutter because taking a photo is a command, not a state change, and we don't want it to trigger a re-render.

Differentiation and Pacing Notes

If class is ahead of schedule

  • Extend the Segment 3 demo: reset the app permission in Settings live and show the denied path running in real time
  • Ask students to predict what happens if you pass null as the facing prop to CameraView — test it live
  • Show Linking.openSettings() and discuss how the production permission-denied screen would look
  • Discuss how you would add a "delete photo" feature — where does state live, what prop goes to which child?

If class is behind schedule

  • Compress Segment 6 (UI and Styles) to 3 minutes — it's the least conceptually dense segment
  • Skip the Think-Pair in Segment 4 and answer the guard question yourself
  • Treat Segment 7 as take-home: direct students to the bugs table in the student doc and move straight to the exit ticket question

Students who are ahead

  • Challenge: add a photo count badge above the list that updates every time a photo is snapped
  • Challenge: make the camera flip happen automatically every 5 seconds using setInterval inside a useEffect
  • Explore: what does the full photo object from takePictureAsync() contain? Log it and report back

Students who are struggling

  • Focus on just two concepts: "useRef gives you a handle to call methods on; useState gives you memory that triggers re-renders." That's the nucleus.
  • Suggest they read the student doc chapter by chapter, not the whole file at once
  • Point them to the Quick Reference table at the end of the student doc — every pattern they need for Project 4 is in that table
