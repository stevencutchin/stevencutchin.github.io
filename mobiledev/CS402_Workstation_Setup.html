Building a Full Expo Project on Your Workstation

Leaving Snack Behind — The Real Development Environment
💻 Node.js + npm 📁 Project structure ⚡ Metro bundler 📱 Expo Go on device 🔄 Fast Refresh 🪟 Windows-specific gotchas

Expo Snack was the appetizer. Fast, convenient, nothing to install — great for learning the basics. But real apps live on real machines. From this point forward, your development environment is your own computer: a local project, a local bundler, and your phone running Expo Go connected over WiFi. This is how professional React Native development actually works.

What changes, what doesn't: Your phone still runs Expo Go. The JavaScript you write is identical. What changes is where the code lives (your hard drive), where it runs (your machine's Metro bundler), and — critically — what you can do with it. Local projects can use any npm package, commit to Git, connect to local servers, and eventually build for the App Store and Play Store. Snack can do none of that.

1. The Architecture Shift

The only thing that changes is where Metro runs — cloud vs. your laptop. Your phone, Expo Go, and the JavaScript you write are identical. The QR code scan still connects them.
1
Install Node.js (LTS)
The JavaScript runtime that powers everything — Metro, npm, the Expo CLI

Everything in the React Native toolchain runs on Node.js. Metro (the bundler), npm (the package manager), and the Expo CLI are all Node programs. You need the LTS (Long-Term Support) version — not the latest. LTS is stable and tested. The latest bleeds.

Download
Node.js LTS
nodejs.org → click the LTS button

Run the installer with all defaults. When it finishes, verify the install:

PowerShell or Command Prompt
C:\Users\you> node --version
v22.11.0 ← any recent LTS is fine
C:\Users\you> npm --version
10.9.0 ← npm comes bundled with Node
🪟 Windows gotcha — PATH not updating: If node --version says "command not found" after the install, close your terminal and open a new one. The installer updates the PATH environment variable, but existing terminal sessions don't see the change. If a fresh terminal still fails, search Windows Settings for "Environment Variables" and verify that the Node.js folder is in your PATH.
🪟 Windows gotcha — PowerShell execution policy: If npm commands fail with a security error about scripts being disabled, run this once in PowerShell as Administrator:
PS> Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
This allows locally-created scripts to run. You only need to do this once.
2
Install Git
Version control — also required by some npm packages

Git manages your project's version history and is required by some npm packages that install from GitHub. You'll also use it to submit assignments and collaborate on the final project.

Download
Git for Windows
git-scm.com — run installer with defaults

During the Git installer, two settings matter:

  • "Adjusting your PATH environment" → select "Git from the command line and also from 3rd-party software"
  • "Configuring line ending conversions" → select "Checkout Windows-style, commit Unix-style" — this prevents CRLF line ending problems in your code

After install, tell Git who you are (use the same email as your GitHub account):

PowerShell
C:\> git config --global user.name "Your Name"
C:\> git config --global user.email "you@boisestate.edu"
C:\> git --version
git version 2.47.0.windows.1

Your editor, your debugger, your terminal — all in one

Visual Studio Code is the standard editor for React Native development. It's free, runs everywhere, and has excellent JavaScript/TypeScript support built in.

Download
VS Code
code.visualstudio.com — System Installer recommended

During install, check "Add to PATH" and "Open with Code" context menu options. These save time daily.

4
Create Your First Local Expo Project
create-expo-app scaffolds everything — no global installs needed

Navigate to where you want your projects to live — something like C:\Users\you\Projects — and run one command. You don't need to install Expo CLI globally first. npx downloads and runs it for you.

PowerShell — in your Projects folder
C:\Projects> npx create-expo-app@latest MyFirstApp
✔ Downloaded and extracted project files.
✔ Installed npm packages.
✅ Your project is ready!
To run your project, navigate to the directory and run one of the following npm commands.
C:\Projects> cd MyFirstApp
C:\Projects\MyFirstApp> code . ← opens VS Code in this folder
🪟 Windows gotcha — long file paths: Node modules nest deeply. Windows has a 260-character path limit by default, and node_modules can exceed it. Enable long paths once and forget about it:
  1. Search Windows for "Group Policy Editor" (or run gpedit.msc)
  2. Navigate to: Computer Configuration → Administrative Templates → System → Filesystem
  3. Enable "Enable Win32 long paths"
  4. Reboot
Alternative: Run this in PowerShell as Administrator:
PS Admin> reg add "HKLM\SYSTEM\CurrentControlSet\Control\FileSystem" /v LongPathsEnabled /t REG_DWORD /d 1 /f
🪟 Windows gotcha — antivirus slowdown: Windows Defender scans every file written to disk. node_modules can contain 50,000+ files. Add your Projects folder to Windows Defender exclusions: Windows Security → Virus & threat protection → Exclusions → Add a folder. This can make npm installs 3–5× faster.
5
Understand the Project Structure
What every file does and which ones you'll actually touch

Open the project in VS Code (code .). Here's what you're looking at:

MyFirstApp/
├── app/ ← your screens (Expo Router)
│ ├── (tabs)/ ← tab layout group
│ ├── _layout.tsx ← root navigation layout
│ └── index.tsx ← home screen ← START HERE
├── assets/ ← images, fonts, icons
├── components/ ← reusable UI components
├── constants/ ← colors, sizes, config
├── node_modules/ ← installed packages (don't touch)
├── app.json ← app name, icon, splash screen, permissions
├── package.json ← dependencies and scripts
├── tsconfig.json ← TypeScript configuration
└── .gitignore ← tells Git to ignore node_modules etc.

Files you edit constantly

  • app/ — your screens and navigation
  • components/ — your reusable pieces
  • app.json — name, permissions, icons
  • constants/Colors.ts — your theme

Files you rarely touch

  • node_modules/ — never edit, managed by npm
  • tsconfig.json — TypeScript settings, set once
  • .gitignore — already set up correctly
  • package-lock.json — auto-generated by npm
app/ is the heart of the project. Expo Router uses file-based routing — the name of a file in the app/ directory becomes the name of the route. app/index.tsx is the home screen. app/profile.tsx would be the /profile route. No manual routing configuration required.

Connecting your phone

  1. Make sure your phone is on the same WiFi network as your laptop. This is the single most common connection failure — phone on mobile data, laptop on campus WiFi, nothing connects.
  2. Android: Open Expo Go → tap "Scan QR code" → point at the terminal QR code.
  3. iOS: Open the default Camera app → point at the QR code → tap the Expo Go notification banner that appears.
  4. Wait 10–20 seconds for the first bundle. After that, changes reload in under 2 seconds.
🪟 Windows gotcha — Windows Firewall blocking Metro: The first time you run Metro, Windows Firewall may pop up asking to allow Node.js network access. Click "Allow access" on both Private and Public networks. If you accidentally blocked it, go to Windows Security → Firewall → Allow an app through firewall → find Node.js and enable both checkboxes.
🪟 Windows gotcha — campus WiFi isolation: Some institutional WiFi networks block device-to-device communication. If your phone can't reach Metro on campus WiFi, run Metro in tunnel mode:
C:\Projects\MyFirstApp> npx expo start --tunnel
Tunnel mode routes through Expo's servers instead of your local network. Slower, but works everywhere. It requires a free Expo account (npx expo login).

7. The Development Loop

Once Metro is running and your phone is connected, development is a continuous loop. Understanding the loop helps you work faster and debug problems faster.

Edit

→
⚡
Metro bundles
Metro detects the file change, re-bundles in ~1 second
→
🔄
Fast Refresh
App updates on your phone without losing state
→
📱
See it live
Result on your physical device immediately

Fast Refresh vs. Full Reload

Fast Refresh updates only the changed component, preserving state — your app doesn't reset. This happens automatically on every save. It's what makes React Native development feel instant.

Sometimes Fast Refresh isn't enough — if you change a navigation structure, a hook's initial value, or something at the module level, you need a full reload. Shake your phone to open the developer menu, then tap "Reload." Or press r in the Metro terminal.

The developer menu

Shake your physical device to open it. Key options:

OptionWhat it does
ReloadFull JavaScript reload — clears all state and restarts the app
Toggle performance monitorShows FPS, memory, and JS thread activity overlay
Toggle element inspectorTap any component to see its layout, styles, and hierarchy
Open JS debuggerOpens Chrome DevTools connected to your running app
7
Make Your First Edit
Prove the loop works — change something, watch it update

Open app/index.tsx in VS Code. You'll see the default Expo welcome screen component. Replace everything inside it with something minimal and your own: 

import { View, Text, StyleSheet } from 'react-native';

export default function HomeScreen() {
  return (
    <View style={styles.container}>
      <Text style={styles.title}>My First Local App</Text>
      <Text style={styles.sub}>Running from my own machine 🎉</Text>
    </View>
  );
}

const styles = StyleSheet.create({
  container: { flex: 1, alignItems: 'center', justifyContent: 'center', backgroundColor: '#1B3A5C' },
  title:     { fontSize: 28, fontWeight: '700', color: '#fff', marginBottom: 12 },
  sub:       { fontSize: 16, color: '#a0c8e8' },
});

Hit Ctrl+S. Within a second or two, your phone updates. No QR scan. No compile step. No deploy. Just save and see. That is the development loop — and it stays this fast as your app grows.

💡 Installing new packages: Whenever you need a new Expo package, use npx expo install instead of npm install. Expo's installer picks the version compatible with your current SDK. Using plain npm install can pull in incompatible versions and break your build in ways that are hard to diagnose.
npx expo install expo-camera ← Expo packages: always use this
npm install lodash ← Plain JS packages: npm is fine

8. Windows Gotchas — Complete Reference

Windows is a perfectly capable React Native development platform. It just has a few rough edges that trip up everyone the first time. Hit them now, fix them once, never think about them again.

ProblemSymptomFix
PowerShell execution policy npm commands fail with "running scripts is disabled" Set-ExecutionPolicy RemoteSigned -Scope CurrentUser in Admin PowerShell
Long file paths npm install fails with ENAMETOOLONG or mysterious errors deep in node_modules Enable long paths in Group Policy or registry. Reboot.
Antivirus scan slowdown npm install takes 5+ minutes. Metro startup is sluggish. Add your Projects folder to Windows Defender exclusions.
Firewall blocking Metro Phone can't connect — "Something went wrong" in Expo Go Allow Node.js through Windows Firewall (both Private and Public).
Phone on different network QR scan works but app shows network error Put phone and laptop on same WiFi. Or use --tunnel flag.
Campus WiFi client isolation Same WiFi but still can't connect npx expo start --tunnel. Requires Expo account.
CRLF line endings Git shows every line as changed after checkout on Windows Git installer setting: "Checkout Windows-style, commit Unix-style." Or git config --global core.autocrlf true
Node not in PATH after install "node is not recognized" in a terminal you already had open Close all terminals and open a new one. PATH changes don't affect existing sessions.
