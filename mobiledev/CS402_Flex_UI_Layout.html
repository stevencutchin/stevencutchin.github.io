1. The Axis Model — Understanding the Two Directions

Every Flexbox layout operates on two perpendicular axes. Confusing which axis a property controls is the source of most layout bugs. Before touching code, lock this model in.

Figure 1: The main axis (controlled by justifyContent) and cross axis (controlled by alignItems) swap when you change flexDirection. This is the source of most layout confusion.
React Native defaults differ from CSS. In a web browser, flexDirection defaults to 'row'. In React Native it defaults to 'column'. This matches how mobile screens scroll (top to bottom) but catches every developer coming from web CSS. If your layout is going sideways when you expected vertical stacking — or stacking when you expected horizontal — check flexDirection first.

justifyContent — Distribution on the Main Axis

Figure 2: All six justifyContent values shown on the main axis. space-between pins items to the edges; space-around gives equal margins; space-evenly gives equal gaps including edges.

2. flex, flexGrow, flexShrink, flexBasis — The Distribution Math

The shorthand flex property is the most common way to control how children share available space. But it wraps three separate concepts, and confusing them leads to layouts that break under real content. Understanding what each does unlocks precise control.

Figure 3: How flex, flexGrow, flexShrink, and flexBasis relate. The shorthand flex: N sets flexGrow: N, flexShrink: 1, and flexBasis: 0.
// These three are equivalent ways to say "take up remaining space equally"
<View style={{ flex: 1 }} />   // shorthand — use this in 95% of cases

<View style={{ flexGrow: 1, flexShrink: 1, flexBasis: 0 }} />  // explicit

// The classic "fill remaining height" pattern:
<View style={{ flex: 1 }}>  // screen root — fills the entire screen
  <Header />             // fixed height from its content
  <View style={{ flex: 1 }}>  // grows to fill whatever Header left
    <ScrollView />
  </View>
  <Footer />             // fixed height from its content
</View>
The flex: 1 chain: For a child to fill available height, every ancestor up to the root must also have flex: 1 or a fixed height. If any ancestor in the chain has no height defined and no flex, it will collapse to zero height and the child will disappear. This is the single most common "my layout is invisible" bug.

3. Common Screen Layout Patterns

Most mobile screens are variations on four fundamental layout patterns. Learn to recognise them and you can construct any screen as a composition of familiar building blocks.

Figure 4: The four canonical mobile screen layout patterns. Most screens you'll build are combinations or refinements of these.

The Header / Scroll / Footer Pattern in Code

This is the most commonly needed pattern and the one most frequently broken by the flex: 1 chain error:

export default function ProfileScreen() {
  return (
    // 1. Root fills the screen
    <View style={{ flex: 1, backgroundColor: '#fff' }}>

      // 2. Header — sized by its own content
      <View style={styles.header}>
        <Text style={styles.headerTitle}>My Profile</Text>
      </View>

      // 3. Scrollable body — flex:1 takes all remaining space
      <ScrollView style={{ flex: 1 }} contentContainerStyle={{ padding: 16 }}>
        <ProfileDetails />
        <ActivityFeed   />
      </ScrollView>

      // 4. Footer — sized by its own content, pinned to bottom
      <View style={styles.footer}>
        <Button title="Edit Profile" onPress={handleEdit} />
      </View>

    </View>
  );
}
contentContainerStyle vs style on ScrollView: The style prop controls the ScrollView's outer container (use flex: 1 here). The contentContainerStyle prop controls the inner scrollable content (use padding and alignItems here). Putting flex: 1 on contentContainerStyle disables scrolling — a very common mistake.

4. Responsive Layouts — Adapting to Any Screen

React Native runs on screens ranging from small phones to large tablets. Hard-coded pixel widths create layouts that look perfect on your test device and broken everywhere else. The tools for writing layouts that adapt are Dimensions, useWindowDimensions, and percentage-based sizing.

Figure 5: The same screen, four device sizes. Notice the layout adapts — single column on small screens, two-column grid on large phones, sidebar + grid on tablet. This is responsive layout.
import { useWindowDimensions } from 'react-native';

export default function PhotoGrid() {
  const { width } = useWindowDimensions();  // updates on rotation

  // Decide columns based on available width
  const numColumns = width > 600 ? 3 : 2;
  const itemSize   = (width - 32) / numColumns;  // subtract padding

  return (
    <FlatList
      data={photos}
      numColumns={numColumns}
      key={numColumns}  // force re-mount when columns change
      renderItem={({ item }) => (
        <Image
          source={{ uri: item.uri }}
          style={{ width: itemSize, height: itemSize }}
        />
      )}
    />
  );
}
useWindowDimensions vs Dimensions.get(): Use useWindowDimensions (a hook) in components — it re-renders when the device rotates or the window size changes. Use Dimensions.get('window') outside of components (in StyleSheet.create or utility functions) where hooks aren't available. The static version does not update on rotation.

5. Absolute Positioning and Overlays

Flexbox positions children relative to each other. Absolute positioning removes a child from the flow entirely and places it at exact coordinates within its parent. Use it for floating action buttons, badges, overlays, and anything that needs to sit on top of other content.

Figure 6: Absolutely positioned children float above normal flow content. Coordinates are relative to the nearest parent with position: 'relative' (the default).
// Floating Action Button — classic absolute positioning pattern
const styles = StyleSheet.create({
  container: { flex: 1 },

  fab: {
    position: 'absolute',
    bottom: 24,
    right: 24,
    width: 56,
    height: 56,
    borderRadius: 28,
    backgroundColor: '#c8890f',
    alignItems: 'center',
    justifyContent: 'center',
    elevation: 6,          // Android shadow
    shadowColor: '#000',  // iOS shadow
    shadowOffset: { width: 0, height: 3 },
    shadowOpacity: 0.25,
    shadowRadius: 4,
  },
});

6. Modular Component Design — Thinking in Layers

A screen full of JSX in a single component is hard to read, impossible to test, and painful to change. Modular design is the discipline of breaking UI into components that are the right size — small enough to be reusable and understandable, large enough to be meaningful. The framework for thinking about this is called Atomic Design.

Figure 7: Atomic design hierarchy. Build small, combine upward. Each level composes the level below it — screens don't build from scratch, they assemble organisms.

7. Component Composition Patterns

Knowing the atomic layers is the design side. These three patterns are the engineering side — the specific techniques for composing components flexibly without coupling them tightly.

Pattern 1: Props as Configuration

The simplest pattern. Pass data and callbacks down; the child renders itself accordingly.

// Flexible Button that adapts through props
function AppButton({ title, onPress, variant = 'primary', disabled = false }) {
  const bg = variant === 'primary' ? '#1B3A5C' : 'transparent';
  const border = variant === 'outline' ? 2 : 0;

  return (
    <Pressable
      onPress={onPress}
      disabled={disabled}
      style={[styles.base, { backgroundColor: bg, borderWidth: border },
               disabled && styles.disabled]}
    >
      <Text style={styles.label}>{title}</Text>
    </Pressable>
  );
}

// Usage — same component, different appearances
<AppButton title="Save"   variant="primary"  onPress={handleSave}   />
<AppButton title="Cancel" variant="outline"  onPress={handleCancel} />
<AppButton title="Submit" disabled={!isValid} onPress={handleSubmit} />

Pattern 2: The children Prop (Slot Pattern)

Instead of accepting content as a prop, accept it as nested JSX. This lets the parent control layout while the caller controls content.

// Card is a layout shell — it doesn't care what goes inside
function Card({ children, style }) {
  return (
    <View style={[styles.card, style]}>
      {children}
    </View>
  );
}

// The caller decides what the card contains — Card never needs to change
<Card>
  <UserTag user={post.author} />
  <Image   source={{ uri: post.imageUri }} style={styles.image} />
  <Text>{post.caption}</Text>
</Card>

// Same Card, totally different content
<Card>
  <Text style={styles.errorText}>Something went wrong.</Text>
  <AppButton title="Try Again" onPress={retry} />
</Card>

Pattern 3: Layout Components vs. Content Components

One of the most useful distinctions in component design. Layout components know about position and spacing but nothing about data. Content components know about data and behaviour but shouldn't hard-code their own position.

✅ Layout Component — knows position, not data

// Positions two items side-by-side
function Row({ left, right, gap = 12 }) {
  return (
    <View style={{
      flexDirection: 'row',
      gap: gap,
      alignItems: 'center'
    }}>
      {left}
      {right}
    </View>
  );
}

✅ Content Component — knows data, not position

// Renders user info — no margin/position
function UserTag({ user }) {
  return (
    <Row
      left={<Avatar uri={user.photo} />}
      right={
        <View>
          <Text>{user.name}</Text>
          <Text>{user.handle}</Text>
        </View>
      }
    />
  );
}
The test of a good content component: Can you drop it into a completely different screen without changing its code? If yes — it doesn't assume its own position, margin, or width — it's well separated. If moving it requires editing its internal styles, layout concerns have leaked into it.

8. StyleSheet Best Practices

Inline style objects create a new object on every render. StyleSheet.create() validates styles at development time and optimises them for the bridge. Beyond the basic rule of "always use StyleSheet.create", several patterns will keep your styles maintainable as your app grows. 

import { StyleSheet } from 'react-native';

// ── Design tokens — define once, reference everywhere ────────────
const tokens = {
  color: {
    navy:    '#1B3A5C',
    blue:    '#2E5D8A',
    text:    '#2c2c2c',
    subtle:  '#888',
    surface: '#f7f9fc',
  },
  spacing: { xs: 4, sm: 8, md: 16, lg: 24, xl: 32 },
  radius:  { sm: 4, md: 8, lg: 16, pill: 999 },
};

// ── StyleSheet that references tokens ────────────────────────────
const styles = StyleSheet.create({
  card: {
    backgroundColor: tokens.color.surface,
    borderRadius:    tokens.radius.md,
    padding:         tokens.spacing.md,
    marginBottom:    tokens.spacing.sm,
  },

  // Conditional styles: compose with arrays in JSX
  // <View style={[styles.btn, isActive && styles.btnActive]} />
  btn:       { padding: tokens.spacing.sm, borderRadius: tokens.radius.sm },
  btnActive: { backgroundColor: tokens.color.navy },
});
PatternDoAvoid
Style objects StyleSheet.create() at the bottom of the file Inline style={{ ... }} except for truly dynamic values
Conditional styles Array syntax: style={[styles.base, active && styles.active]} Ternary building entire style objects: style={active ? {...} : {...}}
Colours and spacing Design token object shared across the app Hard-coded hex strings scattered through multiple files
Platform differences Platform.select({ ios: {...}, android: {...} }) Separate style files per platform — hard to keep in sync
Shadows Both elevation (Android) and shadowColor/Offset/Opacity/Radius (iOS) Only setting one — shadows are invisible on the other platform

Quick Reference

Flex Properties Cheatsheet

  • flexDirection: 'column' — default, vertical
  • flexDirection: 'row' — horizontal
  • justifyContent — main axis spacing
  • alignItems — cross axis alignment
  • alignSelf — override parent alignItems for one child
  • flex: 1 — fill remaining space
  • flexWrap: 'wrap' — allow children to wrap
  • gap — uniform spacing between children (RN 0.71+)

Modular Design Checklist

  • Can I describe this component in one sentence?
  • Does it do only one thing?
  • Could I drop it into another screen unchanged?
  • Are layout concerns separate from data concerns?
  • Does it receive everything it needs through props?
  • Is there any JSX I've written twice that could become a component?
  • Is the component file under ~150 lines?
