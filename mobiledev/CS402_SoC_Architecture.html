System on a Chip — From App to Silicon

How Everything Your App Touches Lives on One Piece of Glass
🔬 Theme 1 — Foundations 📐 Architecture Deep Dive 🏛 LO-1 · LO-2

The Phone in Your Hand Is a Marvel of Compression

A decade ago, building a smartphone required dozens of separate chips soldered to a circuit board — each one a specialist doing one job. Today, all of that logic fits on a single chip smaller than a postage stamp. Understanding what that chip contains, and how your code eventually runs on it, is foundational to thinking seriously about mobile development.

The term for this consolidation is System on a Chip, or SoC. Rather than a motherboard populated with discrete components, the SoC integrates the processor, graphics engine, memory controller, wireless radios, camera processor, and much more into a single silicon die. The result is smaller, faster, cooler-running, and dramatically more power-efficient than what came before.

This document walks through how that architecture is organized — from the JavaScript you write at the top of the stack, all the way down to the transistors at the bottom — and then explores a fascinating edge case: how one company's chip can legally contain another company's secret processor design.

Chapter 1 — The World Before SoC

1
The Old Architecture

To appreciate what the SoC accomplishes, it helps to understand what it replaced. Early smartphones, and desktop computers before them, were built from a collection of specialized chips mounted on a circuit board and wired together. Each chip was designed, manufactured, and sourced independently.

A simplified view of how smartphones and early portable devices were built: many discrete chips, each doing one job, all communicating over a shared bus across the physical circuit board. Power travels between chips. Heat spreads everywhere. Latency accumulates at every crossing.

This approach had real costs. Every time the CPU needed to talk to the GPU, or the camera processor needed to write to memory, signals had to travel across physical wires on the circuit board. That takes time and energy. The chips generated heat independently. The board needed to be large enough to hold them all. And for a portable device powered by a battery, all of that inefficiency matters enormously.

⚡ Power problem

Signals crossing between chips on a PCB burn significantly more energy than signals crossing within a single chip. Every milliwatt matters when your battery holds roughly 15 watt-hours.

📏 Size problem

A board populated with 9+ chips plus their supporting components, capacitors, and traces cannot shrink below a physical floor. Modern phones are 80% battery and screen — the board must be tiny.

Chapter 2 — Enter the System on a Chip

2
The SoC

A System on a Chip takes all of those formerly separate components and fabricates them together on a single piece of silicon — a single die. The result is not merely smaller; it is fundamentally different in how data moves, how power is consumed, and how the hardware can be coordinated.

When the CPU and GPU share the same die, moving image data between them requires no external bus crossing — the data flows across an on-chip interconnect that is orders of magnitude faster and far less power-hungry. When the memory controller is on the same die, it can be coupled tightly to every block that needs memory access.

A modern mobile SoC. Every block that was once an independent chip is now a functional unit — an "IP block" — on the same piece of silicon, connected by high-bandwidth on-chip buses that move data in picoseconds rather than nanoseconds.

The phrase "IP block" here is significant and we will return to it in detail shortly. Each functional unit inside an SoC — the GPU, the CPU cores, the modem — is designed as a reusable piece of logic called an Intellectual Property block. These blocks can be mixed, matched, and licensed between companies. Apple does not design every block in its A-series chips from scratch. Neither does Qualcomm, Samsung, or MediaTek.

The key insight: An SoC is not one design — it is an integration of many designs. The chip manufacturer's job is partly engineering and partly curation: selecting the right IP blocks, arranging them efficiently on silicon, and connecting them with fast-enough on-chip interconnects.

Chapter 3 — The Logical Stack: App Down to Transistors

3
Layer Stack

When your React Native app calls expo-camera to take a photo, a remarkable chain of delegation happens — from your JavaScript, through several layers of abstraction, all the way down to specific circuits on the SoC. Each layer knows nothing about the layers far below or above it. That insulation is what makes the whole system manageable.

The abstraction stack from your JavaScript to the silicon. Each layer speaks only to its immediate neighbors — which is why you can write one app and run it on hardware you've never seen.

Tracing a Camera Capture

Let's walk one specific operation — taking a photo — through the stack:

1
Your app calls camera.takePictureAsync() — a JavaScript function in Expo's SDK. Your code lives entirely in Layer 6, unaware of what happens next.
2
The JS bridge / JSI routes the call to a native Expo module written in Swift (iOS) or Kotlin (Android). The JavaScript VM hands off control at this point.
3
The native module calls the Platform SDK — AVFoundation on iOS, Camera2 on Android. These are high-level, documented, stable APIs that Apple and Google maintain.
4
The platform SDK makes kernel system calls, asking the OS to orchestrate the hardware: "open the shutter, collect sensor data, trigger the ISP pipeline."
5
The kernel invokes the camera driver, which writes commands into hardware registers — specific memory addresses that the camera sensor and ISP on the SoC are listening to.
6
The ISP block on the SoC captures raw sensor data, processes it (demosaicing, noise reduction, HDR), and writes the final pixels to a memory buffer via the on-chip memory controller. All without leaving the die.
7
The result travels back up: the kernel signals the driver, the driver signals the OS SDK, the SDK resolves a callback, the native module resolves a Promise, and your JavaScript receives the image URI.
Why this matters for you as a developer: You never interact with layers 1–4 directly. The abstraction stack is what makes mobile development tractable. But understanding it explains why permissions exist, why certain operations are asynchronous, and why some hardware features are only available on specific OS versions — the API layer simply didn't exist yet.

Chapter 4 — Inside the SoC: Major Components

4
Components

Modern mobile SoCs contain a dozen or more distinct functional blocks. Here are the major ones and what they do — both in isolation and in the context of your app.

Block What It Does Your App Touches It When…
CPU Cluster General-purpose computation. Modern SoCs have heterogeneous clusters: high-performance "big" cores for peak throughput and high-efficiency "little" cores for background work. JavaScript executes. Layout renders. Business logic runs. Essentially always.
GPU Massively parallel processor optimized for graphics: shading, texture sampling, rasterization. Also used for compute tasks (GPGPU). Animations, maps rendering, camera preview, any Animated or react-native-reanimated workload.
Neural Engine / NPU Dedicated accelerator for matrix multiplications that underlie neural networks. Far more efficient than running ML on the CPU or GPU. Face ID, portrait mode segmentation, autocomplete, on-device translation, Core ML inference.
Image Signal Processor (ISP) Transforms raw Bayer-array sensor data into the JPEG/HEIF images you see. Handles demosaicing, white balance, noise reduction, tone mapping, HDR. Any camera operation. The ISP runs before your app ever sees a pixel.
Memory Controller Arbitrates all access to DRAM. In modern Apple chips, CPU, GPU, and NPU all access the same physical memory pool ("unified memory") through this block. Continuously — every read/write your app performs ultimately flows through here.
5G Modem / Baseband Handles all cellular communication: signal encoding/decoding, protocol stack, carrier negotiation. Historically a separate chip; now integrated on the die. Any fetch(), network request, or background sync.
Secure Enclave An isolated processor with its own encrypted memory. Manages biometric templates, encryption keys, and secure boot. The rest of the SoC cannot read its memory. Face ID / Touch ID, Keychain, payments, certificate operations.
Digital Signal Processor (DSP) Optimized for audio: encoding, decoding, noise cancellation. Runs while the main CPU sleeps to save power ("always-on" voice detection). "Hey Siri" / "OK Google" wake words. Audio recording and playback.
Unified memory is a game-changer: On Apple Silicon (M-series and A-series), the CPU, GPU, and NPU all read from and write to the same physical memory pool. There is no "GPU memory" versus "system memory" transfer step — the GPU can read a camera frame the ISP wrote without copying it. This is only possible because everything is on the same die with a coherent memory controller.

Chapter 5 — How NVIDIA's Architecture Lives Inside an Apple Chip

5
IP Licensing & Trust

This is where the story gets genuinely interesting — and where semiconductor engineering intersects with intellectual property law, international business contracts, and a remarkable system of institutional trust.

The GPU inside an Apple A-series chip is not something Apple designed from scratch. For many years, Apple's mobile GPUs used an architecture licensed from Imagination Technologies (the PowerVR architecture). But the broader principle — that one company's chip can run another company's processor design — is even more visible in the PC and server world, and it illuminates how the entire semiconductor industry actually functions.

Consider this: NVIDIA is one of the world's most valuable companies, and its GPU architecture — the shader model, the warp scheduler, the tensor core design — represents decades of engineering and hundreds of billions of dollars in R&D investment. These architectural details are among the most carefully guarded trade secrets in technology. Yet they flow, under contract, into chips designed by other companies. How?

What an IP Block Actually Is

When we say a company "licenses its GPU architecture," what is actually being licensed is a description of the hardware — not a physical chip, but a specification that can be manufactured into one. This description exists at several levels of abstraction:

RTL — Register Transfer Level

Hardware description code (Verilog or VHDL) that describes exactly how logic gates connect and how data moves between registers on each clock cycle. This is the most sensitive form — it is essentially the "source code" of the chip.

GDSII / Layout

The physical layout of transistors on silicon — polygons describing where each gate, wire, and via sits on the die. This is what a fab actually manufactures from. Receiving GDSII means you receive the chip geometry itself.

Soft vs. Hard IP

Soft IP is RTL that the licensee synthesizes for their target process node. Hard IP is pre-synthesized and pre-placed for a specific fab and node. Hard IP is faster to use but less flexible; soft IP is more adaptable.

When NVIDIA licenses its GPU architecture to another chip designer, it is transferring one of these descriptions — the detailed specification of how to build the GPU's logic — into that company's hands. The licensee can then include the block in their SoC, connect it to their own CPU and memory controller, and have it manufactured.

The IP licensing chain: the architecture creator hands over hardware description files under contract; the SoC designer integrates the block with their own components; the foundry manufactures the final chip without needing to understand what any block does.

The Trust Problem — and How It's Solved

This raises an obvious question: why would NVIDIA (or ARM, or Imagination Technologies) hand over the detailed specification of something worth billions of dollars to a competitor or partner? Once the RTL files leave NVIDIA's servers and land on Apple's engineering systems, what stops Apple from copying it, modifying it, or using it in unlicensed ways?

The answer is: a layered system of trust mechanisms, none of which is foolproof on its own, but which together create enough friction and consequence that violation becomes extremely unattractive.

1
Non-Disclosure Agreements (NDAs). Every engineer at the licensee who touches the IP files signs an NDA. These are personal legal obligations, not just corporate ones. Violation exposes the individual to civil and potentially criminal liability. At a company like Apple with thousands of employees and strict internal access controls, the NDA is enforced at the file-access level — only engineers who need the IP can open it.
2
Contract law. The license agreement between the two companies is a binding legal contract that specifies exactly what the licensee can and cannot do. It defines field-of-use (which product categories the IP can go into), geographic scope, and what happens in case of breach. Violating it is grounds for immediate termination of the license, loss of the right to ship products using the architecture, and damages claims.
3
Patent protection. The GPU architecture is protected by hundreds — sometimes thousands — of patents. Even if someone obtained the trade secret specification through legitimate channels and then misused it, the underlying patented techniques give the IP owner a separate, powerful legal hammer. Patent litigation in the US and internationally has produced multi-billion dollar verdicts.
4
Trade secret law. In most jurisdictions, a trade secret is protected as long as the owner takes "reasonable measures" to keep it secret. The NDAs, access controls, and audit rights in the license agreement are precisely those "reasonable measures." Misappropriating a trade secret — even by an employee who had legitimate access — is a federal crime in the US under the Defend Trade Secrets Act.
5
Audit rights. The license typically grants the IP owner the right to inspect the licensee's use of the technology — through third-party audits or design reviews. This is a credible deterrent: if you misuse the IP, the audit process creates the opportunity for detection.
6
Mutual dependency. At the highest tier, the relationship is reciprocal. Apple needs ARM's instruction set architecture. ARM needs Apple to be a flagship customer. Qualcomm needs NVIDIA compute IP for specific products. NVIDIA needs the foundry relationships that Qualcomm and Apple have cultivated with TSMC. Breaking trust in one relationship can cascade through all of them — and everyone in the industry knows it.
The courts as infrastructure: What ultimately makes all of this function is not any individual agreement, but the existence of a legal system that enforces them. Companies share trade secrets with each other because they trust that courts — US federal courts, UK courts, South Korean courts — will enforce the contracts and impose real consequences for violation. The semiconductor IP licensing ecosystem is, at its foundation, a bet on the rule of law.

ARM: The Purest Example of This System at Scale

The company that most clearly illustrates IP licensing in mobile is ARM Holdings (now owned by SoftBank, publicly traded again since 2023). ARM does not manufacture chips. ARM does not sell chips. ARM designs processor architectures and licenses them.

Every mobile processor in every iPhone, Android phone, and tablet in the world runs on ARM's instruction set architecture (ISA). Apple, Qualcomm, Samsung, MediaTek, Amazon, Google — all of them license the ARM ISA. Some (like Apple) go further and design their own microarchitectures that implement the ISA. Others license ARM's complete core designs. Either way, ARM receives a license fee and royalty on every chip shipped.

ARM ISA License

The licensee can design their own processor that runs ARM instruction set code. Apple uses this tier — the M-series and A-series CPUs are Apple-designed but ARM-ISA-compatible. Apple engineers designed the Firestorm core from scratch; they just made it speak ARM's language.

ARM Core License

The licensee takes ARM's complete, verified, tested core design (e.g., Cortex-A78) and integrates it into their SoC. Qualcomm, MediaTek, and many others use this tier for efficiency cores, embedded controllers, and specialized processors where designing a custom core is not worth the investment.

Why This Works — and When It Doesn't

The system works because the economic incentives are well-aligned. It is cheaper for Apple to license ARM's ISA and design their own cores than to invent a new instruction set. It is cheaper for ARM to license rather than manufacture. The license fee and royalty structure makes both parties better off than alternatives.

The system breaks down — occasionally — in ways that are instructive. In 2017, Apple announced it would stop using Imagination Technologies' PowerVR GPU architecture and design its own. Imagination Technologies claimed Apple's replacement GPU was infringing its patents. The resulting legal dispute lasted years and nearly bankrupted Imagination. Eventually Apple acquired a license and the dispute settled — but the episode illustrated both the power of patent protection and the hazard of over-dependence on a single customer.

In 2022, Qualcomm acquired Nuvia (a chip startup founded by former Apple chip architects) and began designing its own ARM-ISA CPU cores. ARM sued, arguing that Nuvia's ARM license did not transfer to Qualcomm and that Qualcomm needed to renegotiate terms. The case is ongoing as of 2025 — a reminder that even the most established licensing relationships involve genuine legal complexity.

The limits of trust: The IP licensing system does not assume perfect trust — it assumes calculated self-interest. Companies share design secrets because the legal, reputational, and business consequences of misuse are worse than the short-term gain. Courts, contracts, and audits are not there because the parties trust each other unconditionally. They are there because unconditional trust is not required — only enough deterrence to make cooperation the rational choice.

Chapter 6 — What This Means for You as a Developer

6
Developer Implications

Understanding the SoC architecture and the IP ecosystem that underlies it has several practical implications for how you think about mobile development.

✦
Performance characteristics differ by hardware block. The CPU runs your JavaScript. The GPU handles animation and rendering. The NPU runs ML inference. If your app is slow, it matters a great deal which block is the bottleneck — and different blocks require different optimization strategies.
✦
The abstraction layers protect you — and hide things from you. You never talk to the GPU directly. You call Animated.timing() and the platform decides whether to use the CPU or GPU. This is mostly a gift — but it also means that understanding why something is fast or slow requires knowing which layer is doing the work.
✦
API availability is determined by hardware availability. When Apple adds a new sensor or hardware block to an A-series chip, it takes time for iOS APIs to expose it, for React Native / Expo SDK to wrap those APIs, and for your app to call them. The SoC is always ahead of the software stack.
✦
Cross-platform means navigating multiple SoC ecosystems. iOS runs on Apple SoCs. Android runs on Qualcomm Snapdragon, Samsung Exynos, MediaTek Dimensity, Google Tensor, and others — each with different GPU architectures, different NPU capabilities, and different camera pipeline implementations. React Native's abstraction layer papers over most of this, but the differences surface in performance, camera behavior, and ML feature availability.
The big picture: Every mobile app runs on hardware that is the product of a global ecosystem of IP licensing, contract law, and trust — connecting ARM Holdings in Cambridge, TSMC's fabs in Taiwan, Apple's chip designers in Silicon Valley, and developers writing JavaScript in Boise. The physical chip in your phone is a compressed artifact of that entire system. When your code executes, it executes on all of that history.
