Hooks and State Variables

useState, useEffect, and useRef in Depth
🪝 React Hooks model 🔄 Re-render cycle ⚡ Side effects & cleanup 📌 Refs without re-renders

Before hooks, sharing stateful logic between React components required awkward patterns — class components, higher-order components, render props. Hooks are a clean solution: ordinary JavaScript functions that let function components opt into React features. This lecture goes beyond the API surface into why each hook behaves the way it does, so you can use all three confidently without fighting the framework.

Where this fits: You've already used useState to build a counter and a todo list. This document explains the underlying model — why calling setState schedules a re-render instead of mutating immediately, what the dependency array in useEffect actually controls, and when useRef is the right tool instead of state.

1. What Are Hooks?

Hooks are functions whose names start with use. When React calls your component function, any hooks inside it run in the same order, every time. React tracks that order internally — it's how it knows which state belongs to which hook call across re-renders.

Think of hooks as a registration system. The first time your component renders, each useState call reserves a slot in React's memory for that component instance. On every subsequent render, React fills those slots back in — in order — with the current values. This is why the rules of hooks exist: the order must be stable.

2. useState — Reactive Variables

useState takes one argument — the initial value — and returns an array of exactly two things: the current value and a setter function. Destructuring gives them readable names. 

import { useState } from 'react';

function Counter() {
  // count starts at 0; setCount is how we change it
  const [count, setCount] = useState(0);

  return (
    <View>
      <Text>{count}</Text>
      <Button title="+" onPress={() => setCount(count + 1)} />
    </View>
  );
}

Calling the setter correctly

When you call setCount(count + 1), you're not mutating count in place. You're telling React: "next time you render this component, give me count + 1 for this slot." React schedules the re-render, and when it runs, the component function executes again with the new value already in place.

⚠ The stale closure trap: If you call the setter multiple times in the same event handler using the current value, only the last call wins — because all three closures captured the same count snapshot. Use the updater function form to avoid this: 
// ❌ All three use the same stale snapshot of count
setCount(count + 1);
setCount(count + 1);
setCount(count + 1); // result: count + 1, not count + 3

// ✅ Each call receives the latest pending value
setCount(prev => prev + 1);
setCount(prev => prev + 1);
setCount(prev => prev + 1); // result: count + 3 ✓

State with objects and arrays

State can be any JavaScript value — strings, numbers, booleans, objects, or arrays. The key rule is that React uses reference equality to detect changes. Mutating an object in place won't trigger a re-render because the reference stays the same. You must always produce a new value. 

const [user, setUser] = useState({ name: 'Alex', score: 0 });

// ❌ Mutating in place — React won't notice
user.score = 10;
setUser(user); // same reference → no re-render

// ✅ Spread to create a new object
setUser({ ...user, score: 10 }); // new reference → re-render ✓

// ✅ Adding to an array
const [items, setItems] = useState([]);
setItems(prev => [...prev, newItem]); // spread, don't push()

3. The Re-render Cycle

Understanding when and why React re-renders is the key to writing components that are both correct and efficient. The cycle has four predictable steps.

The React re-render cycle. Calling a state setter triggers a new render — React runs your component function again, fills hook slots in order, and commits only the UI nodes that changed.

Notice that step ③ runs your entire component function again. All local variables re-initialize. The only things that survive across renders are hook slots managed by React. This is why you cannot rely on a regular variable to remember a value between renders — you need a hook.

✅ Persists across renders

  • State via useState
  • Refs via useRef
  • Context values via useContext
  • Memoized computations via useMemo

❌ Resets on every render

  • Regular const/let variables
  • Inline objects and arrays {} []
  • Function definitions created inside the component
  • Computed values not wrapped in useMemo

4. The Rules of Hooks

React's hook tracking depends on call order being identical every render. Two rules make this possible — and violating either produces bugs that are hard to diagnose.

Rule 1 — Only call hooks at the top level. Never call hooks inside loops, conditionals, or nested functions. If you need to conditionally use state, put the condition inside the hook callback, not around the hook call.
Rule 2 — Only call hooks from React function components or custom hooks. Never call them from regular JavaScript functions, class methods, or event handlers outside a component. 
// ❌ Breaks Rule 1 — hook inside a conditional
if (isLoggedIn) {
  const [profile, setProfile] = useState(null); // slot count changes!
}

// ✅ Correct — hook at top level, condition inside
const [profile, setProfile] = useState(null);
const displayName = isLoggedIn ? profile?.name : 'Guest';
💡 The ESLint plugin: Install eslint-plugin-react-hooks in your project. It catches both rule violations automatically and even catches stale dependency arrays in useEffect. It's the fastest way to learn the rules.

5. useEffect — Managing Side Effects

A "side effect" is anything your component does that reaches outside its own render output: fetching data from an API, subscribing to a sensor, setting a timer, reading from a database, logging to analytics. React's rendering is supposed to be a pure function of props and state — side effects live in useEffect. 

import { useState, useEffect } from 'react';

function WeatherWidget({ city }) {
  const [temp, setTemp] = useState(null);

  useEffect(() => {
    // This runs after render, not during
    fetch(`https://api.weather.example/${city}`)
      .then(r => r.json())
      .then(data => setTemp(data.temp));
  }, [city]); // re-run whenever city changes

  if (temp === null) return <Text>Loading…</Text>;
  return <Text>{city}: {temp}°</Text>;
}

The dependency array

The second argument to useEffect is the dependency array. It controls when the effect re-runs after renders. This is one of the most important things to get right in React Native development.

useEffect(fn)

No array at all. Runs after every single render. Usually a mistake — can cause infinite loops if the effect updates state.

useEffect(fn, [])

Empty array. Runs once — after the first render only. Equivalent to "on mount." Great for initial data fetches.

useEffect(fn, [a, b])

Values listed. Runs after the first render, then again whenever a or b changes. The most precise form.

⚠ Include every value the effect uses. If your effect reads a prop or state variable but you don't list it in the array, the effect captures a stale version of that value. The ESLint plugin will warn you about this. When in doubt, list it.
useEffect lifecycle. The cleanup function (if provided) runs before each new effect execution and on unmount — preventing resource leaks.

The cleanup function

If your effect sets up a subscription, timer, or event listener, it must tear it down when the component unmounts or before the effect re-runs. Return a function from your effect to register the cleanup. 

import { Accelerometer } from 'expo-sensors';

useEffect(() => {
  // Start the subscription
  const subscription = Accelerometer.addListener(data => {
    setAccelData(data);
  });

  // Return cleanup — runs on unmount or before next effect
  return () => {
    subscription.remove();
  };
}, []); // empty array: subscribe once on mount, unsubscribe on unmount
💡 A rule of thumb: If you start something in an effect (timer, subscription, connection), you almost always need to stop it in a cleanup function. Missing cleanups are a leading cause of memory leaks and unpredictable behavior in React Native apps.

Async effects and data fetching

useEffect cannot be an async function directly — but you can define and immediately call an async function inside it. 

useEffect(() => {
  // Define async function inside the effect
  async function loadData() {
    try {
      const response = await fetch('https://api.example.com/data');
      const json = await response.json();
      setData(json);
    } catch (err) {
      setError(err.message);
    }
  }
  loadData(); // call it immediately

  // No cleanup needed here — fetch can't be cancelled this way
  // (for cancellable fetches, use AbortController)
}, [userId]); // re-fetch when userId changes

6. useRef — Persistence Without Re-renders

useRef returns a stable object with a single property: .current. You can read and write .current freely — React will never re-render because of it. The ref object itself is the same object every render; only its .current contents change. 

const renderCount = useRef(0);

useEffect(() => {
  renderCount.current += 1;
  // Updating renderCount.current does NOT cause a re-render
});

This makes useRef the right choice in two distinct scenarios:

📎 Scenario A: DOM / Component Reference

Pass a ref as the ref prop of a component to get a direct handle to the underlying native view — useful for programmatic focus, scrolling, or measuring layout.

🗃 Scenario B: Mutable Instance Variable

Store a value that must persist across renders but should not trigger UI updates: timer IDs, previous prop values, subscription handles, accumulated measurements.

// Scenario A — Focus a TextInput programmatically
const inputRef = useRef(null);

return (
  <View>
    <TextInput ref={inputRef} placeholder="Search…" />
    <Button
      title="Focus"
      onPress={() => inputRef.current?.focus()}
    />
  </View>
);

// Scenario B — Store a timer ID across renders
const timerRef = useRef(null);

function startTimer() {
  timerRef.current = setInterval(() => {
    doWork();
  }, 1000);
}

function stopTimer() {
  clearInterval(timerRef.current);
}

7. Choosing the Right Tool

The three hooks cover distinct scenarios. When you're not sure which to reach for, ask these questions in order.

Question If yes → use Why
Does changing this value need to update the UI? useState State changes trigger re-renders; refs and variables don't.
Do I need to run code after render (fetch, subscribe, timer)? useEffect Effects run after the component renders, not during. Side effects here don't block the UI.
Do I need a value that persists but must not trigger a re-render? useRef Refs are mutable containers with stable identity across renders.
Is the value derived from state or props with no side effects? Regular variable Compute it inside the render function — no hook needed.

8. Putting It Together — A Complete Pattern

Real components use multiple hooks together. Here's a component that fetches a list, stores it in state, runs the fetch in an effect with cleanup for race conditions, and tracks a selected item with a ref to avoid unnecessary re-renders. 

import { useState, useEffect, useRef } from 'react';
import { View, Text, FlatList, ActivityIndicator } from 'react-native';

export function PokemonList() {
  // UI-driving state → useState
  const [pokemon, setPokemon] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError]   = useState(null);

  // Non-UI value → useRef (tracks if component is still mounted)
  const isMounted = useRef(true);

  useEffect(() => {
    async function load() {
      try {
        const res  = await fetch('https://pokeapi.co/api/v2/pokemon?limit=20');
        const data = await res.json();
        // Only update state if still mounted
        if (isMounted.current) setPokemon(data.results);
      } catch (e) {
        if (isMounted.current) setError(e.message);
      } finally {
        if (isMounted.current) setLoading(false);
      }
    }
    load();

    // Cleanup: mark as unmounted so async callbacks are ignored
    return () => { isMounted.current = false; };
  }, []); // run once on mount

  if (loading) return <ActivityIndicator size="large" />;
  if (error)   return <Text>Error: {error}</Text>;

  return (
    <FlatList
      data={pokemon}
      keyExtractor={item => item.name}
      renderItem={({ item }) => <Text>{item.name}</Text>}
    />
  );
}
The isMounted pattern: The isMounted ref guards against "Can't perform a React state update on an unmounted component" — a common warning when async operations complete after the user has already navigated away. Writing to isMounted.current in the cleanup function doesn't cause a re-render, which is exactly what you want.

9. Quick Reference

Hook Signature When it runs Triggers re-render?
useState const [val, setVal] = useState(init) Setter called from event handler or effect ✅ Yes — on every setter call with a new value
useEffect useEffect(fn, deps) After render, when deps have changed Only if the effect calls a setter
useRef const ref = useRef(init) Read/write ref.current any time ❌ Never — ref changes are invisible to React
Dependency array form Effect runs when… Typical use
omitted After every render Rarely — usually a bug
[] Once, after mount Initial data load, subscriptions
[a, b] After mount + whenever a or b changes Re-fetching when a filter or ID changes
💡 Custom hooks: Any function starting with use that calls other hooks is a custom hook. They let you extract and reuse stateful logic — a useLocation that wraps the Expo Location subscription, a useFetch that abstracts the loading/error/data pattern. You'll write your first custom hook when the same three-line useEffect pattern appears in three different components.
