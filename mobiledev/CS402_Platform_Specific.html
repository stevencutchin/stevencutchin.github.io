Building for Specific Platforms

From Inline Branching to True Native Modules
🔀 Platform.OS detection ⚙️ Platform.select() 📁 .ios.js / .android.js files 🔌 Native modules 🛡 Fallback stubs

React Native's promise is "write once, run on both platforms." That promise is mostly true — but not entirely. iOS and Android differ in permissions models, visual conventions, hardware capabilities, and the APIs that expose them. This lecture covers the full toolkit for handling those differences: from a one-line conditional inside a single file all the way to loading completely different native modules per platform.

Two strategies, one principle: Whether you're adjusting a shadow style or loading a biometric authentication module, the underlying approach is the same — detect the platform, provide the appropriate implementation, always have a fallback for the other side. The difference between the two strategies in this lecture is scope, not philosophy.

1. The Detection Spectrum

Platform handling in React Native exists on a spectrum from lightweight to heavyweight. Knowing where your problem sits on that spectrum tells you which tool to reach for.

Lightest
Platform.OS
Inline conditional in a single component. One file, a few different values.
Structural
Platform.select()
Multiple values chosen at once. Style objects, config objects, prop sets.
File-level
.ios.js / .android.js
Separate files per platform. Metro bundler picks the right one automatically.
Heaviest
Native Modules
Truly platform-specific code: Objective-C/Swift or Kotlin/Java behind a JS API.
Your JavaScript sits above two distinct native runtimes. Platform detection is the bridge — it lets your code choose the right path to each runtime at the right level of granularity.
1
Part One
Inline Platform Handling
Platform.OS · Platform.select() · Platform.Version · Permissions · StyleSheet differences

The majority of platform differences you encounter day-to-day don't require separate files or native modules. They're handled right where the difference occurs — inside a single component or style sheet — using React Native's built-in Platform API.

2. Platform.OS — The Simplest Branch

Platform.OS is a string: 'ios', 'android', or 'web' (in an Expo web build). It's evaluated at runtime and never changes during a session. Use it for simple conditional logic anywhere in your code. 

import { Platform } from 'react-native';

// Inline in JSX
const label = Platform.OS === 'ios' ? 'Done' : 'OK';

// Inline style value
const statusBarHeight = Platform.OS === 'ios' ? 44 : 24;

// Inline behavior
async function requestLocation() {
  if (Platform.OS === 'android' && Platform.Version < 23) {
    // Android < 6.0: permissions are granted at install time, not runtime
    return 'granted';
  }
  return await Location.requestForegroundPermissionsAsync();
}

Platform.Version

Platform.Version gives you the OS version — a number on Android (the API level, e.g. 33 for Android 13) and a string on iOS (e.g. "17.2"). This is useful for handling features or permissions that changed at a specific OS version.

🍎 iOS
Platform.Version → "17.2" (string)
Platform.isPad → true on iPad
Platform.isTV → true on Apple TV
Parse with parseInt(Platform.Version) for numeric comparison
🤖 Android
Platform.Version → 33 (number = API level)
API 23 = Android 6.0 (runtime permissions begin)
API 29 = Android 10 (scoped storage begins)
API 33 = Android 13 (granular media permissions)

3. Platform.select() — Choosing Multiple Values at Once

When you have several related values that differ by platform, Platform.select() is cleaner than writing multiple Platform.OS === checks. It takes an object with platform keys and returns the value for the current platform. The default key is used if no matching platform key is found. 

import { Platform, StyleSheet } from 'react-native';

const styles = StyleSheet.create({
  card: {
    borderRadius: 12,
    backgroundColor: '#fff',

    // iOS uses shadow* props; Android uses elevation
    ...Platform.select({
      ios: {
        shadowColor:   '#000',
        shadowOffset:  { width: 0, height: 2 },
        shadowOpacity: 0.12,
        shadowRadius:  6,
      },
      android: {
        elevation: 4,
      },
    }),
  },

  headerText: {
    fontSize:   18,
    fontWeight: '700',
    // Different system font stacks
    fontFamily: Platform.select({
      ios:     'San Francisco',
      android: 'Roboto',
      default: 'System',
    }),
  },
});
💡 Spread syntax + Platform.select() = clean style merging: The spread operator ...Platform.select({...}) inside a StyleSheet.create call is the idiomatic pattern for injecting platform-specific style properties into an otherwise shared style object. You write the shared styles once and inject only the differences.

4. Platform-Specific Permissions

Permissions are where iOS and Android diverge most sharply. The surfaces are different, the timing is different, and the granularity is different. Getting this right is not optional — a permissions crash on either platform is an immediate 1-star review.

🍎 iOS — Permission Model
  • Almost every sensitive resource requires a runtime permission prompt
  • The prompt shows exactly once — if denied, the user must go to Settings to re-enable
  • You must declare usage descriptions in app.json (NSLocationWhenInUseUsageDescription, etc.) — the app is rejected from the App Store without them
  • Photo library access is split: full access vs. add-only vs. limited selection (iOS 14+)
🤖 Android — Permission Model
  • Pre-API 23 (Android 5): all permissions granted at install time — never ask at runtime
  • API 23+ (Android 6+): dangerous permissions require runtime requests
  • API 29+ (Android 10): background location requires a separate follow-up permission after foreground is granted
  • API 33+ (Android 13): granular media permissions — READ_MEDIA_IMAGES instead of READ_EXTERNAL_STORAGE
  • Manifest declarations still required in app.json

Expo's permission packages (expo-location, expo-camera, expo-contacts, etc.) abstract most of this — they check the right permission for each platform internally. But when you need to write custom permission logic, the pattern is: check the current status first, request only if needed, handle every possible outcome explicitly. 

import * as MediaLibrary from 'expo-media-library';
import { Platform } from 'react-native';

async function requestPhotoAccess() {
  // Expo handles the platform difference internally,
  // but we still branch on version for the granular Android 13 case
  if (Platform.OS === 'android' && Platform.Version >= 33) {
    // Android 13+: READ_MEDIA_IMAGES is the correct permission
    const { status } = await MediaLibrary.requestPermissionsAsync();
    return status === 'granted';
  }

  // iOS and older Android: standard flow
  const { status } = await MediaLibrary.requestPermissionsAsync();

  if (status === 'denied' && Platform.OS === 'ios') {
    // iOS: once denied, guide user to Settings — can't re-prompt
    Alert.alert(
      'Photo Access Required',
      'Please enable photo access in Settings to continue.',
      [{ text: 'Open Settings', onPress: () => Linking.openURL('app-settings:') }]
    );
  }

  return status === 'granted';
}
⚠ iOS one-shot permission rule: On iOS, the permission dialog appears exactly once per permission type per app install. If you ask at the wrong moment (before the user understands why you need it) and they deny, they must navigate to Settings > [Your App] to re-enable it. Always ask for a permission at the moment it becomes contextually obvious — not at app launch.
2
Part Two
Platform-Native Modules
File-extension routing · Conditional imports · Native module wrappers · Fallback stubs

Some platform differences run too deep for an inline branch. When entire implementations differ — when you need to call Objective-C APIs on iOS and Kotlin APIs on Android — the solution is to write separate modules and let the build system pick the right one. React Native and Metro have first-class support for this pattern.

5. File-Extension Routing — Metro's Built-In Platform Switch

Metro, the JavaScript bundler used by React Native, resolves module imports with platform-specific file extensions before falling back to the plain .js version. This means you can create MyModule.ios.js and MyModule.android.js and import MyModule — Metro picks the correct file automatically. No runtime check. No conditional. The branching happens at build time.

Metro's platform extension resolution. A single import statement in your component resolves to the platform-correct file at build time — no runtime check required, no dead code shipped to either platform.

The file naming convention

File nameWhen Metro uses it
HapticEngine.ios.jsiOS builds only
HapticEngine.android.jsAndroid builds only
HapticEngine.native.jsBoth iOS and Android (not web)
HapticEngine.jsFallback — any platform with no specific file
HapticEngine.web.jsExpo web builds only
The fallback is your safety net. Always provide a plain .js (or .native.js) fallback. If Metro can't find a platform-specific file, it uses the fallback. If there is no fallback and no platform-specific file, the import fails. A fallback that exports a no-op or throws a descriptive error is far better than a silent crash.

A complete example — HapticEngine

iOS has a rich haptic feedback system (UIImpactFeedbackGenerator) with multiple impact styles. Android has vibration, which is less nuanced. Here's how to expose a unified API from both using the file-extension pattern:

HapticEngine.ios.js
import * as Haptics from 'expo-haptics';

// iOS: full haptic palette — light, medium, heavy, rigid, soft
export const impact = (style = 'medium') => {
  const styleMap = {
    light:  Haptics.ImpactFeedbackStyle.Light,
    medium: Haptics.ImpactFeedbackStyle.Medium,
    heavy:  Haptics.ImpactFeedbackStyle.Heavy,
  };
  return Haptics.impactAsync(styleMap[style] ?? Haptics.ImpactFeedbackStyle.Medium);
};

export const success = () => Haptics.notificationAsync(
  Haptics.NotificationFeedbackType.Success
);

export const error = () => Haptics.notificationAsync(
  Haptics.NotificationFeedbackType.Error
);
HapticEngine.android.js
import * as Haptics from 'expo-haptics';

// Android: vibration only — map haptic concepts to duration
export const impact = (style = 'medium') => {
  // expo-haptics wraps the Vibrator API on Android
  return Haptics.impactAsync(Haptics.ImpactFeedbackStyle.Medium);
};

export const success = () =>
  Haptics.notificationAsync(Haptics.NotificationFeedbackType.Success);

export const error = () =>
  Haptics.notificationAsync(Haptics.NotificationFeedbackType.Error);
MyButton.js ← uses HapticEngine on both platforms
import { impact } from './HapticEngine'; // ← Metro picks .ios or .android
import { TouchableOpacity, Text } from 'react-native';

export function MyButton({ label, onPress }) {
  return (
    <TouchableOpacity
      onPress={() => { impact('light'); onPress(); }}
    >
      <Text>{label}</Text>
    </TouchableOpacity>
  );
}

The consuming component MyButton knows nothing about the platform. It imports a unified interface and calls it. The module boundary is where the platform split lives — and only there.

6. Truly Native Modules — When JavaScript Isn't Enough

Some device capabilities have no JavaScript abstraction at all, or the available abstraction doesn't expose the feature you need. In these cases, you write — or more commonly install — a native module: a package that contains platform-native code (Swift/Objective-C for iOS, Kotlin/Java for Android) with a JavaScript API surface generated by Expo Modules API or the older React Native Bridge.

How native modules work

A native module exposes platform-native code through a JavaScript interface. The Expo Modules API generates the bridge layer — your JavaScript calls the same function name on both platforms; native code does the right thing on each.

Installing and using a native module

Most of the native capabilities you'll need are already wrapped by Expo or the community. Using them looks like any other npm package — but they require a native rebuild because they contain compiled code. 

# Install the package — it contains Swift and Kotlin code
npx expo install expo-local-authentication
import * as LocalAuthentication from 'expo-local-authentication';
import { Platform } from 'react-native';

async function authenticateUser() {
  // Check what biometrics are available on this device
  const types = await LocalAuthentication.supportedAuthenticationTypesAsync();
  const hasBiometric = types.length > 0;

  if (!hasBiometric) {
    // No biometric hardware — fall back to PIN/password
    return promptForPIN();
  }

  // Platform-specific prompt message
  const promptMsg = Platform.select({
    ios:     'Use Face ID to sign in',
    android: 'Use fingerprint to sign in',
    default: 'Authenticate to continue',
  });

  const result = await LocalAuthentication.authenticateAsync({
    promptMessage:         promptMsg,
    fallbackLabel:         'Use passcode',
    cancelLabel:           'Cancel',
    disableDeviceFallback: false, // allow PIN fallback
  });

  return result.success;
}

When you need to write your own native module

Writing a custom native module is necessary when you need a platform capability that no existing package exposes. The Expo Modules API (the modern approach) lets you write Swift and Kotlin code that is automatically exposed to JavaScript. This is an advanced topic covered briefly here for awareness — the full implementation is out of scope for CS 402 — but the pattern is worth understanding:

You write (Swift/Kotlin)

  • An Expo Module class with exported functions
  • Platform-native API calls
  • Event emitters for async callbacks
  • One file per platform (MyModule.swift + MyModuleModule.kt)

Expo generates (JavaScript)

  • A typed JavaScript interface from your module's exports
  • The bridge wiring to call native functions from JS
  • Event subscription helpers
  • TypeScript types if you use the typed API

7. The Fallback Stub — Graceful Degradation

Whenever you write platform-specific code, write a fallback for the other platform — even if that fallback does nothing. A no-op fallback is better than an import error. A descriptive console warning is better than a silent failure. An error with a clear message is better than a crash with a stack trace pointing at a missing module.

BiometricAuth.web.js ← stub for Expo web builds
// Biometric auth is not available on web — export a stub
// that behaves reasonably rather than crashing

export async function authenticateUser() {
  // Web has no biometric API — skip authentication
  console.warn('BiometricAuth: not available on web platform');
  return false;
}

export async function isBiometricAvailable() {
  return false;
}

The consuming code doesn't need to know the difference. It imports the module, calls authenticateUser(), checks the return value, and handles false — which is the same thing it would do if biometric hardware wasn't present on a real device.

💡 Design principle — program to an interface, not an implementation: This is the underlying principle that makes both platform branching strategies work. Your consuming components call a named function with known inputs and outputs. Whether the implementation behind that function is a Platform.select() branch, a .ios.js file, a native module, or a no-op stub — the consumer doesn't care. The platform split is an implementation detail, hidden behind a stable interface.
