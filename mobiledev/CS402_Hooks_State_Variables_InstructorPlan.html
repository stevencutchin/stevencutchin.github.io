Prerequisite Context

Students arriving at this lecture have already seen useState in a counter app (Lecture 10) and had a first look at useEffect and useRef briefly in Lecture 13. This session is the deep dive — explaining the underlying model so students stop cargo-culting the dependency array and start reasoning about it.

📝 Know your room: Do a quick show of hands — "who has hit a bug where their state update didn't seem to do anything, or their useEffect ran too many times?" Almost every hand will go up. Use this to frame the session as answering the question why — not just showing the API again.
1
What Are Hooks — The Mental Model
Concept introduction · No coding yet
0:00 – 0:10 · 10 min

Opening framing

  • Before hooks (2018), stateful logic required class components. Classes had their own problems: confusing this, lifecycle method bloat, difficult reuse of logic between components.
  • Hooks let function components opt into React features — state, side effects, context — without converting to classes.
  • The single biggest thing to internalize: hooks are called in the same order, every render. React tracks them by position — slot 1, slot 2, slot 3. The Rules exist to protect this.
🗣 Opening question (pairs, 60 seconds): "You have a variable inside a function. The function runs again. What happens to the variable?" → Answer: it resets. "So how does a component remember anything between renders?" → Lead into: React's hook slots.

The registration model (brief)

  • Draw (or gesture) a box for the component instance. Inside it, React maintains an ordered list of hook slots.
  • First render: hooks register — slot 1 gets useState(0), slot 2 gets useState(null), etc.
  • Every subsequent render: same function runs, same slots fill in order with current values.
  • If a hook is inside an if, the slot count changes → React panics → Rule 1 exists.
📝 Keep it short: Don't spend more than 10 minutes here. The mental model lands best when students have already seen the consequence of getting it wrong. Plant the seed now; reinforce it after the rules section.

useState — Live Coding
Interactive demo · Students follow along in Expo Snack
0:10 – 0:25 · 15 min

Demo sequence

Open a fresh Expo Snack. Build these three versions in sequence — don't jump ahead.

🖥 Demo 1 (3 min) — Basic counter: Start with useState(0), a Text showing the count, a Button incrementing it. Let students see a re-render happen. Ask: "What changed on screen?" → Only the number. React didn't redraw the whole component, just reconciled the diff.
🖥 Demo 2 (5 min) — Stale closure bug: Modify the Button to call setCount(count + 1) three times in a row. Show that the result is only +1. Ask students to predict before revealing. Then show the updater function form setCount(prev => prev + 1) called three times → result is +3. This moment usually produces audible surprise — let it land.
🖥 Demo 3 (5 min) — Object state: Change state to useState({ name: 'Alex', score: 0 }). Show the mutation bug: mutate and call setState(obj) → nothing happens. Then show the spread fix. Reinforce: React uses reference equality.
✅ Check for Understanding — 2 min
  1. What does calling a state setter actually do immediately? (Schedules a re-render — doesn't mutate the current value)
    → Common wrong answer: "updates the variable." Correct: queues a new render with the new value.
  2. You have an array in state. A student does items.push(newItem); setItems(items). Will the UI update? Why?
    → No. Same reference. React sees no change. Must spread: setItems([...items, newItem]).
📝 Pacing tip: If Demo 1 goes long, cut it — Demos 2 and 3 are the high-value moments. The stale closure bug is the thing students will encounter in their assignments. Make sure that one lands.
3
The Re-render Cycle and Rules of Hooks
Conceptual with diagram · No new coding
0:25 – 0:35 · 10 min

The four-step cycle (use the student document diagram)

  • Trigger: A setter is called, props change, or context changes.
  • Schedule: React batches the update — it doesn't re-render immediately. Multiple setters in one handler fire one re-render, not three.
  • Render: The component function runs again, top to bottom. Hook slots fill. Local variables reset.
  • Commit: React diffs the new output against the previous and patches only the changed native views.
🗣 Discussion prompt: "If I have a console.log('rendered') at the top of my component and a button that increments state, how many times does it log when I press the button?" → Once per press. Emphasize: the whole function runs again, but React only updates the UI diff.

Rules of Hooks (2 min)

  • Rule 1: Only call hooks at the top level — never inside if, loops, nested functions.
  • Rule 2: Only call hooks from function components or custom hooks.
  • Enforcement: the eslint-plugin-react-hooks catches violations before they become runtime bugs. Strongly recommend students install it. It also catches missing dependency array entries.
🚨 Common student violation: Wrapping a useState inside an if (Platform.OS === 'ios') to "only create state on iOS." This breaks the slot count for Android renders. The right move: always declare the state, then use it conditionally.
4
useEffect — Side Effects and Cleanup
Lecture + live coding · The most complex segment
0:35 – 0:55 · 20 min

Concept: why not just put fetch() in the component body?

  • Open with: "What if we just called fetch() at the top of the component, outside any hook?" → Fetches on every single render, including renders caused by the fetch completing. Infinite loop.
  • Side effects need a controlled execution environment. useEffect is that environment: runs after render, with explicit control over when to re-run.
🖥 Demo 4 (7 min) — Three dependency array forms: Build a component with a console.log('effect ran') and demonstrate all three forms back to back. Show the console after each button press. Students need to see the behavior difference before the explanation makes sense, not after.

Sequence: (1) no array → log every render, (2) empty array → log once, (3) [count] → log only when count changes.
🖥 Demo 5 (7 min) — Cleanup: Build a simple interval counter (increments every second). Show it working. Then navigate away and back — show the interval keeps running (check console). Now add the cleanup: return () => clearInterval(id). Navigate away — interval stops. This is the moment the cleanup function clicks.

Talking points for the dependency array

  • Think of the array as a watch list. "Re-run this effect if any of these values has changed since last time."
  • React compares array values using Object.is() — same rules as reference equality for objects.
  • Omitting a value from the array doesn't make the effect "not use it" — it makes the effect use a stale copy. That's the bug.
  • The ESLint rule exhaustive-deps catches this automatically. Trust it.
✅ Check for Understanding — 3 min
  1. You have useEffect(fn, []) and fn reads a prop called userId. What's the bug?
    → Effect runs once with the userId at mount time. If userId changes later, the effect still uses the old value. Fix: add userId to the array.
  2. When does the cleanup function run?
    → Before the next effect run (when deps change) AND when the component unmounts. Not during normal re-renders if deps haven't changed.
  3. Why can't the effect callback itself be async?
    → useEffect expects either a cleanup function or nothing from its callback. An async function returns a Promise, not a cleanup function. Define an async helper inside and call it immediately.
📝 This segment often runs long. If you're at 0:52 and haven't done Demo 5, skip directly to the CFU and make cleanup a talking point rather than a demo. The concept is more important than the live code here.
5
useRef — Mutable Values Without Re-renders
Contrast with useState · Two use cases
0:55 – 1:05 · 10 min

The contrast that makes it click

  • Start with the question: "What if I need to remember something across renders, but I do NOT want a re-render when it changes?" → That's useRef.
  • Classic example: a timer ID. You need to call clearInterval(id) later, but displaying the ID in the UI is meaningless — you don't want a re-render when it's set.
  • useRef returns a stable object: { current: initialValue }. The object reference never changes. Only .current changes.
🖥 Demo 6 (5 min) — Ref for focus: Build a screen with a TextInput and a "Focus" Button. Attach a ref to the input. inputRef.current?.focus() on press. Students immediately understand — they've wanted to do this in assignments and didn't know how.

The three-way decision

Draw this table or say it aloud:

NeedTool
Value that drives UIuseState
Value that persists, doesn't drive UIuseRef
Value that recomputes each renderRegular variable
📝 Don't over-explain useRef. Students rarely need it deeply in their first semester. The timer ID and focus use cases are enough. More advanced uses (storing previous state, animation values) can come up naturally in office hours or the final project.

Synthesis and Wrap-up
Put it together · Q&A · Exit ticket
1:05 – 1:15 · 10 min

The full pattern (PokéAPI demo)

Walk through the PokéAPI example from the student document — or paste it into Snack and run it. Don't type it live; just narrate the three parts:

  • useState for pokemon, loading, error — all three drive UI.
  • useRef for isMounted — persists across renders, doesn't drive UI, protects against the unmounted-component warning.
  • useEffect(fn, []) — runs once on mount, fetches data, sets state, cleans up by marking unmounted.
🏋 Exit Activity (5 min) — Individual or Pairs

Show students this buggy component. Ask them to identify all three problems: 

function BrokenComponent({ userId }) {
  if (userId) {
    const [data, setData] = useState(null); // Bug 1
  }
  useEffect(async () => {             // Bug 2
    const res = await fetch(`/users/${userId}`);
    setData(await res.json());
  }, []);                               // Bug 3
}

Expected answers: (1) Hook inside conditional — violates Rule 1. (2) Async effect callback — returns Promise, not cleanup. (3) Missing userId in dependency array — stale closure.

✅ Final Check for Understanding
  1. Name one thing that persists across renders and one thing that doesn't.
    → Persists: state, refs, context. Doesn't: regular variables, inline objects.
  2. What's the difference between a component unmounting and a component re-rendering?
    → Re-render: function runs again, same instance, hooks slots persist. Unmount: component removed from tree, cleanup runs, all state lost.

Common Student Mistakes

Mistake What goes wrong How to fix
Calling setItems(items.push(x)) push() returns the new length (a number), not the array. State becomes a number. setItems(prev => [...prev, x])
Mutating state object directly Same reference → React sees no change → no re-render. UI appears frozen. Always spread or copy: setUser({ ...user, name: 'new' })
Omitting a dependency from the array Effect reads a stale value — often manifests as "first fetch works, then it stops updating." Trust the ESLint warning. Add the missing dep. If it causes a loop, something else is wrong.
Making the effect callback async Returns a Promise instead of a cleanup function. React issues a warning; cleanup never runs. Define an async helper inside the effect and call it immediately.
Not returning a cleanup for subscriptions Listener accumulates on every re-render. Memory leaks; ghost subscribers fire on unmounted components. Always return a cleanup that calls .remove() or off().
Using useRef for values that should drive UI Value changes but screen doesn't update — ref changes are invisible to React. If the UI needs to show the value, it must be state.

Grading & Assessment Notes

When hooks appear in assignment submissions, the key things to credit generously are correct intent and pattern understanding. A student who writes a useEffect with the right structure but forgets one dependency has demonstrated mastery of the pattern — partial deduction at most.

  • Reserve full deductions for: hooks inside conditionals (Rule 1 violation), no cleanup on subscriptions (systematic absence, not one miss), using regular variables where state is clearly required.
  • Award full credit when: the three-hook structure is correct, loading/error/data states are handled, and the component does what the spec describes — even if the cleanup isn't perfect.
  • Stale dependency array entries: comment and note in feedback, but do not reduce points unless the bug causes the assignment's required behavior to fail.
📝 Grading philosophy reminder: The goal is that students can reason about the hook model. A student who can explain why the cleanup exists — even if they forgot to write it — understands the material. Give them credit for that understanding.
