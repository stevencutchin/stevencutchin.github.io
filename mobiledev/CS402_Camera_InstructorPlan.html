CS 402 — Mobile Application Development

Camera Integration and Media Handling
🕐 Total time: 75 minutes 📅 Semester: Spring 2026 📦 SDK: Expo SDK 52 📄 Student doc: CS402_Camera_Integration.html 🔜 Next class: Project 4 Camera App Introduction

Lesson Overview

This lecture introduces students to hardware-adjacent programming for the first time in the course. Camera access is a natural motivator — nearly every student has an idea that involves photos — so engagement is usually high. The primary challenge is the three-layer mental model: their JS code → Expo SDK → OS native APIs. Students who miss this tend to cargo-cult permission code without understanding why it's structured the way it is.

The second challenge is the permission system. This is often students' first encounter with a gated resource that has a persistent state machine. Plan for confusion here and budget extra time for questions.

Learning Objectives

Understand
Explain why camera access in a mobile app requires permissions and how the OS permission state machine works.
Distinguish
Choose between expo-camera and expo-image-picker for a given use case.
Apply
Write a correct permission request pattern using requestCameraPermissionsAsync().
Apply
Capture a photo using launchCameraAsync() and display it with <Image>.
Analyse
Identify why a captured photo is temporary and choose the correct approach to persist it.
Apply
Configure app.json with the plugin entries required for a production camera build.

Prerequisites Students Should Have

  • useEffect and useState hooks — permission state depends on both
  • Basic async/await — all camera APIs are asynchronous
  • React Native's <Image> component and the source={{ uri }} prop
  • The concept of useRef is helpful but will be introduced during the lesson
⚙️ Prep required before class: Install expo-camera and expo-image-picker in a working Expo project on your own device ahead of time. You'll live-demo from this project. Test the camera on a physical device — the Expo Go simulator camera is unreliable and will cause demo failures. Have a second backup device or a colleague's phone available in case your device misbehaves.

1
Hook and Framing
Why the camera is the most interesting sensor on the phone
⏱ 6 min

Open with a quick show-of-hands or cold call rather than jumping straight into slides. This gets students present and sets up the motivation naturally.

💬 Opening question (cold call 2–3 students): "Think about the last app you used that accessed your camera or photos. What did it do with that image — display it, upload it, process it? Did it ask your permission first? Did you grant it?"
  • The camera isn't just a sensor — it's an output device for user creativity. It motivates nearly every app idea students pitch in project proposals.
  • Unlike GPS or accelerometer data, camera output is user-created content. This raises both technical and privacy stakes: the OS is extremely protective of it.
  • Today is the first time in this course they'll write code that touches hardware. That's a milestone worth naming. The permission system they learn today applies to microphone, location, contacts — everything sensitive on the device.
  • The goal for the session: walk out knowing how to get a photo from the camera into their app and make it persistent. Everything else is detail around that.
📝 Tone note: Keep this opener energetic. Students who have been building list apps or forms for weeks are usually visibly more excited when camera content comes up. Lean into that energy.
2
The Three-Layer Mental Model
Your JS → Expo SDK → OS native APIs
⏱ 12 min

Draw the architecture diagram from the student document on the board before showing the SVG version. Hand-drawing it anchors the concept better than projecting a finished diagram.

Talking Points

  • Sketch three horizontal bands on the board: "Your App," "Expo SDK," "iOS / Android OS." Ask students: where does launchCameraAsync() live? (Expo SDK band.) Where does the actual camera hardware interface live? (OS band.)
  • Emphasize: Expo gives you a JavaScript interface to native code. The same call works on iOS and Android because Expo's native modules translate it into AVFoundation calls on iOS and Camera2 calls on Android. This is the entire value proposition of a cross-platform framework for hardware features.
  • Explain why the three packages exist separately (expo-camera, expo-image-picker, expo-media-library): they're different entry points into different OS capabilities, not overlapping tools for the same thing.
  • Draw the comparison: expo-image-picker is like handing a task to the OS and getting back a receipt. expo-camera is like building your own camera app inside a window in your UI.
Analogize to fetch(): Students understand that fetch() doesn't actually open a TCP connection in their JS — it delegates to the browser's networking layer. Camera APIs work the same way. Their JS delegates to the OS; the OS does the real work.

Package Selection — The Decision Rule

Give students this simple heuristic now; they'll use it in Project 4:

If you need…Use
A photo or video with no custom camera UI — just get me a fileexpo-image-picker ✅ easiest
The camera preview embedded in your own layout (QR scanner, AR overlay, real-time filter)expo-camera
To save a photo to the device's photo galleryexpo-media-library
To let the user browse their existing photosexpo-image-picker (library mode)
📝 Common confusion to pre-empt: Students often think they need expo-camera for everything. Make the point explicitly: for Project 4, expo-image-picker is probably all they need. expo-camera is for when they need to customize the capture experience.
✅ Check for Understanding
  1. "You're building a recipe app. Users tap a button, take a photo of their dish, and the photo appears in their recipe entry. Which package?"
    → expo-image-picker. They don't need custom camera UI.
  2. "You're building a QR code scanner that shows a live viewfinder with a red rectangle overlaid on detected codes. Which package?"
    → expo-camera. They need the live preview embedded in their layout with JSX overlaid.
3
Permissions — The Gate You Must Open First
The state machine, the request pattern, and what "denied" means
⏱ 13 min

This is the segment where students most often get tripped up. Go slowly. The permission state machine is a new concept for most of them, and the consequences of getting it wrong (silent failures, crashes on production builds) are severe.

Talking Points

  • Every sensitive resource on a mobile device — camera, microphone, location, contacts — is gated by the OS. Your app does not have access by default. This is a security model, not a bug.
  • Walk through the four permission states: undetermined (first run — dialog hasn't been shown), granted, denied (user said no), restricted (device policy blocks it, e.g. parental controls or MDM).
  • Critical distinction: once a user denies a permission, you cannot show the system dialog again. Your app must direct them to device Settings. This is an intentional OS design to prevent apps from badgering users. On iOS in particular, there is no programmatic way to re-trigger the dialog.
  • Walk through the permission request pattern step-by-step: (1) check status, (2) request if undetermined, (3) branch on the result. Never assume granted.
  • Three separate permissions exist for camera work: camera, microphone (video with audio), media library. Each must be requested separately.
🖥️ Live demo — show the permission dialog on a real device: Call requestCameraPermissionsAsync() and deliberately show both paths: first run the app and grant the permission (students see the system dialog), then reset the app permission in device Settings and demo what happens when it's denied (the API returns status: 'denied' and shows no dialog). This is more memorable than any diagram.

The Permission Pattern — Code Walkthrough

Walk through this code live (not just show it on a slide). Type it out or narrate while students follow in their student docs.

const { status } = await ImagePicker.requestCameraPermissionsAsync();

if (status === 'granted') {
  // proceed with camera work
} else if (status === 'denied') {
  // inform user, direct to Settings
  Alert.alert('Permission Required', 'Go to Settings to enable camera access.');
}
📝 Ask before moving on: "Why do we check status === 'granted' rather than just calling the camera API and catching an error?" Let students reason through it. The answer is UX: error-catching would work, but the user experience of failing silently or showing a cryptic error is much worse than proactively showing a helpful message.

Requesting at the Right Moment

  • Don't request permissions in useEffect on app startup. Request them at the moment of need — when the user taps "Take Photo."
  • iOS App Store review specifically looks for apps that request permissions without clear justification. Requesting camera on launch with no explanation is a rejection reason.
  • The permission string in app.json is what users read in the system dialog. "Access camera" → rejected by Apple. "Capture photos to attach to your recipe entries" → approved.
⚠️ Production trap: Permission dialogs work in Expo Go without any app.json configuration because Expo Go already declares all permissions. The trap is: when students build their standalone app, it will crash on permission requests if the app.json plugin entries are missing. This bites almost every student on their first production build. Emphasize this clearly now and again at the end of class.
✅ Check for Understanding
  1. "A user denies camera access. Your app calls requestCameraPermissionsAsync() again. What happens?"
    → Returns { status: 'denied' } immediately, no dialog shown. Cannot re-prompt.
  2. "Where in your component lifecycle should you request camera permission?"
    → At the moment the user invokes a camera action, not on app startup.
4
expo-image-picker — The Easy Path
launchCameraAsync, launchImageLibraryAsync, and the asset object
⏱ 15 min

This is the segment most students will use for Project 4. Give it the most code time. The live demo here should produce a visible result — a photo appearing on screen — so students have something concrete to replicate.

Talking Points

  • expo-image-picker offloads the entire capture experience to the OS. The system camera launches, the user takes a photo, the OS dismisses the camera, and your app receives a result object. Your app never renders a camera preview.
  • Two entry points: launchCameraAsync() (opens camera), launchImageLibraryAsync() (opens photo picker). Both return the same shape of result object.
  • The result can be canceled: true — always check this before accessing result.assets.
  • Point out the SDK 48 change explicitly: old code used result.uri. New code uses result.assets[0].uri. Many tutorials online still use the old pattern. If their code gets a uri of undefined, this is why.
🖥️ Live demo — full working example: Build the following in front of the class, narrating as you type. Show it working on a physical device. Students should see the camera launch, the photo get captured, and the image appear in the app. 
import { useState } from 'react';
import { View, Button, Image, StyleSheet } from 'react-native';
import * as ImagePicker from 'expo-image-picker';

export default function App() {
  const [imageUri, setImageUri] = useState(null);

  const openCamera = async () => {
    const { status } = await ImagePicker.requestCameraPermissionsAsync();
    if (status !== 'granted') return;

    const result = await ImagePicker.launchCameraAsync({ quality: 0.8 });
    if (!result.canceled) {
      setImageUri(result.assets[0].uri);
    }
  };

  return (
    <View style={styles.container}>
      <Button title="Take Photo" onPress={openCamera} />
      {imageUri && <Image source={{ uri: imageUri }} style={styles.preview} />}
    </View>
  );
}

const styles = StyleSheet.create({
  container: { flex: 1, alignItems: 'center', justifyContent: 'center' },
  preview:   { width: 300, height: 300, marginTop: 20, borderRadius: 8 },
});

Options Worth Knowing

OptionDefaultNotes
quality1.00–1. Use 0.7–0.85 for a good quality/size balance. 1.0 creates very large files.
allowsEditingfalseShows a crop UI after capture. Combine with aspect for fixed ratios.
mediaTypesImagesMediaTypeOptions.All allows video too. Use Videos for video only.
allowsMultipleSelectionfalseLibrary only. Lets users pick multiple photos at once.
exiffalseIncludes GPS, camera model, exposure data. Useful for location-tagged photo apps.

5
expo-camera — Embedded Camera Preview
CameraView, useRef, takePictureAsync, and when you actually need this
⏱ 12 min

Contextualize this package as "the harder path for when you need it." Most students won't use it for Project 4, but they need to understand what it offers and why useRef appears here.

Talking Points

  • Draw the phone UI on the board: a <CameraView> that fills the screen, with your own buttons and overlays on top. The live camera feed is a native render layer — you can't put JSX "inside" it, you can only stack your JSX above it with absolute positioning.
  • Introduce useRef if students haven't seen it: refs are for when you need to call an imperative command on a component instance — "do this now" — rather than controlling it through state. The camera needs to hear "take a picture now," not "be in picture-taking mode."
  • SDK 50 renamed <Camera> to <CameraView>. The old API is deprecated. Any tutorial using the old component should be treated with suspicion.
  • Use cases that actually warrant expo-camera: QR/barcode scanning (built-in with onBarcodeScanned), real-time overlays, custom shutter UI with animations, apps that need to hold the camera open between captures (like a photo booth).
🖥️ Optional demo — show the anatomy if time permits: If you have time, show <CameraView> running on the device so students can see the live preview embedded in the app. Even 60 seconds of showing it running makes the concept concrete. If pressed for time, the SVG diagram in the student document is sufficient.

The useCameraPermissions Hook

Note that expo-camera ships its own convenience hook, which is cleaner than the manual requestCameraPermissionsAsync() pattern when you're already using <CameraView>:

const [permission, requestPermission] = useCameraPermissions();

// permission.granted — boolean
// requestPermission() — call this to show the dialog
The key mental model for useRef: Tell students to think of a ref as a remote control for a component. You don't re-render the camera to take a photo; you press its shutter button directly. React state is for data that drives the UI. Refs are for directly commanding component instances.
✅ Check for Understanding
  1. "You want to build a barcode scanner. The scanner should be a persistent view in your app, not a full-screen modal. Which package?"
    → expo-camera with CameraView and onBarcodeScanned prop.
  2. "Why does takePictureAsync() need to be called on a ref rather than triggered by setting state?"
    → It's an imperative command, not a state change. There's no "is taking picture" state to toggle — it's a one-time action on the hardware.
6
Media Library and the URI Lifecycle
Temporary vs. permanent storage, saving to the gallery, uploading
⏱ 9 min

This is conceptually short but practically very important. Students who miss the "temp cache" nature of captured URIs will lose photos between app restarts and be very confused.

Talking Points

  • A file:// URI from launchCameraAsync() points to a temporary cache directory. It will survive as long as the app is running in the foreground. After a restart, it may be gone. Never store this URI in a database and assume you can reload it later.
  • There are three valid destinations for a captured photo: (1) display it immediately and discard, (2) save it to the device's permanent media library with createAssetAsync(), (3) upload it to a server. All three are legitimate depending on the app's needs.
  • Draw the URI lifecycle diagram on the board: temp cache → (choice) MediaLibrary asset / Server URL. The temp URI can fan out to either destination.
  • expo-media-library has its own permission (MEDIA_LIBRARY) that is separate from the camera permission. Students often forget to request this and wonder why createAssetAsync() fails silently.
  • iOS has an "Add Photos Only" permission that lets apps save to the library without reading from it. Consider this for apps that don't need to browse the user's existing photos — it's a narrower ask and users are more likely to grant it.
📝 Practical advice for Project 4: Tell students explicitly that for the assignment, saving to the MediaLibrary is the expected way to demonstrate persistence. If they just display the photo and don't save it, it won't be there when the grader reopens the app. This is worth saying very directly.
📱 Quick class exercise (optional, 3 min)
Have students open their own phone's Settings → Privacy → Camera. Ask: which apps have camera access? Which are denied? Did any of them surprise you? This makes the permission state machine tangible and sparks a brief conversation about permission UX design.
7
Pitfalls, app.json, Q&A, and Bridge to Project 4
The production trap, common mistakes, and what comes next
⏱ 8 min

The app.json Production Trap — Say It Again

Revisit this explicitly. It's important enough to mention twice.

  • Expo Go pre-declares every possible permission. Their code works fine in development. When they run eas build, only the permissions listed in app.json plugins get included in the native binary. Missing entries cause crashes.
  • Walk through the three plugin entries they'll need for Project 4: expo-image-picker, expo-camera (if used), expo-media-library.
  • Remind them about permission string quality — "Allow access to camera" is lazy and gets flagged. Write strings that explain the why.

Top Pitfalls Rapid-Fire (project the table from the student doc)

  • result.uri vs. result.assets[0].uri — if images aren't displaying, check this first.
  • Using base64: true on large photos — will freeze the app. Use URI + FormData for uploads.
  • Not handling result.canceled — will crash when user backs out of camera.
  • Displaying a file:// URI on a web view or in an email — URIs are device-local only.

Differentiation and Pacing Notes

If class is running ahead of schedule

  • Add a live demo of recordAsync() for video capture — extend the CameraView segment
  • Show FileSystem.uploadAsync() and how it differs from fetch for large uploads
  • Walk through EXIF data extraction from the asset object — useful for location-tagged photo apps
  • Demo onBarcodeScanned in CameraView — always gets a strong reaction

If class is running behind schedule

  • Cut the CameraView live demo — the student doc SVG is sufficient; defer to office hours
  • Abbreviate the MediaLibrary segment to just the key point: temp cache vs. permanent, and createAssetAsync()
  • Skip the class activity in Segment 6
  • Move the app.json walkthrough to the opening of the next class (Project 4 intro) where it's also directly relevant

Students who are ahead / finish early

  • Challenge: get the photo to upload to a public endpoint (e.g. a test API) using FormData
  • Challenge: add a FlatList that persists a gallery of captured photos using AsyncStorage for URIs + MediaLibrary for files
  • Explore: expo-camera barcode scanning

Students who are struggling

  • Simplify to just the permission check + launchCameraAsync() + display pattern. Everything else is optional once this works.
  • Pair with a stronger student for the demo — camera code is best learned by watching it run first
  • Recommend the student doc Section 10 (pitfalls table) as a debugging checklist
