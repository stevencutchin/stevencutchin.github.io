📋 Instructor Lesson Plan

CS 402 — Mobile Application Development

Building a Full Expo Project on Your Workstation — Leaving Snack Behind
🕐 Total time: 75 minutes 📅 Semester: Spring 2026 🎯 Mode: Live demo — students follow along on their own machines ⚠️ Platform: Windows (primary), macOS (secondary)
🚨 Read this before class: This is a live setup session. Things will go wrong on individual machines — that's expected and normal. Your job is to keep the class moving while handling gotchas without losing the room. The session is structured so that students who hit a blocker can pair with a neighbor who's already running while you help troubleshoot. Have your own laptop running a working project before class starts so you can demo the end state at any moment.

Learning Objectives

Apply
Install Node.js LTS, Git, and VS Code on a Windows workstation and verify each from the terminal.
Apply
Create a new Expo project with create-expo-app and open it in VS Code.
Apply
Start Metro with npx expo start and connect a physical device running Expo Go.
Understand
Explain the difference between Fast Refresh and a full reload, and when to use each.
Apply
Edit a source file, save, and observe the change appear on device without restarting.
Analyze
Diagnose and resolve the most common Windows-specific setup problems (firewall, path length, network isolation).

1
Framing — The Architecture Shift
Why we're leaving Snack · What changes · What stays the same
0:00 – 0:08 · 8 min

Opening

  • "Snack was the appetizer. It was great — nothing to install, instant feedback, worked from day one. But real apps don't live in browsers. They live on machines, in Git repos, with npm packages, connecting to real servers. Today we set that up."
  • Display the before/after architecture SVG from the student document. Walk it: Snack had Metro in the cloud. Now Metro runs on your laptop. Your phone and Expo Go don't change at all — same QR scan, same app. The only thing that moved is where the bundler lives.
  • "By the end of this class period you will have a real project on your hard drive, running on your phone, that you can take home, keep building, put in a GitHub repo, and eventually ship to the App Store. Snack can't do any of that."
🗣 Set expectations honestly: "Setup sessions are the one class where something will go wrong on someone's machine — guaranteed. That's not a sign you did anything wrong. Windows has a few rough edges with Node.js tooling. We have a whole section on those. If you hit a blocker, pair with your neighbor while I come around — don't sit stuck and silent."
📝 The "real meal" framing: Students who've been in Snack for several weeks often feel like they've been learning on training wheels. This session is the moment the wheels come off. Lean into that — it's motivating. By the end of today's class they will have something on their phone that they built from a blank folder on their own machine. That's a real milestone.
2
Install Node.js, Git, and VS Code
LTS only · Verify in terminal · Extensions · Windows gotchas
0:08 – 0:23 · 15 min

Node.js

  • "Go to nodejs.org. You'll see two buttons — LTS and Current. Click LTS. Always LTS. The Current version is for people who like living dangerously, which is not what we're doing today."
  • Run the installer with all defaults. When it finishes, open a new terminal and run node --version and npm --version. Both should respond.
  • Anticipate the PATH issue: "If it says 'command not found,' your terminal was open before the install changed PATH. Close it, open a new one. Try again."
🪟 Watch for — PowerShell execution policy: If npm commands fail immediately with a message about scripts being disabled, run this in Admin PowerShell before continuing. This is the #1 gotcha on fresh Windows installs:
Admin PowerShell
PS Admin> Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

Git

  • git-scm.com → Windows download. Two installer options to call out: PATH setting ("from command line and 3rd-party software") and line endings ("Checkout Windows-style, commit Unix-style"). Everything else: defaults.
  • After install: git config --global user.name and user.email. Use BSU email to match GitHub.

VS Code

  • code.visualstudio.com → System Installer. Check "Add to PATH" and "Open with Code" during install.
  • Extensions: ES7+ snippets, Prettier, ESLint, React Native Tools. Walk through installing them together — Extensions panel, search, install.
  • Set up Prettier as default formatter + format on save. Ctrl+, → search "format on save." This is quality of life that pays off every single day.
📝 Pace this section carefully: Students who've done dev setup before will fly through this. Students who haven't will need more time. Keep everyone moving by telling students who finish early to help their neighbors — peer support here is genuinely faster than waiting for you to reach every machine.
✅ Gate Check — everyone passes before moving on
  1. Run node --version in a terminal. Do you see a version number?
  2. Run git --version. Version number?
  3. Type code . in a folder. Does VS Code open?

Don't advance until most of the room passes all three. Students still stuck pair with someone who's done.

3
Create the Project and Connect the Phone
create-expo-app · project structure · npx expo start · QR scan
0:23 – 0:37 · 14 min

Create the project

  • Have students navigate to a Projects folder first — somewhere shallow like C:\Projects or C:\Users\name\Projects. Not Desktop, not Downloads. Avoid long paths before they even start.
  • npx create-expo-app@latest MyFirstApp — this takes 1–3 minutes. While it runs, walk through what it's doing: downloading the template, installing ~900 packages into node_modules, setting up the project structure.
  • When it finishes: cd MyFirstApp then code . to open VS Code.
🪟 Watch for — long path errors during install: If npm install fails partway with ENAMETOOLONG or a cryptic path error, the 260-char Windows limit is the cause. Guide the student to enable long paths (registry or Group Policy) and rerun. This is worth fixing now — it will happen again on every project otherwise.

Walk the project structure (3 min)

  • Display the file tree from the student document on the projector. Walk: app/ is where they live. node_modules/ they never touch. app.json is the app's identity document. package.json is the ingredient list.
  • "The app/ folder uses Expo Router — file names become route names. index.tsx is your home screen. We'll build on this heavily in future sessions."

Start Metro and connect the phone

  • npx expo start in the project folder (VS Code integrated terminal: Ctrl+`). Wait for the QR code to appear.
  • Android: Expo Go → Scan QR. iOS: Camera app → point at QR → tap banner.
  • Critical: same WiFi network. Call this out loudly. "If your phone is on cellular data and your laptop is on campus WiFi, this will not work. Same network."
  • Wait 15–20 seconds for the first bundle. Show the Metro terminal output as it builds — the first bundle is the slowest. "After this, changes are under 2 seconds."
🪟 Watch for — Windows Firewall popup: First run of Metro will trigger a Firewall permission dialog. Students must click "Allow access" — both Private and Public. If they accidentally clicked Block, guide them to Windows Security → Firewall → Allow an app through firewall → Node.js → check both boxes.
🪟 Watch for — campus WiFi isolation: BSU's campus WiFi may block device-to-device traffic. If phone and laptop are on the same network but still can't connect, switch to tunnel mode:
npx expo start --tunnel
Tunnel requires an Expo account. Have students run npx expo login first. A free account takes 60 seconds to create at expo.dev.
✅ Gate Check — the moment of truth

Ask the room: "How many people can see the Expo default screen on their phone?" Count hands. Students without a connection pair up — phone sharing is fine at this stage. The goal is that everyone has seen a locally-running project before the end of this segment, even if they're sharing with a neighbor.

4
The Development Loop
Fast Refresh · full reload · developer menu · Metro terminal keys
0:37 – 0:49 · 12 min

Fast Refresh — the magic moment

  • Open app/index.tsx in VS Code. Find any text string in the default template. Change it. Hit Ctrl+S. Watch the phone.
  • Give students 10 seconds to watch it update. Then: "That's Fast Refresh. You saved a file on your laptop. In under 2 seconds your phone shows the change. No compile step, no deploy, no QR scan. Just save."
  • Explain what Fast Refresh actually does: it re-evaluates only the changed module and swaps it into the running app, preserving state. It's not a full restart — a counter you were tapping keeps its value.
🖥 Demo — state preservation across Fast Refresh: Add a useState(0) counter with a button that increments it. Tap the button a few times to get to, say, 5. Then change the button label text in the source and save. The label updates on the phone — but the counter still shows 5. State was preserved. Students find this genuinely surprising and it makes the concept concrete.

When Fast Refresh isn't enough

  • Some changes break the Fast Refresh chain: changing a module-level variable, restructuring navigation, modifying a hook's initial value. In these cases, Fast Refresh silently fails or shows stale behavior.
  • Full reload: shake the phone → Reload. Or press r in the Metro terminal. Full reload clears all state and re-runs from scratch.
  • "If something looks wrong and you can't explain why — reload first. Nine times out of ten that's the answer."

The developer menu and Metro terminal

  • Shake the phone: show the developer menu. Walk the options: Reload, element inspector, performance monitor, JS debugger.
  • Metro terminal keyboard shortcuts: r reload, m toggle menu, a open Android emulator, i open iOS simulator (Mac only).
📝 The element inspector is underused: Spend 90 seconds on it. Shake phone → Toggle Element Inspector → tap any component on screen. It shows the component's styles, dimensions, and where it sits in the hierarchy. Students who discover this stop guessing why their layout looks wrong. It's the single best debugging tool for layout problems.
5
Windows Triage — Fixing What's Broken
Dedicated time to resolve remaining issues across the room
0:49 – 1:03 · 14 min

Reserve this segment explicitly for troubleshooting. By this point most students are running, but some will still have issues. Use the triage guide below to diagnose quickly without getting pulled into rabbit holes.

✅ Diagnose first — ask these

  • Does node --version work in a fresh terminal?
  • Phone and laptop on same WiFi?
  • Did Windows Firewall dialog appear? What did they click?
  • Did the npm install finish without errors?
  • Is Metro still running (terminal still open)?

🔧 Most common fixes

  • Execution policy → Admin PS, Set-ExecutionPolicy RemoteSigned
  • Firewall blocked → allow Node.js through Defender Firewall
  • Different networks → npx expo start --tunnel
  • Long path error → enable long paths in registry, rerun install
  • Antivirus stall → add Projects folder to Defender exclusions
📝 Triage strategy — keep the class moving: Students who are fully running should be doing the first-edit activity (Segment 6) while you triage. Don't hold the entire class for one machine. If a student has a deep Windows issue that can't be resolved in 5 minutes, pair them with a neighbor for the rest of class and schedule office hours. The important thing is they've seen the environment working — they can fix their own machine later with the student document as a guide.
Nuclear option — Windows Subsystem for Linux (WSL): For students with persistent, unresolvable Windows toolchain issues, WSL2 with Ubuntu provides a Linux Node.js environment that sidesteps almost all Windows gotchas. Metro runs inside WSL; the phone connects via tunnel. This is a last resort — it adds complexity — but it works when Windows is being actively hostile. Flag this as an option; don't walk through it in class.
6
First Edit, Dev Loop Confirmation, and Wrap-Up
Replace index.tsx · Save · Watch it update · CFU
1:03 – 1:15 · 12 min

The first edit (5 min)

  • Have every student open app/index.tsx and replace the contents with the minimal component from the student document — a dark navy background, their name, and the running-from-my-machine text.
  • Hit save. Watch the phone. "If your phone just updated — you are done. You have a local Expo project running on your own device from your own machine. That is the real development environment."
  • This moment lands. Give it a second. Students who've been trying to get here for 70 minutes feel the payoff.
🏋 Quick Wins — 5 minutes of free exploration

Challenge students to make three visible changes, each one saved and observed on device:

  1. Change the background color to something of their choice
  2. Add a third line of text
  3. Change a font size

The goal is repetitions of the edit → save → see loop. Muscle memory matters here — students who do it three times in a row own it.

✅ Final Check for Understanding
  1. What is Metro? What does it do?
    → Metro is the JavaScript bundler that runs on your local machine. It watches your source files, bundles them when they change, and serves the bundle to Expo Go on your phone over the network.
  2. You changed a component's background color and saved, but your phone still shows the old color. What do you try first?
    → First try Fast Refresh — did the save trigger an update? If not, shake the phone and do a full Reload. Fast Refresh occasionally fails to pick up certain kinds of changes; a reload always works.
  3. When would you use npx expo start --tunnel instead of npx expo start?
    → When the phone and laptop are on different networks, or when the WiFi network blocks device-to-device communication (common on campus/institutional WiFi). Tunnel routes through Expo's servers, bypassing network restrictions.
  4. What's the difference between npx expo install expo-camera and npm install expo-camera?
    → npx expo install picks the version of expo-camera compatible with your current Expo SDK. npm install picks the latest version, which may be incompatible. Always use npx expo install for Expo packages.

Common Student Mistakes

MistakeWhat happensFix
Phone on cellular, laptop on WiFi QR scan starts but app shows connection error. Metro can't reach the phone. Put both on same WiFi. Or use --tunnel.
Clicked "Block" on the Windows Firewall dialog Metro starts fine on laptop but phone never connects. Windows Security → Firewall → Allow app through firewall → Node.js → enable both checkboxes.
npm install expo-camera instead of npx expo install expo-camera Version mismatch. App works until it doesn't — often breaks other packages silently. Always use npx expo install for Expo packages. Delete node_modules, run npm install, reinstall correctly.
Editing files inside node_modules Changes are overwritten next time npm install runs. Also breaks the package in unexpected ways. Never edit node_modules. If you need to modify a package's behavior, use a patch or wrapper.
Closing the Metro terminal and wondering why the app won't update Fast Refresh stops working. App shows last bundled state. Metro must stay running while developing. Keep that terminal open. Rerun npx expo start if it was closed.
Project folder too deep or on Desktop Long path errors during npm install. Mysterious build failures. Keep projects in a short path like C:\Projects\. Enable long paths in Windows registry.
Not running npm install after cloning someone else's project Metro fails immediately — modules not found. node_modules is in .gitignore and isn't cloned. After cloning, always run npm install before npx expo start.

Grading & Assessment Notes

This is a setup session — there's no assignment submission today. The deliverable is a working local environment. Participation credit is appropriate. For future assignments, the fact that code was submitted means the environment works; don't deduct for setup struggles that predate the submission.

  • Full credit: Student has a running local project, made a visible change, and can describe the dev loop.
  • Acceptable: Student paired with a neighbor and observed the environment — still has setup issues at home. Note for office hours follow-up.
  • Flag for follow-up: Student couldn't get Metro running at all by end of class. Reach out before the next session — they will be blocked on everything else.
