The Five-Layer Pyramid
Framing the complete picture · Speed vs. confidence · How layers feed each other
0:00 – 0:09 · 9 min

Opening — situate where students are

  • "You know Jest. You know RNTL. You know how to write a unit test and a component test. Today we zoom out. Those are layers 1 and 2 of a five-layer stack. Let's look at the full picture."
  • Display the pyramid SVG. Walk it bottom to top: fast/cheap at the base, slow/expensive at the apex. The shape is intentional — you have hundreds of unit tests and three E2E journeys. Not the reverse.
  • "If you invert the pyramid — lots of E2E tests, few unit tests — you get a test suite that takes 45 minutes to run, gives you failure messages like 'the login broke' with no detail, and breaks every time you change a button label. That's not confidence. That's misery."

How layers feed each other

  • Display the feed-forward diagram. Walk the arrows: "Each layer catches failures that point down. A failing integration test means a component contract is wrong. A failing component test means a unit-level function is wrong."
  • Point at the "what each layer cannot see" labels at the top. Unit tests can't see layout. Component tests can't see real network. Integration tests can't see real device behavior. Each layer has a blind spot that the next layer covers.
  • The overlap zones are intentional: a login flow tested at both the integration level and the UI level is double-caught. The integration test is faster and more specific; the UI test confirms the navigation wiring is real.
🗣 Question: "What does a unit test failure tell you that a failing E2E test doesn't?" → A unit test pinpoints the exact function and exact input that broke. An E2E failure tells you "something in the login flow is broken" — which could be the form validation, the API call, the navigation, or the success state rendering. Lower layers give more specific failures. That's the argument for the pyramid shape.

Layers 1–3 in Practice
Unit · Component · Integration — what each tests, live code examples
0:09 – 0:21 · 12 min

Layer 1 — Unit: pure functions are the best test targets

  • Show the formatDistance unit test from the student document. Ask: "How long do you think 50 tests like this take to run?" → Under a second. "That's why you have hundreds of them."
  • The discipline: every pure utility function in your app should have a corresponding unit test. Formatters, validators, calculators, transformers — these are all unit-testable and fast to cover.

Layer 2 — Component: RNTL fires callbacks directly

  • Show the TrailCard component test. Point at fireEvent.press — "This is not simulating a tap on a device. This is calling the press callback directly. That's why it runs in milliseconds."
  • The key queries: getByText, getByRole, getByTestId. Query by what the user sees, not by implementation details (component class name, DOM structure). Tests that break when you rename a variable are not useful tests.

Layer 3 — Integration: mock the network, not the logic

  • Show the TrailListScreen integration test. The fetch is mocked — but the component's internal logic for calling fetch, handling loading state, and rendering results is real. That's the integration seam.
  • "You mock what you don't control. The network is external — mock it. Your own component logic is internal — don't mock it."
🖥 Demo (4 min): Open a project with all three layers running. Run the test suite with npm test -- --watchAll. Show the output: unit tests in green in under 1 second, component tests passing, integration test showing the async findByText wait. Then break one unit test intentionally (change a return value). Show the immediate red failure with the exact assertion that failed. This is what fast feedback feels like.

Mobile-Specific Testing Challenges
Six reasons mobile testing is harder than web testing
0:21 – 0:33 · 12 min

Walk the six challenges

  • Fragmentation: Ask students how many Android device models they think exist. The real answer is thousands of active models with different screen sizes, GPU drivers, OS versions, manufacturer skins. A Pixel layout bug is not the same as a Samsung layout bug.
  • OS interruptions: An incoming phone call during a UI test produces a false failure. Automated test frameworks have to be written to tolerate OS overlays — permission dialogs, notifications, system alerts — that appear asynchronously during test runs.
  • Network variability: The single most common mobile app failure mode in production is network-related — but tests almost always run on fast, stable connections. Tools like Network Link Conditioner (iOS) and Android emulator network profiles let you simulate 3G, high latency, or packet loss in tests.
  • Gesture complexity: Swipe-to-delete, pinch-to-zoom, multi-touch — these are fundamentally different from mouse clicks and require platform-specific gesture simulation. Detox and Maestro both have gesture APIs, but they're harder to write than simple taps.
  • Async timing: Animations, transitions, splash screens. Flaky tests are almost always timing issues — the test asserted before the animation finished. The solution is explicit waits and frameworks (Detox) that wait for the JS thread to go idle.
  • Hardware APIs: Camera, GPS, biometrics, sensors. The simulator gives you a synthetic camera feed and injected GPS coordinates. Good enough for most tests. Not good enough for camera-specific bugs like focus failure in low light. Those need real devices.
📝 The implication: These challenges explain why the professional recommendation is to keep the E2E layer small (3–5 journeys) and invest heavily in layers 1–3. Not because E2E isn't valuable — it absolutely is — but because the cost per test is high and the debugging experience when things go wrong is painful. Selective E2E coverage of the highest-value paths, comprehensive unit/component/integration coverage of everything else.
✅ Quick Check
  1. Your E2E test fails intermittently — sometimes passes, sometimes fails with "element not found." What's the most likely cause?
    → Timing. The test is asserting before an animation or async operation completes. Fix: use explicit wait mechanisms (waitFor in Detox, assertVisible in Maestro) or a framework that waits for JS thread idle.

UI Automation — The Three Approaches
Accessibility tree · Visual regression · Device cloud farms
0:33 – 0:47 · 14 min

Approach 1 — Accessibility tree (Detox, Maestro)

  • These tools interact with the app through the same mechanism screen readers use — the accessibility tree. They find elements by testID, accessibilityLabel, or text content, and fire programmatic interactions.
  • Detox is "gray-box" — it has access to the JavaScript runtime inside the simulator. This is why it can wait for the JS thread to go idle before asserting, which eliminates most timing flakiness. The cost: requires macOS for iOS tests.
  • Maestro is the fast-start option — YAML test files, no code required, built-in async tolerance. Show the 8-line login flow from the student document. A junior developer can write this on day one. The tradeoff: less control over complex scenarios.
🖥 Demo (4 min) — Maestro login test: If you have Maestro installed (brew install maestro on Mac, or the Windows installer), run the login YAML from the student document against a running simulator. Show the automated taps firing on screen in real time. Students who have never seen UI automation before are usually visibly impressed by the "ghost hands" tapping the UI.

If live demo is not feasible: show a 60-second Maestro demo video from mobile.dev — the visual is worth the setup cost.

Approach 2 — Visual regression

  • Screenshot the UI at defined checkpoints, pixel-diff against a stored baseline. Catches what accessibility-tree tests miss: a button that moved 4 pixels, a font that stopped loading, a color that drifted.
  • The noise problem: font rendering, anti-aliasing, timestamps, live data. Professional tools solve this with masking (ignore this region), threshold tolerance (allow 0.1% difference), and AI-powered diff classification.
  • Workflow: first run establishes the baseline. Subsequent runs diff against it. You review diffs in a web UI — approve intentional changes, reject regressions. Approved changes become the new baseline.
  • Not a replacement for behavior testing — visual regression catches appearance bugs; accessibility-tree testing catches interaction bugs. They complement each other.

Approach 3 — Cloud device farms

  • AWS Device Farm, Firebase Test Lab, BrowserStack — you upload your app binary and your test suite; they run it in parallel on 50+ real physical devices and return per-device results.
  • The only way to systematically test against fragmentation. A Samsung-specific layout bug, an Android 11 permission behavior change, a Pixel rendering quirk — none of these appear in any simulator. They appear in device farm runs.
  • Cost and speed: expensive ($50–$500 per run) and slow (30–60 minutes). Not a CI tool — a release gate. You run it on release candidates, not on every commit.
📝 Practical positioning for this course: At the CS 402 level, students won't implement Detox or device farm pipelines — that's professional infrastructure. What they should understand is what these tools do, why they exist, and where they fit in the protocol. The actionable takeaway is adding testID props as a habit — this is free cost now and enables all three UI automation approaches later.

Callbacks as Seams — Simulating Entire Sessions
The architectural insight · Hard vs. easy to test components · Live multi-screen demo
0:47 – 1:03 · 16 min

The central insight — every user interaction is a function call

  • "Here is the most important insight in mobile testing: every user interaction in React Native is implemented as a callback function. onPress, onChangeText, onScroll, onSubmitEditing. A user tapping a button calls the onPress callback. A test calling fireEvent.press does exactly the same thing."
  • "Functions don't care who calls them. A user's finger and a test's fireEvent call are identical from the component's perspective. This is why component tests are so powerful — they're not simulating user behavior, they're calling the same code the user's finger calls."
  • "The architectural consequence: a component that accepts its behavior through callback props is testable at every layer. A component that hardwires its own navigation and data fetching requires mocking the entire world around it."
The seam metaphor: A seam is a place in code where you can change behavior without modifying the code itself. Callback props are seams. In production, the real navigation function is passed. In a test, a jest.fn() is passed. The component is identical. The behavior is controlled by whoever passes the prop. Designing for testability and designing for good architecture are the same decision.

Hard vs. easy to test — show the comparison

  • Display the two component variants from the student document side by side. The "hard to test" version owns its own fetch call, its own navigation call, its own routing. To test it, you must mock fetch, mock useRouter, and wait for the effect to fire.
  • The "easy to test" version accepts trail, onBookPress, and onBackPress as props. To test it: render(<TrailDetailView trail={fixture} onBookPress={jest.fn()} onBackPress={jest.fn()} />). One line of setup.
  • "The props-based version is also better architecture. It's composable. You can render it anywhere with any trail data and any navigation behavior. The self-contained version is glued to its context."
🖥 Live Demo — Multi-screen session test (8 min): This is the payoff demo for the entire lecture. Open a Jest test file and live-code the trail booking session from the student document — or walk through it on screen with students following the code in the student doc.

Run it. Show the green output. Then ask: "How long did that take?" The answer is <100ms. "We just simulated a three-screen user session — search, detail, book — with no device, no simulator, no network, and no UI automation tooling. Just Jest and React Native Testing Library. That's what the callback architecture gives you."

Then break it deliberately — change the trail name the mock returns. Show the failure message pointing at the exact assertion. "And this is what fast feedback feels like. You know exactly what broke and exactly where."
✅ Check for Understanding
  1. What is a "seam" in the context of testing, and what creates seams in React Native components?
    → A seam is a place where behavior can be changed without modifying the code itself. In React Native, callback props are seams — the component accepts onPress, onChangeText, etc. as props, and the test passes jest.fn() mocks instead of real functions. The component code is unchanged; the test controls the behavior through the seam.
  2. You have a component with a hardwired fetch() call inside useEffect. To unit test this component, what do you have to mock?
    → You must mock global.fetch (or the specific fetch function), set up the mock response, and use waitFor or findBy queries to handle the async effect. Compare to a component that accepts data as a prop — no mocking needed at all.

The Full Protocol — When Each Layer Runs
CI/CD timing diagram · Strategy table · Annotation activity · Final CFU
1:03 – 1:15 · 12 min

CI/CD timing diagram

  • Display the timeline SVG from the student document. Walk it: "Unit tests run on every save in watch mode. Component tests run on every git push. Integration tests run when a PR merges. UI automation and E2E run only on release candidates."
  • "This is not about laziness. It's about feedback loops. You don't want to wait 45 minutes to find out a function returns the wrong value. You want to know in 3 seconds. The pyramid shape is the tool for that."
🏋 Quick Pairs Activity (5 min) — Layer Identification

Show each scenario and ask pairs to identify which layer(s) should test it:

  1. A function that converts a timestamp to "2 hours ago"
  2. A form component that shows an error message when submitted with an empty field
  3. An API call that returns user profile data, which then populates a screen
  4. The complete sign-up flow: enter email, set password, verify email, see dashboard
  5. Whether the app's card shadows look correct on a Samsung Galaxy S22

Answers: (1) Layer 1 — pure function, unit test. (2) Layer 2 — component test with fireEvent.press. (3) Layer 3 — integration, mock the fetch. (4) Layer 5 — E2E, full journey. (5) Layer 4 — visual regression on a real device (or device farm).

✅ Final Check for Understanding
  1. Why should you use npx expo install instead of npm install for Expo packages? ← Actually, wrong lecture — ask this instead: Why does a testing pyramid have more tests at the base than at the apex?
    → Lower-layer tests are faster, cheaper to write, and more specific in their failure messages. Higher-layer tests are slower, more expensive, and produce less diagnostic information. You run fast tests continuously and expensive tests at release gates. The pyramid shape optimizes for feedback speed and diagnostic quality simultaneously.
  2. You are reviewing a PR. The changed component contains a fetch() call, navigation logic, and business logic all in one function. What testing concern should you raise?
    → The component has no seams — it can't be tested without mocking the entire environment. Suggest extracting the navigation to an onComplete prop and the fetch call to a data hook or parent component. This creates testable seams without changing the component's behavior.
  3. A flaky E2E test fails about 30% of the time with "element 'Sign In' not found." The Sign In button definitely exists in the app. What is the most likely cause and fix?
    → Timing. The test is looking for the Sign In button before an animation completes or before the initial data load finishes. Fix: use waitFor/assertVisible with a reasonable timeout, or use Detox's idle-wait behavior which automatically waits for the JS thread to settle before asserting.

Common Student Mistakes

MistakeWhat happensFix
Testing implementation, not behavior Test checks component's internal state variable name. Breaks on every refactor even when behavior is unchanged. Query by what the user sees (text, role, testID). Assert on output, not internals.
Mocking your own code A test mocks the component's own utility function. The test passes but the real function is never exercised. False confidence. Mock external dependencies (fetch, device APIs). Test your own logic for real.
No testID props anywhere UI automation tests query by text content. Any copy change breaks all tests. Localization support breaks all tests. Add testID to every interactive element as a development habit. It costs nothing and enables every layer of testing.
Async assertion without waitFor/findBy Test asserts on data that arrives via useEffect. Data isn't there yet. Test fails intermittently. Use await screen.findByText('...') or await waitFor(() => expect(...)) for any assertion that depends on async behavior.
Writing E2E tests for everything Test suite takes 45 minutes. Every button label change breaks 20 tests. Nobody runs the suite. Nobody trusts it. E2E tests cover 3–5 critical user journeys. Unit and component tests cover everything else.
Hardwiring navigation and fetch inside components Component can't be tested without mocking the entire world. Test setup takes more time than writing the component. Accept navigation and data functions as callback props. Create seams intentionally. Testability is an architectural decision, not an afterthought.

Grading & Assessment Notes

Testing rarely appears as its own assignment — it shows up as a quality criterion in feature assignments. When evaluating student code for testability, apply the same "start from full credit" philosophy and focus on the structural pattern, not test coverage percentage.

  • Full credit: Key components accept callback props for navigation and side effects. At least one unit test for pure utility functions. At least one component test with fireEvent. testID present on interactive elements.
  • Partial credit: Tests exist but test implementation details rather than behavior. Note this in feedback — "your test will break on any variable rename, which makes it less valuable than testing user-visible output."
  • Comment, not deduction: No visual regression or E2E tests — not expected at this level. Mention them as a professional context note.
  • Significant deduction: Submitted code causes test suite to fail entirely (syntax error in test file, broken test setup). This indicates the student didn't run the tests at all.
📝 The job interview connection: The callback/seam architecture is one of the few testing concepts that comes up directly in mobile developer interviews. "How do you test a component that has navigation logic?" is a real interview question. A student who answers "I extract the navigation to a callback prop so the test can pass a mock" has demonstrated architectural thinking, not just testing knowledge. That's the answer that gets a follow-up offer.
