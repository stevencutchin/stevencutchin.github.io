A Brief History of Mobile

Before we build apps, let's understand how we got here. The path from brick phones to app ecosystems explains why mobile architecture looks the way it does.

WAP — The First Mobile Web Platform
Wireless Application Protocol: stripped-down HTTP for unreliable mobile networks. Used WML instead of HTML with a "card" mentality. The two popular WAP sites? CNN and ESPN.
Early 2000s
In-App Purchases Before Apps: SMS
Before app stores existed, SMS was the monetization platform. Text a short code, get charged on your phone bill. Ring tones, wallpapers, trivia — the first mobile economy.
~2005–2007
Third-Party Apps Begin
Batteries improved, form factors shrank, coverage expanded. Handset manufacturers didn't want to write all the apps — but also didn't want to open their platforms. Phones started running known operating systems (Windows CE, Linux). Now the big players were involved.

2007–Now
The Fractured Mobile Market
Who do you develop for? How do you test for every phone? Which market works best? How do you port between platforms? Which tools can you use? This is the world we inherited.

The Three-Tiered Architecture

The foundation of every modern application

The Three Tiers

For a web application: the browser + dynamically generated HTML is the presentation layer. Middleware files (PHP, Java servlets) contain the business logic. The database server is the data layer.

Why this matters: Keep the presentation separate so it's lightweight, easier to maintain, and can be tested separately. Keep the logic separate so you can change it without changing the UI. Keep the data separate because you should NEVER build a system based on current data values.

The Business Origins of 3-Tier Architecture

Before it was a technical pattern, the 3-tier architecture was a business strategy for managing software teams. Understanding its origins tells you why it's structured the way it is.

In the early days of enterprise software (1990s), large companies discovered that monolithic applications — where UI, logic, and data were all tangled together — couldn't scale organizationally. When the database team needed to optimize queries, they'd break the UI. When designers wanted to refresh the look, they'd introduce logic bugs. The 3-tier architecture solved a people problem: how do you let multiple specialized teams work on the same application without stepping on each other?

The technical separation followed naturally from the organizational one: clear APIs between tiers, independent deployment, separate testing. This is why 3-tier thinking applies everywhere — web apps, desktop apps, mobile apps, microservices. It's not about where the code runs; it's about how you organize the code so that humans can maintain it.

Key takeaway: The 3-tier architecture is a software development methodology — a way to organize your code and your team regardless of platform. It tells you how to structure an application. It applies whether you're building a web app, a desktop app, or a mobile app. It applies across all the mobile architectures we'll discuss next.

Model-View-Controller

The pattern that maps user actions to data and back

MVC — The Pattern

The MVC pattern maps: identify what the user is asking for → load a particular resource → display the pertinent info back to the user. That maps to Controller → Model → View.

Controller — The Traffic Cop

Takes the request from the user. Finds the appropriate model to load. Finds the appropriate view to display. Returns the whole thing back to the user.

Model — Data + API Rules

The representation of the data. Often maps directly to a database table with columns as attributes. Contains relationship rules (a Student has many Classes). Think "class definition with DB backend."

Exercise: If you were building a blog, what might some of the models be? What resources should have addresses? How do they relate to each other? (Posts, Comments, Users, Tags…)

Mobile App Architectures

A mobile-specific design decision that complements the 3-tier model

Two Complementary Decisions

Now we shift from a general software development principle to a mobile-specific design choice. The 3-tier architecture tells you how to organize your code. The mobile architecture spectrum tells you where that code physically runs.

These two ideas are complementary, not competing. You always use the 3-tier architecture to organize your code into presentation, logic, and data concerns — that's good software engineering on any platform. The mobile-specific question is: given that I have these three tiers, where do they physically execute? That answer depends on your app's requirements for connectivity, device features, offline capability, and user experience.

The mobile-specific constraints that drive this choice: Unreliable networks that drop constantly. Limited battery shared across all apps. App store review processes that delay updates. Users who switch between apps constantly and expect instant responsiveness. These constraints don't exist for web or desktop development — they're what make mobile architecture its own discipline.

The Thin-to-Rich Client Spectrum

Where do features live — on the device or on the server?

The Spectrum: A Design Decision, Not a Category

Most apps aren't purely thin or purely rich — they fall somewhere along a spectrum. The question isn't "which type is my app?" but rather: for each feature, where should the computation and data live?

Each position on the spectrum represents a different answer to the same question: how much of the 3-tier architecture lives on the device vs. on the server? Let's look at each one and map the three tiers across them.

Rich Internet Application (RIA)

The lightest possible mobile client. Eschews the use of any phone resources other than a browser. All three tiers live on the server; the phone is just a viewport.

When to use this

Apps that need maximum reach — run on any device with a browser. No app store submission required. Updates are instant (just update the server). Good for information-display apps, internal tools, dashboards.

What you give up

No access to device hardware (camera, GPS, accelerometer). No offline capability whatsoever. Performance limited by network latency. Doesn't feel "native" — no platform-specific gestures or conventions.

Real-world examples: Mobile banking web apps, Google Docs on mobile, many internal enterprise tools. The trade is clear: maximum portability in exchange for zero device integration.

Thin Mobile Architecture

The phone handles presentation natively — it's a real app with native UI — but most business logic and all data services remain on the server. The device is a smart display with some local behavior.

When to use this

Apps that need native feel but centralized control. Social media feeds, email clients, news apps, e-commerce. The server holds the real data; the phone is a beautiful, responsive window into it. Good for apps that require phone services (push, camera) but are data-driven.

What you give up

Requires Internet connectivity for almost everything. Server goes down = app is useless. Latency affects every interaction. You're paying for server infrastructure and scaling.

Rich Mobile Architecture

Business logic and data services live on the phone itself. The device is a self-contained application that can function independently, syncing with a remote server when connectivity is available.

When to use this

Apps that must work "off the grid." Field data collection, GPS/mapping apps (Waze!), note-taking, games, medical apps in remote areas, any app where connectivity can't be guaranteed. The device is the primary computer; the server is a backup and coordination layer.

What you give up

Most complex to build. Data sync conflicts are hard (two devices edit the same record offline — who wins?). Larger app binary. More battery and storage usage. Business logic updates require app store re-deployment instead of a server-side fix.

The 3-Tier Model Applies Across the Entire Spectrum

Here's the key insight: you always organize your code into presentation, logic, and data tiers — that's non-negotiable good engineering. What changes across the mobile spectrum is where each tier physically executes. The 3-tier structure is the constant; the thin-to-rich spectrum is the variable.

In practice, most apps are hybrids. The per-feature question is: for this feature, where should its presentation, logic, and data live? Login logic → server. Photo filters → device. Shopping cart → both (local for speed, server for persistence). The 3-tier discipline ensures each feature is well-structured; the spectrum choice determines where it deploys.

The Three Layers — Mobile Edition

The Presentation Layer

Remember: it's a phone! Simple = good. People have different sized fingers. User actions call functions which execute features — user actions ≠ features. Phones can have varying power/resources and can go on or off the grid at any point.

Approach: Know your client type. Present data coherently. Guard against untrusted input. Factor out business logic. Determine how you'll pass data between layers.

The Business Layer

For the most part, these are your web services and related functionalities. These are your fetched URL items.

Approach: Identify features at this level. Build components that support feature execution. Hide implementation details from the presentation layer. Determine caching strategy. Map out use cases.

The Data Layer

Your module that talks to the database. Will be intertwined with the business layer to some degree. Handles local storage, remote sync, and caching strategies.

Your React Native approach: React UI which calls… your "business logic" code… that connects to some data store (local or remote). This is the 3-tier model in your pocket.

Mobile Application Design Process

A systematic approach to designing mobile applications, applying the three-tier model:

Unique Elements of Mobile Applications

Tiny UI — extremely limited pixels. No real keyboard.

Sharing — users constantly switch apps. Everything is shared.

Unreliable — network drops constantly. Limited storage shared between all apps.

Apps are suspended or killed constantly. Your app is never guaranteed to remain running. Design for interruption from the start.

React Native Application Architecture

Mapping the 3-tier model to your React Native code:

UI Layer (Presentation)

Design user interface and identify state variables. Create state variable update functions. Create callbacks to update UI appearance.

Logic Layer (Business)

What components of the UI will you modify? How will you keep your list of UI callbacks? How will you set up data source callbacks? Not all data may need to be shown on the UI.

Data Layer

Plan for asynchronous updates. Is this a thin app or a feature-rich app? Local storage, remote APIs, or both?
