React Native Components: A Visual Guide

Components are the atoms of every React Native app. This guide walks you through what a component is, how to create one, how to pass data into it, how to compose components together, and the mental models that will make you productive. Every diagram in this guide is a visual you can study offline.

1. What Is a Component?

A component is a JavaScript function that returns a description of what should appear on screen. That description is written in JSX — a syntax that looks like HTML but compiles to JavaScript function calls. React Native takes that description and renders real native views on iOS and Android.

Think of a component as a recipe card: given some inputs (called props), it produces a visual output. The same component can produce different outputs depending on what inputs it receives — just like the same recipe produces different dishes depending on the ingredients you use.

2. Anatomy of a Component

Every React Native component has the same basic structure. Let's build one step by step and label each part.

Figure 2 — The four parts of every component: imports, function with props, JSX return, and styles.

Every component you write will have these four sections. The pattern is always the same — what changes is the specific props you accept, the JSX tree you return, and the styles you apply. Let's look at each piece in detail.

2.1 Imports: Bringing in the Building Blocks

Before you can use any React Native component, you must import it. The most common imports come from two places: 

// Core React Native components
import { View, Text, Image, TextInput, FlatList } from 'react-native';

// React hooks (for state and effects)
import { useState, useEffect } from 'react';

// Your own components
import ContactCard from './ContactCard';

Notice the distinction: UI components come from 'react-native', while hooks come from 'react'. Your own custom components are imported by file path.

2.2 The Function: Your Component's Logic

A component is a regular JavaScript function. The function name must start with a capital letter — this is how React distinguishes your components from built-in HTML-like tags. greeting would be treated as an unknown HTML element; Greeting is recognized as a component. 

// ✅ Correct — capital letter
function Greeting({ name }) { ... }

// ❌ Wrong — React won't treat this as a component
function greeting({ name }) { ... }

2.3 JSX: Describing Your UI

The return statement produces JSX — a tree of components that describes what should appear on screen. JSX looks like HTML but has critical differences:

  • All tags must be closed: <Image /> not <img>
  • JavaScript expressions go inside curly braces: {name}
  • You can only return one root element (wrap siblings in a <View>)
  • Class names don't exist — use the style prop instead

2.4 StyleSheet: Styling Without CSS

React Native does not use CSS files. Styles are JavaScript objects created with StyleSheet.create(). Properties are camelCase (fontSize not font-size), and dimensions are numbers in density-independent pixels — no px, em, or % units.

3. The Core Components

React Native provides a set of built-in components that map to native platform views. You'll use these constantly — they're the atoms from which everything else is built.

Figure 3 — The six core components you'll use in nearly every app.
A critical rule: All visible text must be wrapped in a <Text> component. Writing <View>Hello</View> will crash your app. This is one of the most common errors for developers coming from web development, where bare text inside a <div> is perfectly fine.

4. Props: Configuring Components from the Outside

Props (short for "properties") are how you pass data from a parent component into a child component. They are the inputs to your function. A component should be configurable through its props — the same component with different props produces different UI.

Figure 4 — Same component, different props → different output. This is what makes components reusable.
// Defining a component that accepts props
function Greeting({ name }) {
  return (
    <View style={styles.card}>
      <Text>Hello, {name}!</Text>
    </View>
  );
}

// Using it — the parent passes different props
export default function App() {
  return (
    <View>
      <Greeting name="Alice" />
      <Greeting name="Bob" />
    </View>
  );
}
One-Way Data Flow: Props always flow down — from parent to child. A child cannot change its own props. If the child needs to communicate back up to the parent (e.g., "the user tapped me"), the parent passes a callback function as a prop, and the child calls that function. This keeps data flow predictable.

5. Composition: Building Complex UIs from Simple Parts

The real power of components emerges when you compose them — nesting smaller components inside larger ones to build complex interfaces. Each component handles its own responsibility, and the parent orchestrates the layout.

Figure 5 — A component tree: App → ContactList → ContactCard → Avatar + Name + Phone. Each node is a component.

Here's the code for this composition pattern: 

function Avatar({ uri }) {
  return <Image source={{ uri }} style={styles.avatar} />;
}

function ContactCard({ contact }) {
  return (
    <View style={styles.card}>
      <Avatar uri={contact.photo} />
      <View style={styles.info}>
        <Text style={styles.name}>{contact.name}</Text>
        <Text style={styles.phone}>{contact.phone}</Text>
      </View>
    </View>
  );
}

function ContactList({ contacts }) {
  return (
    <FlatList
      data={contacts}
      keyExtractor={item => item.id}
      renderItem={({ item }) => <ContactCard contact={item} />}
    />
  );
}

Each component is small, focused, and testable in isolation. Avatar only knows how to render an image. ContactCard only knows how to lay out one card. ContactList only knows how to iterate over an array. When something looks wrong, you know exactly which component to inspect.

6. The Component Lifecycle: From Render to Re-Render

Understanding when your component function runs is just as important as knowing what it returns. React calls your function to get the initial output, and then calls it again whenever something changes.

Figure 6 — The render cycle: initial render → on screen → state change → re-render → diff → update only what changed → back on screen.

This is why state matters. When you call setCount(count + 1), React doesn't mutate the screen directly — it re-runs your entire component function with the new state value, compares the new JSX tree to the old one, and surgically updates only the parts that changed. This "diffing" process is what makes React fast enough for mobile.

7. Passing Callbacks: Child-to-Parent Communication

Props flow down, but sometimes a child needs to tell its parent that something happened — "the user tapped me," "the input changed," "the item was deleted." The solution: the parent passes a callback function as a prop, and the child calls it.

Figure 7 — The callback pattern: parent defines the function, passes it down as a prop, child calls it when an event occurs.
// Parent defines the behavior
function TodoList() {
  const [items, setItems] = useState(['Buy milk', 'Walk dog']);

  const handleDelete = (index) => {
    setItems(items.filter((_, i) => i !== index));
  };

  return items.map((text, i) => (
    <TodoItem key={i} text={text} onDelete={() => handleDelete(i)} />
  ));
}

// Child calls the callback — it doesn't know what happens next
function TodoItem({ text, onDelete }) {
  return (
    <View style={styles.row}>
      <Text>{text}</Text>
      <Button title="Delete" onPress={onDelete} />
    </View>
  );
}

Notice that TodoItem doesn't know how deletion works — it just calls onDelete. This separation of concerns is what makes components reusable. You could use the same TodoItem in a different list with a completely different delete behavior.

8. Component Design Patterns

8.1 Container vs. Presentational

A useful mental model is to separate components into two roles. Presentational components focus on how things look — they receive data via props and render UI. Container components focus on how things work — they manage state, fetch data, and pass results to presentational components.

Figure 8 — Container components manage logic; presentational components manage display. Separate concerns for cleaner code.

8.2 Default Props and Defensive Coding

Always design components to handle missing or unexpected props gracefully. Use default parameter values so your component doesn't crash when a prop is omitted: 

function Badge({ label = "New", color = "#2E5D8A" }) {
  return (
    <View style={[styles.badge, { backgroundColor: color }]}>
      <Text style={styles.badgeText}>{label}</Text>
    </View>
  );
}

// All three are valid:
<Badge />                             // "New" in blue
<Badge label="Sale" />                 // "Sale" in blue
<Badge label="Urgent" color="red" />  // "Urgent" in red

9. Common Mistakes and How to Avoid Them

Mistake What Happens Fix
Bare text in a <View> App crashes: "Text strings must be rendered within a <Text> component" Always wrap text in <Text>
Lowercase component name React treats it as a native tag, not your component Always capitalize: MyCard not myCard
Forgetting key in lists Warning; React can't efficiently update the list Add key={item.id} to each list item
Mutating props Unpredictable behavior — React assumes props are immutable Never modify props; create new objects with spread
Calling setState during render Infinite re-render loop Only call state setters in event handlers or useEffect
Missing return in component Nothing renders (returns undefined) Arrow function: use () => (...) not () => {...} unless you include return

10. Putting It All Together: A Complete Example

Here is a complete, working component that demonstrates imports, props, state, events, list rendering, composition, and styling — everything from this guide in one example. 

import { useState } from 'react';
import { View, Text, TextInput, FlatList,
         TouchableOpacity, StyleSheet } from 'react-native';

// Presentational component — just renders one item
function TaskItem({ text, onToggle, done }) {
  return (
    <TouchableOpacity onPress={onToggle} style={styles.item}>
      <Text style={[styles.text, done && styles.done]}>
        {done ? '✓ ' : '○ '}{text}
      </Text>
    </TouchableOpacity>
  );
}

// Container component — manages state and logic
export default function TaskApp() {
  const [input, setInput] = useState('');
  const [tasks, setTasks] = useState([]);

  const addTask = () => {
    if (!input.trim()) return;
    setTasks([...tasks, {
      id: Date.now().toString(),
      text: input,
      done: false,
    }]);
    setInput('');
  };

  const toggleTask = (id) => {
    setTasks(tasks.map(t =>
      t.id === id ? { ...t, done: !t.done } : t
    ));
  };

  return (
    <View style={styles.container}>
      <Text style={styles.title}>My Tasks</Text>
      <TextInput
        style={styles.input}
        value={input}
        onChangeText={setInput}
        onSubmitEditing={addTask}
        placeholder="Add a task..."
      />
      <FlatList
        data={tasks}
        keyExtractor={item => item.id}
        renderItem={({ item }) => (
          <TaskItem
            text={item.text}
            done={item.done}
            onToggle={() => toggleTask(item.id)}
          />
        )}
      />
    </View>
  );
}

const styles = StyleSheet.create({
  container: { flex: 1, padding: 24, backgroundColor: '#fff' },
  title: { fontSize: 28, fontWeight: 'bold', color: '#1B3A5C', marginBottom: 16 },
  input: { borderWidth: 1, borderColor: '#ccc', borderRadius: 8,
           padding: 12, fontSize: 16, marginBottom: 12 },
  item: { paddingVertical: 14, borderBottomWidth: 1, borderBottomColor: '#eee' },
  text: { fontSize: 16, color: '#333' },
  done: { textDecorationLine: 'line-through', color: '#aaa' },
});
What this demonstrates: Imports (lines 1–2), a presentational component (TaskItem), a container component (TaskApp), state management (useState), event handling (onChangeText, onSubmitEditing, onPress), the callback pattern (onToggle), list rendering (FlatList with renderItem), conditional styling (done && styles.done), and immutable state updates (spread operator in toggleTask).
