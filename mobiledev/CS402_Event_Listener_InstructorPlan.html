CS 402 — Mobile Application Development

Event & Listener Patterns in React Native
🕐 Total time: 75 minutes 📅 Semester: Spring 2026 📦 SDK: Expo SDK 52 📄 Student doc: CS402_Event_Listener_Pattern.html 🔗 Prereqs: useState, useEffect, Component Composition

Lecture Overview

This lecture introduces the Event / Listener pattern as a structured way for React Native components to communicate without direct coupling. Students learn to build a small EventEmitter utility from scratch, wire it into components using useEffect, and understand the chain-of-responsibility behavior controlled by listener return values. The pattern generalizes naturally from things students already know — event handlers in the browser DOM — so lean on that analogy early.

Framing for students: Open with the question, "How many of you have ever had to pass a callback through three components just to get it somewhere it needed to go?" Let a few groan or laugh. That lived frustration is the exact motivation for this lecture.

Learning Objectives

Understand
Explain why prop-drilling breaks down in multi-component communication scenarios.
Understand
Describe how the chain-of-responsibility pattern controls event propagation via return values.
Apply
Implement an EventEmitter class with on(), off(), and emit() methods.
Apply
Register and unsubscribe listeners correctly inside useEffect hooks.
Analyze
Distinguish when a listener should return true (pass on) vs. false (consume).
Evaluate
Choose between self-registration and central-registration (composer) strategies for a given screen design.

1
Hook & Motivation — The Prop-Drilling Problem
Open with a relatable pain point before introducing the solution
⏱ 8 min

Opening Hook

Draw (or sketch on-screen) a simple 3-level component tree: App → FormScreen → PasswordField, with a StrengthMeter and SubmitButton as siblings of PasswordField. Ask students: "How does PasswordField tell StrengthMeter what the user typed?"

💬 Quick Poll (show of hands or PollEverywhere): "If you needed PasswordField to update StrengthMeter, SubmitButton, AND log to analytics — how would you do it right now with what you know?" Listen for: lift state up, callback props, context. All are valid, but walk through why each gets painful at scale.

Talking Points

  • Callback props work great for 1-to-1 parent/child — but what about 1-to-many at the same level?
  • Lifting state to the parent means the parent must know about every subscriber — tight coupling.
  • What if the analytics listener lives in a completely different part of the tree?
  • Today's pattern borrows from something students already know: DOM event listeners. addEventListener('click', fn) — multiple listeners, same event, any component can add one.
  • We're going to build that same mechanism ourselves, purpose-built for component-to-component communication in React Native.
📝 Instructor Note: Don't spend more than 3 minutes on the pain-point sketch. The goal is buy-in, not an exhaustive comparison. Students who already know Redux may volunteer it — acknowledge it's valid and note you'll compare it in the final section. Keep momentum toward the solution.
2
Pattern Concepts — Events, Listeners, and the Chain
Establish vocabulary and the chain-of-responsibility contract before touching code
⏱ 13 min

Walk Through the Diagrams

Open CS402_Event_Listener_Pattern.html and project the first SVG diagram ("Listener Chain Execution"). Walk through it left to right. Name each box, trace the event object, and pause at Listener 3.

  • Event object: a plain JavaScript object — { type, payload, timestamp, consumed }. Same object is handed to every listener. Nobody copies it. Nobody owns it until someone returns false.
  • Listeners are just functions. They take the event, do their work, and return a boolean. That's the whole contract.
  • The chain runs in registration order. First registered = first called. This matters and we'll revisit it.
  • return true → "I handled this but I'm done, pass it along." Think of it as stepping aside.
  • return false → "I consumed this event. Nobody else gets it." Think of it as catching a ball — the throw stops.
  • Missing return value (undefined) is falsy — it will stop the chain by accident. We'll show this mistake explicitly.
💬 Analogy — Bouncer at a Club: Each listener is a bouncer in a line. The event is a person trying to get in. If bouncer #1 waves them through (return true), bouncer #2 sees them. If bouncer #2 stops them (return false), bouncer #3 never sees them at all. The bouncers don't talk to each other — they only interact with the person (the event).
✅ Check for Understanding — Verbal Cold Call
  1. "I have three listeners. The second one returns false. How many listeners run total?"
    → Two. The first runs and returns true, the second runs and returns false, the third never executes.
  2. "A listener runs some code but has no return statement. What happens?"
    → It implicitly returns undefined, which is falsy — the chain stops as if it returned false. This is a bug.
  3. "What's in the event object besides the payload?"
    → type (event name), timestamp, consumed flag.
📝 Instructor Note: The "undefined = false" gotcha is the single most common student mistake in assignments that use this pattern. Say it twice, write it on the board. It will save you a lot of grading heartache.
3
Build the EventEmitter — Live Coding
Students should type along; this code is the foundation of everything that follows
⏱ 15 min

Setup

Start with a blank file: utils/EventEmitter.js. Build it incrementally — do not paste the finished version. Students type along in their own editor or a shared Snack session.

🖥 Live Coding Order:
  1. Class skeleton with constructor and this._listeners = {}
  2. The on() method — push to array, return the unsubscribe lambda
  3. The off() method — filter out the specific function reference
  4. The emit() method — build the event object, loop with for...of, break on false
  5. The clear() method — one-liner cleanup

Key Points to Narrate While Typing

  • Why a plain object {} instead of a Map? Either works. Plain object is simpler to read and sufficient for named events. If students ask, acknowledge Map would handle edge-case keys better.
  • The unsubscribe closure: return () => this.off(eventType, listenerFn). Point out this is a closure — it captures both arguments. Call it "the cleanup gift with purchase."
  • for...of not forEach: break works in a for loop. You cannot break out of a forEach. This is one of the few cases where for...of is meaningfully better.
  • Strict === false check: We check for exactly false, not just falsy. This means a listener that returns 0 or null by accident won't incorrectly stop the chain — only an explicit false does.
  • emit() returns event.consumed: Sometimes the caller wants to know if anything stopped the chain — useful for debugging.
📝 Instructor Note: When you type the for...of loop, pause and deliberately write a forEach version first, then try to add break inside it. Show the class it won't work (no error, just silently ignores break). Then switch to for...of. That contrast makes the choice memorable.
✅ Check for Understanding — Think-Pair (2 min)
  1. "Why does on() return a function instead of just registering the listener?"
    → It returns an unsubscribe function so the caller can easily remove the listener later, especially useful in useEffect cleanup.
  2. "What would happen if two different listeners passed the exact same function reference to on()?"
    → It would be registered twice and called twice per emit. This is a real bug to watch for when registering inside loops or unstable closures.
4
Wiring to React Native — useEffect & State
Connect the emitter to real component lifecycle and useState hooks
⏱ 13 min

Project the Component Tree Diagram

Switch to the second SVG in the student doc — the component tree showing FormScreen distributing the emitter to PasswordField, StrengthMeter, and SubmitButton. Walk through the three code examples in sequence.

Step-by-Step Walkthrough

  • Source component (PasswordField): Calls emitter.emit('onChange', { value: text }) inside its onChangeText. It does not know or care who is listening.
  • Listener component (StrengthMeter): The useEffect runs after mount, registers the listener, and returns the unsubscribe function. Walk through the lifecycle diagram at this point (step 1 → 2 → 3 → 4 → 5).
  • Composer (FormScreen): Uses useMemo to create the emitter exactly once. Passes it as a prop. Each child self-registers.
  • Dependency array [emitter]: Because the emitter is stable (same reference every render thanks to useMemo), this effect only runs once. If the emitter were recreated every render, the effect would thrash — re-registering on every keystroke. This is a common bug.
⛔ Live Mistake Demo: Briefly create the emitter with plain const emitter = new EventEmitter() inside the component body (no useMemo). Open React DevTools Profiler or add a console.log('registered') inside the useEffect and type in the field. Show that "registered" fires on every keystroke — the emitter is new each render, the effect re-runs each time. Then add useMemo and show it stops. Students will not forget this.
✅ Check for Understanding — Cold Call
  1. "What happens to the listener if the component unmounts and we forgot the cleanup return?"
    → The listener stays in the emitter's array. It fires on future events, tries to call setState on an unmounted component, and produces a React warning. In older React, this caused memory leaks and potential crashes.
  2. "Could two sibling components register for the same event on the same emitter?"
    → Yes. That's the whole point — multiple independent listeners for the same event. They each call setState on their own state slice independently.
📝 Instructor Note: The lifecycle diagram (segment 3 in the student doc) is worth 2 minutes of direct projection. Point at each numbered step physically while narrating. Students respond well to the "zombie listener" framing — it's memorable and accurate.
5
Live Demo — Password Form With Chain Stopping
Put it all together in a running Expo Snack; demonstrate the false-return stop visually
⏱ 10 min

What to Build

A minimal form with three components wired to a shared emitter on an onChange event. Each listener logs to the console so students can see the chain in action.

🖥 Demo Script — Expo Snack (snack.expo.dev)
  1. Paste in the completed EventEmitter class.
  2. Add a PasswordField that emits on change.
  3. Add StrengthMeter — listener logs "StrengthMeter: strength=" and returns true.
  4. Add SubmitButton — listener logs "SubmitButton: enabled=" and returns true.
  5. Add a third RateLimiter stub — logs "RateLimiter: BLOCKED" and returns false when value contains a space.
  6. Type in the field — show all three logs firing.
  7. Type a space — show only "RateLimiter: BLOCKED" in the log; StrengthMeter and SubmitButton don't see it.
📝 Instructor Note: The RateLimiter returning false when a space is typed is a simple, visually obvious trigger. Students can immediately see the chain stop in the console. If you want a richer demo, make the RateLimiter render a red flash on-screen when it fires — this makes the "consumed" behavior tangible rather than just a console observation.
📝 Instructor Note: Before the demo, open the console panel visibly on the projector. Narrate what you expect to see before you type. This trains students to reason about code before running it — a habit worth building.
💬 Discussion Prompt (after demo): "What if StrengthMeter needed to run even when the rate limiter fires? How would you rearrange the listener order?" Let students suggest moving StrengthMeter before the RateLimiter in the registration order. Confirm: yes, order matters — and the composer pattern gives you explicit control over that order.
6
Scaling Up — Context, Order, and Alternatives
When prop-passing the emitter also becomes pain, and how to compare this pattern to others
⏱ 9 min

React Context Delivery

Walk through the AppEventProvider and useAppEvents() hook from the student doc. Keep it brief — the pattern is identical to any other Context usage students have seen; the only difference is the value being passed is an EventEmitter instance rather than plain state.

  • Use a Context emitter for app-wide events: navigation, auth state changes, notifications.
  • Use a local prop-passed emitter for screen-scoped coordination — keeps the blast radius small.
  • A pure-listener component that renders null is perfectly valid — it's a side-effect observer, not a UI element.

Listener Order Table

Briefly project the ordering table from the student doc. The key insight: when one listener can stop the chain, the order of registration is a design decision, not an implementation detail. Document it.

Quick Comparison

Spend 2 minutes on the comparison table. Frame it as "use the right tool for the job":

  • Callback props — fine for one parent, one child. Doesn't scale to siblings or deep trees.
  • This pattern — clean for one-source-to-many-listeners with optional chain stopping.
  • Context + Reducer — better when many components read shared state; every consumer re-renders on any change.
  • Zustand/Redux — bring in when you need devtools, time-travel debugging, or complex middleware. Overkill for a single form screen.
  • RxJS — when you need async streams, debounce, or operator composition. Heavy dependency, but powerful for sensor-data pipelines.
💬 Quick Scenario Exercise: Give students 60 seconds: "You're building a music player. When a song changes, the mini-player thumbnail, the lock screen metadata, and the Now Playing screen all need to update. Which pattern do you reach for, and why?" Take 2-3 answers. There's no single right answer — look for students reasoning about scope and coupling.
7
Wrap-Up, Checklist & Q&A
Consolidate key takeaways and give students the implementation checklist
⏱ 7 min

Seven-Point Recap (project the checklist from student doc)

Read through the implementation checklist in the student doc with the class. For each item, ask a student to explain it in their own words before you confirm. This is a fast-paced round — about 30 seconds per item.

  • Every on() call saves its returned unsubscribe function.
  • Every useEffect that registers a listener returns that unsubscribe function.
  • Every listener returns explicit true or false — never undefined.
  • The emitter is stable across renders (useMemo or useRef).
  • Listener order is intentional when order matters for correctness.
  • A consumed event doesn't leave later listeners with stale UI.
  • Tests cover the chain-stop case — listeners after false are never called.
📝 Instructor Note: Leave the checklist projected during Q&A. Students often ask questions that are directly answered by one of the seven items. Pointing at the checklist item while answering reinforces that it's a reference they should keep open during lab work.

Exit Ticket (optional — 2 min if time allows)

🖊 Exit Ticket — Index Card or Canvas Quiz

Ask students to write one sentence answering each of:

  1. What does returning false from a listener do?
  2. Where in the component does listener registration happen, and why?
  3. What happens if you forget to return the unsubscribe function from useEffect?

Collect or post to Canvas discussion for a quick formative pulse. No grade required.

Common Student Mistakes to Watch For

MistakeSymptomCorrection
Listener has no return statement Chain stops unexpectedly after the first listener that "forgot" Require explicit return true or return false always
Emitter created in component body without useMemo Listeners re-register on every render; console floods with "registered" Wrap in useMemo(() => new EventEmitter(), [])
Missing cleanup in useEffect React warning: "Can't perform state update on unmounted component"; zombie listeners accumulate Always return unsubscribe from the effect
Registering same function reference twice Listener fires twice per event; double state updates Ensure registration happens only once; check effect dependency array
Calling emitter.off() with a new lambda Listener never removed; off() receives a different function reference than was registered Always use the unsubscribe function returned by on()
Assuming listeners fire synchronously "later" Student writes async listener that awaits inside the loop, creating ordering surprises The emitter loop is synchronous; async side effects should be fire-and-forget inside the listener, not awaited in emit()

Grading & Assessment Notes

When this pattern appears in assignment submissions, credit the intent generously. A student who builds a functioning emitter but forgets cleanup in one component has demonstrated clear understanding of the pattern — partial deduction at most. Reserve full deductions for:

  • No unsubscribe at all (not just missing in one place — systematically absent).
  • Emitter recreated on every render, causing listener accumulation that breaks functionality.
  • Listener return values completely absent, causing all chains to stop after the first listener.
📝 Grading Philosophy: If a student clearly understands the chain concept (fires events, multiple listeners, correct behavior) but has a minor implementation issue (one missing cleanup), treat it as a code quality note rather than a correctness failure. The pattern understanding is the learning objective, not memorizing the cleanup syntax.
