Integrated Service Support

URL Schemes and the Linking API — Reaching Beyond Your App
🔗 URL scheme protocol 📞 Phone, SMS, Mail 🗺 Maps & Directions 📷 Media & Contacts 🔄 Deep links

A mobile app doesn't live in isolation. The phone around it contains a phone dialer, a messaging app, a camera, a contacts database, a map, a browser, and dozens of other services — all accessible to your React Native app through a single, elegant mechanism: URL schemes. This lecture explains how that mechanism works, what services are available, and how to use Expo's Linking API to hand off to native services gracefully and safely.

The core idea: On the web, a URL like https:// means "open this page in a browser." Mobile operating systems extend this idea: tel: means "dial this number," sms: means "open a message thread," mailto: means "compose an email." Your app constructs the URL and asks the OS to handle it — you never touch the native app directly.

1. The Mechanism — URL Schemes as Inter-App Protocol

Every app on iOS and Android can register one or more URL schemes — prefixes that the OS will route to that app when another app opens a URL with that prefix. Apple Maps registers maps://. The Phone app registers tel://. Gmail registers googlegmail://. Your own app can register a custom scheme too.

When your React Native app calls Linking.openURL('tel:+12085551234'), the sequence looks like this:

URL scheme handoff sequence. Your app constructs a URL, the OS looks up which app handles that scheme, and hands off — your app remains in memory and the user can return to it.

Notice what your app does not do directly: it doesn't access the phone's call hardware, it doesn't read the user's contacts. It constructs a URL and the OS takes it from there — your app steps back, and the user sees a native interface they already trust.

Why this matters for UX: Using the phone's native Phone app to place a call gives the user the interface they already know, with their contacts, their audio preferences, their Do Not Disturb settings. You get the capability for free; the OS provides the polish.

2. Triggering URL Schemes — Concept First, Then the Right Tool

The concept — with a tool you already know

The fastest way to see URL schemes in action is with fetch() — a function you already know. Try this in a React Native component: 

// You already know fetch. Now watch what it can do.
await fetch('sms:+12085551234');   // opens Messages
await fetch('tel:+12085551234');   // opens the dialer
await fetch('mailto:help@example.com'); // opens Mail

These work because React Native's fetch() runs through the device's native networking layer, not a Node.js HTTP stack. When that native layer encounters a non-HTTP scheme, it behaves exactly like the built-in browser following a link — it looks up the scheme in the OS registry and hands off to the registered app. One line. No imports. The phone's entire service ecosystem is reachable.

That's the concept. Your app constructs a URL string. The mobile OS reads the scheme prefix — sms:, tel:, mailto:, maps: — looks up which app registered for that scheme, and launches it with your URL as the payload. You never touch the native app directly; the OS mediates everything.

The proper implementation — Linking.openURL()

In production code, use Linking.openURL() from expo-linking. It does the same thing as the fetch() approach above — same OS handoff, same result — but it adds a critical capability: canOpenURL(), which lets you check whether the device supports a scheme before you try to open it. This is what makes your app behave gracefully on simulators, tablets, and restricted devices where some services aren't available. 

import * as Linking from 'expo-linking';

async function openService(url) {
  const supported = await Linking.canOpenURL(url);

  if (supported) {
    await Linking.openURL(url);
  } else {
    // Graceful fallback — the scheme isn't available on this device
    Alert.alert('Not available', `This device cannot open: ${url}`);
  }
}

Every scheme URL in the rest of this document can be opened with either approach. All the examples use Linking because of the canOpenURL() safety check — but the URLs themselves are the same either way.

The Linking API provides three methods you'll use regularly:

MethodWhat it doesReturns
Linking.openURL(url) Hands the URL to the OS. The OS finds the registered handler and launches it. Promise<void> — resolves when handoff succeeds, rejects if no handler found.
Linking.canOpenURL(url) Asks the OS whether any installed app handles this scheme. Use before openURL to avoid errors. Promise<boolean>
Linking.getInitialURL() Returns the URL that launched your app (if it was opened via a URL scheme). Used for deep link handling on startup. Promise<string | null>

The basic pattern for any URL scheme handoff is the same:

import * as Linking from 'expo-linking';

async function openService(url) {
  const supported = await Linking.canOpenURL(url);

  if (supported) {
    await Linking.openURL(url);
  } else {
    // The scheme is not supported on this device
    Alert.alert('Not available', `This device cannot open: ${url}`);
  }
}
⚠ iOS requires scheme registration: On iOS, canOpenURL() only returns true for schemes you have declared in your app.json under ios.infoPlist.LSApplicationQueriesSchemes. Add every scheme you intend to query. Android has no equivalent requirement for the built-in schemes.

3. The Built-In Scheme Catalog

The following schemes are available on both iOS and Android for the system's built-in services, unless noted. Third-party app schemes (FaceTime, Google Maps, Spotify, etc.) depend on the app being installed.

📞
Phone Call
tel:

Opens the dialer pre-filled with a number. User confirms before dialing. Works on physical devices only.

💬
SMS / Text
sms:

Opens the native Messages app with a pre-filled recipient. Body text can be pre-filled on most devices.

📧
Email
mailto:

Opens the default mail client. Supports pre-filling to, cc, bcc, subject, and body via query parameters.

🗺
Maps
maps: / geo:

Opens Apple Maps (iOS) or Google Maps (Android). Supports addresses, coordinates, and turn-by-turn directions.

🌐
Browser
https: / http:

Opens the user's default browser. Alternatively, use expo-web-browser to open in-app.

📅
Calendar
calshow:

Opens the system Calendar app to a specific date (iOS). Android uses intent-based URIs instead.

🎵
Music / Media
music: / audio:

Opens the system music app (iOS: Apple Music via music://). Spotify, YouTube Music use their own schemes.

📍
Directions
maps:?daddr=

Launches navigation to a destination. Use google.navigation: for Android users with Google Maps.

📸
Camera / Photos
Expo API

Camera and photo library access require Expo's expo-camera and expo-image-picker — not URL schemes.

📱
App Store
itms-apps:

Opens the iOS App Store directly to an app listing. Useful for "Rate this app" prompts. Android uses market://.

⚙️
Settings
app-settings:

On iOS, opens your app's Settings page. Useful after a permission denial — guide users to re-enable.

🤝
FaceTime
facetime:

Initiates a FaceTime call (iOS only). facetime-audio: for audio-only. Requires the other device to have FaceTime.

💡 Contacts access is different: Reading and writing the system contacts database is not done via URL schemes — it requires the expo-contacts package and a user permission grant. URL schemes can only open the dialer or Messages with a pre-filled number; they cannot enumerate the address book.

4. URL Anatomy — Building Scheme URLs

URL scheme strings follow the same structure as HTTP URLs: a scheme, optionally a host, and query parameters for additional data. Understanding the anatomy lets you construct any URL correctly without guessing.

Anatomy of a URL scheme string. The scheme identifies the handler; everything after the colon is scheme-specific. Query parameters use standard URL encoding.

Special characters in query parameter values — spaces, ampersands, non-ASCII — must be URL-encoded. Use encodeURIComponent() to encode individual values, then assemble the URL with string concatenation or a template literal. 

// Always encode user-supplied values
const subject = encodeURIComponent('Help with my order #1042');
const body    = encodeURIComponent('Hello,\n\nI need help with…');
const url     = `mailto:[email protected]?subject=${subject}&body=${body}`;

await Linking.openURL(url);

5. Platform Differences

iOS and Android handle URL schemes slightly differently. Your app needs to account for both, especially for maps and sms body text.

🍎 iOS
  • Maps scheme: maps:?q=address
  • SMS with body: sms:+1555...&body=Hello
  • FaceTime: facetime://+15551234
  • App Settings: app-settings:
  • canOpenURL() requires scheme in LSApplicationQueriesSchemes
  • App Store: itms-apps://itunes.apple.com/app/id123
🤖 Android
  • Maps scheme: geo:lat,lng?q=address
  • SMS with body: sms:+1555...?body=Hello (note ? not &)
  • No FaceTime equivalent via URL scheme
  • App Settings: use expo-intent-launcher
  • canOpenURL() works without pre-declaration for system schemes
  • Play Store: market://details?id=com.app.id

For maps in particular, the scheme syntax differs enough that cross-platform code needs a small helper. A common pattern is to branch on Platform.OS: 

import { Platform } from 'react-native';
import * as Linking from 'expo-linking';

function getMapsURL(address) {
  const encoded = encodeURIComponent(address);
  return Platform.OS === 'ios'
    ? `maps:?q=${encoded}`       // Apple Maps
    : `geo:0,0?q=${encoded}`;    // Android (Google Maps)
}

async function openInMaps(address) {
  const url = getMapsURL(address);
  const ok  = await Linking.canOpenURL(url);
  if (ok) await Linking.openURL(url);
}

6. Complete Example — A Contact Actions Card

The following is a real-world component: a "Contact Card" that exposes seven different native service handoffs for a single person — call, SMS, email, directions, FaceTime, share contact, and view on a map. This pattern appears in nearly every business-facing mobile app. 

import { useState } from 'react';
import {
  View, Text, TouchableOpacity, StyleSheet,
  Alert, Platform, ActivityIndicator
} from 'react-native';
import * as Linking from 'expo-linking';

// ─── helpers ──────────────────────────────────────────────

async function safeOpen(url, label) {
  try {
    const supported = await Linking.canOpenURL(url);
    if (!supported) {
      Alert.alert(
        'Not available',
        `${label} is not available on this device.`
      );
      return;
    }
    await Linking.openURL(url);
  } catch (err) {
    Alert.alert('Error', `Could not open ${label}: ${err.message}`);
  }
}

function getMapsURL(address) {
  const q = encodeURIComponent(address);
  return Platform.OS === 'ios' ? `maps:?q=${q}` : `geo:0,0?q=${q}`;
}

function getSMSURL(phone, message = '') {
  const body = encodeURIComponent(message);
  // iOS uses & to separate number and body; Android uses ?
  return Platform.OS === 'ios'
    ? `sms:${phone}&body=${body}`
    : `sms:${phone}?body=${body}`;
}

// ─── component ────────────────────────────────────────────

export function ContactCard({ contact }) {
  // contact shape: { name, phone, email, address, facetime }
  const [busy, setBusy] = useState(null); // tracks which action is loading

  async function handle(key, url, label) {
    setBusy(key);
    await safeOpen(url, label);
    setBusy(null);
  }

  const actions = [
    {
      key:   'call',
      icon:  '📞',
      label: 'Call',
      url:   `tel:${contact.phone}`,
    },
    {
      key:   'sms',
      icon:  '💬',
      label: 'Text',
      url:   getSMSURL(contact.phone, 'Hi, I saw your listing…'),
    },
    {
      key:   'email',
      icon:  '📧',
      label: 'Email',
      url:   `mailto:${contact.email}?subject=${encodeURIComponent('Inquiry')}`,
    },
    {
      key:   'maps',
      icon:  '🗺',
      label: 'Directions',
      url:   getMapsURL(contact.address),
    },
    {
      key:   'facetime',
      icon:  '📹',
      label: 'FaceTime',
      url:   `facetime:${contact.facetime}`,
    },
  ];

  return (
    <View style={styles.card}>
      <Text style={styles.name}>{contact.name}</Text>
      <Text style={styles.phone}>{contact.phone}</Text>

      <View style={styles.actions}>
        {actions.map(({ key, icon, label, url }) => (
          <TouchableOpacity
            key={key}
            style={styles.actionBtn}
            onPress={() => handle(key, url, label)}
            disabled={busy !== null}
          >
            {busy === key
              ? <ActivityIndicator size="small" color="#2E5D8A" />
              : <Text style={styles.actionIcon}>{icon}</Text>
            }
            <Text style={styles.actionLabel}>{label}</Text>
          </TouchableOpacity>
        ))}
      </View>
    </View>
  );
}

const styles = StyleSheet.create({
  card:        { backgroundColor: '#fff', borderRadius: 12, padding: 20, elevation: 3, shadowColor: '#000', shadowOpacity: 0.1, shadowRadius: 8 },
  name:        { fontSize: 20, fontWeight: '700', color: '#1B3A5C', marginBottom: 4 },
  phone:       { fontSize: 15, color: '#666', marginBottom: 20 },
  actions:     { flexDirection: 'row', justifyContent: 'space-around' },
  actionBtn:   { alignItems: 'center', padding: 10 },
  actionIcon:  { fontSize: 28, marginBottom: 4 },
  actionLabel: { fontSize: 12, color: '#2E5D8A', fontWeight: '600' },
});
What makes this pattern powerful: Five lines of data in the actions array give the user five different native services — phone, messaging, email, maps, and video call — all using interfaces they already know. Your app contributed the data and the trigger; the OS contributed everything else.

7. Deep Links — Other Apps Calling Into Yours

URL schemes are a two-way street. Just as your app can open other apps, other apps (and websites, emails, and QR codes) can open yours — at a specific screen, with specific data. This is called a deep link.

Expo Router handles deep links automatically when you configure your app's scheme in app.json. A URL like myapp://product/42 will open your app and navigate to the /product/42 route — exactly as if the user had navigated there manually. 

// app.json — register your scheme
{
  "expo": {
    "scheme": "myapp",   // enables myapp://... links
    "name": "My App"
  }
}
// In a component — handle a deep link that arrives while app is running
import * as Linking from 'expo-linking';
import { useEffect } from 'react';

useEffect(() => {
  // Handle links that arrive while app is already open
  const subscription = Linking.addEventListener('url', ({ url }) => {
    const { path, queryParams } = Linking.parse(url);
    console.log('Opened via deep link:', path, queryParams);
    // Navigate to the relevant screen using your router
  });

  return () => subscription.remove(); // cleanup
}, []);
💡 Universal Links (iOS) and App Links (Android): Both platforms support a more powerful form of deep linking where a standard https:// URL opens your app instead of the browser — if your app is installed. This requires a verified apple-app-site-association (iOS) or assetlinks.json (Android) file hosted on your server. Expo's documentation covers this under "Linking."

8. Design Principles — When and How to Use URL Schemes

✅ Good uses

  • Tapping a phone number in a profile opens the dialer
  • A "Get directions" button opens the user's preferred maps app
  • "Email support" pre-fills a support address with a subject
  • A "Rate us" prompt links to the App Store listing
  • Permission denied → guide user to app Settings with one tap

❌ Things to avoid

  • Opening native services without user intent (no surprise dials)
  • Skipping canOpenURL() and letting the app crash on unsupported devices
  • Pre-filling SMS/email body text that sounds like it came from the user when it didn't
  • Relying on a third-party scheme (e.g. Spotify) without a fallback for users who don't have it
⚠ You are not sending the SMS or making the call. Linking.openURL('sms:...') opens the user's Messages app with a pre-addressed draft. The user sees it, reads it, and decides whether to send. This is the correct behavior — it preserves user agency and avoids the legal and trust problems that come with sending communications silently on someone's behalf.
