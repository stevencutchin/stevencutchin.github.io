Composable Components in React Native

Building Reusable Panels, Pickers, and Full UI Modules
📋 Topic: Architecture & Component Design ⏱ 75 min lecture 🔗 Prerequisite: Event & Listener Patterns, useState, useEffect

What Composability Means

A composable component is one that can be combined with others — or reconfigured through props — to produce new behavior, without modifying its source code. The goal is to write UI building blocks once and deploy them many times, in many contexts, for free.

React Native's component model is inherently compositional — every <View> accepts children, and every component you build can do the same. The difference between a composable component and a tightly-coupled one is whether the structure is fixed in the component or handed in from outside.

❌ Not Composable

A UserCard that hard-codes its avatar, title, and action button. To make a ProductCard you copy the file and change the internals. Now you have two files to maintain and they'll drift apart.

✅ Composable

A Card that accepts a header, body, and footer as slots. A UserCard and a ProductCard are now just Card with different content passed in — one component, many uses.

Composability works at multiple scales. At the small end, a LabeledInput that wraps a TextInput with a Text label. At the large end, a full DatePicker module that assembles a calendar grid, a month navigator, and a confirm button — all from smaller, individually testable pieces.

The Four Levels of Composition

A useful mental model is to think of UI components as existing at four levels of abstraction. Each level builds on the one below, and each has a different scope of reuse.

⚛️ Level 1
Atoms
Single-responsibility primitives. They accept no children and have a focused, minimal prop surface. They know nothing about the screen they appear on.
Examples: Text, Icon, Badge, Divider, Avatar, Pill
🧩 Level 2
Molecules
Compositions of atoms with a unified purpose. They expose slots (via children or named props) for their inner atoms, and add a thin layer of layout and behavior.
Examples: LabeledInput, IconButton, SearchBar, ListItem, FormRow
🏗️ Level 3
Panels
Functional UI containers that manage their own state. They accept content through slots but also maintain internal logic — validation, selection, animation — that their children can participate in via context or events.
Examples: Panel, Accordion, TabBar, DatePicker, ColorChooser
📦 Level 4
Modules
Full self-contained units deployed as a single import. A module encapsulates one complete user workflow: it manages state, composes multiple panels, handles its own error boundaries, and exposes a narrow, stable public API.
Examples: CardManager, ProfileEditor, CheckoutFlow, MediaBrowser
You don't have to use all four levels on every project. The model is a thinking tool. The key question at any level is: "Does this component know more than it needs to?" If a DatePicker knows about your app's user object or API client, it's overreaching — strip that knowledge out.

The Composition Hierarchy in Practice

A CardManager module owns shared state and composes two panels: CardForm for editing and CardList for browsing. Each panel wires its own EventEmitter for internal coordination. Both panels are independently usable outside of CardManager.

Building the Panel Primitive

Before you can build a DatePicker or a ColorChooser, you need a reusable Panel — a visual container that manages its own layout, optional header/footer, and a slot for arbitrary content. This is the workhorse of Level 3 composition.

The Children Slot — Containment

The simplest slot is React's built-in children prop. Any JSX placed between a component's opening and closing tags is passed as props.children. This is the containment pattern: the Panel provides structure; the caller provides content. 

// components/Panel.js  — the reusable container primitive
import React from 'react';
import { View, Text, StyleSheet } from 'react-native';

export default function Panel({
  title,           // optional header text
  children,        // main content slot — accepts anything
  footer,          // named slot: JSX for the bottom action area
  style,           // escape hatch for caller-controlled outer style
  contentStyle,    // escape hatch for the inner scroll/content area
}) {
  return (
    <View style={[styles.panel, style]}>

      {title && (
        <View style={styles.header}>
          <Text style={styles.title}>{title}</Text>
        </View>
      )}

      <View style={[styles.content, contentStyle]}>
        {children}        {/* ← the main slot: whatever caller passes */}
      </View>

      {footer && (
        <View style={styles.footer}>
          {footer}         {/* ← named slot: JSX passed as a prop */}
        </View>
      )}

    </View>
  );
}

const styles = StyleSheet.create({
  panel:   { backgroundColor: '#fff', borderRadius: 12, shadowColor: '#000',
             shadowOpacity: 0.08, shadowRadius: 8, elevation: 3 },
  header:  { padding: 16, borderBottomWidth: StyleSheet.hairlineWidth, borderBottomColor: '#e0e8f0' },
  title:   { fontSize: 16, fontWeight: '700', color: '#1B3A5C' },
  content: { padding: 16 },
  footer:  { padding: 12, borderTopWidth: StyleSheet.hairlineWidth, borderTopColor: '#e0e8f0',
             flexDirection: 'row', justifyContent: 'flex-end', gap: 8 },
});

Using the Panel

The caller decides what goes inside. The Panel provides visual consistency and structural discipline — header, scrollable body, action footer — across the whole app. 

// Anywhere in your app — no boilerplate, no duplication
<Panel
  title="Edit Profile"
  footer={
    <>
      <Button label="Cancel" variant="ghost" onPress={onCancel} />
      <Button label="Save"   variant="primary" onPress={onSave}   />
    </>
  }
>
  <LabeledInput label="Name"  value={name}  onChangeText={setName}  />
  <LabeledInput label="Email" value={email} onChangeText={setEmail} />
</Panel>
💡 Named Slots via Props: React Native doesn't have HTML's <slot> element, but the same idea works through named props that accept JSX. The footer prop above is a named slot. You can have as many named slots as your component needs — header, footer, sidebar, emptyState, loadingState, etc.

Slots, Children, and Render Props

These three patterns are complementary and can be combined in the same component. FlatList in React Native itself uses all three simultaneously.

Render Props — When the Component Needs to Feed Data Back

Sometimes a slot isn't passive — the component needs to provide data to whatever fills the slot. A CardList knows each card's data; the caller decides how to render each row. This is the render prop pattern: a prop that is a function, receiving data from the component and returning JSX. 

// components/CardList.js — render prop for the row design
export default function CardList({
  cards,
  renderItem,          // render prop: (card, index) => JSX
  emptyState,          // named slot: shown when cards.length === 0
  onSelect,
}) {
  if (cards.length === 0) return emptyState || null;

  return (
    <FlatList
      data={cards}
      keyExtractor={(item) => item.id}
      renderItem={({ item, index }) =>
        renderItem(item, index)  // call the render prop with card data
      }
    />
  );
}

// Usage — the caller owns the row design completely:
<CardList
  cards={myCards}
  emptyState={<EmptyState message="No cards yet" />}
  renderItem={(card) => (
    <CardRow title={card.title} color={card.accentColor} onPress={() => onSelect(card)} />
  )}
/>

Module Walkthrough 1 — DatePicker

A DatePicker is a self-contained module: it manages which month is visible, which day is selected, and emits a single clean onDateChange callback. Internally it composes several panels. Externally it looks like a black box with one output.

🗓 MonthNavigator

  • Displays current month/year
  • Prev / Next chevron buttons
  • Fires internal onMonthChange event
  • Has no knowledge of selected day

📅 CalendarGrid

  • Renders 6×7 day cells
  • Receives month, year, selectedDate props
  • Fires internal onDayPress event
  • Highlights today, grays out adjacent months

✅ ConfirmBar

  • Shows selected date formatted
  • Clear and Confirm buttons
  • Listens for onDayPress to update display
  • On Confirm, calls parent onDateChange
// modules/DatePicker.js — the public-facing module
import React, { useState, useMemo, useCallback } from 'react';
import { View } from 'react-native';
import { EventEmitter } from '../utils/EventEmitter';
import MonthNavigator from './MonthNavigator';
import CalendarGrid   from './CalendarGrid';
import ConfirmBar     from './ConfirmBar';

export default function DatePicker({
  initialDate,     // Date | null
  minDate,         // Date | null — disables days before this
  maxDate,         // Date | null — disables days after this
  onDateChange,    // (date: Date) => void — the single public output
  renderDayCell,   // render prop — optional custom day cell design
}) {
  const emitter = useMemo(() => new EventEmitter(), []);

  // Module-level state — owned here, distributed to children via props
  const [viewMonth, setViewMonth] = useState(
    initialDate || new Date()
  );
  const [selected, setSelected] = useState(initialDate || null);

  const handleConfirm = useCallback(() => {
    if (selected) onDateChange(selected);
  }, [selected, onDateChange]);

  return (
    <View>
      <MonthNavigator
        viewMonth={viewMonth}
        emitter={emitter}         // fires 'onMonthChange'
        onMonthChange={setViewMonth}
      />
      <CalendarGrid
        viewMonth={viewMonth}
        selectedDate={selected}
        minDate={minDate}
        maxDate={maxDate}
        emitter={emitter}         // fires 'onDayPress'
        onDayPress={setSelected}
        renderDayCell={renderDayCell} // render prop passes through
      />
      <ConfirmBar
        selectedDate={selected}
        emitter={emitter}         // listens to 'onDayPress'
        onConfirm={handleConfirm}
        onClear={() => setSelected(null)}
      />
    </View>
  );
}
Public API Surface: The caller of DatePicker only needs to know about four props: initialDate, minDate, maxDate, and onDateChange. Everything else — the month navigator, the grid, the confirm bar — is an internal implementation detail. If you redesign the grid later, the caller's code doesn't change.

State flows down as props; events flow up as emitter calls; the module's single public output is the onDateChange callback. The parent screen sees none of the internal complexity.

Module Walkthrough 2 — ColorChooser

A ColorChooser is a good example of a module that manages multiple sub-views — different UI modes for picking a color — and exposes a unified value via one callback. Internally it switches between a swatch grid, an HSL slider set, and a hex input, all sharing the same selected color state.

🎨 SwatchGrid

  • Renders a grid of preset color chips
  • Fires onColorSelect event
  • Highlights the currently selected swatch
  • Caller can pass a custom swatches array

🎚 HSLSliders

  • Three range sliders: H, S, L
  • Fires onColorSelect on every slider change
  • Syncs to selected value from SwatchGrid
  • Shows real-time color preview strip

#️⃣ HexInput

  • TextInput accepting 6-digit hex codes
  • Validates on blur; ignores invalid input
  • Fires onColorSelect on valid entry
  • Syncs to slider/swatch changes

All three sub-views share the same internal event — onColorSelect — via an emitter. When any one of them changes the color, all three update to reflect the new value. This is the event listener pattern from the previous lecture applied inside a module. 

// modules/ColorChooser.js
export default function ColorChooser({
  initialColor = '#3A7AB5',
  swatches,          // optional override of the preset palette
  onColorChange,     // (hexColor: string) => void
  showHex = true,
  showSliders = true,
}) {
  const emitter = useMemo(() => new EventEmitter(), []);
  const [color, setColor] = useState(initialColor);

  useEffect(() => {
    return emitter.on('onColorSelect', (event) => {
      setColor(event.payload.hex);
      onColorChange(event.payload.hex);
      return true;  // all sub-views see the same event to sync themselves
    });
  }, [emitter, onColorChange]);

  return (
    <Panel title="Choose a Color">
      <SwatchGrid
        swatches={swatches || DEFAULT_SWATCHES}
        selectedColor={color}
        emitter={emitter}
      />
      {showSliders && <HSLSliders color={color} emitter={emitter} />}
      {showHex     && <HexInput    color={color} emitter={emitter} />}
    </Panel>
  );
}
💡 Boolean Feature Flags on a Module: Props like showHex and showSliders let the caller configure the module's capabilities without needing to import and assemble the sub-components manually. The module is the single import; the props are the configuration interface. This is sometimes called a "headful/headless" spectrum — the more you strip away, the more the caller composes.

Module Walkthrough 3 — CardManager

A CardManager is a multi-view module — it contains a list view and an editor view, and manages transitions between them. It's the most architecturally complex example: sub-views communicate through a shared event bus, module-level state coordinates both views, and the caller only interacts with a clean public API.

State Architecture

The CardManager owns three pieces of state that both panels depend on: 

// The three state slices inside CardManager
const [cards,        setCards]        = useState(initialCards || []);
const [selectedCard, setSelectedCard] = useState(null);    // card being edited
const [view,         setView]         = useState('list'); // 'list' | 'edit' | 'create'

The Full Module

// modules/CardManager.js
export default function CardManager({
  initialCards = [],
  renderCard,        // render prop — how each card row looks in the list
  emptyState,        // named slot — shown when no cards exist
  header,            // named slot — optional panel header override
  onCardsChange,     // (cards[]) => void — called whenever the collection changes
  maxCards,          // optional limit
}) {
  const emitter = useMemo(() => new EventEmitter(), []);
  const [cards, setCards] = useState(initialCards);
  const [selectedCard, setSelectedCard] = useState(null);
  const [view, setView] = useState('list');

  // Central event wiring — all mutations flow through here
  useEffect(() => {
    const unsub1 = emitter.on('onSelect', (e) => {
      setSelectedCard(e.payload.card);
      setView('edit');
      return true;
    });
    const unsub2 = emitter.on('onSave', (e) => {
      const updated = cards.map(c => c.id === e.payload.card.id ? e.payload.card : c);
      setCards(updated);
      onCardsChange(updated);
      setView('list');
      return true;
    });
    const unsub3 = emitter.on('onDelete', (e) => {
      const updated = cards.filter(c => c.id !== e.payload.id);
      setCards(updated);
      onCardsChange(updated);
      return true;
    });
    const unsub4 = emitter.on('onAdd', () => {
      setSelectedCard(null);  // null = create mode
      setView('create');
      return true;
    });
    return () => { unsub1(); unsub2(); unsub3(); unsub4(); };
  }, [emitter, cards, onCardsChange]);

  return (
    <Panel title="Card Manager" header={header}>
      {(view === 'list') && (
        <CardList
          cards={cards}
          emitter={emitter}
          renderItem={renderCard}
          emptyState={emptyState}
          canAdd={!maxCards || cards.length < maxCards}
        />
      )}
      {(view === 'edit' || view === 'create') && (
        <CardForm
          card={selectedCard}
          emitter={emitter}
          mode={view}
        />
      )}
    </Panel>
  );
}

The CardManager operates as a mini state machine: the view state variable determines which panel renders. Events from child panels trigger transitions. The module's single public output is onCardsChange.

Designing a Clean Prop Surface

A module's prop surface is its public API contract. A well-designed prop surface is small, stable, and unsurprising. These four categories cover the props that most UI modules need:

Category
Prop Type
Purpose & Examples
Data in
Value props
Initial or controlled data the module displays. initialCards, initialDate, initialColor, value
Data out
Callback props
Called when the user produces a result. One primary output per module. onDateChange, onColorChange, onCardsChange
Slots
children / named JSX props
Customizable content regions. children, header, footer, emptyState, renderItem, renderDayCell
Config
Boolean / primitive flags
Feature toggles and constraints. Keep these additive — default to fully-featured, allow callers to strip down. showHex, showSliders, maxCards, minDate, readOnly
⚠ Prop Inflation: When a module's prop list exceeds about 10–12 props, it is often a sign that the module is doing too much, or that its internals have leaked into the public API. Before adding another prop, ask: could this be a named slot instead? Could it be a sub-component the caller assembles themselves rather than a config flag?

Controlled vs. Uncontrolled Modules

React Native follows the same distinction for modules that it does for input components: uncontrolled modules manage their own state; controlled modules have their state driven entirely by the parent.

Uncontrolled (defaultValue pattern)

The module owns its state. The parent provides a starting value and listens for changes via a callback, but does not re-drive the state. Simpler for most use cases.

<DatePicker
  initialDate={today}
  onDateChange={setDue}
/>

Controlled (value + onChange pattern)

The parent owns the state and passes it back down every render. The module fires onChange and the parent updates its state, causing a re-render. Required when parent logic must validate or transform values.

<DatePicker
  value={dueDate}
  onChange={(d) => setDue(d)}
/>
Best Practice: Support both modes in the same module using a standard convention: if value is provided, operate in controlled mode; if only initialValue is provided, operate in uncontrolled mode. React's own TextInput works exactly this way.

Quick Reference — Composition Techniques

Technique How It Works Best Used For
children prop JSX between tags becomes props.children Wrapping layout containers where content is unknown at design time
Named slot props Props that accept JSX values (footer={<...>}) Specific, positioned regions in a structured layout (header, footer, sidebar)
Render props Props that are functions returning JSX; component feeds data back to the slot Lists and grids where the component owns data but caller owns the row design
Specialization A specific component wraps a generic one with fixed prop values Creating themed variants — PrimaryButton wraps Button
EventEmitter (internal) Shared emitter wires sub-components without prop drilling Panels where sibling sub-components need to react to each other
View state machine A state variable ('list' | 'edit' | 'create') switches which panel renders Multi-step or multi-mode modules like wizards, managers, editors

Summary

Composable components are the most important skill in scaling a React Native codebase. The techniques in this lecture work together — you will use several simultaneously in any real module:

  • Think in four levels: atoms build molecules, molecules build panels, panels build modules.
  • The Panel primitive provides visual structure; content comes through slots.
  • Use children for unknown content, named slot props for positioned regions, and render props when the component must feed data back to the slot.
  • Modules own a narrow public API — data in, one primary callback out, slots for customization, boolean flags for configuration.
  • Use an internal EventEmitter to wire sub-component communication without exposing that wiring to the caller.
  • A view state variable turns a module into a mini state machine, switching which panel is active.
  • Support both controlled and uncontrolled modes using the value / initialValue convention.
