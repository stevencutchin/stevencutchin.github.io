Final Project

A Real App. On Your Phone. In a Job Interview.
👥 Teams of 3 ⭐ 250 points 📱 Live demo required 📅 Presentations: May 7

The final project is your capstone — a complete, working mobile application built by a team of three, presented live during finals week. This is not a prototype or a mockup. It runs on a real phone. A stranger should be able to pick it up and use it without explanation. That's the bar.

Everything you've built toward this semester — components, navigation, sensors, databases, networking, platform handling, testing — comes together here. The project is deliberately open-ended: you choose what to build, you decide the architecture, and you own the result. The deliverables along the way keep you on track and give you a professional document trail to go with the code.

The goal behind the goal: You will walk into a technical job interview with this app on your phone. The interviewer will tap it, something will happen, and you will explain how you built it. That moment — the working app plus the ability to talk about what's under the hood — is what this project is for. Build something you're proud to show.

2. Teams

Teams are three people. Team composition is your choice — pick people you can work with, not just people you know. The project is large enough that uneven contribution will hurt the entire team's grade and make the final weeks miserable.

🏗 Architecture Lead
  • Owns the technical structure
  • Navigation and data flow decisions
  • Writes the Architecture Document
  • Ensures components have clean interfaces
🎨 Feature Lead
  • Drives feature implementation
  • UI consistency and polish
  • Coordinates work session demos
  • Writes the User Manual
🔗 Integration Lead
  • APIs, databases, external services
  • Testing and error handling
  • Manages submission package
  • Coordinates the final presentation
⚠ On contribution: All team members are expected to make meaningful code contributions visible in your Git history. "I did the design document" is not a substitute for writing code. Peer evaluations are part of the final grade — you will rate your teammates, and they will rate you. Significant contribution imbalance will be reflected in individual scores.

3. Deliverables and Timeline

The project is structured as a series of deliverables that build on each other. Each one exists for a reason — the proposal forces you to decide what you're building before you start building it; the design document forces you to think before you code; the architecture document captures decisions you'll be asked about in the presentation.

Due: Tuesday, April 7
Project Proposal
5 points
2 pages minimum. This document answers three questions: what are you building, why will it be impressive to a future employer, and how will your team divide the work.
  • App name, concept, and the problem it solves
  • The three (or more) course topics it demonstrates
  • Team member names and their primary responsibilities
  • Three core features you commit to finishing, plus two stretch goals
Submit as PDF via Canvas. Both team members confirm submission in comments.
Due: Tuesday, April 21 — Project Work Session II
Design Document
35 points
A complete design document covering the user experience and visual design of your app. Think of this as the document a new team member would read to understand what the app does and how it should feel.
  • Screen inventory — one paragraph per screen, what it does and how the user reaches it
  • Navigation map — a diagram of how screens connect
  • Data model — what data the app stores and how it's structured
  • UI wireframes or mockups for each major screen (hand-drawn is fine)
  • External services and APIs the app will use
Bring a printed copy to the April 21 work session for peer review.
Due: Tuesday, April 28
Architecture Document
35 points
A technical document describing how your app is built — not what it does, but how. This is what you'll refer to during your presentation Q&A.
  • Component hierarchy — major components, their props, and their relationships
  • Data flow — where state lives and how it moves between components
  • Where your app sits on the thin-to-rich client spectrum and why
  • Platform-specific decisions — any iOS/Android differences you handled
  • Testing strategy — which parts are tested and at what layer
  • Key technical decisions and the alternatives you considered
Due: Monday, May 4
User Manual
35 points
A clear, friendly guide to using your app — written as if for someone who has never seen it. Screenshots required. If a user needs a workaround to use a feature, document the workaround (then fix the feature if you can).
  • Getting started — installation or first-launch instructions
  • Feature walkthrough — one section per major feature with screenshots
  • Known limitations — anything that doesn't work perfectly yet
  • Troubleshooting — the three most likely things to go wrong and how to fix them
Due: Monday, May 4
Source Code + Final Package
70 points — 35 code + 35 package
Submit a clean, working codebase and a complete submission package.
  • Source code: GitHub repo link with full commit history. All three team members must have commits. Include a README.md with setup instructions — a TA should be able to clone and run with no additional guidance.
  • Final package: A single zip or folder containing your proposal, design document, architecture document, user manual, and any additional assets (demo video, screenshots). Organized and labeled.
Thursday, May 7 · 9:30–11:30 AM · Finals Week
Final Presentation
35 points
A live demo and technical walkthrough. Every team member speaks. The app runs on a real physical device during the demo — not a simulator, not a video.
  • Demo (5 min): Show the app working. Let it speak for itself before you explain it.
  • Architecture walkthrough (5 min): Explain one technically interesting decision — what you chose, what you considered, and why.
  • Q&A (5 min): Any team member may be asked about any part of the codebase. You are expected to know your own app.
Each team has approximately 15 minutes total. Be ready when called — we keep to the schedule.

5. Technical Requirements

Every project must demonstrate breadth across the course topics. A project that is entirely local with no sensors and no networking misses the point. These are the minimum technical requirements — you are expected to exceed them.

📱 Must use (at minimum)

  • React Native with Expo SDK
  • Expo Router for navigation (2+ screens)
  • At least one device sensor or camera
  • Local persistence (AsyncStorage or SQLite)
  • At least one external API or network call
  • Platform-appropriate UI (iOS and Android)

✅ Quality expectations

  • Runs on a real physical device without crashing
  • No placeholder data or "TODO" labels in the demo
  • Error states handled gracefully (no blank screens)
  • Unit tests for core utility functions
  • Git history shows all three team members contributing
  • README enables a stranger to clone and run the app

🏆 To aim for

  • Something a stranger would actually want to use
  • Loading and empty states that feel intentional
  • Consistent visual style across all screens
  • App icon and splash screen (not the Expo defaults)
  • Works offline or degrades gracefully when offline

🚫 Automatic deductions

  • App crashes during the demo — significant
  • Simulator demo instead of real device — significant
  • Team member can't explain their own code — individual
  • No README or broken setup instructions — package points
  • Missing Git history for a team member — code points

7. How to Submit

All deliverables are submitted via Canvas. One team member submits on behalf of the team — include all team member names in every submission.

  • Documents: PDF preferred. Name files clearly: TeamName_Proposal.pdf, TeamName_Design.pdf, etc.
  • Source code: Submit the GitHub repo URL. Make sure the repo is public or the instructor has been added as a collaborator before the deadline.
  • Final package: A single Canvas submission containing all documents organized into a folder or zip. Label everything.
  • Peer evaluations: Submitted individually, privately, via the Canvas peer evaluation tool. Due by end of finals day.
Questions? Start with your team. Then Canvas discussion. Then office hours. The earlier you surface a scope or technical concern, the more options you have. A concern raised in Week 12 is solvable. The same concern raised the night before the presentation is not.
