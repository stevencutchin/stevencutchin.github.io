Final Project — Expectations & Walkthrough

Build Something Real. Document It Honestly. Present It Proudly.
🏁 250 Points 📅 Teams of 3 🎤 Presentations — Finals Week, May 7 📦 4 Deliverable Documents + Source Code

What the Final Project Is

The final project is your team's chance to build a real mobile application from scratch — something you conceived, designed, implemented, and can demo live. This is the kind of work you can point to in a portfolio or walk through in a job interview.

Every team picks an original app idea and carries it from proposal through working code and documentation. The course has given you all the building blocks: components, state, navigation, sensors, networking, persistence, and architecture. The final project is where you assemble them into something coherent.

The core goal: A working React Native (Expo) app that a real user could install and use, accompanied by documentation that shows what it does, how it is built, and how to operate it.

Below you will find a breakdown of every deliverable, annotated examples drawn from past student projects, and guidance on what distinguishes strong submissions from weak ones.

The Five Deliverables at a Glance

📋 Project Proposal
Submitted early — gates team approval
  • App concept and purpose
  • Target users and problem solved
  • Core features (MVP scope)
  • Proposed tech stack
  • Rough development timeline
🎨 Design Document
UI/UX decisions and screen-level layout
  • Screen-by-screen UI walkthrough
  • Navigation flow description
  • Screenshots or mockups of each view
  • Key UX decisions explained
🏗 Architecture Document
Technical design and component structure
  • Tech stack overview and rationale
  • Component / class / module breakdown
  • Data flow and API communication
  • Backend or services description
📖 User Manual
How to install and use the app
  • Installation / setup instructions
  • Feature-by-feature usage guide
  • Screenshots illustrating each step
  • Known limitations or caveats
Source Code is submitted alongside the documents — a zip of your complete Expo project, or a link to your team's repository. The grader will attempt to run it. Make sure it starts without hidden configuration steps.

How the Deliverables Flow

The four documents are produced in roughly this order as development progresses. The Presentation synthesizes all of them.

Chapter 1 — The Project Proposal

1
Proposal

The proposal is a short document — typically one or two pages — that shows the instructor your team has a viable idea and a reasonable plan to execute it. It is submitted before the bulk of development begins, so any feedback you receive can actually shape what you build.

What to Include

1
Overview paragraph. What does the app do? Who would use it, and what problem does it solve for them? Concrete descriptions work best — "an app for K-12 schools that helps teachers and parents coordinate student pick-up" is more memorable than "a communication app."
2
Core features — MVP scope. List the features you are planning to build. Being honest about what is core versus what is stretch is actually a strength here — a working app with three solid features tends to make a better impression than an ambitious app that crashes on launch.
3
Proposed tech stack. What frameworks and services will you use? React Native + Expo is the expected frontend. If you need a backend or a third-party API, naming them here and explaining why you chose them goes a long way.
4
Development timeline. A rough week-by-week plan helps you notice gaps before they become crises. It is completely fine if the plan shifts as development progresses — the value is in thinking through what needs to happen and when.
EXAMPLE 1
From the "Amber" Check-in/out App Proposal (Fall 2021)
"Amber is made for parents and faculty in a K-12 learning environment. Our primary goal is to increase security by helping ensure kids don't wander off or worse, get kidnapped. Our secondary goal is to speed up pick-up queues after school ends. The main functionality would be when a parent with the app gets close to the school, our app ensures that they are within a certain distance from the school (using a geofence)."
✅ What works: The overview immediately states two concrete goals in plain language. It names the target audience (parents, teachers), the mechanism (geofence), and the problem (safety + queue time). Someone who has never heard of this app could explain it back to you after reading this paragraph.
EXAMPLE 1
Tech Stack and Timeline from the Same Proposal
"For our front-end, we will be using React Native. We chose this because React Native reduces development time for cross-platform functionality… For our back-end functionality, we will be using the Azure cloud. This is mainly because we are most familiar with a Microsoft tech stack."
✅ What works: The team gives a reason for each choice, not just a list of technologies. "We are most familiar with a Microsoft tech stack" is a perfectly valid reason — honest rationale beats fabricated technical justification every time.
On timelines: The timeline is a thinking tool, not a contract. If you write "Connect Frontend/Backend — Week 3" and it slips to Week 4, that is completely fine. What the timeline really does is help you ask: do we have enough time for everything we want to build? If the answer feels uncertain, it is better to narrow scope in the proposal than discover the problem the night before submission.

Chapter 2 — The Design Document

2
Design

The design document is a screen-by-screen tour of your app's user interface. Think of it as what you would hand to a new team member on their first day so they could understand what the app looks like and why each screen is built the way it is.

What to Include

One section per distinct screen or major UI state works well. For each screen, you might cover:

  • What the user sees and what actions are available
  • Key layout decisions and why you made them
  • How the UI communicates state or feedback to the user (e.g., loading indicators, error text, color changes)
  • Screenshots of the actual running app (or high-fidelity mockups if the app is not yet complete)
EXAMPLE 1
Teacher View — Amber App Design Document
"The teacher view shows a list of all of the students that are in their class. In addition there is a button at the top that allows them to open and close a session. The button is placed outside of the list container which means it will always be at the top no matter how far the teacher scrolls down the student list. The color of the checkout button also changes to help tell the teacher if the session is currently opened or closed."
✅ What works: This paragraph explains a specific layout decision (button outside the scroll container) and the reason behind it (always visible regardless of scroll position). It also describes a UX affordance (color change communicates session state). This is the level of specificity a good design doc has — not just "there's a button" but "here's why the button is placed there and what it communicates."
EXAMPLE 2
Landing Page — Relatable Weather App Design Document
"The main centerpoint of our application is the funny quip provided when first opening the application. The first content on the landing page is the funny quip which changes every time the application is re-opened. The quip is also meant to be tailored to the reported weather and use up to date slang and funny quotes."
✅ What works: This team led with the defining characteristic of their app — the thing that makes it unique. A design doc should reflect the app's identity, not just list generic "there is text here and a button there" descriptions. Their design section for extra weather info cards is also strong: they explain that the cards are intentionally minimal because "the purpose of the landing page is to provide minimal information to convey the weather while also trying to be funny and relatable." Purpose-driven design choices are what elevate a design doc.

A strong design doc section doesn't just describe what a screen looks like — it explains the reasoning behind layout choices, state feedback, and interaction design.

Something to watch for: It's easy to drop in a grid of screenshots without much explanatory text — but screenshots without commentary leave the reader doing all the work. Even a sentence or two per screen about what the user can do there and why it's designed that way makes the document significantly more useful.

Chapter 3 — The Architecture Document

3
Architecture

The architecture document describes how your app is built technically. While the design document answers "what does it look like and how does the user experience it?", the architecture document answers "how does it work under the hood?" A useful mental model: it's what a developer would want to read before extending or maintaining your codebase.

What to Include

Tech Stack Overview

List the key technologies and services your app depends on. For each one, a brief explanation of why you chose it adds a lot. Not just "we used Firebase" — but what Firebase is doing for you and why it was the right fit.

Component / Module Breakdown

Describe each major piece of your codebase: the screens, helper modules, data classes, and any custom components. Explain what each one is responsible for and how it connects to the others.

Data Flow

Trace how data moves through your app: from a user action, to a state update, to a network request, to a database write, and back to the UI. Diagrams are particularly effective here.

Backend / Services

If your app communicates with a backend or third-party API, describe what each endpoint does, how you authenticate, and how the mobile client consumes the responses.

EXAMPLE 1
Tech Stack + Data Flow — Amber App Architecture Document
"These services end up interacting with each other in three main ways. The first is when a user performs an action on the device. It updates the local state and UI and then sends a request off to the backend. The backend handles the request, normally updating a record in our database and telling Firebase to send a message to registered devices. The second way is when the user changes screens… The third way is when the user receives a notification from Firebase. Throughout all these interactions the data in the database acts as the source of truth."
✅ What works: This team identified three distinct interaction patterns and described each one as a flow — action → state update → backend → Firebase → devices. Calling out the "source of truth" concept shows genuine architectural thinking, not just code narration.
EXAMPLE 2
Module Breakdown — Relatable Weather App Architecture Document
"The WeatherObject data class is used primarily as an interface to assign the data received from the weather API for easy manipulation and data assignment. This covers the most important fields received from the API and stored the fields that we require to be accessed and manipulated later… The APIHelper class is just a small helper class with a companion object that contains a single function called LoadItems(). This function's purpose is to make a request to our API using some query passed in as a parameter which it does on a new thread."
✅ What works: Each class gets its own paragraph that answers: what is it, what does it do, and who uses it. The "companion object / new thread" detail shows the team actually understands why the class exists, not just that it exists. Your architecture doc should reach this level of specificity for each major file or module in your Expo project.

Not all apps need all three tiers. A lightweight app might skip a custom backend and talk directly to a third-party API. Your architecture document should show what your app actually uses — not an idealized diagram.

Architecture vs. Design: These two documents cover different ground. The design document says "there is a button at the top of the teacher screen that changes color." The architecture document says "the button's color is determined by the sessionOpen boolean in component state, which is set by a POST request to /api/session/toggle." One is about the user experience; the other is about what is happening in the code. Both are interesting — just in different ways.

Chapter 4 — The User Manual

4
User Manual

The user manual is written for someone who has never seen your app before. The goal is to give them everything they need to install it, start it, and explore every feature without getting stuck. Writing for a non-technical reader is a helpful mindset — if your roommate could follow it, a grader certainly can.

What to Include

1
Installation instructions. How does a new user get the app onto their device? Do they need Expo Go? A TestFlight link? An APK? Can they clone and run it? Be precise — include any dependencies or setup steps they need to complete first.
2
Feature walkthrough. One section per major feature. Explain what the user does, what happens, and what they see. Annotated screenshots are enormously helpful here.
3
Known limitations. Being upfront about what your app cannot do, or what might not work in certain conditions, is actually a sign of professional maturity. Readers appreciate honesty, and it's much better than leaving them confused when something doesn't work as expected.
EXAMPLE 1
Feature Walkthrough — Amber App User Manual
"When a parent logs in, they will be met with a list including their Child/Children attending the school. The Parent will be notified when school has ended. After this notification and after they have entered the specified geofenced area (predetermined by the school), they will have the ability to check out their child (via pressing on Request Student). Finally, after picking up the child they will be prompted with a button (Confirm Student Has Arrived) to communicate with the Teacher that the pickup has happened."
✅ What works: This paragraph follows the user through a complete workflow from login to confirmation. It names the actual UI elements ("Request Student," "Confirm Student Has Arrived") in parentheses so the reader knows exactly what to look for on screen. This is significantly more useful than "the parent can check out their child."
EXAMPLE 2
Installation Instructions — Relatable Weather App User Manual
"If you don't have an android device, Install Android Studio… Otherwise, you can sideload the APK file by plugging your device into a computer and dragging / dropping the APK into your phone's internal storage… Unfortunately, this app isn't on the app store so you will need to ensure that you can open the APK file for the app from Android Studio."
✅ What works: The team acknowledges upfront that the app isn't on the App Store and gives the user two alternative paths (Android Studio or sideloading). The honest "unfortunately" is a good sign — they are meeting users where they are. Your manual should be equally candid about any friction in getting your app running.
Distribution tip: For Expo-based apps in 2026, the easiest path for the grader is usually Expo Go with a shared project link, or a published EAS build. Including a QR code or URL that lets someone run your app on a real device in under two minutes is a really nice touch.

Chapter 5 — The Source Code Submission

5
Code

Your source code is submitted as either a zip of the entire Expo project directory or a link to a private GitHub repository. Either way, the grader should be able to clone or unzip it and run npx expo start without any missing steps.

Before You Submit

1
Remove secrets. API keys, passwords, and tokens should not be in your source code. A .env file added to .gitignore is the standard approach. If your app needs secrets to run, noting that in the README and providing a way for the grader to supply their own is much appreciated.
2
Include a README. A short file explaining how to install dependencies (npm install) and start the app (npx expo start) is really helpful. Any quirks worth knowing — simulator vs. real device, which platform is tested — are good things to mention there too.
3
Test a clean install. Unzipping your own submission into a fresh directory and following your own README is the single best way to catch any issues before the grader does. It only takes a few minutes and can save a lot of confusion.
4
Leave out node_modules. Including node_modules in a zip produces enormous archives and the grader will run npm install anyway. A .gitignore or a quick folder delete before zipping takes care of this.

Chapter 6 — The Final Presentation

6
Presentation

Presentations happen during the finals week exam slot on May 7, 9:30–11:30 AM. Every team presents. The audience includes the instructor and your classmates.

The presentation is a live event — not a slide reading. Walking the audience through your app as if giving a product showcase demo tends to work really well. Show real features on a real device or simulator, talk through the architecture in your own words, and enjoy the Q&A — it's a chance to show how deeply you understand what you built.

🎬 What to Cover

Brief intro to the problem your app solves; live demo of the core user workflow; architecture diagram walkthrough; what you learned building it; Q&A.

⏱ Timing

Aim for 8–12 minutes of presentation followed by a short Q&A. Practice your demo at least twice — live demos have a way of failing when it matters most.

A good backup plan for demos: Recording a short screen capture of your app working before presentation day is really worth the ten minutes it takes. If the network goes down or a device misbehaves, you can show the recording instead — and keep things moving smoothly.

Grading at a Glance

Component What We Are Looking For
Project Proposal Clear problem statement, realistic scope, named tech stack, rough timeline
Design Document Screen-by-screen coverage, layout rationale, UX decisions explained, screenshots
Architecture Document Tech stack with rationale, component/module breakdown, data flow, backend described
User Manual Works for a non-technical reader, complete installation steps, full feature walkthrough
Source Code Runnable without hidden steps, no secrets committed, README present, node_modules excluded
Working App Core features function as documented; app launches without crashing; UI is navigable
Presentation Live demo, architecture walkthrough, all team members speak, responds to Q&A
Grading philosophy: Credit is awarded for what is there, not deducted for what is missing. A complete, honest, modest app tends to do really well — often better than an ambitious app with a missing document or a broken build. Acknowledging what your app does not yet do in your documentation is never penalized — it's actually appreciated.

What Makes a Strong Project

Looking across the example projects and the strongest submissions from past semesters, a few patterns distinguish the great from the merely adequate:

✦
Coherent identity. The strongest apps tend to have a clear point of view. "Amber" is for school safety. "Relatable Weather" is about personality-driven weather data. A coherent identity makes every design and architecture decision easier to explain — and easier to read.
✦
Honest scope management. Both example projects built exactly what they said they would in their proposals. They did not overpromise. A delivered MVP is worth more than an ambitious specification with half the features implemented — and it's a lot less stressful to work toward.
✦
Documents that match the code. The most confusing submissions are ones where the documents describe features the app doesn't have, or overlook features it does. Writing the documents once the code is in a stable state makes this much easier to get right.
✦
Explanations, not just lists. "We used Azure Functions because they are lightweight and the documentation is better than AWS" is architecture thinking. "We used Azure Functions" is a bullet point. Sharing the reasoning behind your choices is what makes a document genuinely interesting to read.
✦
A user manual a stranger could follow. Handing your user manual to a friend who hasn't seen your app and watching them try to install and use it is a great way to spot gaps. If they get stuck anywhere, that section probably needs a little more explanation.
The big picture: The final project is a capstone. It is meant to show that you can take an idea from concept to working software and communicate what you built to both technical and non-technical audiences. That combination of building and explaining transfers directly to industry — it is genuinely what professional development looks like. The documentation is as much a part of the work as the code itself.
