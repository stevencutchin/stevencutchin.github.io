CS 402 — Mobile Application Development

Flex Layout and Modular UI Composition
🕐 Total time: 75 minutes 📅 Semester: Spring 2026 🎓 Type: Code-heavy / Live demo throughout 📄 Student doc: CS402_Flex_UI_Layout.html 🔙 Prior context: Theme 2 covered Flexbox basics 🔜 Next class: Final Project Introduction

Lesson Overview

This is the second time students encounter Flexbox in the course — but for many, it's the first time it will make sense. The Theme 2 introduction gave them the property names. This lecture gives them the mental model that makes those properties predictable. The central shift is from memorising properties to understanding the axis system — after which every Flexbox behavior becomes deducible rather than guessable.

The second half of the lecture introduces modular component design through the Atomic Design framework. This transition from layout mechanics to design discipline is the most conceptually demanding part of the session. Budget the first half of class for building confident Flexbox intuition, then use that foundation to motivate why structure matters as components grow.

This lecture comes just before the Final Project Introduction, which makes it the last major technical foundation session before students scope their own work. Ending with the modular design checklist gives them a framework to apply immediately when planning their final project component structure.

What Makes This Lecture Different

  • Students have prior exposure but fragile understanding. Most can copy-paste Flexbox properties that work. Far fewer can explain why a layout broke or predict what a change will do. The goal is closing that gap.
  • Live coding is essential. Flexbox is a visual system. Describing properties verbally is the least effective way to teach it. Every major concept needs a running code demo students can watch break and fix in real time.
  • The flex: 1 chain bug is universal. Every student in the room has seen their layout collapse to zero height at least once. Opening with a live demo of that specific bug and its fix creates immediate buy-in.
  • Atomic design is new vocabulary for existing instincts. Students already know that big components feel wrong. This lecture gives them the language and a principled framework to act on that instinct.

Learning Objectives

Explain
Describe how the main axis and cross axis work in Flexbox and identify which properties control each.
Predict
Given a set of Flexbox properties, predict how children will be positioned and sized without running the code.
Apply
Build the header/scroll/footer layout pattern correctly, including the full flex: 1 ancestor chain.
Apply
Write responsive layouts using useWindowDimensions that adapt column count or layout pattern to screen width.
Analyse
Decompose a given screen design into atomic layers (atoms, molecules, organisms, screens).
Apply
Separate layout concerns from content concerns when composing components, using the children prop and layout component patterns.

Prerequisites Students Should Have

  • Basic Flexbox exposure from Theme 2: flexDirection, justifyContent, alignItems, flex
  • Props and component composition from lectures 4–5
  • StyleSheet.create() and the difference between React Native styling and CSS
  • At least one completed assignment where layout bugs were encountered — the more frustrated they were, the more receptive to this lecture
⚙️ Prep required before class: Create a bare Expo project with a single screen containing a broken layout — specifically the flex: 1 chain bug where a ScrollView collapses to zero height because a parent is missing flex: 1. This will be your opening live demo. Also prepare a second file containing the four canonical layout patterns from Section 3 of the student doc, ready to toggle between. Demoing from pre-built files rather than typing from scratch keeps the energy up and eliminates typo-induced detours.

1
Hook — The Layout Bug Everyone Has Hit
Open with a live demo of the most common Flexbox failure before explaining anything
⏱ 7 min

Start with the broken code, not the explanation. Students who have hit this bug — and nearly all of them have — will lean forward immediately when they recognise it.

🖥️ Open the pre-built broken project. Show this on screen: A screen with a header, a ScrollView that should fill the middle, and a footer pinned at the bottom. The ScrollView is invisible — collapsed to zero height. The footer is floating up against the header. Ask the class: "Who has seen this before?" Then: "Who knows why it's happening?"
  • Let silence sit for 10–15 seconds after "who knows why?" Students who think they know will volunteer. Students who don't will become curious. Both are useful states to be in.
  • Don't fix it yet. Instead, use the brokenness to motivate the axis model: "To understand why this is broken, you need to understand how Flexbox actually calculates where things go. That's what the first part of today is about. We'll come back and fix this in about fifteen minutes."
  • Frame the session goal clearly: "By the end of today you should be able to look at any layout bug and explain in one sentence why it's happening. Right now most of you are guessing. That ends today."
💬 Quick poll before moving on: "Show of hands — how many of you have solved a Flexbox layout problem by randomly adding flex: 1 to things until it worked?" Most hands will go up. Acknowledge it without judgment: "That's the sign of someone who knows the properties but not the model. Today we fix the model."
2
The Axis Model
Main axis vs. cross axis, flexDirection, justifyContent, alignItems
⏱ 12 min
🖊 Draw before showing the diagram
Draw two rectangles side by side on the board — one labeled "column" (arrow pointing down), one labeled "row" (arrow pointing right). Label the arrows "MAIN AXIS." Draw a perpendicular arrow on each and label it "CROSS AXIS." Then write: justifyContent = main, alignItems = cross. This takes 60 seconds and gives students a spatial anchor before you project Figure 1 from the student doc.

Talking Points

  • The single most important sentence in this lecture: "justifyContent controls the main axis. alignItems controls the cross axis. Those axes swap when you change flexDirection." Say it, write it on the board, and say it again after the diagram. This one sentence explains 80% of Flexbox layout decisions.
  • React Native defaults to flexDirection: 'column' — so the main axis points down by default. This is the opposite of web CSS, where flexDirection defaults to 'row'. Students coming from any web background will have this reflex burned in wrong. Name it explicitly.
  • Walk through Figure 2 (justifyContent variants) from the student doc. For each value, ask students to predict the result before revealing it. space-between and space-around are consistently confused — give the mnemonic: "between pins to edges, around wraps each item."
  • alignItems: 'stretch' is the default and the source of the "all my children are the same height even though they have different content" question. Name it. It stretches all children to fill the cross axis dimension of the tallest sibling.
  • alignSelf on a child overrides the parent's alignItems for that child only. Quick to mention, occasionally very useful.
🖥️ Live demo — toggle flexDirection in a running app: Have three colored boxes inside a View. Toggle flexDirection between 'column' and 'row' live. Then keep it on 'row' and toggle justifyContent through all six values. Students watching the boxes move in real time internalise the model faster than any diagram. Keep this under 4 minutes — it's a demonstration, not a deep dive.
✅ Check for Understanding
  1. "I have flexDirection: 'row' and I want my children centered vertically. Which property do I use?"
    → alignItems: 'center' — because when direction is row, the cross axis is vertical.
  2. "I have flexDirection: 'column' and I want equal space between children but no space at the edges. Which justifyContent value?"
    → 'space-between' — pins items to the edges of the main axis with equal gaps between.
3
flex, flexGrow, flexShrink, flexBasis — The Distribution Math
What flex: 1 actually does, and why the chain must be unbroken
⏱ 10 min

This is the segment that resolves the bug from Segment 1. Keep the broken code visible in a corner of the screen throughout this explanation — students should be able to see exactly how what you're explaining maps to what's broken.

Talking Points

  • Establish the mental model for flex: 1 before anything else: "Flex: 1 means 'take up my share of whatever space is left over after fixed-size siblings are placed.' If there is no space left over, there is nothing to claim." This is the key to understanding why a container with no height produces invisible children.
  • Walk through Figure 3 from the student doc: three siblings with flex: 1, flex: 2, flex: 1. Total parts = 4. The middle child gets 2/4 = half the available space. This arithmetic is all Flexbox is doing under the hood.
  • Introduce the flex chain rule: for any child to fill available height, every ancestor up to the screen root must have either a defined height or flex: 1. An ancestor with neither has zero computed height, which means its children have zero space to claim, which means flex: 1 children inside it are zero height. Draw this as a chain of containers on the board — break one link and the whole chain fails.
  • Keep flexGrow, flexShrink, flexBasis brief. Students almost never need to set them separately. The message: flex: 1 is shorthand for { flexGrow: 1, flexShrink: 1, flexBasis: 0 }, which means "start from zero, grow proportionally, shrink proportionally." That's it.
🖥️ Fix the broken layout from Segment 1 live: Return to the broken project. Add flex: 1 to the screen root. Still broken? Show that the intermediate container also needed it. Add it. Fixed. This two-step fix — and watching the layout snap into place — is more instructive than ten minutes of explanation. Students remember the fix because they watched it happen.
⚠️ The contentContainerStyle trap: Students frequently put flex: 1 on contentContainerStyle in a ScrollView instead of on the style prop. This silently disables scrolling — the content container expands to fill available space and never overflows, so there is nothing to scroll. Mention this explicitly. It is one of the most time-consuming bugs to diagnose without knowing to look for it.
✅ Check for Understanding
  1. "Three sibling Views have flex values of 1, 3, and 1. The parent is 500px tall. How tall is the middle View?"
    → Total parts = 5. Middle gets 3/5 × 500 = 300px.
  2. "A View with flex: 1 inside a View with no height set and no flex. What happens?"
    → The parent has zero height (nothing forces it taller). The child has flex: 1 = "a share of zero available space" = 0px. Both are invisible.
4
Four Canonical Layout Patterns
Header/scroll/footer, sidebar/content, grid, card row — and when to use each
⏱ 13 min

This segment is the payoff for the axis model work. Students should be able to recognize each pattern, name the Flexbox properties that produce it, and know which pattern to reach for in a given design situation. Demo all four from the pre-built file — switching between them is far faster than building each one.

Talking Points

  • Header / Scroll / Footer: The most common mobile screen structure. Root is flex: 1, flexDirection: 'column'. Header and footer are sized by their own content (no flex). The middle ScrollView has flex: 1 — it claims all remaining space. Project Figure 4 from the student doc and point to each zone.
  • Sidebar / Content: Root is flex: 1, flexDirection: 'row'. The sidebar has a fixed width (no flex). The content area has flex: 1. This is the tablet layout pattern and also common for master-detail navigation on larger phones in landscape.
  • Grid: Two approaches: FlatList with numColumns (handles virtualization for large datasets), or flexWrap: 'wrap' with percentage widths on items (simpler, better for small fixed grids). If the dataset could grow large, always use FlatList. Remind students that numColumns requires a key prop change when the column count changes, or React will not re-render correctly.
  • Card row (avatar + content): A card with an inner flexDirection: 'row'. The avatar has a fixed width. The text content column has flex: 1. This is the list item pattern that appears in almost every app — contact lists, message threads, notification feeds.
📱 Prediction exercise — 3 minutes
Project a screenshot of a real app screen (Instagram feed, Settings, or Maps — something students recognise). Ask: "What layout pattern is the root of this screen? What's the flexDirection of the list items? Where do you see flex: 1 happening?" Give them 90 seconds to discuss with a neighbor, then cold-call 2–3 pairs. This makes pattern recognition active rather than passive.
📝 Connect to the final project explicitly: Say directly: "Next class you're getting your Final Project brief. Every screen you design will be one of these four patterns or a combination. Before you write any code, sketch the screen and label the pattern. It takes 30 seconds and saves an hour of debugging."
✅ Check for Understanding
  1. "You're building a chat app. Each message bubble has a user avatar on the left and message text on the right. What pattern and flexDirection do you use inside each message component?"
    → Card row pattern. flexDirection: 'row' inside the message. Avatar has fixed width; text container has flex: 1 to fill remaining space.
  2. "You have a grid of photos. The dataset comes from an API and could be thousands of items. Should you use flexWrap or FlatList numColumns?"
    → FlatList with numColumns. flexWrap renders all items at once; FlatList virtualizes and only renders what's on screen. For large datasets, flexWrap will crash or freeze the app.
5
Responsive Layouts
useWindowDimensions, adapting column count, and handling rotation
⏱ 8 min

Keep this segment tight. The concept is straightforward once the pattern is demonstrated — students don't need to master responsive design today, they need to know it's achievable and know the right hook to reach for.

Talking Points

  • Hard-coded pixel widths are the enemy of responsive layout. Show the specific failure: a width: 300 that looks perfect on iPhone 15 and overflows on iPhone SE. Width should almost always be a percentage, flex: 1, or calculated from useWindowDimensions.
  • useWindowDimensions() is a hook that returns { width, height } and re-renders the component when the device rotates or the window size changes. It's the correct tool inside components. Dimensions.get('window') is the static version — use it outside components, in StyleSheet.create() or utility functions, but it won't update on rotation.
  • The column count pattern from Section 4 of the student doc: calculate numColumns based on width threshold, then calculate itemSize as (width - padding) / numColumns. Simple arithmetic that produces a grid that always fits. The key={numColumns} on FlatList is easy to forget — without it, React doesn't know to rebuild the grid when columns change.
  • Breakpoint heuristic for this course: under 600dp = phone layout, over 600dp = tablet layout. Students don't need a full responsive design system for their projects — a single breakpoint covers all practical cases.
🖥️ Quick demo — rotate the simulator: If using a simulator, rotate it while a useWindowDimensions demo is running. Students watching the layout reflow in real time on rotation — especially a grid changing from 2 to 3 columns — makes the concept immediately concrete. This takes under 2 minutes and is worth doing even if you skip other demos.
6
Absolute Positioning and Overlays
FABs, badges, overlays — when to step outside the flow
⏱ 7 min

Absolute positioning is a narrow but important tool. Students frequently reach for it too early (trying to solve layout problems that flex handles better) or too late (not knowing it exists when they genuinely need an overlay). The goal here is calibrating when it's the right choice.

Talking Points

  • Absolute positioning removes an element from the normal flow. Its siblings don't know it exists — it doesn't push them or take space from them. It sits at coordinates relative to the nearest parent with position: 'relative' (which is every View by default).
  • The three canonical use cases: (1) Floating Action Button — pinned to a corner of the screen over everything else. (2) Notification badge — small indicator overlaid on an icon. (3) Loading overlay — semi-transparent cover over a screen while data loads.
  • The four positioning properties — top, right, bottom, left — are offsets from the corresponding edge of the containing parent, not the screen. Students often expect them to be screen coordinates and are confused when nothing moves where expected.
  • Shadow properties: elevation for Android, shadowColor/shadowOffset/shadowOpacity/shadowRadius for iOS. You need both. Mention this clearly — "I guarantee someone's final project will have shadows on one platform and not the other."

❌ Common mistake — position without dimensions

// Forgot width and height —
// the FAB is 0×0 and invisible
fab: {
  position: 'absolute',
  bottom: 24,
  right: 24,
  // no width, no height!
}

✅ Correct — explicit dimensions required

// Absolute elements must define
// their own size — no parent to
// stretch or constrain them
fab: {
  position: 'absolute',
  bottom: 24, right: 24,
  width: 56, height: 56,
  borderRadius: 28,
}
7
Atomic Design — Thinking in Layers
Atoms, molecules, organisms, templates, screens — and the discipline of decomposition
⏱ 12 min

This is the conceptual pivot of the lecture — from "how does layout work" to "how do I structure components as my app grows." The transition should be made explicitly: "Now that you understand how to lay things out, let's talk about how to organise what you're laying out." Students who are engaged with the layout content will carry that energy into this section.

🖊 Draw the hierarchy before projecting Figure 7
Write five boxes left to right on the board: Atoms → Molecules → Organisms → Templates → Screens. Ask students to predict what might belong in each level before you fill them in. Most students can correctly guess "screens" and "atoms" — the middle three are where interesting discussion happens. Let them argue about whether a SearchBar is an atom or a molecule. (It's a molecule — it combines a TextInput atom with an Icon atom.)

Talking Points

  • Atoms are single-purpose primitives: a Text styled as a heading, a circular Avatar, an Icon button, a Divider. They can't be meaningfully broken down further. They take props, they render a single thing, they know nothing about the app's data structure.
  • Molecules combine 2–4 atoms into a unit with a single clear purpose. A SearchBar combines a TextInput and a search icon. A UserTag combines Avatar, name Text, and timestamp Text. The rule of thumb: if you can name what it does in two words, it's a molecule.
  • Organisms are the first components that manage their own state. A PostCard knows how to display a post. A NavBar knows about routes. A CommentThread fetches and renders its own data. Organisms are the primary reusable building blocks of screens.
  • Templates are layout scaffolds with no data — a two-column layout, a modal structure, an auth screen frame. They accept organisms as children and define where things go without caring what those things are.
  • Screens are the top level. They fetch data, wire it to organisms, and handle navigation. A screen component should be thin — mostly composition, minimal rendering logic of its own.
  • Project the PostCard decomposition diagram (Figure 8). Walk through each layer. Ask: where would you add a "like count" to this component hierarchy? (Into the ActionBar molecule — it already groups the action icons.) Where would you add a "promoted post" badge? (Probably the PostCard organism — it knows the post type.)
💬 Discussion prompt: "Think of a screen you've already built in this course. Pick one component that's doing too much — more than one job. What would you split it into? Name the two new components." Give students 90 seconds, then collect 3–4 answers. This grounds the abstract framework in their own code immediately.
📝 Don't let this become a purity debate. Students occasionally latch onto atomic design as a strict taxonomy and start arguing about whether something is a molecule or an organism. Redirect: "The lines between levels are less important than the habit of asking the question. If you're asking 'should this be its own component,' you're already thinking the right way. The framework is a guide, not a rule set."
8
Composition Patterns and Close
The children prop, layout vs. content separation, and bridge to Final Project
⏱ 6 min

This segment is deliberately short — a fast synthesis and bridge rather than new material. Students are absorbing a lot today; the goal is to land two concrete takeaways before the final project introduction next class.

The Two Patterns Worth Naming Today

  • The children prop (slot pattern): Briefly demonstrate that a Card component that accepts children and renders them inside a styled container is infinitely reusable. The card never changes; the content slot does. This is the single most useful pattern for building a consistent visual language across a project.
  • Layout vs. content separation: A component that knows about position/spacing and a component that knows about data are better kept separate. The quick test: "Can I drop this component into a different screen without editing it?" If yes — it's cleanly separated. If no — layout concerns have leaked into it.
🖥️ 90-second demo — the children prop: Show a Card component with {children} being used twice on the same screen with completely different inner content. The visual consistency without code duplication is immediately obvious. Students who have been repeating card-shaped styling across their projects will have a small revelation.

Bridge to Next Class

🔜 Next class — Final Project Introduction: Students will receive the final project brief and begin scoping their apps. Connect today's material directly: "Before you write any code for your final project, I want you to sketch your main screens and label: (1) which canonical layout pattern each screen uses, and (2) what your organisms are — the 3–5 self-contained components that do the real work. Bring that sketch to the next class."
💬 Exit question (cold call 1–2 students): "Name one component in your existing project code that you would restructure based on today's lecture. What would you split it into, and why?" This closes the learning loop between new concepts and prior work, and seeds the mindset they'll need for final project planning.

Differentiation and Pacing Notes

If class is running ahead of schedule

  • Extend the pattern recognition exercise from Segment 4 — give students a second, more complex screen (e.g. a Settings screen with sections and grouped rows) and ask them to identify all the layout patterns present simultaneously
  • Introduce the gap property (React Native 0.71+) as a cleaner alternative to marginBottom on list items — quick and immediately useful
  • Show Platform.select() for cross-platform style differences — shadow properties are a natural example since they were just covered in Segment 6
  • Introduce the design token pattern from Section 8 of the student doc — creating a shared tokens object for colours and spacing as a precursor to the final project

If class is running behind schedule

  • Cut Segment 6 (absolute positioning) entirely — it's important but not foundational; direct students to Section 5 of the student doc and the FAB code example as self-study
  • Abbreviate Segment 5 (responsive layouts) to just the key hook: useWindowDimensions exists, it updates on rotation, use the width to calculate column count; skip the demo
  • In Segment 7, cover only Atoms, Molecules, and Organisms — skip Templates and give Screens a one-sentence treatment; the hierarchy table in the student doc covers the rest
  • Combine Segments 7 and 8 into a single 10-minute block, skipping the discussion prompt and going straight to the bridge to final project

Students who are ahead / want to go deeper

  • Challenge: refactor an existing project component file using the atomic design hierarchy — document which components are atoms, molecules, and organisms in a comment header
  • Explore: react-native-reusables or NativeBase as examples of published component libraries built on exactly these patterns
  • Research: Brad Frost's original Atomic Design methodology (atomicdesign.bradfrost.com) — the framework behind Figure 7
  • Implement: the full design token pattern from Section 8 of the student doc across a project, then measure how many files needed to change when one colour value was updated

Students who are struggling

  • Ground every flex concept in a physical metaphor: the main axis is a conveyor belt; justifyContent decides the spacing between boxes on the belt; alignItems decides how tall each box is relative to the tallest
  • For the flex: 1 chain, suggest they always add it from the root downward — "start at the screen root, confirm it has flex: 1, then work inward" — rather than hunting for the missing link bottom-up
  • Defer atomic design to independent reading; focus all available time on ensuring the axis model and the four canonical patterns are solid, since those unblock immediate project work
  • Recommend the Flexbox Froggy web game (flexboxfroggy.com) as a 20-minute reinforcement exercise — it teaches the same axis model through puzzles in a way that complements today's lecture well

Common Bugs to Anticipate in Student Work

These are the layout bugs that will appear in student projects and office hours in the days following this lecture. Being able to diagnose them by description speeds up debugging help significantly.

Student's descriptionLikely causeQuick diagnosis
"My ScrollView doesn't scroll" flex: 1 on contentContainerStyle instead of style, OR parent has no height Check that style={{ flex: 1 }} is on the ScrollView itself, not contentContainerStyle. Check the flex: 1 ancestor chain.
"My layout looks fine on my phone but broken on the grader's phone" Hard-coded pixel widths, or fixed heights that assume a specific screen size Search the codebase for numeric width/height values. Replace with flex, '%', or useWindowDimensions calculations.
"My footer is floating in the middle of the screen" ScrollView between header and footer is missing flex: 1 Classic flex: 1 chain bug. The ScrollView needs flex: 1 to push the footer down.
"My grid items are different widths" Using flexWrap with percentage widths and forgetting to account for padding/gap in the calculation Item width should be (screenWidth - horizontalPadding * 2 - gap * (cols - 1)) / cols.
"My absolute-positioned FAB is invisible" Missing explicit width and height on the absolutely positioned View Absolute elements must define their own dimensions — no parent to stretch or constrain them.
"My component is 500 lines and I don't know where to start refactoring" No decomposition discipline — everything is in one file Apply the modular design checklist from Section 6. Find the first JSX block that could have a one-sentence description and extract it.
