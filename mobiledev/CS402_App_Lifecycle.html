The App Lifecycle in Detail

Create · Render · Undisplay · Delete
🌱 Component mount & init 🔄 Re-render cycle 📴 Background & suspend 🗑 Unmount & cleanup 📱 AppState API

Every component in your app is born, lives an active life, goes dormant, and eventually dies. So does the app itself. Understanding this lifecycle — at both the React component level and the OS level — is the difference between apps that behave predictably and apps that leak memory, lose user data, and drain batteries. This lecture maps the complete lifecycle using a four-phase model you already have intuition for.

The organizing model: Database systems have CRUD — Create, Read, Update, Delete. The app lifecycle has its own version: Create, Render, Undisplay, Delete. Every component and every app passes through these four phases in order. The hooks you already know — useState, useEffect, useRef — map directly onto these phases. By the end of this lecture, you'll see them as lifecycle tools, not just API calls.

1. The Four Phases at a Glance

This model operates at two levels simultaneously. At the React level, individual components move through these phases as they mount, re-render, hide, and unmount. At the OS level, the entire app moves through equivalent states as it launches, runs, goes to the background, and is terminated. Understanding both levels — and how they interact — is the full picture.

The CRUD lifecycle at two levels. React components (top) and the OS app state (bottom) move through equivalent phases in parallel. Dashed vertical lines show the correspondence. Curved arrows show reversible transitions — a component can go from Undisplay back to Render when re-focused, just as the app moves from Background back to Active when the user returns.
C
Phase One
Create — Birth of a Component
Mount · Initialize · First render · Effects fire

When React adds a component to the screen for the first time, three things happen in strict order: the component function runs, state slots are initialized with their starting values, and the first render output is committed to the UI. Then — after the UI is painted — any useEffect calls with an empty dependency array fire. This is the Create phase.

What happens during Create

  1. Component function runs. React calls your function for the first time. Every useState(initialValue) reserves a slot and returns the initial value. Every useRef(initial) creates its mutable container.
  2. JSX is evaluated. React builds the element tree from your return value and commits it to the native view hierarchy. The user sees the first render.
  3. Mount effects fire. useEffect(fn, []) callbacks run after the first paint. This is where data fetching, subscriptions, and one-time setup belong.
import { useState, useEffect } from 'react';

export function TrailMap() {
  // Step 1: State slots created with initial values
  const [trails, setTrails] = useState([]); // slot 1 → []
  const [loading, setLoading] = useState(true); // slot 2 → true

  // Step 3: Mount effect fires AFTER first paint
  useEffect(() => {
    fetchTrails().then(data => {
      setTrails(data);
      setLoading(false);
    });
  }, []); // ← empty array = Create phase only

  // Step 2: First render committed — shows spinner
  if (loading) return <Spinner />;
  return <TrailList data={trails} />;
}
Why effects fire after paint, not before: React commits the UI first so the screen is never blank while your effect is running. The user sees the loading state immediately; the data fetch happens in the background. This is intentional — it keeps the Create phase fast and the UI responsive from the very first frame.

At the OS level

When the app first launches, the OS creates a new process, loads the JavaScript bundle, and mounts the root component. AppState is already 'active' by the time your root component function runs. The Create phase at the OS level is a one-time event per app launch.

R
Phase Two
Render — The Active Life
State changes · Re-renders · Effects with dependencies · User interaction

After Create, the component enters the Render phase — the phase it spends most of its life in. Every state update, every prop change, every context change triggers a re-render. React re-runs the component function, fills hook slots with current values, diffs the new output against the previous, and patches only what changed. This cycle can happen dozens of times per second during animations.

Effects during Render

useEffect calls with a non-empty dependency array belong to the Render phase. They run after the first render (like all effects) and then again whenever a listed dependency changes. This is how you react to specific state transitions rather than all re-renders. 

useEffect(() => {
  // Runs on Create AND whenever searchQuery changes
  searchTrails(searchQuery).then(setResults);
}, [searchQuery]); // ← dependency = re-run on this value

AppState: 'active'

At the OS level, the Render phase corresponds to AppState === 'active': the screen is on, your app is in the foreground, and the user is interacting. This is the normal operating mode.

⚠ Render phase ≠ visible: A component can be in the Render phase — mounted and capable of re-rendering — even when it is not the currently displayed screen. Navigation stacks keep previous screens mounted. A tab you're not looking at is still in the Render phase. This matters for effects: they keep running unless you handle the navigation lifecycle explicitly.
U
Phase Three
Undisplay — Gone From View, Not From Memory
Background · Suspended · Navigation blur · State preserved

Undisplay is the most misunderstood phase. When the user presses the home button, switches apps, or navigates away from a screen, the component is not destroyed. It is still mounted, its state is still intact, and its effects are still registered. What changes is the OS context: the app moves to the background, CPU allocation drops, and the screen goes dark. This is the Undisplay phase.

AppState — listening to the OS transition

The AppState API from react-native tells you which phase the app is in at the OS level. Subscribe to it in a useEffect to respond to the transition.

'active'

Screen on, app in foreground. The user is interacting. Full CPU allocation.

'background'

App moved to background. Screen may be off. Limited execution time. Save state now.

'inactive'

Transitional state on iOS only — briefly between active and background (e.g. during a phone call overlay, or swipe-to-switch animation).

import { useEffect, useRef } from 'react';
import { AppState } from 'react-native';

function useAppStateChange(onBackground, onForeground) {
  const appState = useRef(AppState.currentState);

  useEffect(() => {
    const subscription = AppState.addEventListener('change', nextState => {
      const wasActive = appState.current === 'active';
      const isActive  = nextState === 'active';

      if (wasActive && !isActive) {
        onBackground(); // active → background: U phase begins
      }
      if (!wasActive && isActive) {
        onForeground(); // background → active: R phase resumes
      }

      appState.current = nextState;
    });

    return () => subscription.remove(); // cleanup on unmount
  }, []); // register once on Create
}

The reference appState.current uses useRef — not useState — because we want to track the previous state without triggering a re-render on every change. This is the exact use case useRef was designed for.

What to do in the Undisplay phase

✅ Do when entering background

  • Save unsaved user data to AsyncStorage or SQLite
  • Pause timers, animations, and polling intervals
  • Release camera / microphone / GPS if not needed
  • Log analytics: session duration, last screen
  • Checkpoint any in-progress work

✅ Do when returning to foreground

  • Re-sync stale data (time-sensitive: prices, availability)
  • Resume paused timers and animations
  • Re-check permissions (user may have changed them in Settings)
  • Refresh auth tokens that may have expired
  • Restore scroll position or focused field

Navigation blur vs. AppState background

There are two distinct ways a screen can be "not visible." Understanding the difference prevents subtle bugs:

EventWhat happenedComponent stateAppState
Navigation blur User navigated to another screen inside your app Still mounted — state preserved, effects running Still 'active'
AppState background User pressed home or switched apps Still mounted — state preserved, effects still registered 'background'
Component unmount Screen was removed from the navigation stack Unmounted — state gone, cleanup fired Either — independent of AppState
💡 Expo Router and focus events: Expo Router exposes useFocusEffect from @react-navigation/core — a hook that runs a callback when the screen comes into focus and cleans up when it blurs. Use this for effects that should run only when the screen is actively visible, not for the entire time the component is mounted. 
import { useFocusEffect } from '@react-navigation/core';
import { useCallback } from 'react';

useFocusEffect(
  useCallback(() => {
    // Runs every time this screen comes into focus (R phase)
    const interval = setInterval(refreshPrices, 30000);

    return () => {
      // Runs every time this screen loses focus (U phase)
      clearInterval(interval);
    };
  }, [])
);
D
Phase Four
Delete — Cleanup and Termination
Unmount · Cleanup functions · Process kill · No warning

The Delete phase is the end of a component's life. React unmounts the component, tears down its hook slots, and — critically — runs every cleanup function returned from useEffect. After this, the component's state is gone. There is no way to recover it unless you persisted it explicitly during the Undisplay phase.

The cleanup function is the Delete hook

The function returned from useEffect is React's Delete-phase hook. It runs in two situations: when a dependency changes (before the next effect run) and when the component unmounts (the final Delete). Write it as if unmounting is imminent — because it often is. 

useEffect(() => {
  const sub = Accelerometer.addListener(handleData);
  const timer = setInterval(poll, 1000);

  return () => {
    // Delete phase — runs before next effect AND on unmount
    sub.remove();           // stop sensor subscription
    clearInterval(timer);  // stop polling timer
  };
}, []); // ← registered once; cleaned up on Delete

OS-level Delete — the process kill

At the OS level, Delete happens when the OS reclaims memory from a backgrounded app. There is no AppState event for this. The process simply ends. Any data not persisted during the Undisplay phase is lost — there is no "last chance to save" callback. This is why the Undisplay phase's save-on-background pattern is essential: you cannot rely on getting a signal before deletion.

⚠ You will not be notified before your app is killed. iOS and Android both reserve the right to terminate backgrounded apps without warning. If a user has 20 apps open and opens a large game, your app may be killed in seconds. Never assume you'll get a chance to save data at the last moment — save it the moment the Undisplay phase begins.

What causes Delete at the component level

  • The user navigates back past a screen, and the router removes it from the stack
  • A conditional renders null instead of the component — {isLoggedIn && <Dashboard />}
  • A list item is removed and its component is no longer in renderItem
  • The root component unmounts because the app process terminates

6. All Four Phases in One Component

Real components handle all four lifecycle phases. Here is a location tracker that illustrates every phase in a single component — annotated to show exactly which phase each piece of code belongs to. 

import { useState, useEffect, useRef } from 'react';
import { AppState, Text, View } from 'react-native';
import * as Location from 'expo-location';

export function LocationTracker() {

  // ── C: Create — state initialization ──────────────────────
  const [coords, setCoords]       = useState(null);
  const [paused, setPaused]       = useState(false);
  const appState                  = useRef(AppState.currentState);
  const locationSub               = useRef(null);

  // ── C: Create — start tracking on mount ───────────────────
  useEffect(() => {
    async function startTracking() {
      const { status } = await Location.requestForegroundPermissionsAsync();
      if (status !== 'granted') return;
      locationSub.current = await Location.watchPositionAsync(
        { accuracy: Location.Accuracy.High },
        pos => setCoords(pos.coords)  // R: triggers re-render
      );
    }
    startTracking();

    // ── D: Delete — cleanup on unmount ────────────────────────
    return () => locationSub.current?.remove();
  }, []);

  // ── U/R: Undisplay / Render — respond to app state changes ─
  useEffect(() => {
    const sub = AppState.addEventListener('change', next => {
      if (appState.current === 'active' && next !== 'active') {
        // U: Undisplay — entering background
        locationSub.current?.remove(); // release GPS
        setPaused(true);
      }
      if (appState.current !== 'active' && next === 'active') {
        // R: Render — returning to foreground
        startTracking(); // restart GPS
        setPaused(false);
      }
      appState.current = next;
    });
    return () => sub.remove(); // D: cleanup
  }, []);

  // ── R: Render — UI output ─────────────────────────────────
  return (
    <View>
      {paused && <Text>Tracking paused (background)</Text>}
      {coords && (
        <Text>{coords.latitude.toFixed(4)}, {coords.longitude.toFixed(4)}</Text>
      )}
    </View>
  );
}

7. Lifecycle Quick Reference

PhaseReact triggerOS stateYour hookAction to take
C — Create Component mounts App launches → 'active' useEffect(fn, []) Fetch initial data, subscribe, request permissions
R — Render State/props change AppState === 'active' useEffect(fn, [deps]) React to specific state changes, sync derived data
U — Undisplay Screen blurs / nav away 'background' / 'inactive' AppState listener · useFocusEffect Save state, pause timers, release sensors
D — Delete Component unmounts Process killed (no event) return () => {} from effect Cancel subscriptions, clear timers, prevent leaks
💡 The memory of the lifecycle: A component's state exists only while the component is mounted — from Create to Delete. If you need data to survive the Delete phase (across navigation or app kill), it must leave the component: write it to AsyncStorage, SQLite, or a remote API during the Undisplay phase. State is memory; persistence is storage. They are not the same thing.
