CS 402 — Mobile Application Development

Integrated Service Support — URL Schemes and the Linking API
🕐 Total time: 75 minutes 📅 Semester: Spring 2026 📚 Lecture: Phase C / D boundary 🎯 Themes: Theme 1 (Foundations) & Theme 2 (Building)

Learning Objectives

By the end of this session, students should be able to:

Understand
Explain what a URL scheme is and how the OS routes it to the correct native app.
Understand
Distinguish between fetch() (network request) and Linking.openURL() (OS handoff) — and why they are not interchangeable.
Apply
Use Linking.canOpenURL() and Linking.openURL() safely with error handling.
Apply
Construct correct scheme URLs for phone, SMS, email, and maps on both platforms.
Analyze
Identify which services require URL schemes vs. which require dedicated Expo packages (camera, contacts).
Create
Build a component that hands off to at least two native services with appropriate fallbacks.

1
Framing — Apps Don't Live Alone
Concept introduction · The fetch() misconception · URL scheme model
0:00 – 0:10 · 10 min

Opening hook

  • Ask: "How many apps are installed on your phone right now? 30? 60? More?" Pause. "Every one of them can be opened by your app — instantly, with data pre-filled — with a single line of code."
  • The phone isn't a container for your app. It's an ecosystem. URL schemes are the protocol that lets apps cooperate without knowing about each other.
  • Contrast with web: on the web, every page lives in a browser sandbox. On mobile, every app can be a first-class participant in the platform.
How it works under the hood: React Native's fetch() runs through the device's native networking layer — not a Node.js HTTP stack. When that native layer encounters a non-HTTP scheme like sms: or tel:, it passes the URL to the OS's registered protocol handler, exactly the way clicking a link in the built-in browser does. So fetch('sms:+15551234') genuinely does open the Messages app. Linking.openURL() is the more explicit API and adds canOpenURL() for graceful fallback — both are worth knowing.
📝 Pedagogical sequence: The lecture uses fetch() to introduce the concept and Linking.openURL() as the production implementation. This works because students already know fetch() — no new API to learn before the concept lands. Once they've seen the concept with a familiar tool, introducing Linking as "the right way to ship it" feels like a natural upgrade rather than an arbitrary rule.

The URL scheme model

  • Every native app can register one or more scheme prefixes with the OS: tel:, sms:, maps:, spotify:, myapp:.
  • When any app calls Linking.openURL('tel:+15551234'), the OS looks up which app registered tel: and launches it — passing the URL.
  • Your app hands off and steps back. The user sees the native dialer (or Maps, or Messages). When they're done, they can return to your app.
  • You never touch the phone hardware, the contacts database, or the user's messages. The OS mediates everything.
🗣 Quick class question: "When you tap a phone number link in a website on your phone, what happens?" → The dialer opens. That's the browser doing exactly what we're doing — opening a tel: URL. Your React Native app has the same power.
2
The Linking API — canOpenURL + openURL
API walkthrough · The defensive open pattern · iOS caveat
0:10 – 0:22 · 12 min

Two approaches — both valid

  • Start with fetch() — students already know it. Three lines, no imports, open the Snack and type them live. The reaction when Messages or the dialer opens is the teaching moment.
  • Then explain why it works: React Native's native networking layer intercepts non-HTTP schemes and routes them to the OS, exactly like the built-in browser following a link.
  • Then introduce Linking.openURL() as the production-quality version: same result, but adds canOpenURL() — the ability to check before you try.
  • Frame it clearly: "fetch() is how we introduce the concept. Linking is how we ship it."
🖥 Demo 1 (4 min) — fetch() as concept introduction: Open a fresh Expo Snack. Add a Button with onPress={() => fetch('sms:+12085551234')}. No imports. Press it on a physical device or show the simulator response. Let the simplicity land — one familiar function, one string, the entire messaging system is reachable. Then do the same with tel: and mailto:. Three lines, three different native apps.
🖥 Demo 2 (5 min) — Linking as the proper implementation: Now introduce Linking.openURL() and the canOpenURL() check. Show what happens on the simulator when you try tel: — without the check it fails silently or throws; with the check you get a clean Alert. Ask: "Which experience do you want your users to have on a device that doesn't support calling?"
🖥 Demo 1 (5 min) — The minimal safe-open helper: Open Expo Snack. Type the safeOpen(url, label) helper function from the student document live. Run it with tel:+12085551234. On simulator: show it gracefully fails and shows the Alert. On a physical device (if available): show the dialer opening. Emphasize: the check before the open is the pattern — always.
📝 iOS LSApplicationQueriesSchemes: Mention this at this point but don't dwell on it. Say: "iOS requires that you declare in app.json every scheme you intend to query with canOpenURL — otherwise it always returns false. Android doesn't require this for built-in schemes. We'll see the app.json entry in the full demo." Students don't need to memorize the property name — they need to know it exists and where to find it.
🗣 Why check first? "Could we just skip canOpenURL and try openURL, catching the error?" Technically yes — but the error experience is different. canOpenURL lets you show a polite Alert with a real message. An uncaught openURL rejection is usually a generic crash. Defensive code is better UX.
3
Phone Calls and SMS
Live coding · Platform URL difference for SMS body
0:22 – 0:37 · 15 min

Phone calls — the simple case

  • tel:+12085551234 — always include the country code (+1 for US) for reliability. The format without it works locally but breaks internationally.
  • Works on physical devices. On the iOS Simulator and Android Emulator, canOpenURL('tel:...') returns false — simulators have no phone hardware. This is expected.
  • The user sees the dialer with the number pre-filled. They must press Call. Your app never initiates the call unilaterally.
🖥 Demo 2 (5 min) — Phone button: In Snack, add a Button that calls safeOpen('tel:+12085551234', 'Phone'). Run on simulator — show the Alert ("not available"). If a student has a physical device on the same Snack URL, have them run it. The dialer opening on a real device is visually compelling. Even just showing "the alert fires correctly" is enough.

SMS — the platform difference that trips everyone

  • Basic SMS — just the number — is identical on both platforms: sms:+12085551234.
  • Pre-filling the body is where they diverge: iOS uses & as the separator; Android uses ?. This is the one URL scheme gotcha that students hit repeatedly.
  • Always use a helper function — getSMSURL(phone, body) — so the branch is in one place.
🖥 Demo 3 (7 min) — SMS with pre-filled body: Build the getSMSURL helper live. Show what happens if you use the wrong separator on a given platform (the body text either doesn't appear or the URL fails to parse). Then show the correct version. Ask: "Would a user know the body text was pre-filled by the app?" → No — it appears in the compose field just like they typed it. Discuss: is this ethical? When is it appropriate?
📝 Ethics moment — pre-filling text: This is worth a 2-minute aside. Pre-filling is legitimate when it saves user time ("Hi, I'm calling about your listing"). It's problematic when it makes the message sound like it came entirely from the user. The user always sees the text before sending — they can edit or delete it. But your app should never pre-fill content that misrepresents the user's intent. Students appreciate this being named explicitly.
✅ Quick Check — 2 min
  1. What does sms:+15551234?body=Hello do on iOS?
    → On iOS, the ? separator is wrong — the body may not pre-fill. iOS expects &body=. This is a common cross-platform bug.
  2. A student's app opens the SMS URL, but no pre-filled text appears. What are two possible causes?
    → (1) Wrong separator for the platform. (2) The body text is not URL-encoded — special characters break parsing.
4
Email, Maps, and Other Services
mailto: anatomy · geo: vs maps: · encodeURIComponent · service catalog
0:37 – 0:52 · 15 min

Email — the most parameter-rich scheme

  • mailto: supports to, cc, bcc, subject, and body as query parameters. All standard URL query string syntax.
  • The single biggest mistake: forgetting encodeURIComponent(). Spaces become + or break the URL; ampersands in the body text terminate the parameter early.
  • The email opens in the user's default email client — not necessarily the system Mail app. Gmail app, Outlook, etc. all register mailto:.
🖥 Demo 4 (4 min) — mailto: with encoding: Show a mailto URL constructed without encoding first — include a subject like "Help & Feedback" and watch it break. Then add encodeURIComponent() and show it working correctly. This is the fastest way to make encoding feel necessary rather than optional.

Maps — the platform branch

  • iOS: maps:?q=Boise+ID opens Apple Maps. maps:?daddr=Boise+ID starts navigation to a destination.
  • Android: geo:0,0?q=Boise+ID is the standard. The geo: scheme is the Android-native approach.
  • Coordinates instead of address: geo:43.6187,-116.2146 (Android) or maps:?ll=43.6187,-116.2146 (iOS).
  • Google Maps app on iOS also accepts comgooglemaps:// if installed — useful if you know your audience has it, but always fall back.
🖥 Demo 5 (5 min) — getMapsURL helper + live open: Build the getMapsURL(address) helper from the student document. Open "Boise State University, Boise ID" in Maps on the simulator — it works on both iOS and Android simulators because they have Maps apps. Students often don't realize this until they see it working.

The rest of the catalog — quick tour (3 min)

  • FaceTime: facetime:+15551234 (iOS only). facetime-audio: for voice. Graceful fallback required — Android users don't have FaceTime.
  • App Store / Play Store: itms-apps:// and market://. Use these for "Rate this app" prompts rather than linking to the store website.
  • App Settings: app-settings: (iOS). Indispensable for the permission-denied UX flow — one tap gets the user to where they need to go.
  • Camera/Photos/Contacts: Not URL schemes. These require expo-camera, expo-image-picker, expo-contacts — dedicated packages with permission flows.
📝 Camera/Photos distinction: Students frequently ask "can I open the camera with a URL scheme?" The answer is no — and for good reason. Camera access requires explicit permission; URL schemes bypass permission flows. The distinction helps students understand why the permission system and the URL scheme system are separate.

Full Demo — ContactCard Component
Live build · Five services · Loading state · Disable during handoff
0:52 – 1:07 · 15 min

Don't type the full ContactCard component from scratch — it's long. Instead, paste the completed version from the student document into Snack and walk through it section by section. The goal is pattern reading, not typing speed.

Points to narrate as you walk through the code

  • The actions array: Data-driven approach — adding a sixth service means adding one object to the array, not writing a new handler. This is how good mobile UIs scale.
  • The busy state: One piece of state tracks which action is in flight. The ActivityIndicator replaces the icon while waiting. The disabled prop prevents double-taps. Small but critical — OS handoffs have latency on some devices.
  • The platform helpers: All the cross-platform logic is isolated in getMapsURL() and getSMSURL(). The component itself doesn't know about platform differences.
  • The safeOpen helper: Reused across all five actions. One place for error handling. This is the pattern — consolidate defensive logic, don't repeat it.
🏋 5-Minute Pair Activity

Fork the Snack. Add a sixth action to the actions array that opens the user's email app with a pre-filled support message. Students must: (1) construct the mailto URL, (2) URL-encode the subject and body, (3) add it to the actions array. Most pairs finish in 4–5 minutes.

📝 What to watch for in the activity: The most common mistake will be forgetting encodeURIComponent() on the subject and body. Let students hit this and discover it themselves — the debugging experience of "my body text is truncated at the ampersand" is memorable and reinforces encoding more than any explanation.

Deep Links, Design Principles & Wrap-Up
The other direction · Good/bad use patterns · CFU · Exit
1:07 – 1:15 · 8 min

Deep links — your app as the target (3 min)

  • URL schemes work in both directions. Register a scheme in app.json ("scheme": "myapp") and other apps, emails, QR codes, and web pages can open your app at a specific route.
  • Expo Router handles this automatically — a link to myapp://product/42 navigates to your /product/42 screen.
  • Use Linking.addEventListener('url', handler) to catch links that arrive while the app is already running. Always remove the listener in cleanup.
  • Universal Links / App Links (https:// → your app) are more powerful and more complex — good topic for independent reading for students working on their final project.

Design principles (2 min)

  • URL scheme handoffs should always be user-initiated — a tap, not a background effect.
  • Always check canOpenURL() and handle the negative case gracefully.
  • Pre-filled text is a time-saver, not a deception — the user always reviews before sending.
  • For third-party apps (Spotify, Google Maps), always provide a fallback for users who don't have the app.
✅ Final Check for Understanding
  1. A student writes fetch('sms:+15551234'). Does this work? What does Linking.openURL() give you that fetch() doesn't?
    → fetch() does work — React Native's native networking layer routes non-HTTP schemes to the OS handler, just like a browser link. What you lose is canOpenURL(): fetch() has no built-in way to check whether the scheme is supported before attempting, so graceful fallback requires a try/catch instead.
  2. On iOS, canOpenURL('maps:?q=Boise') always returns false in your app. What's likely missing?
    → The scheme "maps" is not listed in ios.infoPlist.LSApplicationQueriesSchemes in app.json. iOS requires pre-declaration for canOpenURL to work.
  3. Name two services accessible via URL scheme and two that require a dedicated Expo package.
    → URL scheme: phone (tel:), SMS (sms:), email (mailto:), maps (maps:/geo:). Expo package: Camera (expo-camera), Contacts (expo-contacts), Photos (expo-image-picker).

Common Student Mistakes

MistakeWhat goes wrongHow to fix
Using fetch() without a fallback fetch() works for URL schemes via the native layer, but has no canOpenURL() equivalent — unhandled rejections surface as uncaught errors on unsupported devices. Either wrap in try/catch with a graceful Alert, or switch to Linking.openURL() preceded by canOpenURL().
Skipping canOpenURL() App crashes or shows an unhandled rejection on devices/simulators that don't support the scheme. Always check first, handle the false case with an Alert or fallback UI.
Wrong SMS body separator on iOS vs Android Body text doesn't appear in the compose field. Student thinks the feature is broken. Use &body= on iOS, ?body= on Android. Isolate in a helper function.
Forgetting encodeURIComponent() Spaces, ampersands, or special characters break the URL. Subject/body is truncated or garbled. Always encode user-supplied strings: encodeURIComponent(text).
Using maps: scheme on Android Android doesn't have an Apple Maps app. The URL fails silently or opens the browser. Branch on Platform.OS — use maps: for iOS, geo: for Android.
Assuming URL schemes open the camera No such scheme exists. There is no camera: URL scheme on either platform. Use expo-camera or expo-image-picker for camera/photo library access.
Missing iOS LSApplicationQueriesSchemes canOpenURL() always returns false on iOS for unregistered schemes, even for system apps. Add schemes to expo.ios.infoPlist.LSApplicationQueriesSchemes in app.json.

Grading & Assessment Notes

When URL scheme usage appears in assignments, grade generously on the pattern and more critically on the fallback. A component that opens the correct service but crashes on a simulator (because there's no canOpenURL() check) has demonstrated the concept — the defensive code is a polish issue, not a failure of understanding.

  • Full credit: correct scheme, correct encoding, canOpenURL check with graceful fallback.
  • Partial credit (−5–10%): correct scheme and it works, but no canOpenURL check and no error handling.
  • Significant deduction: fetch() used instead of Linking.openURL() — this indicates a conceptual misunderstanding of how the mechanism works, not just a polish issue.
  • Do not penalize for simulator-only testing (tel: always fails on simulator) — it's a tooling constraint, not a code error.
📝 Grading philosophy: The learning objective is understanding the handoff model. A student who builds a working email button with correct encoding has understood it. A missing canOpenURL() call is feedback, not a failure. Reserve meaningful deductions for the conceptual gap (fetch vs Linking) or missing encoding that causes the feature to actually malfunction.
