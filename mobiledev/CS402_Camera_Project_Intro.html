What the App Looks Like

The three screens of the completed app: a photo grid gallery, an empty state for first-time users, and a full-screen detail view (stretch goal). All three live within a single PhotoGallery module.

The Core Insight: A Gallery Is a Virtual List

The photo gallery you need to build is, at its heart, a FlatList. You already know FlatList. The job of this assignment is not to learn a new list component — it is to extend the one you have into a purpose-built gallery module using the composable component techniques from the previous lecture.

📋 Raw FlatList

General-purpose. Accepts any data, any row. Knows nothing about photos. No empty state. No camera integration. No persistence.

→

🖼 PhotoGallery Module

Wraps FlatList. Owns photo-shaped data. Provides a grid layout by default. Has an empty state. Has a camera FAB. Persists its data across restarts.

Wrapping, not subclassing. React Native components can't be traditionally subclassed the way you might extend a class in Java. Instead, you wrap FlatList inside a new component, supply it with the default configuration that makes sense for a photo grid, and expose only the props that callers actually need to vary. The inner FlatList is an implementation detail.

Wrapping FlatList — The Extension Architecture

PhotoGallery wraps FlatList with sensible photo-grid defaults, adds state management, persistence, and camera integration, then exposes only the props that genuinely vary between use sites. FlatList itself is untouched.

The Photo Record — Your Data Shape

Before writing any component code, define what a single photo record looks like. Every photo in the gallery is an object in the photos array. Agreeing on this shape up front means the rest of your code can be written against a stable contract. 

// The shape of a single photo record — define this in types.js or at the top of PhotoGallery
const photoRecord = {
  id:          'uuid-v4-string',    // unique, stable identifier
  uri:         'file:///...',       // MediaLibrary permanent URI (not the temp capture URI)
  capturedAt:  1711830400000,        // Date.now() at capture time — for display and sorting
  width:       4032,                 // native pixel dimensions (from asset.width)
  height:      3024,                 // used to compute aspect ratio for the thumbnail
  location:    { latitude: 43.61, longitude: -116.20 },  // optional — if exif: true was set
};
Two different URIs to understand: launchCameraAsync() returns a temporary cache URI like file:///var/mobile/tmp/.... This URI is valid right now but will disappear after the app restarts. You must call MediaLibrary.createAssetAsync(tempUri) to save it permanently, then store the asset's URI — that permanent one — in your record. Never persist the temp URI.

Generating a Unique ID

Every photo needs a stable ID so FlatList's keyExtractor and your delete operations work correctly. Two common approaches: 

// Option A — use expo-crypto (recommended for production)
import * as Crypto from 'expo-crypto';
const id = await Crypto.randomUuidAsync();

// Option B — simple timestamp + random suffix (fine for this assignment)
const id = `photo_${Date.now()}_${Math.random().toString(36).slice(2)}`;

Data Flow — Capture to Persistent Gallery

Every photo takes two persistence paths simultaneously: the file goes to the device's MediaLibrary; the record metadata goes to AsyncStorage or SQLite. On restart, loadPhotos() reads the record index, restores the photos array, and FlatList renders from it.

Building PhotoGallery Step by Step

Step 1 — The Persistence Layer

Write your persistence helpers first, before touching any UI. This gives you a clean, testable layer that the component will call into. 

// utils/photoStorage.js — the persistence contract
import AsyncStorage from '@react-native-async-storage/async-storage';

const STORAGE_KEY = '@cs402_photos';

export async function loadPhotos() {
  try {
    const raw = await AsyncStorage.getItem(STORAGE_KEY);
    return raw ? JSON.parse(raw) : [];
  } catch {
    return [];   // always return an array, never throw to the UI
  }
}

export async function savePhotos(photos) {
  await AsyncStorage.setItem(STORAGE_KEY, JSON.stringify(photos));
}

export async function addPhoto(newRecord) {
  const existing = await loadPhotos();
  const updated  = [newRecord, ...existing];  // newest first
  await savePhotos(updated);
  return updated;
}

export async function removePhoto(id) {
  const existing = await loadPhotos();
  const updated  = existing.filter(p => p.id !== id);
  await savePhotos(updated);
  return updated;
}
💡 Why write this as a separate file? If you later switch from AsyncStorage to SQLite (or even a cloud backend), only this file changes. Your component code — PhotoGallery.js — stays identical. This is the separation of concerns principle from the architecture lectures applied directly.

Step 2 — The Camera Capture Helper

Extract the camera work into its own async function. Keeping this out of the component body makes it easier to test and reuse. 

// utils/capturePhoto.js
import * as ImagePicker   from 'expo-image-picker';
import * as MediaLibrary  from 'expo-media-library';

export async function captureAndSavePhoto() {
  // 1. Request camera permission (shows dialog only once if undetermined)
  const camPerm = await ImagePicker.requestCameraPermissionsAsync();
  if (camPerm.status !== 'granted') return null;

  // 2. Request media library permission (separate from camera)
  const libPerm = await MediaLibrary.requestPermissionsAsync();
  if (libPerm.status !== 'granted') return null;

  // 3. Launch the camera — this hands control to the OS
  const result = await ImagePicker.launchCameraAsync({
    quality:       0.8,
    exif:          false,
    allowsEditing: false,
  });

  if (result.canceled) return null;   // user pressed Cancel

  const tempUri = result.assets[0].uri;  // ⚠ temporary — must save now

  // 4. Save to device MediaLibrary — this makes the file permanent
  const asset = await MediaLibrary.createAssetAsync(tempUri);

  // 5. Build and return the photo record — use the permanent asset URI
  return {
    id:         `photo_${Date.now()}_${Math.random().toString(36).slice(2)}`,
    uri:        asset.uri,         // permanent MediaLibrary URI ✓
    capturedAt: Date.now(),
    width:      result.assets[0].width,
    height:     result.assets[0].height,
  };
}

Step 3 — The PhotoThumbnail Atom

The row component that FlatList renders for each photo. Keep it simple — just the image, a timestamp label, and an optional press handler. This is a Level 1 atom. 

// components/PhotoThumbnail.js
import { Image, Text, TouchableOpacity, StyleSheet, Dimensions } from 'react-native';

const SCREEN_WIDTH = Dimensions.get('window').width;
const COLUMNS     = 2;
const GAP         = 4;
const THUMB_SIZE  = (SCREEN_WIDTH - GAP * (COLUMNS + 1)) / COLUMNS;

export default function PhotoThumbnail({ photo, onPress, onLongPress }) {
  const date = new Date(photo.capturedAt).toLocaleDateString();

  return (
    <TouchableOpacity
      onPress={() => onPress?.(photo)}
      onLongPress={() => onLongPress?.(photo)}
      style={styles.container}
    >
      <Image
        source={{ uri: photo.uri }}
        style={styles.image}
        resizeMode="cover"
      />
      <Text style={styles.date}>{date}</Text>
    </TouchableOpacity>
  );
}

const styles = StyleSheet.create({
  container: { width: THUMB_SIZE, margin: GAP / 2 },
  image:     { width: THUMB_SIZE, height: THUMB_SIZE, borderRadius: 6 },
  date:      { fontSize: 11, color: '#666', marginTop: 4, textAlign: 'center' },
});

Step 4 — The PhotoGallery Module

Now assemble everything. The module owns state, handles camera capture, manages persistence, and wraps FlatList with photo-grid defaults. 

// modules/PhotoGallery.js — the full composable module
import React, { useState, useEffect, useCallback } from 'react';
import { FlatList, View, TouchableOpacity, Text, StyleSheet, ActivityIndicator } from 'react-native';
import PhotoThumbnail             from '../components/PhotoThumbnail';
import { captureAndSavePhoto }   from '../utils/capturePhoto';
import { loadPhotos, addPhoto, removePhoto } from '../utils/photoStorage';

export default function PhotoGallery({
  // ── Data slots ──
  initialPhotos,       // optional seed data (for testing)
  onPhotoAdded,        // (photo) => void — notify parent of additions

  // ── Composition slots ──
  renderItem,          // render prop — override the default thumbnail
  emptyState,          // named slot — custom empty view

  // ── Config ──
  columns = 2,         // grid columns — 2 or 3
  showDates = true,   // whether thumbnails show capture date
}) {
  const [photos,  setPhotos]  = useState(initialPhotos || []);
  const [loading, setLoading] = useState(!initialPhotos);

  // Load persisted photos on mount
  useEffect(() => {
    if (initialPhotos) return;      // skip if seed data was provided
    loadPhotos().then(saved => {
      setPhotos(saved);
      setLoading(false);
    });
  }, []);

  // Camera capture action — wired to the FAB
  const handleCapture = useCallback(async () => {
    const record = await captureAndSavePhoto();
    if (!record) return;            // canceled or permission denied

    const updated = await addPhoto(record);   // persist and get full updated list
    setPhotos(updated);                        // update state → FlatList re-renders
    onPhotoAdded?.(record);                    // notify parent if it cares
  }, [onPhotoAdded]);

  // Delete action — called by long-press handler or selection mode
  const handleDelete = useCallback(async (id) => {
    const updated = await removePhoto(id);
    setPhotos(updated);
  }, []);

  if (loading) {
    return <ActivityIndicator style={styles.loader} size="large" />;
  }

  return (
    <View style={styles.container}>

      {/* ── The Virtual List ── */}
      <FlatList
        data={photos}
        keyExtractor={(item) => item.id}
        numColumns={columns}            // grid layout
        contentContainerStyle={styles.grid}
        ListEmptyComponent={              // named slot with default
          emptyState || (
            <View style={styles.empty}>
              <Text style={styles.emptyText}>📷 No photos yet</Text>
              <Text style={styles.emptyHint}>Tap the button to take your first photo</Text>
            </View>
          )
        }
        renderItem={({ item }) =>       // render prop with default thumbnail
          renderItem
            ? renderItem(item, { onDelete: handleDelete })
            : <PhotoThumbnail
                photo={item}
                showDate={showDates}
                onLongPress={() => handleDelete(item.id)}
              />
        }
      />

      {/* ── Floating Action Button ── */}
      <TouchableOpacity style={styles.fab} onPress={handleCapture}>
        <Text style={styles.fabIcon}>📷</Text>
      </TouchableOpacity>

    </View>
  );
}

const styles = StyleSheet.create({
  container: { flex: 1 },
  grid:      { padding: 2 },
  loader:    { flex: 1, justifyContent: 'center' },
  empty:     { flex: 1, alignItems: 'center', marginTop: 80 },
  emptyText: { fontSize: 20, color: '#888', marginBottom: 12 },
  emptyHint: { fontSize: 14, color: '#aaa', textAlign: 'center' },
  fab:       { position: 'absolute', bottom: 24, right: 24,
               width: 60, height: 60, borderRadius: 30,
               backgroundColor: '#2E5D8A', alignItems: 'center',
               justifyContent: 'center', elevation: 6, shadowOpacity: 0.3 },
  fabIcon:   { fontSize: 26 },
});

Full App Architecture

The screen is thin — it just mounts PhotoGallery. The module owns all state and wires to two utility modules: one for persistence, one for camera capture. FlatList renders the grid; the FAB triggers the full capture pipeline.

Why FlatList (Virtual List) — Not ScrollView

A photo gallery can easily contain hundreds of images. On a mobile device with limited RAM, rendering all of them simultaneously would be catastrophic. This is exactly the problem FlatList — a virtual list — solves.

❌ ScrollView — All in memory

Every item rendered and held in memory simultaneously. 200 photos × average thumbnail size = the entire gallery in GPU texture memory. Smooth on 5 items, jank and crash on 100.

✅ FlatList — Only visible items

Only the items visible on screen plus a small buffer are rendered. As the user scrolls, off-screen items are recycled. Memory usage stays roughly constant regardless of gallery size.

FlatList adds two specific behaviors that matter for a photo grid:

  • numColumns — renders items in a grid instead of a single column. Set this to 2 or 3 for a photo gallery. When using numColumns > 1, each item must have a fixed width (not flex: 1), calculated from the screen width divided by the column count.
  • onEndReached — called when the user scrolls near the bottom of the list. Use this for pagination if your app eventually loads photos from a cloud backend — it's the hook point for "load more."
⚠ numColumns and item width: When you use numColumns={2}, FlatList lays items out in rows of two. Each item must have an explicit pixel width or it will fill the full row. Calculate the thumbnail size like this: const thumbSize = (screenWidth - gap * (columns + 1)) / columns. Using flex: 1 on items inside a multi-column FlatList produces layout bugs that are frustrating to debug.

Stretch Goal — Selection Mode

The selection mode (long-press to select, then delete) is the most architecturally interesting stretch goal. It demonstrates a pattern you will use in many real apps: a mode variable that changes what UI elements render and what gestures do. 

// Inside PhotoGallery — add a selection mode state
const [selectedIds, setSelectedIds] = useState(new Set());
const isSelecting = selectedIds.size > 0;   // true when any photo is selected

const toggleSelect = (id) => {
  setSelectedIds(prev => {
    const next = new Set(prev);
    next.has(id) ? next.delete(id) : next.add(id);
    return next;
  });
};

const deleteSelected = async () => {
  let updated = [...photos];
  for (const id of selectedIds) {
    updated = await removePhoto(id);
  }
  setPhotos(updated);
  setSelectedIds(new Set());
};

// In renderItem — pass mode context to each thumbnail:
// onPress:     isSelecting ? toggleSelect : navigate to detail
// onLongPress: enter selection mode by toggling first item
💡 Using a Set for selections: A JavaScript Set is the right data structure here — it guarantees uniqueness, has O(1) has() and add(), and size tells you instantly whether you're in selection mode. The only caveat: React doesn't know a Set mutated, so always create a new Set(prev) when updating state.

Implementation Tips & Common Mistakes

TrapSymptomFix
Persisting the temp capture URI Gallery is empty after app restarts; stored URI is gone Always call createAssetAsync(tempUri) and store asset.uri, not result.assets[0].uri
Using flex: 1 on thumbnails with numColumns Items stretch to full row width; grid breaks Set explicit pixel width: (screenWidth - gaps) / columns
Not handling result.canceled App crashes when user backs out of the camera Check if (result.canceled) return null before accessing result.assets
Forgetting the MediaLibrary permission createAssetAsync() silently fails; no photo saved Request both requestCameraPermissionsAsync() AND requestPermissionsAsync() from MediaLibrary
Creating emitter or storage calls on every render Infinite re-render loop; storage read on every keystroke Put loadPhotos() in useEffect([], []); wrap action handlers in useCallback
Passing numColumns as dynamic state that changes React Native FlatList error: numColumns cannot change on the fly without remounting If you need a column toggle, unmount and remount FlatList with a key change
Not testing with a cold app restart Photos appear during the session but vanish when grader reopens the app Test persistence explicitly: capture → force-quit app → reopen → gallery should be populated

Recommended Starting Order

Don't try to build the whole app at once. This sequence gets you to a working demo as fast as possible, then adds features incrementally.

1
Get a photo to appear on screen. Write only captureAndSavePhoto() and a single Image component. Don't worry about lists yet. Just prove the camera works and returns a persistent URI.
2
Add persistence for a single photo. Call addPhoto() after capture, loadPhotos() in useEffect. Prove the URI survives a restart before building the list.
3
Build the raw FlatList. Add FlatList with numColumns={2} and a simple image-only render. Get the grid rendering correctly with your persisted photos array.
4
Wrap into PhotoGallery. Move the FlatList, FAB, and state into the module component. Add the empty state. This is the composable component extraction step.
5
Polish and stretch goals. Add the timestamp label to thumbnails, add the full-screen detail view, implement selection and delete. These only make sense once the core flow is solid.
You should be able to demo a working version of steps 1–3 in the next class session. Come to office hours early if you hit the permission or URI lifecycle issues — those are the two most common blockers, and both have quick fixes once you know what to look for.

Summary

Project 4 is not just a camera assignment — it is a composition exercise built on a virtual list. Everything you need is already in your toolkit:

  • A photo gallery is a FlatList with grid defaults and photo-shaped data — wrap it, don't rewrite it.
  • The persistence layer is a two-part system: MediaLibrary saves the file permanently; AsyncStorage or SQLite saves the record index. Both are required for true persistence.
  • The temp URI trap is the single most common mistake — always createAssetAsync() before storing anything.
  • Composable design pays off here: photoStorage.js, capturePhoto.js, PhotoThumbnail, and PhotoGallery are each independently understandable, testable, and replaceable.
  • Build in order: camera first, persistence second, list third, module wrapper fourth. Don't skip steps.
  • FlatList's numColumns requires explicit item widths — calculate from screen width, not flex: 1.
