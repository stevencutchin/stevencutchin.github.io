Load Remote List — App Architecture Guide

This guide deconstructs the "Load Remote List" app — a React Native application that manages a selectable list with remote persistence. We'll trace every communication path: between components, between the app and the server, between user actions and state changes, and between React's rendering engine and VirtualizedList's virtualization layer.

1. High-Level Architecture: The Big Picture

This app is a two-tier system: a React Native client that manages the UI and local state, and a remote PHP server that provides persistent storage. The client can operate independently (adding, removing, toggling items), but relies on the server for loading saved data and persisting changes across sessions.

Figure 1 — Two-tier architecture: the React Native client manages UI and state; the PHP server provides persistent JSON storage via GET (load) and POST (save).
Where This Fits in the 3-Tiered Model (Theme 4): This app collapses the logic tier into the client — all business rules (adding, deleting, toggling, deduplication) run in JavaScript on the phone. The server is purely a storage layer. In a production app, you'd move rules like deduplication and access control to the server so they can't be bypassed by a modified client.

2. File Structure and Responsibilities

The app is split across three files. Each has a single, clear responsibility — a basic application of separation of concerns. This mirrors the container/presentational/service pattern from the React Native Components guide.

Figure 2 — Three files, three roles: container component (App.js), presentational component (ListItem.js), and service module (remoteAccess.js).

3. Component Tree and Data Flow

React apps form a tree of components. In this app, VirtualList is the root. It renders a button row and a VirtualizedList, which in turn renders many <Item /> components. Data flows down through props; user actions flow up through callback functions.

Figure 3 — Component tree: data flows down via props (blue), user taps flow back up via callback functions (gold).
One-Way Data Flow in Action: The <Item /> component doesn't know how to toggle selection — it just calls the onPress callback it received as a prop. The parent (VirtualList) owns the state and the logic. This keeps the Item component simple, reusable, and easy to test.

4. State: The Single Source of Truth

All application data lives in three useState calls owned by the VirtualList component. No child component holds state — everything flows down from here. The list array is the primary state that drives the entire UI.

Figure 4 — Three state variables. The list array is the primary state — every UI element and server interaction reads from or writes to it.
Immutable Updates: State is never modified in place. Every handler creates a new array — using [newItem, ...list] (spread), list.filter(), or list.map() — and passes it to setList(). This is how React detects changes and triggers a re-render.

5. User Actions → Handlers → State → Re-Render

Every button and every row tap maps to a handler function. Each handler creates a new list array and calls setList(), which triggers React to re-render the component. VirtualizedList then queries the new state to display the updated list.

Figure 5 — Every user action maps to a handler → state update → re-render cycle. Save is the only action that doesn't change local state.

The remoteAccess.js module encapsulates all network I/O. Both functions use the Fetch API with async/await.

7. VirtualizedList: How Virtualization Works

A ScrollView renders every item into the component tree, even items off-screen. With 10,000 items, that's 10,000 components in memory — the app would freeze. VirtualizedList solves this by only rendering the items currently visible in the scroll viewport, plus a small buffer above and below.

Figure 8 — VirtualizedList only creates Item components for visible rows. The "Explode" button adds 10,000 items to demonstrate that rendering stays smooth.
The data Prop Trick: VirtualizedList requires a data prop, but this app ignores it — the real data lives in list state. The getItemCount and getItem callback functions read directly from state instead of from data. This is a valid pattern but somewhat unusual; FlatList (a simpler wrapper around VirtualizedList) avoids this indirection by reading from data directly.

8. The Item Component: Anatomy of a Presentational Component

ListItem.js exports a single component — <Item /> — that is purely presentational: it receives all its data and behavior through props, has no state of its own, and delegates user interaction back to the parent via a callback.

Figure 9 — Item receives four props: the data, a callback, and two style objects. Selected items invert to white-on-black.

What's Missing (and Why That's Instructive): This example deliberately omits error handling (what if fetch() fails?), loading indicators (the user doesn't know data is being fetched), authentication (anyone can read/write this user's data), and input validation.
