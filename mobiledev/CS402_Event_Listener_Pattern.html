Event & Listener Patterns in React Native

Decoupled Communication Between Components
📋 Topic: Architecture & State Patterns ⏱ 75 min lecture 🔗 Prerequisite: useState, Component Composition

The Problem This Pattern Solves

When a deeply nested component needs to communicate a change upward — or sideways — passing callbacks through every intermediate layer ("prop drilling") becomes unmanageable. The Event / Listener pattern gives components a clean, decoupled channel for broadcasting state changes without requiring direct parent-child wiring.

Consider a form with a PasswordField component. When the user types, the field needs to notify a StrengthMeter, a SubmitButton, and maybe an analytics tracker — all of which may live in completely different parts of the component tree. Passing a single callback down and back up would require changes to every layer in between.

Core Idea: A component exposes named events. Other components (or a parent composer) register listeners for those events. When the component's state changes, it fires the event — iterating through every registered listener in order until one signals it has fully consumed the event by returning false.

🔌 Observer / PubSub

One sender, many receivers. Receivers don't know about each other. Classic decoupling.

⛓ Chain of Responsibility

Handlers form a sequence. Each can handle-and-stop or pass-and-continue.

🎯 This Pattern

Both combined: a list of listeners called in order, chainable, stoppable on false.

How the Chain Works

Each listener receives the same event object. Returning true passes the event to the next listener. Returning false marks the event consumed and stops the chain.

The Return Value Contract

The return value of each listener function is the signal that controls chain propagation. This is a deliberate, explicit contract every listener author must understand.

✅ return true — Pass It On

The listener processed or observed the event but does not claim exclusive ownership of it. The dispatcher continues to the next listener in the list.

  • Logging / analytics observers
  • UI state updates that don't conflict with others
  • Validation checks that want further handling

🛑 return false — Consumed

The listener has fully handled this event and no further listener should act on it. The dispatcher stops immediately after this listener returns.

  • A modal that intercepts a "back" gesture
  • An error handler that swallows a failed submit
  • A rate-limiter that blocks repeated rapid actions
⚠ Common Mistake: Forgetting to return true at the end of a listener that only observes. A function with no return statement returns undefined, which is falsy — it will silently stop the chain as if it consumed the event. Always be explicit.

Building the Event Emitter

The foundation is a small utility class — EventEmitter — that lives outside the React component tree. It manages named listener lists and handles dispatch logic. You instantiate one per component type (or share one app-wide bus for global events). 

// utils/EventEmitter.js
export class EventEmitter {
  constructor() {
    // Map of eventType → ordered array of listener functions
    this._listeners = {};
  }

  /** Register a listener for a named event. Returns an unsubscribe fn. */
  on(eventType, listenerFn) {
    if (!this._listeners[eventType]) {
      this._listeners[eventType] = [];
    }
    this._listeners[eventType].push(listenerFn);

    // Return an unsubscribe function for easy cleanup
    return () => this.off(eventType, listenerFn);
  }

  /** Unregister a specific listener. */
  off(eventType, listenerFn) {
    if (!this._listeners[eventType]) return;
    this._listeners[eventType] =
      this._listeners[eventType].filter(fn => fn !== listenerFn);
  }

  /** Fire an event. Calls each listener in order.
   *  Stops if a listener returns false (event consumed). */
  emit(eventType, eventData = {}) {
    const listeners = this._listeners[eventType] || [];
    const event = {
      type:      eventType,
      payload:   eventData,
      timestamp: Date.now(),
      consumed:  false,
    };

    for (const listener of listeners) {
      const result = listener(event);
      if (result === false) {
        event.consumed = true;
        break;                 // chain stops here
      }
    }
    return event.consumed;   // useful for callers to know
  }

  /** Remove all listeners for one event type, or all events. */
  clear(eventType) {
    if (eventType) delete this._listeners[eventType];
    else this._listeners = {};
  }
}
Why return the unsubscribe function from on()?
React Native components mount and unmount frequently. If you register a listener in useEffect and forget to remove it when the component unmounts, you accumulate zombie listeners that fire on stale state — a classic memory leak. Returning the cleanup function makes it trivial to wire up with useEffect's return slot.

Wiring Listeners Inside a Component Tree

The FormScreen composer creates all listener registrations in one place. Child components remain unaware of each other — they only know about the shared emitter.

Connecting to React Native State

In React Native, listeners typically call setState (via useState hooks) on their own component. This means each subscriber owns its own reactive slice of UI state and updates independently.

Step 1 — The Source Component Fires Events

// components/PasswordField.js
import React, { useState } from 'react';
import { TextInput, View } from 'react-native';

export default function PasswordField({ emitter }) {
  const [value, setValue] = useState('');

  const handleChange = (text) => {
    setValue(text);           // update own local state

    // Fire the event — all registered listeners are called in order
    emitter.emit('onChange', { value: text });
  };

  return (
    <View>
      <TextInput
        value={value}
        onChangeText={handleChange}
        secureTextEntry
      />
    </View>
  );
}

Step 2 — Each Listener Component Hooks Into Its Own State

// components/StrengthMeter.js
import React, { useState, useEffect } from 'react';
import { View, Text } from 'react-native';

function scorePassword(value) {
  if (value.length < 6)  return 'weak';
  if (value.length < 10) return 'medium';
  return 'strong';
}

export default function StrengthMeter({ emitter }) {
  const [strength, setStrength] = useState('weak');

  useEffect(() => {
    // Register listener; save the returned unsubscribe function
    const unsubscribe = emitter.on('onChange', (event) => {
      setStrength(scorePassword(event.payload.value));
      return true;  // ← observe but don't consume; let chain continue
    });

    return unsubscribe;  // ← cleanup on unmount (critical!)
  }, [emitter]);

  return <Text>Password strength: {strength}</Text>;
}

Step 3 — The Composer Registers All Listeners in One Place

// screens/FormScreen.js  — the "composer" that wires everything together
import React, { useMemo } from 'react';
import { View } from 'react-native';
import { EventEmitter } from '../utils/EventEmitter';
import PasswordField from '../components/PasswordField';
import StrengthMeter from '../components/StrengthMeter';
import SubmitButton  from '../components/SubmitButton';

export default function FormScreen() {
  // useMemo keeps the same emitter across re-renders
  const emitter = useMemo(() => new EventEmitter(), []);

  return (
    <View>
      {/* Each child receives the shared emitter */}
      <PasswordField emitter={emitter} />
      <StrengthMeter  emitter={emitter} />
      <SubmitButton   emitter={emitter} />
    </View>
  );
  // No explicit registration here — each child handles its own useEffect cleanup
}
💡 Composer Strategy: Two valid patterns exist: (1) Self-registration — each child subscribes to the emitter in its own useEffect, as shown above. Clean and self-contained. (2) Central registration — the composer registers all listeners itself and passes callbacks as props. Useful when you need fine-grained control over listener order. Both are valid; pick one and stay consistent within a screen.

Lifecycle of a Listener

The unsubscribe function returned by emitter.on() must be called in the useEffect cleanup to prevent zombie listeners.

Scaling Up — Using React Context

Passing emitter as a prop works well for shallow trees. For deep trees or app-wide events, wrap the emitter in a React Context to avoid prop drilling. 

// context/AppEvents.js
import React, { createContext, useContext, useMemo } from 'react';
import { EventEmitter } from '../utils/EventEmitter';

const AppEventContext = createContext(null);

export function AppEventProvider({ children }) {
  const emitter = useMemo(() => new EventEmitter(), []);
  return (
    <AppEventContext.Provider value={emitter}>
      {children}
    </AppEventContext.Provider>
  );
}

// Convenience hook — any component can call useAppEvents() instead of prop drilling
export function useAppEvents() {
  const emitter = useContext(AppEventContext);
  if (!emitter) throw new Error('useAppEvents must be inside AppEventProvider');
  return emitter;
}
// Any deeply nested component can now subscribe with zero prop drilling:
function DeepAnalytics() {
  const emitter = useAppEvents();

  useEffect(() => {
    return emitter.on('onChange', (event) => {
      trackEvent('password_typed', event.payload);
      return true;  // analytics always passes through
    });
  }, [emitter]);

  return null;  // pure listener, no UI
}

Controlling Listener Order

Listeners fire in the order they were registered. In React, component useEffect hooks run after render — which means registration order follows the order components mount. For predictable ordering, use the central-registration pattern in the composer instead of self-registration.

Scenario Recommended Order Reasoning
Validation + UI update Validator first, UI second If validation fails and returns false, UI need not react to an invalid state.
Analytics + business logic Business logic first, analytics last Analytics should observe the final processed event, not an intermediate state.
Rate-limiter + normal handler Rate-limiter first The limiter must run first so it can return false to block subsequent handlers.
Error boundary + success handler Error check first Error listener returns false on bad state; success listener never sees it.

Implementation Checklist

Before shipping a feature that uses this pattern, verify each item:

  1. Every emitter.on() call saves the returned unsubscribe function.
  2. Every useEffect that registers a listener returns the unsubscribe function.
  3. Every listener function explicitly returns true or false — never undefined.
  4. The emitter instance is stable across renders (useMemo or useRef, not raw useState).
  5. Listener order is intentional and documented when order matters for correctness.
  6. A single "consumed" event does not produce stale UI in later listeners that expected to run.
  7. Tests cover the chain-stop case: assert that listeners after an false return are never called.
⚠ React Re-render Pitfall: Do not create the emitter inside the component body without useMemo or useRef. Every re-render would produce a new emitter instance, wiping all registered listeners and causing every child's useEffect (whose dependency is [emitter]) to re-run unnecessarily.

When to Use This Pattern vs. Alternatives

Approach Best For Trade-off
Event / Listener (this pattern) One source, multiple independent receivers; chain-stoppable; loose coupling needed Requires discipline on return values and cleanup
Callback Props Single parent-child communication; simple cases Prop drilling pain at scale; tight coupling
React Context + Reducer App-wide shared state that many components read All consumers re-render on any state change
Zustand / Redux Complex state with time-travel, devtools, middleware Added dependency; overkill for local component coordination
RxJS / Observable Async streams, debounce, complex composition operators Steep learning curve; heavy for most mobile use cases

Summary

The Event / Listener pattern gives you a clean, composable way to connect React Native components without tangling their dependencies. The key ideas to carry forward:

  • An event emitter is the shared bus — one instance per scope (screen or app).
  • Components fire events by calling emitter.emit(type, payload).
  • Components listen by calling emitter.on(type, fn) inside useEffect.
  • Returning true from a listener passes the event to the next in line.
  • Returning false consumes the event and stops the chain immediately.
  • Always unsubscribe on unmount — the cleanup function returned by on() does this.
  • Use Context to distribute the emitter to deeply nested components without prop drilling.
