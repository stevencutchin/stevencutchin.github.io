CS 402 — Mobile Application Development

Composable Components in React Native
🕐 Total time: 75 minutes 📅 Semester: Spring 2026 📦 SDK: Expo SDK 52 📄 Student doc: CS402_Composable_Components.html 🔗 Prereqs: Event & Listener Patterns, useState, useEffect

Lecture Overview

This lecture teaches students how to design React Native components that genuinely scale — components that can be reconfigured, extended, and combined without modifying their source code. The core ideas are the four-level composition model (atoms → molecules → panels → modules), three slot patterns for accepting external content, and a systematic approach to designing clean public APIs for full UI modules. The lecture is example-driven: a Panel primitive, then a DatePicker, a ColorChooser, and a multi-view CardManager show the ideas at increasing complexity.

This lecture builds directly on the previous Event / Listener lecture. The EventEmitter pattern appears inside each module walkthrough as the wiring mechanism for sub-component communication. Students do not need to re-learn it — they need to see it applied at a larger architectural scale.

Framing for students: Open with a question students feel: "How many times have you copy-pasted a component and just changed a few strings inside it?" Pause for recognition. That habit is the target of this entire lecture — composable design eliminates the need for it.

Learning Objectives

Understand
Explain the four levels of component composition and what distinguishes a panel from a module.
Understand
Describe when to use children, named slot props, and render props respectively.
Apply
Build a reusable Panel primitive with named slots for header, body, and footer.
Apply
Implement a self-contained UI module with a minimal public prop surface and a single primary callback output.
Analyze
Decompose a feature requirement into atoms, molecules, panels, and a module boundary.
Evaluate
Critique a component's prop surface for leaking internal concerns, and refactor it using slots or sub-components.

Hook & Motivation — The Copy-Paste Trap
Surface a pain point students have already felt before introducing the solution
⏱ 7 min

Opening Hook

On the projector, show two files side by side: UserCard.js and ProductCard.js. They are nearly identical — same layout, same styles — except one says "Username" and the other says "Product Name." Ask the class: "If the card's corner radius needs to change, how many files do you update?"

💬 Quick Discussion (2 min): "Think of the last time you duplicated a component in your own project. What was the reason — was the original component not flexible enough? What would you have needed to avoid the copy?" Take 2–3 answers. Listen for: "I needed a different header," "I needed a different button at the bottom," "the colors were different." Each of these is a slot.

Talking Points

  • Copy-paste code creates maintenance debt: two places to fix every bug, two places where styles drift apart.
  • The root cause is almost always a component that has its content baked in rather than passed in.
  • Composability solves this by making the structure reusable and the content variable — the same component, many configurations.
  • Today we will build the vocabulary and techniques to design components this way from the start — and refactor ones that aren't.
  • The payoff is real: a well-composed Panel and CardList you write today may never need to be rewritten, regardless of how the app grows.
📝 Instructor Note: Keep this segment tight — 7 minutes maximum. The goal is motivation, not an exhaustive critique of bad design. If students start offering detailed alternative patterns (Redux, Context), acknowledge them briefly and redirect: "Great — some of those connect to what we're about to see. Let's build the vocabulary first."

The Four Levels of Composition
Establish shared vocabulary before touching any code
⏱ 11 min

Walk the Layer Stack

Project the layer-stack table from the student doc. Walk through each level top to bottom, naming real examples students have already built or used. The goal is to anchor abstract vocabulary to concrete things they recognize.

  • Atoms — no children, one purpose. Text, Image, TouchableOpacity from React Native itself are atoms. So is a custom Badge or Avatar you write.
  • Molecules — two or three atoms doing one thing together. A search bar is a TextInput atom plus an Icon atom plus a clear button, laid out as a row. The molecule adds layout and a unified interaction model.
  • Panels — this is the key new concept today. A panel manages its own state and wires its children together, but accepts its content from outside. A Panel with a header, body slot, and footer slot is a panel.
  • Modules — a complete user workflow behind a single import. From the outside it looks like a black box with a clean API: data in, one callback out. The DatePicker we're about to build is a module.
  • The model is a thinking tool, not a rigid taxonomy. The key question at any level: "Does this component know more than it needs to?"
💬 Quick Classification Exercise (2 min): Call out these components one at a time and ask students to vote: Atom, Molecule, Panel, or Module?
  • TextInput — Atom
  • LabeledInput (label + input + error text) — Molecule
  • A form with validation state, submit button, and field layout — Panel
  • A full checkout flow with address, payment, confirmation steps — Module
There are no trick answers here. The point is to practice using the vocabulary, not to find the one correct answer.
✅ Check for Understanding — Cold Call
  1. "What's the difference between a molecule and a panel?"
    → A molecule groups atoms for a unified purpose but doesn't manage significant state. A panel manages its own state and wires its children together, while accepting its content through slots.
  2. "If a DatePicker component imported your app's Redux store to get the logged-in user's timezone, which rule does that break?"
    → It knows more than it needs to. Timezone should be passed as a prop. The component should be ignorant of where the data comes from.
📝 Instructor Note: The "does it know more than it needs to?" question is the most portable idea in this lecture. Write it on the board and leave it there for the rest of class. Students should be able to self-apply it to their own code long after they forget the specific vocabulary.

The Panel Primitive — Children, Named Slots & Render Props
Build the foundational container live, then introduce all three slot patterns
⏱ 12 min

Live Build — Panel.js

Start with a blank file. Build the Panel component incrementally — do not paste finished code. Students type along or watch closely.

🖥 Live Coding Order:
  1. Basic View wrapper with a style prop pass-through
  2. Add optional title → conditional header View with Text
  3. Add children in the content area — show that any JSX placed between tags appears here
  4. Add the named footer prop — show it's just a prop that holds JSX, rendered in a bottom row
  5. Add the contentStyle escape hatch — explain this is the "caller controls override"

Introduce the Three Slot Patterns

After building Panel, project the three-pattern diagram from the student doc. Walk left to right: children, named slot props, render props. Use the following framing for each:

  • children (containment): The component says "put whatever you want here." The caller places JSX between the tags. The component provides structure; the caller provides content. Best when the component doesn't care what it's wrapping — a modal, a card, a scrollable area.
  • Named slot props (positioned regions): The component says "I have specific named areas — header, footer, emptyState — and you can fill them in." The prop holds JSX. Best when the component has a known layout with variable regions. React Native's FlatList uses ListHeaderComponent and ListEmptyComponent this way.
  • Render props (data-back slots): The component says "I'll give you the data, you decide how to render it." The prop is a function that receives data and returns JSX. Best for lists and grids where the parent owns the data but the caller owns the row design. FlatList's renderItem is the canonical React Native example — students already know it.
Anchor to what students know: Spend 30 seconds explicitly pointing out that FlatList uses all three patterns simultaneously — ListHeaderComponent (named slot), renderItem (render prop), and implicitly children-like behavior in some wrappers. Students have used FlatList already; this reframes something familiar as intentional design.
✅ Check for Understanding — Think-Pair (90 sec)
  1. "You're building a Modal component. Which slot pattern do you use for the modal's content, and which do you use for the action buttons at the bottom?"
    → children for the body content (unknown, arbitrary). Named slot prop (e.g., actions) for the button row — it has a specific position in the layout.
  2. "You're building a Leaderboard that fetches and owns the scores array, but the caller wants to design each row differently in different screens. Which pattern fits?"
    → Render prop — the Leaderboard owns the data; the caller provides renderItem={(score, rank) => <MyRow .../>}.
4
Module 1 — DatePicker
The clearest example of a module: distinct sub-components, one public output, render prop pass-through
⏱ 12 min

Walk the Anatomy Grid First

Before showing any code, project the three anatomy cards from the student doc: MonthNavigator, CalendarGrid, ConfirmBar. Ask students: "Before I show you how these connect — what state do you think the module needs to own?" Let the class suggest. Expected answers: current month, selected date. Confirm, then proceed.

Talking Points — Walking the Code

  • Module owns state, children receive it as props. viewMonth and selected live in DatePicker and are passed down. Sub-components do not maintain their own copy of this data — there is one source of truth.
  • EventEmitter wires the sub-components internally. MonthNavigator fires onMonthChange; CalendarGrid and ConfirmBar both listen. The module's top-level onMonthChange={setViewMonth} handler updates state, causing a re-render. Point out: neither CalendarGrid nor ConfirmBar knows that MonthNavigator exists.
  • The render prop passes all the way through. renderDayCell is accepted by DatePicker and forwarded to CalendarGrid. DatePicker doesn't know what it renders — it just passes it along. This is how modules stay extensible without growing their own prop surface.
  • One primary output: onDateChange(date). The caller of DatePicker never interacts with MonthNavigator, CalendarGrid, or ConfirmBar directly. The module boundary is the entire public API.
🖥 Project the Data Flow Diagram (2 min): Point at each component in the SVG diagram and narrate: "MonthNavigator fires — state updates — CalendarGrid re-renders with the new month. User taps a day — CalendarGrid fires onDayPress — ConfirmBar listens and updates its display. User hits Confirm — ConfirmBar calls the module's onDateChange — that's the only thing the parent ever sees." Trace the arrows physically on the projector.
💬 Quick Discussion: "What would need to change if the client wanted to swap the calendar grid for a horizontal week-strip view instead of a month grid?" Expected answer: only CalendarGrid changes. The DatePicker module's prop surface and the parent screen's code don't change at all. That's the value of the module boundary.
✅ Check for Understanding — Cold Call
  1. "Why does the DatePicker module use useMemo to create its internal emitter rather than creating it in the component body?"
    → To keep the same emitter reference across re-renders. If the emitter is recreated each render, all useEffect hooks in child components that depend on [emitter] re-run, causing listeners to re-register on every state change.
  2. "The minDate prop disables days before a certain date. Where does that logic live — in the module or in the CalendarGrid?"
    → In CalendarGrid — it's the component that renders the day cells and applies the disabled style. The module just passes the prop through. Keeping logic where the rendering happens avoids leaking CalendarGrid's concerns into the module's code.

Module 2 — ColorChooser
Multiple sub-views sharing one event; boolean feature flags; controlled sync across siblings
⏱ 10 min

Key Concept: Three Sub-views, One Shared Event

The ColorChooser introduces a specific challenge: three different UI modes — a swatch grid, HSL sliders, and a hex input — all represent the same underlying value. When any one of them changes the color, all three need to update. Ask students before revealing the solution: "How would you keep three different input widgets synchronized to one value?"

  • All three fire the same event: onColorSelect. Every sub-view emits this event when the user interacts with it. The module listens at the top level, calls setColor, and passes the new color back down as a prop to all three. This is a controlled update cycle — the emitter triggers state, state drives re-renders, re-renders sync all views.
  • The module listens to its own emitter. Point out that useEffect in ColorChooser registers a listener on the shared emitter. The module is both the composer and a listener. This is valid and useful — the module uses the event to update its own state, which flows back down.
  • Boolean feature flags keep the API clean. showHex and showSliders default to true, so the full-featured module works with no configuration. Callers who want a simplified picker can strip it down without touching internals. Always default to the most capable version — let callers reduce, not add.
  • The module wraps its own Panel. Point out: ColorChooser renders inside a <Panel title="Choose a Color">. This is level 3 building level 4 — a panel used as a structural building block inside a module. Students should recognize the layer model in action.
📝 Instructor Note: Students sometimes ask why not just use three separate useState hooks (one per sub-view) and sync them manually. The answer: you'd need synchronization logic between three state variables every time any one changes, and any one getting out of sync produces a subtle inconsistency bug that's hard to trace. One state variable, one event, one update path is simpler and more correct. The emitter is the synchronization mechanism.
💬 Scenario Discussion (2 min): "Imagine a client asks you to add a fourth input mode: an eyedropper that lets the user sample a color from a photo. Where does this new sub-component plug in, and what does it need to do?" Expected answer: it fires onColorSelect on the shared emitter with the sampled hex color. The rest of the module handles it automatically — no changes needed to ColorChooser's core logic.

Module 3 — CardManager
Multi-view state machine; central event wiring; combining everything learned so far
⏱ 12 min

Introduce the State Machine First

Before showing code, project the state machine diagram (the three-node diagram with LIST → EDIT → CREATE transitions). Walk the arrows and name the events that trigger each transition. Ask: "What does a state machine give you that a collection of booleans doesn't?"

  • One state variable, never conflicting. view: 'list' | 'edit' | 'create' guarantees only one panel renders at a time. Compare this to three booleans: isListVisible, isEditVisible, isCreateVisible — these can all be true simultaneously if a bug sets them wrong. The string enum makes illegal states unrepresentable.
  • Events drive transitions. The module's useEffect registers four listeners at once: onSelect, onSave, onDelete, onAdd. Each handler updates cards state and/or view state. All mutations flow through this one place — there's no mutation logic scattered across child components.
  • Central registration by the module. Point out: in previous lectures, children self-registered in their own useEffect. Here, the module registers all listeners centrally. This gives explicit control over which events the module handles and in what order — especially important when events have side effects like calling onCardsChange.
  • Cleanup returns all four unsubscribes in one function. The pattern return () => { unsub1(); unsub2(); unsub3(); unsub4(); } is the standard way to clean up multiple listeners registered in a single useEffect. Write this on the board.
  • Render props and named slots both appear. renderCard (render prop) lets callers design the list rows. emptyState (named slot) lets callers supply a custom empty view. header (named slot) lets callers override the panel title area. All three patterns from segment 3 appear in one module.
🖥 Walk the Event Table (2 min): Project the bottom row of the state machine diagram and read across: "CardList fires these events. CardForm fires these events. The module catches all of them and decides what state to update. Neither CardList nor CardForm knows about the other — they only know about the emitter." This is the payoff of the event pattern applied at module scale.
📝 Instructor Note: If time is tight, stop after walking the state machine diagram and the event table. Students can read the code walkthrough in the student doc independently. The conceptual insight — events drive a state machine that controls which panel renders — is the key takeaway, not the code itself.
✅ Check for Understanding — Cold Call
  1. "The CardManager needs to enforce a maximum number of cards. Where does that logic live — in CardList, CardForm, or the module itself?"
    → In the module — it owns the cards array and the maxCards prop. It controls whether to allow onAdd to proceed by checking cards.length < maxCards before transitioning to 'create' view. CardList just uses the canAdd prop to show or hide the add button.
  2. "Why does the cleanup function need to call all four unsubscribes, not just one?"
    → Because four separate listeners were registered with emitter.on() — one per event type. Each registration is independent. Calling unsub1() only removes the onSelect listener; the other three remain active until their own unsubscribes are called.

Prop API Design, Controlled vs. Uncontrolled & Wrap-up
Give students portable design principles they can apply to any component they build
⏱ 11 min

The Four Prop Categories (3 min)

Project the prop surface table from the student doc. Walk through each row by naming it for the CardManager we just built:

  • Data in: initialCards — the starting dataset.
  • Data out: onCardsChange — the single primary callback. One output per module.
  • Slots: renderCard, emptyState, header — customizable regions.
  • Config: maxCards — a constraint. Boolean and primitive flags that tune behavior.
📝 Instructor Note: Emphasize "one primary output per module" as a rule students can apply immediately. When a module has three or four callbacks (onSave, onDelete, onAdd, onReorder all exposed to the parent), it is a sign that the module's state management has leaked out. Those should be internal events; the parent only needs the final state.

Controlled vs. Uncontrolled (3 min)

Briefly contrast the two modes. Frame it as: "Uncontrolled is easier to use; controlled is necessary when the parent needs to validate or transform the value before it comes back in."

  • Uncontrolled: parent provides initialValue and listens to onChange. Module owns state. Simpler — use this by default.
  • Controlled: parent provides value and onChange. Parent owns state and passes it back every render. Required when the parent needs to intercept — e.g., blocking a date selection that conflicts with another field.
  • Best practice: support both with a single convention — if value is passed, use controlled mode; if only initialValue is passed, use uncontrolled mode. Check once with a conditional in the component.

Prop Inflation Warning (1 min)

Read the prop inflation warning from the student doc aloud: "When a module's prop list exceeds about 10–12 props, it is often a sign that the module is doing too much, or that its internals have leaked into the public API." Give students the diagnostic question: "Could this prop be a slot instead? Could it be a sub-component the caller assembles themselves?"

Seven-Point Summary Round-robin (2 min)

Project the bullet summary from the student doc. Ask a different student to read and briefly explain each point. Keep it fast — about 15 seconds per bullet.

🖊 Exit Ticket — Canvas Quiz or Index Card

One sentence answering each:

  1. Name the four levels of composition and one example of each from today's lecture.
  2. What is a render prop and when would you use it instead of children?
  3. What does "one primary output per module" mean and why does it matter?

No grade required — use responses to identify students who need follow-up before the next assignment.

Common Student Mistakes to Watch For

MistakeSymptomCorrection
Hard-coding content inside a panel instead of using slots Student copies the panel for every screen variation; styles diverge over time Identify the variable region and convert it to a children slot or named prop
Exposing internal events as module props Module has onMonthChange, onDayPress, onConfirm all in the public API Collapse internal events; only expose the final value callback (onDateChange)
Using multiple boolean state flags instead of a view state variable Impossible states become possible (isEditVisible && isCreateVisible === true); bugs are hard to reproduce Replace booleans with a string enum: view: 'list' | 'edit' | 'create'
Module manages state that belongs to the parent Two sources of truth; parent and module disagree after an async update Identify who truly owns the data; implement controlled mode so the parent drives the value
Render prop returns undefined for some items List renders blank rows where the render prop returned nothing; no error thrown Always return a valid React element from render props; add a fallback: renderItem?.(item) ?? <DefaultRow />
Prop inflation — too many config props on a module Every new requirement adds a prop; the module becomes impossible to use without reading all docs Audit props: convert display variations to slots; convert behavioral variants to sub-components the caller assembles

Grading & Assessment Notes

When evaluating student work that implements composable components, prioritize architectural intent over syntactic completeness. A student who designs a clean module boundary and thoughtful slot API — even with a minor implementation bug — has demonstrated the learning objective more clearly than one who produces working but monolithic code.

  • Credit generously for correct use of any slot pattern (children, named slot, render prop) even if the implementation has minor errors.
  • Full credit for modules with a clear single-output callback, even if additional minor callbacks are also present.
  • Partial deduction for modules where all state is managed externally by the parent when the module should own it — this reverses the encapsulation intent.
  • Full deduction for components with no slot mechanism at all — hard-coded content throughout — when the assignment specifically asks for a composable design.
📝 Grading Philosophy: A working monolith that solves the visible problem but ignores the composability requirement misses the learning objective entirely, even if every pixel looks correct. A composable component with a render bug in one edge case is closer to the goal. Grade the architecture, not just the runtime result.
